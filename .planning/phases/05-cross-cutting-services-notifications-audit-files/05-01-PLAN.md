    `NotificationSecurityConfig` (Security 7 lambda DSL): wire shared-lib `JwtAuthenticationFilter`; `/api/v1/notifications/**` requires authentication; permitAll actuator.

    `ProcessedEventService`: implements the idempotent consumer pattern (RESEARCH Pattern 1 / Phase 1 SC5 gap). `boolean tryProcess(String consumerName, UUID eventId, Runnable action)` → in a `@Transactional`: check `processedEventRepository.existsByConsumerNameAndEventId(consumerName, eventId)`; if true → skip (return false); else execute `action`, then INSERT processed_events row. This is the FIRST real implementation of the dedup table shipped in Phase 1.

    `TemplateRenderService`: uses Thymeleaf `SpringTemplateEngine` to render a template by code (e.g. "tenant-welcome") with a `Map<String,Object>` context → returns rendered HTML string for email body. Also generates a plain-text subject from the rule/template metadata.

    `NotificationDispatchService`: the core rules engine (RESEARCH Pattern 2). Mark `dispatch(...)` as `@Async` — consumer records `processed_events` then returns; SMTP/Feign must NOT block the RabbitMQ listener thread (RESEARCH Pitfall: synchronous email in @RabbitListener).

    `dispatch(EventEnvelope<?> envelope, String eventType)`:
      1. Lookup `NotificationRule` by eventType (+ tenantId with platform-default fallback where tenant_id IS NULL).
      2. For each channel in `rule.channels`:
         - EMAIL: render template via TemplateRenderService, resolve recipient (from envelope payload adminEmail or via target_roles — for v1, extract from payload directly; full role→user resolution via AuthClient is a v2 enhancement noted in SUMMARY). Send via `JavaMailSender`. Persist `notification_deliveries` row status=SENT (or FAILED on exception).
         - IN_APP: persist `in_app_notifications` row with title/body/severity for the target user(s) from the payload.

    `TenantProvisionedConsumer` (`@RabbitListener(queues="notification.tenant-provisioned.queue")`): parse `EventEnvelope<Map<String,Object>>`, call `processedEventService.tryProcess("notification.tenant-provisioned", envelope.eventId(), () -> { tenantAwareMessageProcessor.process(envelope, env -> notificationDispatchService.dispatch(env, "TENANT_PROVISIONED")); })`.

    `AuthEventsConsumer` (`@RabbitListener(queues="notification.auth-events.queue")`): filter for `USER_LOCKED` event type from the envelope, dispatch similarly. Other auth events are no-op for v1.

    `LowStockConsumer` (`@RabbitListener(queues="notification.low-stock.queue")`): dispatch `INVENTORY_STOCK_LOW`.

    `NotificationDtos`: `InAppNotificationResponse{ id, title, body, severity, eventType, isRead, createdAt }`, `NotificationPageResponse`.

    `NotificationController` (`@RestController @RequestMapping("/api/v1/notifications")`):
      - `GET /` — paginated in-app notifications for the authenticated user (filter by `tenantId` from TenantContext + `userId` from JWT SecurityContext), sorted by createdAt desc. Returns `ApiResponse<List<InAppNotificationResponse>>` with `PageMeta`. This feeds the Phase 4 TopBar bell.
      - `PATCH /{id}/read` — mark notification as read (set is_read=true). Scoped to the user.
      - `GET /unread-count` — count of unread notifications for the badge.

    Gateway route: add to `gateway/src/main/resources/application.yml`:
