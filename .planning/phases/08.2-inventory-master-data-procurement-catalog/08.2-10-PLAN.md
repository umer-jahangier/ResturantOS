<objective>
Make a purchase-order line reference the vendor's catalog on the server side: accept a
`vendorItemId`, derive ingredient / unit / price from it, validate it belongs to the order's vendor,
and persist the link.

Purpose: the backend half of ROADMAP Success Criterion 6 and PUR-08. Today a PO line requires a
hand-typed ingredient UUID and a free-text unit, and the Zod message on the frontend literally says
no ingredient picker exists yet. Plan 08.2-19 removes that input; this plan gives it something
correct to post.
Output: `vendorItemId` on the line entity, request and response, catalog-driven derivation and
validation in `PurchaseOrderService`, and a dedicated integration test.
</objective>

In `PurchaseOrderService`'s create path, before building lines, collect every non-null
`vendorItemId` in the request and load them in ONE batched lookup scoped to the caller's tenant.

For each line:
- When `vendorItemId` is present: reject with a 422-mapped domain exception if the catalog row does
  not exist under the caller's tenant, if it is archived, or if its `vendorId` differs from the
  purchase order header's `vendorId` (this is the cross-vendor guard — a line must come from the
  chosen vendor's catalog). Then set `ingredientId` from the catalog row, set `uom` from the
  catalog row's `orderUom` when the request's `uom` is blank, and set `unitPricePaisa` from the
  catalog row's current effective price when the request supplies no price. When the request DOES
  supply a price, use it and mark `priceOverridden = true` on the response line — a manager can book
  a one-off deal, and the fact that they did must be visible.
- When `vendorItemId` is absent: behave exactly as today from `ingredientId` + `uom` +
  `unitPricePaisa`.

`lineTotalPaisa` and the header total are computed by the existing arithmetic, unchanged.

When mapping to `PurchaseOrderDto`, resolve `vendorSku` and `packDescription` from the already-loaded
catalog map — do not issue a second per-line query.

Do not consult vendor category tags anywhere in this file. Category tags filter and suggest in the
UI; they are not a validation input and must not become one.

No controller signature changes are required; verify the controller still compiles and that no new
permission code was introduced.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → `POST /api/v1/purchasing/purchase-orders` | Untrusted `vendorItemId`, `unitPricePaisa` and `qty`. |
| PurchaseOrderService → vendor_items / vendor_item_prices | Tenant-scoped reads that determine what a line costs. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-08.2-101 | Tampering | A line referencing another vendor's catalog entry, corrupting three-way match | high | mitigate | The catalog row's `vendorId` must equal the PO header's `vendorId` or the request is rejected with 422; `vendorItemFromAnotherVendorIsRejected` gates it. |
| T-08.2-102 | Information Disclosure | Cross-tenant catalog entry priced onto a PO | high | mitigate | The batched catalog lookup is tenant-scoped, so a foreign id resolves to nothing and is rejected as unknown; `catalogItemFromAnotherTenantIsInvisible` gates it. |
| T-08.2-103 | Elevation of Privilege | Category tags silently gating what may be ordered | high | mitigate | `PurchaseOrderService` references no category at all (grep-gated); tags remain a UI filter only, per D-01. |
| T-08.2-104 | Repudiation | An off-contract price booked without a trace | medium | mitigate | A supplied price sets `priceOverridden = true` on the response line, so the override is visible on the PO rather than indistinguishable from the catalog price. |
| T-08.2-105 | Denial of Service | Large line arrays causing per-line catalog queries | low | mitigate | Catalog rows for the whole request are loaded in one batched lookup before the line loop. |
</threat_model>
