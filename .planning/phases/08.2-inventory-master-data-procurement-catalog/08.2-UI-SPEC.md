## Modal vs. Full-Page Decision (per the 07.3 precedent)

The 07.3 precedent — heavier operational surfaces move from `Dialog` to a routed page or large
in-place panel — **applies to recipe revision authoring, does not apply to PO line entry, and
applies to vendor catalog management**:

- **Recipe view + revision authoring → NEW dedicated full page** (`/app/inventory/recipes/
  [menuItemId]`), replacing the current create-only `RecipeFormDialog` flow as the primary
  surface. Reason: the live plate-cost panel needs a persistent, non-scrolling side column
  (research §10.2's two-pane layout: component list left, sticky cost panel right) — a
  `sm:max-w-2xl` dialog with `max-h-[65vh] overflow-y-auto` cannot host that without becoming
  cramped or requiring the cost panel to scroll out of view, which defeats "live." `RecipeFormDialog`
  itself is **retired as the primary authoring surface**; its `useFieldArray` line-row pattern is
  reused inside the new page, not thrown away.
- **PO line entry → stays inside `PurchaseOrderFormDialog`, unchanged shape.** Replacing the
  hand-typed `ingredientId` text input with a search-as-you-type catalog picker is a
  component-level swap inside an existing line row, not a structural size increase — the dialog
  does not need to grow. No conversion to a page.
- **Vendor catalog → NEW dedicated full page** (`/app/purchasing/vendors/[id]`), because a
  vendor's catalog (grid of SKUs with effective-dated price history) does not fit inside the
  existing flat vendor list + inline-edit dialog, and per research §10.2 a vendor needs multiple
  sections (header, catalog, price changes) that a dialog cannot reasonably host. The existing
  flat `/app/purchasing/vendors` list stays as-is for header CRUD; only catalog management moves
  to the new detail route.
- **Category tree, ingredient grid, stock operations → new pages** (they have no prior
  dialog-based form to migrate from; built directly as pages per INV-13/14/15's own scope).

**No Radix Tabs primitive exists.** Where research recommends tabbed sections (vendor detail:
Catalog / Price Changes), build a simple `aria-controls`-wired button-group section switcher (the
same raw-element pattern 07.3 used for `order-type-toggle`) — do not add a new dependency for
this.

## Screens & Interaction Contracts

### 1. Ingredient Categories (`/app/inventory/categories`) — INV-13

- **Layout:** an indented recursive tree list (no tree primitive exists — build a small recursive
  component with `Button variant="ghost"` rows + `ChevronRight`/`ChevronDown` expand toggle), max
  3 levels enforced both by the DB trigger (D-02) and client-side (an "Add subcategory" action is
  hidden/disabled once a node is at level 3).
- **Row shows:** name, item count (how many ingredients have this as primary category), default GL
  account chips (inherited-vs-own indicated with a subtle "(inherited)" suffix in
  `text-muted-foreground`), and a row action menu (`DropdownMenu`, already in the primitive set):
  Edit, Add subcategory (hidden at level 3), Move to… (reparent), Archive.
- **Create/Edit:** a `Dialog` form (small — name, parent selector limited to valid targets at
  level < 3, default Cost/Inventory/Waste GL account selects, sort order). Mirrors
  `VendorFormDialog`'s create-or-edit-in-one-dialog shape.
- **Reparent ("Move to…"):** a `DropdownMenu` listing only valid targets — excludes the node
  itself, its descendants (cycle prevention), and any node already at level 3 (would push the
  moved subtree past the cap). Confirms with a small inline toast summarizing the GL-default
  impact count ("12 ingredients' inherited GL account may change").
- **Archive:** confirm dialog per the Destructive Confirmation copy above. **If the server
  responds with "category in use"** (in-use = ≥1 ingredient has it as primary category, or it has
  non-archived children), the button re-enables, the dialog stays open, and an inline error banner
  inside the dialog shows the exact copywriting-contract error string with the live count — never
  a silent toast-only failure the user can miss.
- **Empty state:** per Copywriting Contract, with a "Create category" action button (the
  `EmptyState` primitive already supports an `action` prop).
- **Loading:** `Skeleton` rows matching the tree row height (per DS-02, no spinner).
- **Validation timing:** on blur for text fields, on submit for the full form (matches
  `RecipeFormDialog`/`VendorFormDialog`'s existing react-hook-form + Zod pattern via
  `createZodResolver`).

### 2. Ingredient Master Data (`/app/inventory/ingredients`) — INV-14

- **Layout:** `DataTable` (the existing primitive — sorting, pagination, skeleton, empty-state all
  built in) with a search input above it (server-side or client-side fuzzy filter — research §7(a)
  flags typo-tolerant search as the primary defense against duplicate-item creation; at minimum a
  case-insensitive substring filter on name/short-name is required, fuzzy matching is a stretch
  goal, not a hard gate).
- **Columns:** name, category (leaf name, clickable → filters the list), stock UOM, par level,
  reorder point, storage location, status (`ACTIVE`/`ARCHIVED` chip), row actions (Edit, Archive).
- **Filter bar:** category dropdown (raw `<select>`, matching `RecipeFormDialog`'s existing
  select-element convention), status filter (Active/Archived/All — default Active), allergen
  filter (multi-select via the pill-toggle pattern below).
- **Create/Edit dialog:** a larger `Dialog` (`sm:max-w-2xl`, scrollable body — mirrors
  `RecipeFormDialog`'s `max-h-[65vh] overflow-y-auto` shape) with grouped sections: General (name,
  short name, primary category — **required, single-select**, description), Units (stock UOM,
  purchase UOM read-only note "set per vendor item", recipe UOM, item-level conversions as a small
  dynamic `useFieldArray` list mirroring `RecipeFormDialog`'s line-array pattern), Stock (par
  level, reorder point, storage location, shelf-life days), Compliance (allergens as a **pill
  toggle-group**, not checkboxes — no checkbox primitive exists; each allergen renders as a
  `button` with `aria-pressed`, active state using the accent border/fill, same visual language as
  07.3's `order-type-toggle` segmented control).
- **Measure-type lock:** once any stock transaction exists for the ingredient, the stock-UOM /
  measure-type field renders **disabled** with a tooltip ("Locked — stock transactions already
  exist") rather than a submit-time error, so the constraint is visible before the user tries.
- **Archive vs. edit:** archiving never requires reassignment (unlike categories) — an archived
  ingredient simply drops out of active pickers (recipe lines, PO catalog) while existing
  references keep working, per D-04. The archive confirm copy reflects this (no "in use" refusal
  path for ingredients).
- **Empty / loading / validation:** same conventions as Screen 1.

### 3. Recipe Detail + Revision Authoring (`/app/inventory/recipes/[menuItemId]`) — INV-15

- **Layout:** two-column page (research §10.2's recommended shape). Left column: version-history
  strip (chips: "v1", "v2 · current", "v3 · scheduled from 2026-08-01") — selecting a version
  shows its lines read-only below; a **"Create new revision"** primary CTA pre-fills a new
  editable line-array form from the currently-selected version (never blank, per D-05 — this is a
  revision, not a fresh start). Right column: the **sticky plate-cost panel** (position: sticky
  within the viewport on desktop; stacks below the form on narrow viewports — no fixed-width
  breakage).
- **Version banner** (research §10.2): "v3 · active since {date}" or "v3 · scheduled from {date}"
  or "Draft — unsaved", always visible at the top of whichever version is selected.
- **Cost panel contents:** batch cost, cost per portion, food-cost % (portion cost ÷ menu-item
  price, only shown if a price is known — otherwise "—" with a tooltip "No menu price to compare
  against"), and per-line "% of plate cost" with a small horizontal bar next to each ingredient
  line in the form (research §10.2's mini-bar pattern).
- **"Live" contract (precise, since this is new ground):**
  - **Recompute trigger:** any change to a line's qty, UOM, or yield %, or to the header's
    portions/batch-yield fields.
  - **Debounce:** 400ms after the last keystroke before firing a recompute — this is a **server
    round-trip** (a new preview endpoint this phase must add), not a client-side calculation.
    Cross-dimension UOM conversion and moving-average costing (D-06/D-08) are backend-owned
    invariants; duplicating that arithmetic in the browser risks silently drifting from the
    production depletion math. The panel is explicitly a **cost estimate for authoring, not the
    ledger source of truth** — state this in a small caption under the panel: "Estimated from
    current ingredient cost — actual cost is set at depletion time."
  - **Rounding/display:** money values render via `MoneyDisplay` (paisa integers, HALF_UP,
    formatted PKR) exactly as elsewhere in the app; food-cost % renders to 1 decimal place;
    per-line share renders as a whole-number percentage.
  - **Loading (recompute in flight):** previous values stay visible, dimmed (`opacity-60`), with a
    small inline "Recalculating…" caption — never a blank panel or a full-panel spinner (DS-02
    skeleton-first rule applies to recompute-in-progress too, not just first load).
  - **Error (a line's UOM can't be converted / ingredient has no known cost):** that line gets an
    inline warning (copy: "Couldn't price this line — check the unit conversion for {ingredient}")
    and its contribution is excluded from the batch-cost total with a footnote: "N line(s)
    excluded from cost pending a fix" — the panel degrades gracefully, it never blocks saving or
    throws.
- **Saving a revision:** always creates a new version (D-05) — there is no "save in place" action
  anywhere on this page. `effectiveFrom` uses the same local-midnight `calendarDateToInstant`
  helper already fixed in `RecipeFormDialog.tsx:71-76` (do not regress the bug this phase's own
  CONTEXT.md origin section documents) — reuse that function, don't reimplement it.
- **Empty (no versions yet for this menu item):** same `EmptyState` as the current
  `recipes/page.tsx:78-81`, with the "Create new revision" CTA relabeled "Create first recipe
  version" for a menu item with zero versions.

### 4. Recipe Coverage (`/app/inventory/coverage`) — INV-15 (3-state fix)

- Extends the existing `coverage/page.tsx`. The stat cards (Total / Covered / Missing) gain a
  third figure: **Scheduled** (recipes with `effectiveFrom` in the future — currently silently
  merged into "missing", per this phase's own origin bug). Four cards total: Total Active Menu
  Items, Covered, Scheduled, No Recipe.
- The missing-items table splits into two sections (or one table with the `SCHEDULED`/`NO_RECIPE`
  chip from the Status System table above distinguishing rows) — each scheduled row shows the
  chip label "Scheduled from {date}" inline, not a separate column, so the date is legible at a
  glance without adding table width.
- **Regression guard tie-in (not a UI concern, but the UI must not silently disagree with the
  backend again):** the frontend's coverage-state derivation must consume the same three-state
  field the backend now returns (not re-derive "current" client-side from a boolean) — flag this
  to the planner as a hard dependency, since CONTEXT.md's origin bug was exactly a frontend/backend
  definition mismatch (MSW mock used `r.current`, backend used `effective_from <= now()`).

### 5. Purchase Order Line Entry — catalog picker (`PurchaseOrderFormDialog`) — PUR-08

- **New shared component — `CatalogItemCombobox`:** an anchored (non-full-screen) search-as-you-
  type combobox, built on the existing `cmdk` `Command` primitive (already used by
  `command-palette.tsx`) but **not** wrapped in a full-screen `Dialog` — instead anchored via a
  new `components/ui/popover.tsx` wrapper over `radix-ui`'s `Popover` export (available today from
  the already-installed unified `radix-ui` package; no new dependency, just a new thin wrapper
  file following the same style as the other `components/ui/*` primitives). This is the one new
  shadcn-style primitive this phase adds — record it under Registry Safety if a future audit adds
  it via `npx shadcn add popover` instead of hand-building it.
- **Behavior:** typing filters the **selected vendor's catalog only** (scoped the moment a vendor
  is chosen on the PO header — matches PUR-08's "filtered to the vendor's catalog/categories"
  requirement). Each result row shows: item name, pack size/description, vendor SKU, and current
  contract price (`MoneyDisplay`). Selecting a result fills the line's ingredient, unit (from the
  vendor item's order UOM), and unit price — all editable afterward (a manager can still override
  price for a one-off deal) but the picker is the only way to set the `ingredientId`; the free-text
  UUID input is deleted.
- **Zero matches:** per Copywriting Contract's PO-picker empty-state row (rendered inside the
  popover's list area, not a full `EmptyState` block — keep it compact since it's an inline
  overlay, not a page).
- **No vendor selected yet:** the combobox is disabled with a placeholder "Select a vendor first."
- **Validation timing:** the line-level Zod schema changes from `z.string().uuid(...)` to
  requiring a picker-confirmed selection (a hidden field or a discriminated shape) — surfaced as a
  field error only on submit attempt, matching the rest of the form's existing validation timing.

### 6. Vendor Catalog (`/app/purchasing/vendors/[id]`) — PUR-07

- **Header:** vendor name, payment terms, contact info (moved from the flat list row — matches
  `VendorFormDialog`'s existing fields, just promoted to a page header instead of a dialog-only
  view).
- **Section switcher (no Tabs primitive — button group):** "Catalog" / "Price Changes". Catalog
  section is a `DataTable` of `vendor_item` rows: vendor SKU, description, pack size, order UOM,
  current price, MOQ, lead time, status (`ACTIVE`/`ARCHIVED`), linked ingredient (name, clickable
  → filters `/app/inventory/ingredients`). A row action "Edit price" opens a small dialog that
  **always inserts a new effective-dated `vendor_item_price` row** — there is no in-place price
  edit anywhere in this UI, per D-01/locked decision; the dialog's copy makes this explicit: "This
  creates a new price effective {date} — the old price is kept for history, never overwritten."
- **Price Changes section:** a simple chronological table per catalog item (item, old price, new
  price, % delta, effective date) — read-only, no actions.
- **Category tags (filter-only, never authorizing):** shown as plain pills on the vendor header
  with a caption "Used to filter and suggest — does not restrict what can be ordered" so the UI
  itself communicates the D-01 invariant, not just the data model.
- **Add catalog item:** a `Dialog` form (vendor SKU, linked ingredient — via the same
  ingredient-search combobox used in Screen 5's PO picker, reused not reimplemented — pack size,
  order UOM, initial price, MOQ, lead time).
- **Empty (no catalog items):** per Copywriting Contract.

### 7. Stock Operations (`/app/inventory/stock`) — INV-15 item 8

- **On-hand read view (the new endpoint this phase must add):** `DataTable` of ingredient × branch
  on-hand qty, filterable by ingredient/category/branch. Rows at/below reorder point get the
  warning row-wash; rows at/below zero get the destructive row-wash (Status System table above).
  This is the "read model with nothing to call today" gap CONTEXT.md flags — the page cannot ship
  without the new `GET` endpoint; note this as a hard backend dependency for the planner.
- **Entry points to the four write actions**, each a `Dialog` form in the same
  `VendorFormDialog`/`PurchaseOrderFormDialog` shape (header fields + a `useFieldArray` line list
  using the shared ingredient combobox for line selection, replacing any hand-typed ID):
  - **Opening balance:** ingredient + qty + unit cost, one-time per ingredient/branch.
  - **Receipt:** vendor/PO reference (optional — a receipt can be logged without a PO), lines
    (ingredient, qty, stock UOM, unit cost).
  - **Transfer — ship:** destination branch + lines. **Transfer — receive:** a list of
    in-transit transfers awaiting this branch, each with a "Receive" row action opening a
    confirm/adjust dialog (variance shown live as counted − shipped, matching the count-sheet
    variance-highlight pattern below).
  - **Count:** a count-session flow grouped by storage location (research §10.2/§7(e) — mirrors
    physical shelf order since ingredients now carry `storage_location` from Screen 2). Each row:
    ingredient, expected qty (dimmed, informational only), counted-qty input, live variance
    (counted − expected, colored via the warning/destructive row-wash once past a threshold).
    Submitting posts the count; mobile-first entry is a target but not a hard gate this phase.
- **Loading/empty/error:** same conventions as every other screen in this contract — `Skeleton` on
  load, `EmptyState` with an actionable CTA when a branch has no stock/transfers/counts yet, and
  the generic `SERVICE_UNAVAILABLE` copy from the Copywriting Contract if the module is briefly
  down (this is exactly the class of defect CONTEXT.md's "carried-over defects" section flags —
  the UI copy for it lives here so the executor doesn't invent ad hoc text per screen).

  invoice-learning pattern) is **out of scope** — this phase's vendor catalog is manually
  maintained, no OCR/invoice-line auto-mapping exists yet.
