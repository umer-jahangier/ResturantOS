<objective>
Replace the fabricated spend-by-category analytics with real ingredient categories. Today the
purchasing analytics "spend by category" table is computed from a static classpath YAML that maps a
handful of ingredient UUIDs to names and defaults everything else to "Uncategorized" — it never calls
inventory-service at all.

Purpose: ROADMAP Success Criterion 7 and locked decision D-09, which is explicit that the mock is
DELETED, not bypassed. Plan 08.2-09 shipped the internal batch endpoint this consumes.
Output: a Feign-backed batch resolver behind the existing integration-mode switch, the mock
implementation and its fixture removed from the repository, and a test proving a real category name
reaches the report.
</objective>

Change `IngredientCategoryResolver` to a batch interface:
`Map<UUID, String> resolveAll(Collection<UUID> ingredientIds)`. Remove the single-id `resolve` method
entirely — leaving it in place is how per-line resolution creeps back. Update the interface javadoc
to state that this is the real inventory-service-backed seam and to drop the "Phase 8 will add a
feign-backed resolver" note, which is now done.

`InventoryCategoryClient`: a `@FeignClient(name = "inventory-service", configuration =
FeignClientConfig.class)` interface declaring
`@GetMapping("/internal/inventory/ingredient-categories") List<IngredientCategoryResponse>
getIngredientCategories(@RequestParam("ingredientIds") List<UUID> ingredientIds)` with a nested
record `IngredientCategoryResponse(UUID ingredientId, UUID categoryId, String categoryName, String
categoryPath)` mirroring the inventory-side DTO field-for-field.

`FeignIngredientCategoryResolver implements IngredientCategoryResolver`, selected by the same
mechanism the GRN adapter pair uses so `restaurantos.inventory.integration-mode` governs both.
`resolveAll`:
- returns an empty map immediately for a null/empty input;
- de-duplicates the input ids;
- chunks the id list into batches of at most 500 (the internal endpoint's cap) and issues one call
  per chunk;
- builds `Map<UUID, String>` from `ingredientId` to `categoryName`, substituting the literal
  `Uncategorized` for a null/blank name;
- catches Feign/connection failures, logs a WARN naming the failure and the affected id count, and
  returns a map that resolves every requested id to `Uncategorized`. The spend report must degrade,
  never 500 — a purchasing analytics page must not go dark because inventory-service is restarting.

Update `InventoryIntegrationProperties`' javadoc so the switch's documented scope covers both GRN
data and ingredient-category resolution.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| purchasing-service → inventory-service `/internal/**` | Service-to-service call carrying tenant context; a missing or wrong tenant header would read another tenant's categories. |
| Feign response → spend report labels | Untrusted string values rendered into an analytics table. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-08.2-111 | Information Disclosure | Tenant context not propagated on the outbound Feign call | high | mitigate | The new client reuses `FeignClientConfig`, the same interceptor chain the GRN client already relies on for tenant/service headers; `spendBucketsUseCategoryNamesReturnedByInventoryService` exercises the wired path. |
| T-08.2-112 | Denial of Service | Analytics page failing hard when inventory-service is restarting | high | mitigate | `resolveAll` catches transport failures and degrades every bucket to `Uncategorized`; `inventoryServiceFailureDegradesInsteadOfFailingTheReport` gates it. |
| T-08.2-113 | Denial of Service | Per-line Feign call fan-out on a large report | medium | mitigate | The interface takes a batch, the single-id method is deleted, and `categoryResolutionIsCalledOncePerReportNotOncePerLine` asserts one call per report. Ids are chunked at the endpoint's 500 cap. |
| T-08.2-114 | Repudiation | Fabricated analytics presented as real spend data | high | mitigate | The static fixture and its resolver are deleted from the repository; grep gates in the acceptance criteria fail the build if either returns. |
</threat_model>
