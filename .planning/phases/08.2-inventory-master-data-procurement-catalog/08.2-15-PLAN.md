`IngredientFormDialog({ ingredient, trigger })` — one dialog for create and edit, driven by whether
`ingredient` is defined, exactly like `VendorFormDialog`. Dialog size and scroll behaviour copy
`RecipeFormDialog`'s (`sm:max-w-2xl`, scrollable body).

Four grouped sections, each introduced by a sub-heading matching `RecipeFormDialog`'s established
`text-sm font-medium` shape:
- General: name, short name, SKU, primary category (a raw styled select populated from
  `useCategories`, REQUIRED — the Zod input schema already enforces it, and the field carries a
  required marker), description, item type (PURCHASED / PREPARED / BOTH).
- Units: measure type, stock unit (this is the existing `baseUomCode` field — do not introduce a
  separate stock-UOM field), recipe unit, default yield percentage, and a read-only muted note
  reading that the purchase unit is set per vendor item, not here (D-06). Then a `useFieldArray`
  conversions list whose rows are from-unit, to-unit, factor and an optional note, with add and
  remove buttons, copying `RecipeFormDialog`'s line-row grid shape.
- Stock: par level, reorder point, storage location, shelf-life days, and a perishable toggle button
  using the `aria-pressed` pattern.
- Compliance: `AllergenPillToggle` bound to the form's `allergenCodes` array.

Measure-type lock: when `ingredient?.measureTypeLocked` is true, render the measure-type and stock-
unit fields `disabled` and wrap them in a `Tooltip` whose content is the UI contract's lock copy. The
lock is read from the server-computed flag; never re-derive it in the browser.

Submit maps the string-typed form values to the input schema shape in a `toIngredientInput()`
function, trimming strings, converting numerics, and dropping empty conversion rows. On success fire
a `sonner` toast and close; on error toast the server message.

`ingredients/page.tsx` is a client component using the existing `DataTable` primitive.

Header: an h1 reading "Ingredients" at `text-2xl font-semibold`, a muted description, and a
`PermissionGuard require="inventory.item.manage"`-wrapped `IngredientFormDialog` trigger labelled
"Add ingredient" per the Copywriting Contract.

Filter bar above the table: a search `Input` (case-insensitive substring on name and short name — the
hard gate; fuzzy matching is an explicit stretch goal, not required), a category select populated
from `useCategories`, a status select defaulting to Active with Archived and All options, and an
allergen multi-select using `AllergenPillToggle`. Debounce the search input with `useDebouncedValue`
at 300ms before it reaches the query, so typing does not fire a request per keystroke. Pass search,
category and status to the server through `useIngredients`; apply the allergen filter client-side
over the returned rows.

Columns: name (with the short name as a muted second line when present), category name (rendered as
a button that sets the category filter), stock unit, par level, reorder point, storage location,
status chip (`active` / `archived`), and a row-action `DropdownMenu` with Edit and Archive (or
Restore for an archived row).

Archive flow: a confirmation `Dialog` using the Copywriting Contract's ingredient archive copy, which
states that existing recipes, stock and purchase history keep referencing the ingredient and it only
disappears from pickers for new use. Unlike categories there is no in-use refusal path, so a plain
toast on error is acceptable here — but still close the dialog only on success.

Loading uses `DataTable`'s built-in skeleton; empty uses the `EmptyState` with the Copywriting
Contract's ingredient strings and an "Add ingredient" action.

Do not modify `inventory/layout.tsx`.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser user → ingredient mutations | Write actions gated by permission client-side and authorized server-side. |
| Search / filter input → query parameters | Untrusted strings forwarded to the API. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-08.2-151 | Elevation of Privilege | Create / edit / archive controls visible to a view-only role | medium | mitigate | Mutating controls are wrapped in `PermissionGuard require="inventory.item.manage"`; the server authorizes independently. |
| T-08.2-152 | Tampering | A measure-type change slipping through on an ingredient with transaction history | high | mitigate | The field is disabled from the server-computed `measureTypeLocked` flag, and the server returns 409 regardless; both are tested. |
| T-08.2-153 | Repudiation | An ingredient hard-deleted, orphaning stock and recipe history | high | mitigate | No delete control exists in the UI and no delete endpoint exists on the server; archive is the only path. |
| T-08.2-154 | Denial of Service | A list request per keystroke in the search box | low | mitigate | The search value is debounced at 300ms before reaching the query key. |
</threat_model>
