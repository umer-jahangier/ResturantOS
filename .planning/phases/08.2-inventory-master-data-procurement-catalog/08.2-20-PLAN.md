<objective>
Close the two carried-over infrastructure defects that CONTEXT.md commits to this phase and that no
other plan owns: the gateway's incomplete `resilience4j.circuitbreaker.instances` map, and the bash
dev-stack script's omission of purchasing-service.

Purpose: this phase puts real traffic on the gateway's `inventory-route` and `purchasing-route` for
the first time. Both routes carry a `CircuitBreaker` filter naming an instance that is not declared,
so both silently run on resilience4j's defaults — a 60-second open-state window and the reactive
Spring Cloud CircuitBreaker's 1-second call timeout. CONTEXT.md records the observed consequence,
reproduced twice: after any upstream restart the gateway returns 503 persistently, with the upstream
healthy on its own port and one UP instance in Eureka, until the gateway process itself is
restarted. Shipping eight new inventory/purchasing screens on top of that is shipping them onto a
route that cannot self-heal. The bash script defect is the same failure the 2026-07-22 vendor-create
bug came from, still live for anyone running the stack outside PowerShell.

Requirements rationale: INV-15 and PUR-08 are the two requirements whose deliverables ride entirely
on these routes — INV-15's recipe revision, cost preview, stock read model and stock-operation
screens all call `inventory-route`; PUR-08's catalog-backed PO line picker and real spend-by-category
analytics call `purchasing-route` and its internal seam into inventory-service. A route whose
breaker cannot recover makes both requirements intermittently unverifiable, and the bash-script gap
makes PUR-08 unrunnable on a non-Windows machine. This plan unblocks them; it does not itself
deliver either requirement's user-visible surface.

Output: four declared circuit-breaker instances with matching call timeouts, a config-drift test that
makes the omission impossible to repeat, and a bash dev-stack at parity with the PowerShell one.
</objective>
