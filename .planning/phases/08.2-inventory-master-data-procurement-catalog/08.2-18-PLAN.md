`VendorItemFormDialog({ vendorId, vendorItem, trigger })` — one dialog for add and edit. Fields:
linked ingredient (a `CatalogItemCombobox` populated from `useIngredients`, disabled in edit mode
because the backend forbids re-mapping an existing catalog row to a different ingredient — render a
muted note saying a re-map means creating a new catalog item), vendor SKU, description, pack
description, pack quantity, pack unit, order unit, minimum order quantity, order multiple, lead-time
days, a preferred toggle button and a catch-weight toggle button (both `aria-pressed` buttons — no
checkbox primitive exists). In create mode only, an initial price in rupees plus an initial
effective-from date.

`VendorItemPriceDialog({ vendorItem, trigger })` — a small form with the new price in rupees, a price
unit, an effective-from date converted with the shared `calendarDateToInstant` helper, an optional
branch scope select, and a contract-price toggle. Its description text states verbatim that this
creates a new price effective from the chosen date and that the old price is kept for history and
never overwritten. Show the current price above the input for comparison using `MoneyDisplay`. On
submit call `useRecordVendorItemPrice` — there is no update-price hook to call, by design.

`VendorCategoryTags({ vendorId, editable })` — renders the vendor's category tags as pills using the
`aria-pressed` toggle pattern when `editable`, or as plain muted pills when not, followed by a caption
in `text-muted-foreground` stating that the tags are used to filter and suggest and do not restrict
what can be ordered. When editable, changes are saved through `useReplaceVendorCategories`. The
caption is not decorative — it is how the D-01 invariant reaches the user, so it must always render,
including when the tag list is empty.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → vendor catalog and price endpoints | Untrusted SKU, pack, price and effective-date input. |
| Vendor catalog view | Contract pricing is commercially sensitive and must not be visible to roles without vendor.view. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-08.2-181 | Elevation of Privilege | Catalog and price write controls visible to a view-only role | high | mitigate | Every mutating control is wrapped in `PermissionGuard require="vendor.manage"`; the server enforces `@PreAuthorize` independently. |
| T-08.2-182 | Tampering | A UI affordance that overwrites a historical price | high | mitigate | No update-price hook, repository method or control exists; the only price action opens the create-a-new-effective-dated-price dialog, and the dialog copy states the invariant. |
| T-08.2-183 | Elevation of Privilege | Users believing category tags restrict what can be ordered | medium | mitigate | `VendorCategoryTags` always renders the filter-only caption, including when the tag list is empty. |
| T-08.2-184 | Tampering | Re-mapping a catalog row to a different ingredient, rewriting the meaning of historical purchase lines | high | mitigate | The linked-ingredient field is disabled in edit mode with an explanatory note, and the server's update request record has no `ingredientId` field at all. |
| T-08.2-185 | Repudiation | Price-delta direction conveyed by colour alone | low | mitigate | Each delta carries a directional icon alongside the semantic token. |
</threat_model>
