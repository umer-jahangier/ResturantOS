<objective>
Turn `ingredients` into real, manageable master data: additive V6 columns for the industry-standard
field set, per-item UOM conversions, an allergen set, a required primary category, a measure-type
lock, and archive-not-delete. Also expose the internal batch category-resolution seam that lets
purchasing-service compute spend by real ingredient categories.

Purpose: ROADMAP Success Criterion 2, INV-14, the re-opened INV-01, and the backend half of PUR-08's
Success Criterion 7. Today ingredients can only be created by raw SQL, `is_active` exists in the
database but not in the update request, and there is no DELETE anywhere in inventory-service.
Output: `V6__ingredient_master_data.sql`, an expanded ingredient domain and API, an internal
category endpoint, and two integration test suites.
</objective>

`ALTER TABLE recipes ADD COLUMN net_yield_pct NUMERIC(6,2) NOT NULL DEFAULT 100.00;`
This is the recipe-header cooking/reduction loss — the yield a batch loses between the sum of its
raw lines and the finished quantity a portion is cut from. `recipes` is the version-header table
from `V1__inventory_schema.sql` (D-01: one row per version, resolved by `effective_from`), so the
number is versioned for free and a revision can change it without rewriting history. Add a comment
above the statement recording three things: (a) it is D-07's recipe-header number and completes the
set of three; (b) the `100.00` default means every existing version is explicitly "no reduction
loss", so no shipped plate cost moves; (c) authoring and cost consumption of this number are
DEFERRED with the prep/sub-recipe builder that CONTEXT.md's "Out of scope" section already defers —
`RecipeCostPreviewService` (plan 08.2-07) does not read it in this phase, and no UI in plans
08.2-16/08.2-17 exposes it. Modelling it now is what keeps the additive-migrations-only constraint
from forcing a second `recipes` migration later. Because this is DDL with a constant default, RLS
does not filter it and no backfill statement is needed.
Scope guard: `recipes` is the only pre-existing table V6 touches besides `ingredients`. The
recipe-line table and its per-line waste column stay entirely out of this migration — neither its
name nor its columns may appear anywhere in the V6 file, including in comments.
