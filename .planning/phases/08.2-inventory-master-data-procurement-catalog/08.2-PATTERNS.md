### `Vendor` / `VendorItem` / `VendorItemPrice` (purchasing-service, new + extend)

**Analog:** `Vendor.java`, `VendorService.java`, `VendorController.java`

**Entity pattern** (`Vendor.java:10-58`) — plain `TenantAuditableEntity` extension, boolean `active` flag, nullable optional fields. Copy directly for `VendorItem` (add `vendorId`, `itemId` FK-pair, `vendorSku`, `packQty`, `packUomId`, `orderUomId`, `moq`, `leadTimeDays`, `isPreferred`, `isActive`).

**Service pattern — search + paginated list + create/update via one `apply()` helper** (`VendorService.java:35-58`):
```java
@Transactional(readOnly = true)
public Page<VendorDto> list(String search, Pageable pageable) {
    Page<Vendor> page = (search == null || search.isBlank())
            ? vendorRepository.findAll(pageable)
            : vendorRepository.findByNameContainingIgnoreCase(search.trim(), pageable);
    return page.map(this::toDto);
}
@Transactional
public VendorDto create(CreateVendorRequest req) {
    UUID tenantId = tenantContext.requireTenantId();
    Vendor vendor = new Vendor();
    vendor.setTenantId(tenantId);
    apply(vendor, req);
    return toDto(vendorRepository.save(vendor));
}
```
`VendorItemService.list(vendorId, pageable)` follows the same `Page<>` shape (needed for the vendor-catalog `DataTable`). **Critically, `VendorItemPriceService` must NOT follow this `apply()`-mutate-in-place pattern** — per D-01/locked decision, price changes are always `INSERT`, never `UPDATE`. Model `VendorItemPriceService.recordNewPrice(vendorItemId, price, effectiveFrom)` on the append-only precedent in `RecipeService.createVersion` (below), not on `VendorService.update`.

**Controller pattern — `@PreAuthorize` + `ApiResponse.paginated`** (`VendorController.java:36-57`):
```java
@GetMapping
@PreAuthorize("hasAuthority('vendor.view')")
public ApiResponse<List<VendorDto>> list(...) {
    Page<VendorDto> result = vendorService.list(search, PageRequest.of(page, size));
    return ApiResponse.paginated(result.getContent(), new PageMeta(...));
}
```
Note: `VendorController` uses `@PreAuthorize`, while `IngredientController` uses the `InventoryAuthorizationService`/OPA seam — **these are two different authz mechanisms in two different services.** New purchasing-service controllers (`VendorItemController`) must follow `VendorController`'s `@PreAuthorize("hasAuthority('vendor.manage')")` convention, reusing the existing `vendor.manage`/`vendor.po.approve` permission codes (no new codes per CONTEXT.md).

### Append-only effective-dated pricing/versioning (`VendorItemPrice`, `RecipeService` revision)

**Analog:** `Recipe.java` + `RecipeService.createVersion` (`RecipeService.java:94-136`)

```java
@Transactional
public RecipeDto createVersion(CreateRecipeVersionRequest request) {
    List<Recipe> priorVersions = recipeRepository.findByMenuItemIdOrderByVersionDesc(request.menuItemId());
    int nextVersion = priorVersions.stream().findFirst().map(r -> r.getVersion() + 1).orElse(1);
    priorVersions.stream().filter(Recipe::isCurrent).forEach(prior -> {
        prior.setCurrent(false);
        recipeRepository.save(prior);
    });
    Recipe recipe = new Recipe();
    recipe.setEffectiveFrom(request.effectiveFrom() != null ? request.effectiveFrom() : Instant.now());
    ...
    return toDto(recipeRepository.save(recipe), ...);
}
```
And the D-01 resolution seam (`RecipeService.java:56-65`):
```java
public Optional<Recipe> resolveEffectiveRecipe(UUID menuItemId, Instant atInstant) {
    UUID tenantId = tenantContext.requireTenantId();
    return recipeRepository.findEffectiveVersionsDesc(tenantId, menuItemId, atInstant).stream().findFirst();
}
```
Copy this exact shape for `VendorItemPrice`: never `UPDATE` a price row; always `INSERT` a new row with `effectiveFrom = now()` (or a supplied future date), and resolve "current price" via `findEffectiveVersionsDesc`-style query ordered by `effective_from DESC` filtered `<= now()`. This is also the direct analog for the *new* recipe edit-as-revision UI flow (INV-15/D-05) — `RecipeService.createVersion` is already the "revision, not mutation" pattern the phase must extend to a `PUT`-style "create new version pre-filled from current" endpoint.

**Migration comment convention to copy** (`V1__inventory_schema.sql:106-113` + `RecipeService.java:17-22` javadoc): document the invariant directly above the entity/table, exactly as done for `recipes` ("D-01: versioned, resolved by effective_from at order time").

### Recipe cost-preview endpoint (new)

**Analog:** `MacCalculator` (moving-average costing) + `UomConverter` (cross-dimension conversion) + `RecipeService`'s line-assembly loop (`RecipeService.java:123-133`)

The UI-SPEC.md's "live" plate-cost panel needs a **new, non-persisting** endpoint (e.g. `POST /api/v1/inventory/recipes/preview`) that takes a draft line array and returns computed batch/portion cost + per-line share, without ever calling `recipeRepository.save(...)`.
