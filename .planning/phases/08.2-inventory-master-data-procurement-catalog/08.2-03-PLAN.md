<objective>
Close the origin bug this phase was created from: a recipe scheduled for a future date renders
"Current: Yes" in the versions table while the coverage report counts it as missing, because the two
surfaces use different definitions. Make coverage three-state (COVERED / SCHEDULED / NO_RECIPE),
fix the N+1 in `getCoverage()`, and add the tenant predicate CONTEXT.md flags as missing on
`RecipeRepository.findByMenuItemIdOrderByVersionDesc`.

Purpose: ROADMAP Success Criterion 4 and INV-15's coverage clause. Also lays the server-side field
the UI contract requires so the frontend never re-derives coverage state client-side again.
Output: three-state `CoverageResponse`, a single-query `getCoverage()`, a tenant-scoped repository
query, and a `RecipeCoverageIT` case that seeds a FUTURE `effectiveFrom` â€” the exact case no test
covered.
</objective>
