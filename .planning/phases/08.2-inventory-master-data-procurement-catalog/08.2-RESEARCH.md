# Restaurant Inventory Master-Data & Procurement Domain — Research Brief

**Purpose:** design an industry-standard (not MVP) inventory master-data + procurement domain for ResturantOS.
**Method:** vendor documentation, help centers, and industry guidance from Restaurant365, Toast/xtraCHEF, MarketMan, Crunchtime, Oracle (Simphony/SimVen Inventory), Apicbase, Craftable, Lightspeed, Supy, meez.
**Date:** 2026-07-22

---

## 0. Executive summary (read this if nothing else)

Every serious platform converges on the same skeleton, and the differences are mostly naming:

1. **Three distinct item concepts, never one.** `InventoryItem` (purchasable/countable) → `Recipe` (prep/sub-recipe, which may itself be countable) → `MenuItem` (POS-sellable). R365 calls them Purchased Item / Recipe / Menu Item; Apicbase calls them Ingredient / Semi-finished product / Finished product; Crunchtime calls them Company Product / Recipe / POS Item.
2. **Vendor identity is separated from item identity.** The `VendorItem` (vendor SKU, pack size, price) is its own entity linked many-to-one to the internal item. This is the single most important structural decision in the whole domain. R365 states it explicitly: "Purchased Items ... are **not** Vendor-specific, while Vendor Items are vendor-specific Items associated directly with a single Purchased Item."
3. **Categories are a shallow hierarchy (2–3 levels), and they carry GL/COGS semantics.** R365 uses exactly three levels with a most-specific-wins GL override. Crunchtime uses category / subcategory / microcategory. Not a deep arbitrary tree.
4. **UOM is a 3-role model over a typed measure system** (Weight / Volume / Each), with global conversions plus item-specific bridging conversions (density, each-weight, pack size).
5. **Yield/waste is a per-recipe-line percentage AND/OR a prep recipe.** Both exist; prep recipes are the higher-fidelity mechanism.
6. **Purchasing is PO → Receipt(GRN) → Invoice with tolerance-based 3-way match**, and the PO line is selected from the **vendor catalog**, not from the item master.
7. **Master data is archived, never deleted, once referenced.**

## 1. Ingredient / Item Master

### 1.1 What the platforms actually store

**Restaurant365 — Purchased Item record** ([docs](https://docs.restaurant365.com/docs/adding-a-new-purchased-item)):

| Section | Fields |
|---|---|
| General | Item Categories 1, 2, 3; Name; Identifying Number; Description; Active; **Key Item**; **Usage Type** ("switch to actual usage for this item in place of theoretical usage") |
| Item Accounts | **Cost Account** (GL for COGS), **Inventory Account** (GL asset), **Waste Account** (GL debited by waste log) |
| Units of Measure | **Measure Type** (locked after save), Reporting U of M, Inventory U of M 1/2/3, Measurement Conversions |
| Costing | Cost Method (avg/last/etc.) |
| Multi-unit | Location Groups; Location-Specific Settings |
| Flags | Sales (AR invoicing), Catering, Prep |

Note also `Donation` GL account and `Actual as Theo` toggle from the [Items Grid](https://docs.restaurant365.com/doc/docs/items-grid) bulk-edit list, plus **barcodes** as a first-class multi-value column.

**Oracle Simphony / SimVen inventory item** ([Oracle docs](https://docs.oracle.com/cd/E93646_01/doc.311/e93648/t_masterfile_setup_inventory_items.htm)):
Item Number, Item Category (SALE/NONS/COMP), Family Group, Item Class, Major Group, POS Name, Long Description; Purchase Units / Transfer Units / Selling Units with **Parts per Unit Purchased** and **Parts per Transfer Unit**; Purchase Cost; Sell Price; **Warehouse Location**, **Minimum Level / Maximum Level / Par Level**; **Primary Vendor ID**; flags for Metered Item, Perishable.

**Toast Inventory** — the "Inventory Stock Data" import template columns are: *ingredient name, category, unit of measure, unit price, par amount, unit conversions, supplier, pack size, storage area* ([Toast support](https://support.toasttab.com/en/article/Get-Started-with-Toast-Inventory)). That template is effectively the industry's minimum viable field set.

**Apicbase — Ingredient** ([field guidance](https://support.apicbase.com/help/general-recommendations-for-filling-out-ingredients)):
Product Name; **Simplified Product Name** (short display name used inside recipes); Category / Subcategory; **Accounting Category** (separate from operational category!); Brand; Description/Specifications; **Supplier**; **Unique Article Number (EAN/UPC)**; package mass/volume; price; **Ingredient Verification status (Approved/Allowed)**; Allergens; Dietary info; Nutritional info; **Shelf Life**; **Storage Instructions**; custom fields; images; **outlet assignments**; "Used in" (reverse recipe references).

**Crunchtime** — split between **Company Product** (global item/ingredient) and **Location Product** (per-site assignment carrying storage location, **Min Par / Max Par**, Issue & Inventory Cost, count-sheet sequence, transaction eligibility flags) ([developer docs](https://developer.crunchtime.com/docs/inventory-recipe), [support](https://support.crunchtime.com/hc/en-us/articles/360013143173-How-do-I-create-or-add-a-new-Location-Product)).

**MarketMan** — item = name + category + purchase info (supplier, product code, size, pack nickname, price) ([MarketMan](https://www.marketman.com/platform/restaurant-purchasing-software-and-order-management)). Note that MarketMan powers Square for Restaurants' inventory offering ([Square](https://squareup.com/us/en/inventory-management/restaurants)), so "Square for Restaurants inventory" ≈ MarketMan's model.

### 1.2 Consolidated canonical field set for an inventory item

- **Identity:** id, tenant/org, internal item code/number, name, short/display name, description, barcodes[] (GTIN/UPC/EAN), brand, image
- **Classification:** category (leaf, FK into hierarchy), tags[], accounting category (optional override), item type (see 1.3), key-item flag
- **Costing/GL:** inventory (asset) account, cost (COGS) account, waste account, donation/comp account, cost method (moving average / last cost / FIFO), current average cost, last cost
- **UOM:** measure type (WEIGHT | VOLUME | COUNT), stock/inventory UOM, reporting UOM, recipe/usage UOM, alternate count UOMs[]
- **Ops:** default storage location, storage instructions, shelf-life days, perishable flag, temperature class, count-sheet sequence (per location), min/max par, reorder point, reorder qty/multiple
- **Purchasing:** default/preferred vendor item, purchasable flag, lead time (usually on vendor item), tax code/VAT class
- **Compliance:** allergens[], dietary attributes, nutrition (per 100g/base unit), country of origin, spec sheet URL
- **Lifecycle:** active/archived, effective from/to, created/modified audit, per-location availability

### 1.3 Inventory item vs prep item vs menu item

| | Inventory Item | Prep / Sub-recipe Item | Menu Item |
|---|---|---|---|
| Origin | Purchased from a vendor | Produced in-house from a recipe | Sold through the POS |
| Has a cost from | Invoices / vendor price | Rolled up from its recipe components ÷ yield | Rolled up from its recipe(s) |
| Counted in inventory | Yes | Optionally (R365: "Available in Inventory"; Crunchtime: prep item production) | No |
| Depletes on sale | Only via a recipe | Via parent recipe, or directly if it is the countable item | Triggers depletion of its recipe |
| Has a vendor | Yes | No (unless also purchasable, e.g. buy-or-make marinara) | No |
| Apicbase term | Ingredient | Semi-finished product | Finished product |
| R365 term | Purchased Item | Recipe (Prep) | Menu Item + Menu Item Link |

**Key subtlety:** a prep item can be *both* produced and purchased (buy-vs-make). The clean modeling answer is that "stockable thing" and "how it is sourced" are different axes — see the recommended model in §9.

Apicbase draws the line as: finished products "designated for sale, requiring portioning specifications" vs semi-finished products, "intermediate recipes like sauces or doughs used as sub-recipes... without specific portioning requirements" ([Apicbase](https://support.apicbase.com/help/create-a-new-recipe)).

## 2. Categories

### 2.1 Hierarchy vs flat — the verdict is *shallow hierarchy*

- **Restaurant365:** exactly **three** levels — Item Category 1 / 2 / 3. "Category 1 typically represents broad level categories. Category 2s typically represent more detailed categories within each Category 1, and usually line up with P&L Accounts in the food categories." Examples given: Cat1 = Food, Liquor, Beer; Cat2 = Dairy, Vodka, Bottle Beer; Cat3 = IPA, Merlot, Beef ([Item Categories](https://docs.restaurant365.com/docs/item-categories)).
- **Crunchtime:** "product categories, subcategories, and **microcategories**" — again three levels ([developer docs](https://developer.crunchtime.com/docs/inventory-recipe)).
- **Apicbase:** Category + Subcategory + a *separate* Accounting Category ([Apicbase](https://support.apicbase.com/help/create-custom-fields-categories)).
- **Supy's guidance:** use "generic" parent categories — **7–8 maximum** — with specific leaf-level groupings, organized by usage pattern rather than arbitrary divisions ([Supy](https://supy.io/blog/how-to-manage-items-list-in-restaurant-inventory-software)).
- **MarketMan:** flat-ish operational list (proteins, bread, dairy, produce, canned goods, dry goods, alcohol, cooking items) aligned to physical shelf organization for shelf-to-sheet counting ([MarketMan](https://www.marketman.com/blog/restaurant-inventory-categories)).

Nobody ships an arbitrary-depth tree. Two or three levels is the ceiling in every product examined.

### 2.2 What categories drive

Confirmed uses across products:

1. **GL account defaulting** — R365: the item's Cost Account "will be auto-filled with the cost account associated with its lowest item category assignment," and "Item Category 3 will override Category 2, and Category 2 will override Category 1"; if a level has no linked account it falls back up the chain to Category 1.
2. **COGS / P&L reporting groups** — category 2 "usually line up with P&L Accounts."
3. **Variance thresholds** — R365 supports per-Item-Category acceptable variance caps for actual-vs-theoretical ([R365](https://help.restaurant365.net/support/solutions/articles/12000086222-item-category-variance-caps)).
4. **Purchase-order suggestion filters** — categories can be excluded from PO suggestions.
5. **Count-sheet grouping/ordering** — MarketMan/Toast shelf-to-sheet: sheet categories mirror shelf categories.
6. **Invoice line coding** — xtraCHEF maps invoice line items to a purchase Category, and "Categories themselves are mapped to GL codes which your accounting system uses"; mappings are remembered for future invoices ([Toast](https://support.toasttab.com/en/article/xtraCHEF-How-to-Map-Products)).
7. **Vendor catalog browsing/filtering** — categories filter the order guide, but note this is a *filter*, not an authorization.

### 2.3 One category or category + tags?

Standard practice: **one primary category assignment (the leaf of a shallow tree) + free-form tags/custom fields for everything else.** Apicbase explicitly separates the operational Category/Subcategory from the **Accounting Category** and then adds arbitrary **custom fields** for the rest. R365 gets three orthogonal-ish slots but the "one path" pattern dominates because GL derivation requires a single deterministic answer.

Do **not** make the category multi-valued: GL derivation, COGS rollup, and variance caps all need exactly one answer per item.

### 2.4 Naming conventions worth copying

R365 recommends "using ALL CAPS for item prefixes to easily distinguish item prefixes from item names," e.g. `PROD` (Produce), `MEAT`, `BEER`, and auto-prefixing item names by category. Supy recommends encoding discriminating spec into the name itself ("Shrimps HOSO 50/60", not "Shrimps"), one language, and a standardized abbreviation list.

## 3. Units of Measure

### 3.1 The typed-measure + equivalence model (R365 is the clearest reference implementation)

R365 defines **three Measure Types: Weight, Volume, Each**. "Weight has a base of OZ-wt... Volume has a base of OZ-fl"; Each is its own dimension. Every UOM equates back to one of these three bases. "By converting one UofM to another and defining the equivalency between different Units of Measure, Restaurant365 can calculate all Units of Measure for those Measure Types." Conversions carry an **Equivalent Qty** — e.g. Equivalent Qty of 4 means there are 4 `Bottle-750ML` in the new UOM `Case-4/750ML` ([R365 UOM](https://docs.restaurant365.com/docs/unit-of-measure), [UOM Conversions](https://docs.restaurant365.com/docs/unit-of-measure-conversions)).

Critically, **Measure Type is locked once the item record is saved** — because changing dimension retroactively would invalidate every historical transaction and recipe line. Copy this constraint.

Oracle's SimVen expresses the same idea with explicit factors: Purchase Units / Transfer Units / Selling Units + **Parts per Unit Purchased** + **Parts per Transfer Unit**.

### 3.2 The 3-UOM roles

| Role | Example | Where used |
|---|---|---|
| **Purchase UOM** (a.k.a. order/pack unit) | Case 6×#10 can | PO lines, receiving, invoices — lives on the **VendorItem**, not the item |
| **Stock/Inventory UOM** (base) | lb, or "can" | On-hand balances, valuation, counts. Alternate count UOMs allowed on count sheets |
| **Recipe/Usage UOM** | oz, fl-oz, each | Recipe lines, depletion |

Crunchtime's practical note: use **Alternate Count Units** (cases or cans, not just pounds) on count sheets "to minimize mathematical errors during physical counts," but avoid *location-specific* count units because they hurt multi-site scalability ([Crunchtime](https://www.crunchtime.com/blog/5-best-practices-for-your-restaurants-inventory-workflow)).

### 3.3 Cross-dimension conversions (the hard part)

Global conversions handle within-dimension (lb↔oz, gal↔fl-oz). Cross-dimension needs **item-specific** bridges:

- **Density** (volume ↔ weight): 1 cup flour = 4.25 oz-wt
- **Each-weight** (count ↔ weight): 1 avocado = 6 oz-wt
- **Pack size** (count ↔ count): 1 case = 6 cans; 1 can = 102 fl-oz

So: a global `uom_conversion` table for dimension-internal factors, plus an `item_uom_conversion` table scoped to a single item for the bridges. R365 confirms conversions can be defined at item and recipe level, and "are available to Items and Recipes whether they are Inventoried or not."

### 3.4 Yield / waste — AP vs EP

Standard culinary math ([CIA Kitchen Calculations](https://www.ciachef.edu/wp-content/uploads/2024/07/kitchen-calculations.pdf), [Intro to Food Production ch.7](https://psu.pb.unizin.org/hmd329/chapter/ch7/)):

```
yield % = EP weight / AP weight
conversion factor = 100 / yield %
EP cost per unit = AP cost per unit × conversion factor
```

Two mechanisms exist in real products, and **mature systems support both**:

1. **Per-recipe-line waste %** — Apicbase's `Prep. Waste%` on each ingredient line; Apicbase's plate cost formula is literally `Σ (Quantity used × Unit price) ÷ Yield %` ÷ portions ([Apicbase](https://get.apicbase.com/food-cost-calculator/)).
2. **A prep recipe that converts units** — the higher-fidelity approach. Per FCM: "Using a prep recipe to convert units — like turning a whole fish bought by the Pound into fillets used by the Ounce or Each — is the best way to track your true costs and account for any product lost during prep" ([FCM](https://foodcostmanagement.zendesk.com/hc/en-us/articles/49478936542867-Prep-Recipes-and-Yields)).

There is also a third, subtler one: Apicbase's **"Net Measurement info"** on a recipe, used "when finished product weight differs from ingredient totals — such as when a sauce reduces during cooking." This is *cooking* yield (evaporation/shrink) as opposed to *trim* yield, and it belongs on the recipe header, not the line.

**Recommendation: model three distinct numbers** — item-level default yield %, recipe-line waste % override, and recipe-header net-yield. Do not collapse them.

## 4. Recipes / BOM

### 4.1 Recipe types

- **Prep / batch / sub-recipe (semi-finished):** has a batch yield qty + UOM, may be inventoried and counted (R365 "Available in Inventory"), can be a component of other recipes. No sell price required.
- **Plate / menu recipe (finished product):** has portions, links to POS menu item(s), has sell price, target margin, plate cost.

Apicbase's `Product Type` field is the discriminator; `#portions` defaults to 1 if blank. Crunchtime's Recipe Enhanced Service creates recipes "for inventory depletion through POS sales and prep item production" — i.e. recipes serve two consumers: sales depletion and production ([Crunchtime dev docs](https://developer.crunchtime.com/docs/inventory-recipe)).

### 4.2 Recipe fields observed

R365 recipe record surfaces: recipe name, **Yield**, **Cost per batch**, ingredients with measurements and individual costs, **Prep minutes / Cook minutes / Finish minutes**, **Shelf life days**, Serving plate, Container, Tools, Steps, plus a print date warning: "Always review the dates of printed recipes to ensure that the most up-to-date recipe is being viewed" ([R365 recipes](https://docs.restaurant365.com/docs/recipes)).

Apicbase adds a **Financial tab**: Sell price (incl./excl. tax), VAT %, **Target Profit Margin**, and production cost including labor when hourly wages are configured. Allergens auto-populate from ingredients but can be manually supplemented.

### 4.3 Nesting and cost roll-up

A recipe line references *either* an inventory item *or* another recipe. Cost rolls up recursively:

```
component_cost = qty_in_recipe_uom × unit_cost(component) ÷ (yield% × (1 - waste%))
batch_cost     = Σ component_cost  (+ optional labor, packaging, overhead)
unit_cost(prep recipe) = batch_cost ÷ batch_yield_qty
plate_cost     = batch_cost ÷ portions
```

Guard against **cycles** (recipe A → B → A). This must be an explicit graph validation on save, not discovered at cost-calculation time.

### 4.4 Versioning and effectivity dating

The industry stance: recipes must be version-controlled, and cost history must remain accurate after a recipe changes. "Backdating recipe changes ensures historical cost reports remain accurate even after a recipe is modified"; the system "must be able to indicate which recipe variation is currently being used as the control recipe" ([FoodDocs](https://www.fooddocs.com/post/recipe-management), [Supy](https://supy.io/blog/recipe-costing-software-guide)).

Practical implementation pattern:
- `Recipe` (stable identity, name, type) → `RecipeVersion` (version no., `effective_from`, `effective_to`, status draft/active/archived, approved_by) → `RecipeLine` (belongs to a version).
- Theoretical usage for a sales date resolves the version effective on that date.
- **Snapshot cost onto transactions.** Never recompute historical theoretical cost from live master data.

### 4.5 Theoretical vs actual, and recipe coverage

- **Theoretical food cost (TFC):** Σ (recipe portion cost × portions sold) ÷ revenue.
- **Actual food cost (AFC):** (opening inventory + purchases − closing inventory) ÷ revenue.
- **Variance = AFC − TFC**, "the single most useful diagnostic number in back-of-house operations"; acceptable band is typically **2–3%** ([meez](https://www.getmeez.com/blog/actual-vs-theoretical-food-costs), [Supy](https://supy.io/blog/learn-theoretical-food-cost-vs-actual-guide), [Chefs Resources](https://www.chefs-resources.com/kitchen-management-tools/whiteboard-chef-management-series/theoretical-food-cost-vs-actual-food-cost-acceptable-variances/)).

**Why recipe coverage matters:** every menu item sold *without* a linked recipe contributes revenue and real consumption but **zero theoretical cost**, which silently inflates the variance and makes the number useless as a diagnostic. Industry commentary flags exactly this: give-away items (bread, butter, ketchup, seasonings) "won't be represented in theoretical food cost but will show up in actual food cost and create variance" ([R&R FMG](https://rrfmg.com/could-my-food-cost-issue-be-the-recipe-itself/)).

R365 mediates POS→recipe with **Menu Item** records (auto-created from POS integration; carry display name, estimated cost, price, target margin), **POS Items** that group differently-named menu items across locations, and **Menu Item Rules** (`WHEN Menu Item is X THEN deplete Recipe Y`) ([R365](https://docs.restaurant365.com/docs/menu-items-menu-item-links)). Notably R365's docs do *not* prominently surface an unlinked-menu-item report — **this is a genuine gap you can beat them on.** Ship a first-class "Recipe Coverage" dashboard: % of menu items with an active recipe, weighted by sales volume, with an explicit "uncosted revenue %" number.

## 5. Vendors & Vendor Catalogs

### 5.1 The two candidate designs

**(a) Vendor ↔ Category assignment** — "Vendor X supplies Produce and Dairy." Cheap to set up; used only as a soft filter. Gofrugal mentions categorizing vendors "for easier selection" ([Gofrugal](https://www.gofrugal.com/restaurant/procurement-software.html)). No serious back-office platform uses this as the *purchasing* constraint, because it can't answer "what does this cost, in what pack, under what SKU?"

**(b) Explicit Vendor Item catalog** — this is the industry standard, unambiguously.

R365 states the split directly: "Purchased Items are counted during the Inventory Count process and used as Ingredients in Recipe Costing and **are not Vendor-specific**, while Vendor Items are vendor-specific Items associated directly with a single Purchased Item for purposes of AP Invoicing and Purchase Orders." Cardinality: "multiple Vendor Items can be associated with a single Purchased Item. Vendor Items may not be associated with more than one Purchased Item." Fields include vendor, vendor item number/SKU, description, UOM, pack size, **Contract Price** ("the current expected vendor price used for purchase orders" and price verification), plus **Split U of M / Split Price** for EDI ([R365 Vendor Items](https://docs.restaurant365.com/docs/vendor-items-link-items-sidesheet)).

MarketMan: purchase info per item = supplier + product code + size + pack nickname + price; "manage catalogs, pricing, and order history for every supplier, compare prices across vendors, and send orders to multiple suppliers simultaneously" ([MarketMan](https://www.marketman.com/platform/restaurant-purchasing-software-and-order-management)).

Oracle: **Vendor Purchase Catalog** under Master Data, with receiving-unit maintenance and internal barcode generation for vendor purchase items ([Oracle](https://www.oracle.com/food-beverage/restaurant-pos-systems/restaurant-inventory-management-software/)).

Apicbase: ingredient carries **Supplier** + **Unique Article Number (EAN/UPC)**, and the recipe ingredient picker displays "package sizes available, supplier name, and the supplier article-number."

xtraCHEF: invoice line items (vendor products) get **mapped** to internal categories/items on first sight and the mapping is remembered — an implicit vendor-catalog materialized from invoices ([Toast](https://support.toasttab.com/en/article/xtraCHEF-How-to-Map-Products)).

Crunchtime: order recommendations powered by PAR levels, consumption history, forecast, and on-hand — which requires vendor-item pack sizes to convert a needed quantity into orderable cases.

### 5.2 Verdict and tradeoffs

**Do (b) — the explicit VendorItem catalog — as the system of record. Add vendor↔category only as a UX filter / default-vendor heuristic, never as a constraint.**

| Design | Pros | Cons |
|---|---|---|
| Vendor↔Category only | Trivial setup; works for tiny operators | Can't price a PO line; can't do 3-way match; can't compare vendors; no pack-size math; forces free-text PO lines |
| VendorItem catalog only | Exact pricing/pack/SKU; enables 3-way match, price comparison, EDI, price-change alerts | Setup burden; needs import + invoice-learning to bootstrap |
| **Both (catalog canonical, category as filter)** | Best of both: fast pickers and defaults, exact commerce | Slightly more model surface |

Mitigate the setup burden the way the market does: (i) CSV/Excel catalog import per vendor, (ii) **auto-create vendor items from received invoices/OCR and prompt to map** (xtraCHEF's pattern), (iii) "freehand ordering" — Craftable lets users "quickly populate order lines complete with SKUs and GL Coding" for items not yet in the system, then reconcile later ([Craftable](https://support.craftable.com/hc/en-us/articles/360012051893-Creating-a-Mixed-Vendor)).

### 5.3 How this constrains PO line selection

PO lines must be **VendorItem** references, not item references. Once vendor is chosen on the PO header, the line picker is scoped to that vendor's catalog. Qty is entered in the vendor item's **order UOM** (case), and the system converts to stock UOM via pack size for the receipt.

Consequences to design for:
- **Preferred/primary vendor** flag per item (Oracle has `Primary Vendor ID`) drives suggested-order vendor split.
- **Multi-vendor price comparison**: because N vendor items point at one internal item, you can rank by normalized price-per-stock-UOM. NxtEdge markets exactly this: "compare pricing, pack sizes, and alternatives from one screen... compare identical products, substitute pack sizes, and approved alternatives in real time" ([NxtEdge](https://nxtedge.net/restaurant-vendor-price-comparison-software/)).
- **Contract pricing**: price with `effective_from`/`effective_to` (+ optional per-location contract), and flag invoice lines that deviate. Supy: "enforcing approved prices across every order, with automatic invoice discrepancy detection" ([Supy](https://supy.io/platform/restaurant-procurement-software)).
- **Min order qty, order multiple, lead time, delivery days, cut-off time** live on the vendor item / vendor-location relationship.

## 6. Purchasing flow

### 6.1 Canonical chain

```
Suggested Order (par/forecast) → Purchase Order → Receipt / GRN (1..n, partial) → Vendor Invoice → 3-way match → AP posting → GL
                                                    ↘ Credit memo / return
```

**Three-way match** compares PO, goods receipt note, and supplier invoice, releasing payment only when all three agree within tolerance ([Ramp](https://ramp.com/blog/accounts-payable/3-way-match), [ininvoice](https://www.ininvoice.com/en/three-way-matching)).

**Tolerances:** typical configuration is a ±2% price/qty variance auto-approving with 3%+ flagged for review; invoice qty must equal GRN qty and must not exceed PO qty ([Ramp GRN guide](https://ramp.com/blog/accounts-payable/goods-receipt), [Mindsprint](https://www.mindsprint.com/resources/blogs/2-way-vs-3-way-po-matching)). Common exceptions: partial shipments, price errors, missing PO number, receipt data-entry mistakes.

Design tolerances as **configurable policy** at tenant → vendor → category level, with separate price tolerance (% and absolute) and qty tolerance (% and absolute), plus an over-receipt policy (block / warn / allow).

### 6.2 Receiving specifics that matter in restaurants

- **Partial receipts** are the norm (produce shorts, substitutions). Model `ReceiptLine.qty_received` against `POLine.qty_ordered` with a running `qty_outstanding`; close-short is an explicit action.
- **Substitutions**: vendor ships a different pack size / different SKU. The receipt line must be able to point at a *different vendor item* than the PO line, with a link back.
- **Catch weight**: proteins ordered by case but billed by actual weight. The receipt needs both `qty_units` and `qty_weight`.
- **Price at receipt vs price on PO**: receiving is where cost enters inventory; capture landed unit cost per receipt line and use it to update moving-average cost.
- **Credits / returns / damaged goods** must be first-class, since they feed AP and inventory in opposite directions.

### 6.3 How the catalog feeds PO UX

Suggested ordering: on-hand + on-order vs par → shortfall in stock UOM → divide by vendor item pack size → round up to order multiple → PO line in cases. Crunchtime's recommendations use "PAR levels, historical consumption, sales forecast, on-hand inventory." R365 lets you exclude categories from PO suggestions entirely.

## 7. UX patterns worth copying

### (a) Item list management
- **Dense grid with 30+ columns, show/hide columns, saved views, CSV export** (R365 Items Grid).
- **Inline editing right in the grid**: toggles (Active, Key Item), dropdowns (Categories 1–3, GL accounts, cost method), text (item number), and a barcode sub-editor.
- **Bulk edit via checkbox selection + pencil icon**: R365 bulk-edits Location Group, Active, Key Item, Actual-as-Theo, Item Categories 1/2/3, and GL accounts (Inventory, Cost, Waste, Donation). Bulk category/GL reassignment is the single highest-value bulk operation.
- **CSV/Excel import template** as a first-class onboarding path (Toast's "Inventory Stock Data" template: name, category, UOM, unit price, par, unit conversions, supplier, pack size, storage area).
- **Fuzzy search** so "Apple Red" finds "Red Apple" and "Selery" finds "Celery" — Supy explicitly calls out typo-tolerant search as the primary defense against duplicate item creation.
- **"Used in" reverse references** on the item (Apicbase) so you can see which recipes break before you change something.

### (b) Recipe builder with live plate cost
- Search-as-you-type component picker that returns **both items and sub-recipes** in one list, showing "package sizes available, supplier name, and supplier article-number" inline (Apicbase).
- Per-line: qty, unit, waste %, line cost, **% of total plate cost** (drives the "what's actually expensive here" insight).
- Sticky cost panel: batch cost, cost per portion, sell price, **food cost %, gross margin, target margin delta**. Apicbase exposes Target Profit Margin; Supy triggers alerts when a dish exceeds target food cost.
- Scale/resize: change portions or batch size and recalculate everything.
- Expand/collapse nested sub-recipe to show its own roll-up.
- Print/station view with prep/cook/finish minutes, shelf life, plating, and an explicit **version + date stamp** on the printout.

### (c) Category management
- Small tree editor (2–3 levels), each node carrying default Cost/Inventory/Waste GL accounts, variance cap %, and "exclude from PO suggestions."
- Show item counts per node and block/redirect on delete of a non-empty node (merge-into flow).
- Enforce/suggest naming prefixes (`PROD`, `MEAT`, `BEER`).

### (d) Vendor setup with catalog
- Vendor header: code (must match accounting system exactly — xtraCHEF is emphatic about this), contacts, terms, default GL/AP account, delivery days, cut-off times, order method (email/EDI/portal), min order value.
- **Catalog tab as a grid**: vendor SKU, description, pack size, order UOM, price, effective dates, lead time, min qty, order multiple, and a **Linked Item** column with a mapping status chip (Mapped / Unmapped / Ambiguous).
- **Unmapped queue**: everything arriving off invoices lands here for one-click mapping (xtraCHEF's remembered mapping).
- **Price change report / alerts** per vendor item over time — this is what operators actually open weekly.

### (e) Counting
- Count sheets **ordered by storage location, then by a per-location sequence number**, mirroring physical shelf order — Crunchtime: "Storage Locations represent the physical places where inventory is counted (walk-ins, dry storage, bar rails)" and dictate count-sheet appearance; users "enter/edit the appropriate number to indicate where it should appear on inventory count sheets."
- Distinguish **primary** storage location (receive + deplete) from **secondary** (count only) — Crunchtime's model, worth copying verbatim.
- Alternate count units (case/can/bottle) per line to reduce mental math.
- Mobile-first entry, resumable, with variance highlighting on submit.

## 8. Pitfalls (what teams get wrong)

1. **Collapsing vendor identity into item identity.** Putting `vendor_sku`, `pack_size`, `price` directly on the ingredient row. It works until the second supplier, then you get duplicate ingredients ("Tomatoes – Sysco", "Tomatoes – US Foods"), broken recipe costing, and impossible price comparison. R365's explicit statement that purchased items "are not Vendor-specific" is the corrective.
2. **Deep/over-normalized category trees.** Nobody in the market ships arbitrary depth. Deep trees make GL derivation ambiguous, break report subtotals, and invite one-item leaves. Cap at 3 levels; use tags for the long tail.
3. **Multi-valued primary category.** GL defaulting, COGS grouping, and variance caps each need exactly one deterministic answer.
4. **Not versioning recipes / not snapshotting cost.** If theoretical cost is recomputed from live master data, last month's variance report changes every time someone edits a recipe or a price lands. Snapshot unit cost and recipe version onto every transaction; keep `effective_from/to` on recipe versions.
5. **Letting measure type change after transactions exist.** R365 hard-locks it. Copy that.
6. **Storing conversions as floats and re-rounding at every hop.** lb→oz→g→recipe-line→depletion accumulates error across millions of depletion events. Use `DECIMAL` with generous scale, canonicalize to a base unit per dimension, convert **once** from source to base and once from base to target (never chain), and round only at display and at GL posting.
7. **Ignoring the count↔weight/volume bridge.** "Each" is not convertible to weight globally; it's per item. Systems that skip item-level density/each-weight force users into fake UOMs and wrong costs.
8. **One yield number for everything.** Trim yield (AP→EP), cooking/reduction yield, and per-recipe waste are three different phenomena. Collapsing them makes prep recipes lie.
9. **Deleting master data instead of archiving.** R365's rule: a purchased item can only be deleted if it is not an ingredient on any recipe and not tied to vendor items on any transaction; otherwise **deactivate**. Crunchtime's advice: uncheck "Storage Active" and reassign primary location rather than deleting, so it drops off count sheets without orphaning history. Enforce this with FK restrict + an `archived_at` column; never cascade-delete.
10. **No cycle detection in the recipe graph.** A → B → A will hang or stack-overflow cost roll-up. Validate on save.
11. **Duplicate items from weak search.** Fuzzy/typo-tolerant search plus a duplicate-detection prompt at create time is a data-model-preserving feature.
12. **Location-specific units and location-specific costs done too early.** Crunchtime warns location-specific count units "reduce scalability across multiple locations." Model global item + per-location overrides, and default the overrides to null.
13. **No un-costed-revenue metric.** Without a recipe-coverage number, the AvT variance is a mystery number rather than a diagnostic.
14. **Treating PO/receipt/invoice as one document.** Partial receipts, substitutions, catch weight, and credits all require them to be separate entities with explicit links.
15. **Not separating "operational category" from "accounting category."** Apicbase keeps both. Chefs want Produce > Leafy Greens; accounting wants a P&L line. Usually aligned, occasionally not — allow an override.

## 9. RECOMMENDED DATA MODEL

### 9.1 Entity map

```
Tenant
 └─ Location (site)                                UomDimension (WEIGHT|VOLUME|COUNT)
      ├─ StorageLocation                            └─ Uom (code, name, dimension, factor_to_base)
      │                                                  └─ UomConversion (global, within dimension)
ItemCategory (self-ref, max depth 3)
 ├─ default_inventory_account / cost_account / waste_account / donation_account
 ├─ variance_cap_pct, exclude_from_po_suggestions
 └─ Item ──────────────────────────────────────────────────┐
      ├─ ItemUomConversion (item-scoped bridges: density, each-weight)
      ├─ ItemLocation (per-site: storage loc, par min/max, reorder pt/qty, count seq, active)
      ├─ ItemTag (m:n)
      ├─ ItemAllergen (m:n)
      ├─ ItemBarcode (GTIN/UPC/EAN)
      └─ 1:n VendorItem ◄── Vendor
                 ├─ VendorItemPrice (effective-dated, optional per-location, contract flag)
                 └─ referenced by PurchaseOrderLine / ReceiptLine / VendorInvoiceLine

Recipe (stable identity)
 └─ RecipeVersion (version_no, effective_from/to, status, yield_qty, yield_uom, portions, net_yield_pct)
      └─ RecipeLine (component_type ITEM|RECIPE, component_id, qty, uom, waste_pct, sequence, note)

MenuItem (POS-sourced)  ──  MenuItemRecipeLink (effective-dated, qty/multiplier)  ──►  Recipe
ModifierItem            ──  same link table                                       ──►  Recipe

PurchaseOrder → PurchaseOrderLine → GoodsReceipt → GoodsReceiptLine → VendorInvoice → VendorInvoiceLine
                                                                       └─ MatchException
InventoryTransaction (RECEIPT|COUNT|WASTE|TRANSFER|PRODUCTION|DEPLETION|ADJUSTMENT)
```

### 9.2 Core entities and key fields

**`item`** — the single stockable/purchasable master record. *One table, discriminated by `item_type`, not three tables.*

| Field | Notes |
|---|---|
| `id`, `tenant_id`, `item_code` (unique per tenant), `name`, `short_name`, `description` | `short_name` for recipe-line display (Apicbase's Simplified Product Name) |
| `item_type` | `PURCHASED` \| `PREPARED` \| `BOTH` — replaces the buy-vs-make dilemma. `PREPARED`/`BOTH` items carry `produced_by_recipe_id`. |
| `category_id` | FK to leaf `item_category`; single-valued, required |
| `accounting_category_id` | nullable override |
| `measure_type` | WEIGHT/VOLUME/COUNT — **immutable once any transaction exists** |
| `stock_uom_id`, `recipe_uom_id`, `reporting_uom_id` | roles, all in/derivable within `measure_type` (+ bridges) |
| `inventory_account_id`, `cost_account_id`, `waste_account_id`, `donation_account_id` | nullable; resolved from category chain most-specific-first when null |
| `cost_method` | MOVING_AVG \| LAST \| FIFO |
| `avg_unit_cost`, `last_unit_cost`, `last_cost_at` | denormalized, in stock UOM |
| `default_yield_pct` | AP→EP default (100 = no trim loss) |
| `shelf_life_days`, `is_perishable`, `storage_instructions`, `temperature_class` | |
| `preferred_vendor_item_id` | nullable FK; the "primary vendor" |
| `tax_code_id` | purchase tax treatment |
| `is_key_item`, `use_actual_as_theoretical` | R365's Key Item / Actual-as-Theo |
| `is_active`, `archived_at`, audit cols | never hard-delete |

**`item_category`** — `id, parent_id, level (1..3), code, name, prefix, default_*_account_id, variance_cap_pct, exclude_from_po_suggestions, sort_order, is_active`. Enforce `level ≤ 3` and `parent.level = level - 1` in a check/trigger. GL resolution walks self → parent → grandparent.

**`uom` / `uom_conversion` / `item_uom_conversion`** —
`uom(id, code, name, dimension, factor_to_base DECIMAL(18,9), is_base)`.
`item_uom_conversion(item_id, from_uom_id, to_uom_id, factor DECIMAL(18,9))` for cross-dimension bridges. Rule: **convert source → dimension base → target in exactly two multiplications**, never chained hops.

**`vendor`** — `id, code (must match accounting system), name, legal_name, tax_id, contacts, payment_terms, currency, ap_account_id, default_expense_account_id, order_method (EMAIL|EDI|PORTAL|MANUAL), min_order_value, delivery_days_mask, order_cutoff_time, is_active, archived_at`.

**`vendor_item`** — the catalog. This is the load-bearing entity.

| Field | Notes |
|---|---|
| `id, vendor_id, item_id` | **many vendor_items → one item; a vendor_item belongs to exactly one item** (R365's cardinality) |
| `vendor_sku`, `vendor_description`, `gtin` | |
| `order_uom_id`, `pack_description` ("6/#10 CAN"), `pack_qty`, `pack_uom_id` | `pack_qty × pack_uom` converts an order unit to stock UOM |
| `qty_per_order_unit_in_stock_uom` | denormalized conversion, recomputed on save |
| `current_price`, `price_uom_id`, `is_catch_weight` | |
| `lead_time_days`, `min_order_qty`, `order_multiple` | |
| `is_preferred`, `is_contract_item`, `is_active`, `mapping_status` (MAPPED\|UNMAPPED\|AMBIGUOUS) | |

**`vendor_item_price`** — `vendor_item_id, location_id (nullable = all), price, price_uom_id, effective_from, effective_to, source (CATALOG|CONTRACT|INVOICE|MANUAL)`. Never overwrite price in place — insert a new effective-dated row. This is what makes price-change reporting and contract-compliance checks trivial.

**`item_location`** — `item_id, location_id, storage_location_id (primary), par_min, par_max, reorder_point, reorder_qty, count_sheet_sequence, is_storage_active, is_orderable, cost_override`. Plus `item_secondary_storage(item_id, location_id, storage_location_id, count_sheet_sequence)` for count-only locations.

**`recipe` / `recipe_version` / `recipe_line`** —
`recipe(id, code, name, recipe_type PREP|PLATE, produces_item_id nullable, category_id, is_inventoried)`.
`recipe_version(id, recipe_id, version_no, status DRAFT|ACTIVE|ARCHIVED, effective_from, effective_to, batch_yield_qty, batch_yield_uom_id, portions, portion_size_qty, portion_size_uom_id, net_yield_pct, prep_minutes, cook_minutes, finish_minutes, shelf_life_days, method_steps, labor_cost, packaging_cost, computed_batch_cost, computed_portion_cost, cost_computed_at, target_food_cost_pct, approved_by, approved_at)`.
`recipe_line(recipe_version_id, sequence, component_type ITEM|RECIPE, item_id, recipe_id, qty, uom_id, waste_pct, is_optional, note)`.
Constraint: exactly one of `item_id`/`recipe_id`; unique active version per date range; cycle check on save.

**`menu_item` / `menu_item_recipe_link`** — `menu_item(id, pos_external_id, location_id nullable, name, price, target_margin_pct, category)`; link table carries `recipe_id, qty, effective_from, effective_to`, so a menu item can deplete multiple recipes (entrée + side + sauce) and modifiers link the same way.

**Purchasing** —
`purchase_order(id, location_id, vendor_id, po_number, status DRAFT|SUBMITTED|PARTIAL|RECEIVED|CLOSED|CANCELLED, expected_date, currency, subtotal, tax, total, created_by, submitted_at)`
`purchase_order_line(po_id, line_no, vendor_item_id, item_id (snapshot), qty_ordered, order_uom_id, unit_price, qty_received, qty_outstanding, expected_date, notes)`
`goods_receipt(id, po_id nullable, vendor_id, location_id, receipt_number, received_at, received_by, status)`
`goods_receipt_line(receipt_id, po_line_id nullable, vendor_item_id, qty_received, order_uom_id, qty_in_stock_uom, catch_weight_qty, unit_cost, substitution_of_vendor_item_id, condition OK|DAMAGED|REJECTED)`
`vendor_invoice(id, vendor_id, location_id, invoice_number, invoice_date, due_date, subtotal, tax, freight, total, status PENDING|MATCHED|EXCEPTION|APPROVED|POSTED, source OCR|EDI|MANUAL)`
`vendor_invoice_line(invoice_id, vendor_item_id, item_id, qty, uom_id, unit_price, extended, gl_account_id, receipt_line_id nullable)`
`match_exception(invoice_line_id, type PRICE_VARIANCE|QTY_VARIANCE|NO_RECEIPT|NO_PO|DUPLICATE|UOM_MISMATCH, expected, actual, variance_pct, status OPEN|ACCEPTED|DISPUTED|RESOLVED, resolved_by)`
`match_tolerance_policy(scope TENANT|VENDOR|CATEGORY, scope_id, price_tolerance_pct, price_tolerance_abs, qty_tolerance_pct, over_receipt_policy BLOCK|WARN|ALLOW)` — default ±2% price, 0% over-receipt-block, per the industry norm.

**Ledger** — `inventory_transaction(id, location_id, item_id, txn_type, txn_date, qty_stock_uom, unit_cost_snapshot, extended_cost, source_doc_type, source_doc_id, recipe_version_id nullable, gl_batch_id)`. Every quantity movement, including theoretical depletion, goes here. **Cost is snapshotted, not referenced.**

### 9.3 The five opinionated calls

1. **Both vendor↔category and vendor↔item — but only the catalog is authoritative.** Build `vendor_item` as the system of record for PO/receipt/invoice. Add `vendor_category(vendor_id, category_id, is_preferred)` purely to (a) pre-filter pickers, (b) default the vendor on suggested orders for items with no preferred vendor, (c) route unmapped invoice lines. Never let a category assignment authorize a purchase.
2. **One `item` table with `item_type`, not separate ingredient/prep tables.** A prep item that is sometimes purchased (buy-vs-make marinara) is common enough that a separate PrepItem table forces duplication. `item_type = BOTH` + `produced_by_recipe_id` handles it, and inventory/valuation code stays uniform.
3. **Version recipes from day one; snapshot costs onto transactions.** Retrofitting versioning after you have a year of AvT history is a migration you cannot do correctly.
4. **Three-level max category tree + tags.** Enforce the depth in the schema. Store default GL accounts on category nodes with most-specific-wins resolution — copy R365 exactly, it's a proven design.
5. **Effective-dated prices, never in-place price updates.** `vendor_item_price` rows are append-only. This one decision gives you price-change reports, contract-compliance alerts, invoice price validation, and correct historical costing for free.

### 9.4 Invariants to enforce in code/DB

- `item.measure_type` immutable once an `inventory_transaction` exists.
- Recipe graph acyclic (validate on version save, DFS from the edited node).
- `vendor_item.item_id` immutable once referenced by a posted transaction (re-map creates a new vendor item).
- No hard delete of `item`, `vendor`, `vendor_item`, `recipe`, `item_category` — FK RESTRICT + `archived_at`.
- `recipe_version` effective ranges per recipe must not overlap for ACTIVE status.
- All quantity/cost columns `DECIMAL(18,6)` minimum (cost `DECIMAL(18,6)`, qty `DECIMAL(18,4)`); round only at display/GL boundary, banker's rounding.
- PO line qty in order UOM; receipt writes both order-UOM qty and derived stock-UOM qty (store both — do not re-derive later, the pack size may change).

## 10. RECOMMENDED UX

### 10.1 Information architecture

```
Inventory
  ├─ Items            (grid: saved views, inline edit, bulk edit, import, fuzzy search)
  ├─ Categories       (3-level tree editor with GL defaults)
  ├─ Units & Conversions
  ├─ Storage Locations
  ├─ Counts           (count sheets by storage location)
  ├─ Waste / Transfers / Production
Recipes
  ├─ Recipe Library   (prep + plate, filter by coverage/margin)
  ├─ Recipe Builder   (live cost panel)
  └─ Recipe Coverage  ← the differentiator
Purchasing
  ├─ Vendors          (header + Catalog tab + Unmapped queue + Price changes)
  ├─ Suggested Orders
  ├─ Purchase Orders
  ├─ Receiving
  └─ Invoices & Matching (exception queue)
```

### 10.2 Screen-by-screen

**Items grid.** Dense table, column chooser, saved views, CSV export/import, server-side fuzzy search. Inline edit for toggles/dropdowns/short text. Multi-select → bulk edit for category, GL accounts, active, key-item, location group. Row expander shows vendor items (with price-per-stock-UOM normalized for comparison) and "used in N recipes." Duplicate-detection prompt at create ("3 similar items exist — did you mean…?").

**Item detail.** Tabs: General · Units & Conversions · Costing & GL · Vendors (catalog rows) · Locations & Par · Recipes (reverse refs) · Compliance (allergens/nutrition/spec) · History (price + cost trend, transactions).

**Recipe builder.** Two-pane: left = component list with search-as-you-type that returns items *and* sub-recipes, showing pack size / supplier / SKU inline; right = sticky cost panel with batch cost, cost/portion, sell price, food cost %, margin, and delta-to-target with a colored chip. Each line shows qty, UOM, waste %, line cost, and **% of plate cost** with a mini bar. Nested sub-recipe rows expand in place. Version banner at top ("v3 · active since 2026-05-01 · [New version] [History]"). Scale control (portions / batch multiplier). Unsaved-cost preview so chefs see the impact before committing.

**Recipe Coverage dashboard (ship this — competitors under-serve it).** Headline: "% of sales revenue backed by a costed recipe." Table of menu items with no active recipe, sorted by units sold and revenue descending; secondary list of recipes whose components have no cost (never purchased / no vendor price); third list of recipes above target food cost %. One-click "create recipe" from a row. Tie the number directly to the AvT report so users understand why their variance is untrustworthy.

**Category management.** Compact tree (max 3 levels), each node showing item count + default GL accounts inline. Drag to reparent (with a confirmation showing how many items' GL derivation changes). Delete blocked when non-empty; offer "merge into…".

**Vendor setup.** Header form + tabs: Catalog (grid with inline price edit → creates a new effective-dated price row), Unmapped Items queue (badge count), Price Changes (last 30/90 days, % delta), Orders, Invoices, Terms & Delivery. Catalog import wizard with column mapping and a preview-diff before commit.

**Suggested order → PO.** Start from par/forecast shortfall grouped by preferred vendor, showing needed qty in stock UOM and the resulting case count after pack-size rounding, with an editable qty. Vendor swap on a line shows the price comparison inline (price per stock UOM across all vendor items for that item). Split into per-vendor POs on submit.

**Receiving.** Mobile-friendly. PO lines pre-filled, "receive all" default, per-line short/over entry with tolerance chips (green within tolerance, amber over). Substitution picker. Catch-weight entry where flagged. Photo attach. Partial receipt leaves the PO open with visible outstanding qty.

**Invoice matching.** Three-column diff view (PO | Receipt | Invoice) per line with variance columns and a status chip. Exception queue as the primary work surface; bulk-accept within tolerance; require a reason code to accept outside tolerance.

**Count sheets.** Grouped by storage location, ordered by the per-location sequence number, matching physical shelf order. Alternate count units per line. Progress indicator, resumable, offline-tolerant, variance flagged on submit against theoretical on-hand.
