<objective>
Land the additive purchasing-service Flyway V5 migration that gives vendors a real item catalog:
`vendor_items` (vendor SKU, pack size, purchase UOM, MOQ, lead time) linked one-to-one to an
ingredient, append-only effective-dated `vendor_item_prices`, filter-only `vendor_categories` tags,
and a nullable `purchase_order_lines.vendor_item_id`.

Purpose: PUR-07 and ROADMAP Success Criteria 5 and 6. `vendor_catalogues` has existed in SQL since
purchasing V1 and is 100% dead code with no entity, repository, controller or frontend; this
migration supersedes it with a model that can actually price a PO line and three-way match.
Output: `V5__vendor_item_catalog.sql` and a new `PurchasingSchemaMigrationIT`.

Verified on disk 2026-07-22: purchasing-service migrations are V1, V2, V3, V4 — the next free
version is **V5**, not V3 as CONTEXT.md's constraints section states.
</objective>

Create the migration named exactly `V5__vendor_item_catalog.sql`.

`vendor_items`: `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`, `tenant_id UUID NOT NULL`,
`vendor_id UUID NOT NULL REFERENCES vendors(id) ON DELETE RESTRICT`, `ingredient_id UUID NOT NULL`
(deliberately NOT a foreign key — ingredients live in inventory_db, this is a cross-service soft
reference; state that in a comment above the column), `vendor_sku VARCHAR(80)`,
`vendor_description VARCHAR(240)`, `gtin VARCHAR(20)`, `order_uom VARCHAR(20) NOT NULL`,
`pack_description VARCHAR(80)`, `pack_qty NUMERIC(18,4) NOT NULL DEFAULT 1`,
`pack_uom VARCHAR(20) NOT NULL`, `qty_per_order_unit_in_stock_uom NUMERIC(18,6)`,
`min_order_qty NUMERIC(18,4)`, `order_multiple NUMERIC(18,4)`, `lead_time_days INT`,
`is_preferred BOOLEAN NOT NULL DEFAULT false`, `is_catch_weight BOOLEAN NOT NULL DEFAULT false`,
`archived_at TIMESTAMPTZ`, plus the standard audit block. Add
`CONSTRAINT uq_vendor_item_tenant_vendor_sku UNIQUE NULLS NOT DISTINCT (tenant_id, vendor_id, vendor_sku)`.
Do NOT add a unique constraint on `(tenant_id, vendor_id, ingredient_id)` — a vendor legitimately
sells the same ingredient in more than one pack size. Add indexes
`idx_vendor_items_tenant_vendor (tenant_id, vendor_id)` and
`idx_vendor_items_tenant_ingredient (tenant_id, ingredient_id)`.

`vendor_item_prices`: `id`, `tenant_id UUID NOT NULL`,
`vendor_item_id UUID NOT NULL REFERENCES vendor_items(id) ON DELETE RESTRICT`,
`branch_id UUID` (nullable = applies to all branches), `unit_price_paisa BIGINT NOT NULL`,
`price_uom VARCHAR(20) NOT NULL`, `effective_from TIMESTAMPTZ NOT NULL DEFAULT now()`,
`effective_to TIMESTAMPTZ`, `source VARCHAR(20) NOT NULL DEFAULT 'MANUAL'` with a CHECK constraining
it to `CATALOG`, `CONTRACT`, `INVOICE`, `MANUAL`, `is_contract_price BOOLEAN NOT NULL DEFAULT false`,
plus the audit block. Add index
`idx_vendor_item_prices_item_effective (tenant_id, vendor_item_id, effective_from DESC)`.
Add a header comment stating the append-only invariant in the same style the `recipes` table is
documented in inventory V1: a price change inserts a new row and closes the prior row's
`effective_to`; there is no in-place price UPDATE anywhere in the application.

`vendor_categories`: `id`, `tenant_id UUID NOT NULL`,
`vendor_id UUID NOT NULL REFERENCES vendors(id) ON DELETE RESTRICT`, `category_id UUID NOT NULL`
(cross-service soft reference to inventory-service's `item_categories`), `category_name VARCHAR(160)`
(denormalized label so the vendor page renders without a cross-service call),
`is_preferred BOOLEAN NOT NULL DEFAULT false`, plus the audit block, and
`CONSTRAINT uq_vendor_category UNIQUE (tenant_id, vendor_id, category_id)`. Add a header comment
carrying the D-01 invariant verbatim in meaning: these tags exist only to filter pickers and suggest
a default vendor; they never authorize or restrict a purchase. Do not add any column that names a
permission, allowance or restriction.

`ALTER TABLE purchase_order_lines ADD COLUMN vendor_item_id UUID REFERENCES vendor_items(id) ON
DELETE RESTRICT;` — nullable, because historical PO lines predate the catalog and must keep working.
Add `CREATE INDEX idx_po_lines_vendor_item ON purchase_order_lines (vendor_item_id);`

For each of the three new tables emit
`ALTER TABLE x ENABLE ROW LEVEL SECURITY;` + `CREATE POLICY tenant_isolation ON x USING (tenant_id =
current_setting('app.current_tenant_id', TRUE)::UUID);` + `GRANT SELECT, INSERT, UPDATE, DELETE ON x
TO purchasing_user;`

Leave `vendor_catalogues` completely untouched. Add a single comment line in the header noting that
`vendor_items` supersedes it and that dropping it is deliberately out of scope for this additive
migration.
