<objective>
Land the additive Flyway V5 migration that turns ingredient categories into first-class master data:
a self-referencing `item_categories` tree hard-capped at 3 levels by a database trigger, and a
required `ingredients.category_id` foreign key backfilled from the legacy free-text
`ingredients.category` column.

Purpose: INV-13 / Success Criterion 1 needs one deterministic category answer per ingredient before
any API, UI, or spend-analytics work can be built on top of it. This is the riskiest change in the
phase (it mutates a FORCE-RLS table that the proven depletion loop reads), so it ships alone, ahead
of everything else.
Output: `V5__item_categories.sql`, extended `SchemaMigrationIT`, new `ItemCategoryMigrationIT`.
</objective>

Create the migration file named exactly `V5__item_categories.sql` (V1..V4 are taken on disk; V5 is
the next free inventory-service version — verified 2026-07-22).

Statement order is load-bearing. Emit in exactly this order:

(1) `CREATE TABLE item_categories` with: `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`,
`tenant_id UUID NOT NULL`, `parent_id UUID REFERENCES item_categories(id) ON DELETE RESTRICT`,
`level SMALLINT NOT NULL CHECK (level BETWEEN 1 AND 3)`, `code VARCHAR(40)`,
`name VARCHAR(160) NOT NULL`, `default_inventory_account_code VARCHAR(20)`,
`default_cost_account_code VARCHAR(20)`, `default_waste_account_code VARCHAR(20)`,
`variance_cap_pct NUMERIC(6,2)`, `exclude_from_po_suggestions BOOLEAN NOT NULL DEFAULT false`,
`sort_order INT NOT NULL DEFAULT 0`, `archived_at TIMESTAMPTZ`, and the standard audit columns used
by every other inventory table (`created_at`, `updated_at`, `created_by`, `updated_by`,
`deleted_at`). Add `CONSTRAINT uq_item_category_tenant_parent_name UNIQUE NULLS NOT DISTINCT
(tenant_id, parent_id, name)` — `NULLS NOT DISTINCT` is required (Postgres 16) so two sibling roots
with the same name under a NULL parent still collide. Add
`CREATE INDEX idx_item_categories_tenant_parent ON item_categories (tenant_id, parent_id)`.

(2) `CREATE OR REPLACE FUNCTION enforce_item_category_depth() RETURNS trigger LANGUAGE plpgsql` that
raises `EXCEPTION` (SQLSTATE `23514`) when: `NEW.parent_id IS NULL AND NEW.level <> 1`; or
`NEW.parent_id IS NOT NULL` and the parent row's `level` is not `NEW.level - 1`; or the parent row's
`level >= 3`; or the parent row's `tenant_id <> NEW.tenant_id`; or `NEW.parent_id = NEW.id`. Attach
it as `CREATE TRIGGER trg_item_category_depth BEFORE INSERT OR UPDATE ON item_categories FOR EACH
ROW EXECUTE FUNCTION enforce_item_category_depth()`. Include a header comment stating that this is
the D-02 hard cap and that the service layer mirrors it for a friendly error, but the database is
the enforcement point.

(3) `ALTER TABLE ingredients ADD COLUMN category_id UUID;` (nullable at this point).

(4) The backfill, wrapped in the FORCE-RLS escape hatch. In production the Flyway user owns
`ingredients` and `ingredients` is `FORCE ROW LEVEL SECURITY`, so with no `app.current_tenant_id`
GUC set the isolation predicate evaluates to NULL and both the read and the write would silently
touch zero rows. Therefore: `ALTER TABLE ingredients NO FORCE ROW LEVEL SECURITY;` then
(a) `INSERT INTO item_categories (tenant_id, parent_id, level, name) SELECT DISTINCT tenant_id,
NULL, 1, COALESCE(NULLIF(btrim(category), ''), 'Uncategorized') FROM ingredients;`
(b) `UPDATE ingredients i SET category_id = c.id FROM item_categories c WHERE c.tenant_id =
i.tenant_id AND c.parent_id IS NULL AND c.name = COALESCE(NULLIF(btrim(i.category), ''),
'Uncategorized');` then `ALTER TABLE ingredients FORCE ROW LEVEL SECURITY;`. Add an inline comment
above the escape hatch explaining exactly why it exists and that the re-FORCE on the following line
is mandatory. Note the ordering constraint: this backfill runs BEFORE step (5) enables RLS on
`item_categories`, otherwise the INSERT would be filtered the same way.

(5) `ALTER TABLE ingredients ALTER COLUMN category_id SET NOT NULL;` then
`ALTER TABLE ingredients ADD CONSTRAINT fk_ingredient_category FOREIGN KEY (category_id) REFERENCES
item_categories(id) ON DELETE RESTRICT;` and
`CREATE INDEX idx_ingredients_tenant_category ON ingredients (tenant_id, category_id);`

(6) `ALTER TABLE item_categories ENABLE ROW LEVEL SECURITY; ALTER TABLE item_categories FORCE ROW
LEVEL SECURITY; CREATE POLICY tenant_isolation ON item_categories USING (tenant_id =
NULLIF(current_setting('app.current_tenant_id', TRUE), '')::UUID);` followed by
`GRANT SELECT, INSERT, UPDATE, DELETE ON item_categories TO inventory_user;`

Do NOT drop, rename or retype `ingredients.category`. Leave it in place and add a comment marking it
as retained-for-history and no longer read by application code (plan 08.2-09 removes it from the
DTO). Do not touch `stock_lots`, `inventory_movements`, `ingredient_branch_stock`, `recipes` or
`recipe_lines` in any way.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Flyway migration process → inventory_db | Runs outside any tenant request context; the `app.current_tenant_id` GUC is unset. |
| Tenant A rows ↔ Tenant B rows inside item_categories | Only the RLS policy separates them at runtime. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-08.2-011 | Information Disclosure | `item_categories` table | high | mitigate | ENABLE + FORCE ROW LEVEL SECURITY with the `NULLIF(current_setting('app.current_tenant_id', TRUE), '')::UUID` `tenant_isolation` policy, asserted in `SchemaMigrationIT#itemCategories_hasForceRlsAndTenantIsolationPolicy`. |
| T-08.2-012 | Tampering | `ingredients` FORCE-RLS posture | high | mitigate | The backfill's `NO FORCE` window is closed by an explicit re-FORCE in the same migration; `ItemCategoryMigrationIT#ingredientsStillForceRlsAfterBackfill` fails the build if it is ever dropped. |
| T-08.2-013 | Elevation of Privilege | `enforce_item_category_depth()` cross-tenant parent lookup | medium | mitigate | The trigger explicitly rejects a parent whose `tenant_id` differs from the child's, so a crafted `parent_id` cannot graft a node onto another tenant's tree even if RLS were misconfigured. |
| T-08.2-014 | Denial of Service | Unbounded category recursion | low | mitigate | `level BETWEEN 1 AND 3` CHECK plus the trigger make cycles and deep trees unrepresentable, so tree walks terminate. |
</threat_model>
