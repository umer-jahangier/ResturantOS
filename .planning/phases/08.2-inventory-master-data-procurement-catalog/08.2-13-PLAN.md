Add one schema per row of the artifacts table, each with the "Mirrors {JavaRecord} exactly (field
list)" comment, `qtyField` for every BigDecimal (`packQty`, `minOrderQty`, `orderMultiple`,
`deltaPct`) and plain `z.number().int()` for paisa values.

`apiVendorItemSchema` includes `currentUnitPricePaisa`, `currentPriceUom` and
`currentPriceEffectiveFrom` as nullable — a catalog row may exist before its first price.

`recordVendorItemPriceInputSchema` has `unitPricePaisa` (int, non-negative), `priceUom` (non-empty),

`source`. There is deliberately no update-price input schema.

is removed from the write contract entirely. The UI contract is explicit that the picker becomes the
only way to set the line's item and the free-text UUID input is deleted; keeping an `ingredientId`
escape hatch in the client schema would let the old field quietly come back. Make `uom` and
`unitPriceRupees` optional in the input schema, since the server fills both from the catalog entry
when they are absent, and keep the existing `qty` validation. Add a `.refine` producing the message
the form should show when no catalog item has been picked.

Extend the purchase-order response schema's line shape with `vendorItemId`, `vendorSku`,
`packDescription` and `priceOverridden`, all nullable for legacy lines.

Mirror everything with `z.infer` type aliases and identity `adaptX` functions in
`purchasing.adapter.ts`, matching the existing file's shape.

`listVendorItems(vendorId, page?, size?)` (paginated GET),

`archiveVendorItem(vendorItemId)` (POST to `/vendor-items/{id}/archive`, never `del`),
`listVendorItemPrices(vendorItemId)`, `recordVendorItemPrice(vendorItemId, input)` (POST),
`listVendorPriceChanges(vendorId, since?)`,
`listVendorCategories(vendorId)`, `replaceVendorCategories(vendorId, input)` (PUT).

Hooks — delete the three local key arrays and repoint all existing purchasing hooks plus the nine new

Invalidation rules:

  `vendorItems(branchId, vendorId)`.
- `useRecordVendorItemPrice` → invalidate `vendorItemPrices(branchId, vendorItemId)`,
  `vendorItems(branchId, vendorId)` (the row's current price changes) and the vendor's price-changes
  key.
- `useReplaceVendorCategories` → invalidate `vendorCategories(branchId, vendorId)`.
- `useCreatePurchaseOrder` keeps its existing invalidations.

`useVendorItems(vendorId)` must be `enabled` only when `vendorId` is truthy — the PO line picker
mounts before a vendor is chosen and must not fire a request with an empty id.

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |

| T-08.2-132 | Information Disclosure | Catalog and contract prices persisting across a branch switch | high | mitigate | All purchasing keys are branch-scoped through `queryKeys.purchasing.*`. |
