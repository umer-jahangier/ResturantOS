# Phase 08.2 — Inventory Master Data & Procurement Catalog — CONTEXT

## Summary

Phases 8, 08.1 and 10 delivered a near-complete inventory/purchasing **backend** and proved the
`ORDER_CLOSED → depletion` loop end to end. What they deliberately deferred (08.1 D-04: "vertical-slice
UI first") is now the bottleneck: **11 backend endpoints have no frontend consumer at all**, ingredients
can only be created by raw SQL, `category` is a free-text column rather than an entity, and the
vendor↔ingredient relationship is a table with no Java behind it. This phase turns the module into
something a restaurant manager can actually operate, at industry-standard fidelity.

This is master-data + UI work. It must not disturb the depletion loop.

## Origin — user-reported defects (2026-07-22)

Three bugs were reported, root-caused and **fixed before this phase was created**. They are recorded
here because two of them expose design gaps this phase must close properly:

1. **Vendor create → "service temporarily unavailable."** `purchasing-service` was never added to any
   dev-stack script, so it wasn't running; the gateway's `lb://purchasing-service` couldn't resolve →
   503 → circuit-breaker fallback text. Fixed in `scripts/start-dev.ps1`, `restart-service.ps1`,
   `local-service-env.ps1`. Secondary blocker: Postgres `max_connections=100` was fully consumed by
   12 services × Hikari pool 10, so the 13th failed Flyway with `FATAL: remaining connection slots are
   reserved for roles with the SUPERUSER attribute` — now `max_connections=300` in
   `deploy/docker-compose.yml`.
2. **Vendor create → 403 for every role including OWNER.** Liquibase drift: `vendor.manage` and
   `vendor.po.approve` are declared in `030-create-roles-permissions.xml:78-79,115-116` but were added
   *after* that `runOnChange="false"` changeset had executed, and `031` explicitly skipped re-inserting
   them on the false assumption 030 had landed them. Both codes were absent from the `permissions`
   table entirely. Fixed by `045-repair-vendor-manage-approve-grants.xml` (`ON CONFLICT DO NOTHING` —
   no-op on fresh DBs, repair on drifted ones).
3. **Coverage stuck at "1 covered" after adding 2 recipes.** `RecipeFormDialog.tsx` converted the
   `<input type="date">` value to **midnight UTC**; at +05:00 "today" became a *future* instant.
   `RecipeService.getCoverage()` counts an item only when `effective_from <= now()`, but the versions
   table keys off `is_current` — so the recipe rendered "Current: Yes" while coverage ignored it.
   Fixed by parsing the calendar date as **local** midnight. **The UX half is in scope here** (SC-4):
   a scheduled recipe must be visibly pending, not silently uncounted.

## Verified current state — DO NOT REBUILD (confirmed by code audit)

**Backend that already exists and works:**
- inventory-service: `GET/POST/PUT /ingredients`, `GET /ingredients/{id}`, `GET/POST /uom`,
  `GET /menu-items`, `POST/GET /recipes`, `GET /recipes/{menuItemId}/effective`, `GET /recipes/coverage`,
  `POST /receipts`, `POST /transfers/ship`, `POST /transfers/receive`, `POST /counts`,
  `POST /opening-balance`, `GET /internal/grn/pending-count`. All `@RequiresFeature("FEATURE_INVENTORY")`
  + OPA via `InventoryAuthorizationService`.
- purchasing-service: full vendor / PO lifecycle / GRN / invoice 3-way-match / AP payment / analytics
  surface (21 endpoints). Vendor bank account is field-encrypted (`bankAccountLast4` only on read).
- `MacCalculator`, `UomConverter`, `DepletionService` (FEFO + aggregate-MAC), transactional outbox,
  `processed_events` idempotency, `IdempotencyService`.

**Frontend that already exists:** `/app/inventory/recipes` (create-only), `/app/inventory/coverage`,
and the full `/app/purchasing/**` set (vendors, POs incl. detail + workflow, invoices, payments, analytics).

**The actual gaps:**
- **No ingredient UI whatsoever.** `useIngredients()` exists solely to populate a `<select>` inside
  `RecipeFormDialog`. No list, no create, no edit, no archive.
- **No recipe viewing.** The API returns `lines`; `recipes/page.tsx:72` renders only `lines.length`.
  No edit, no delete — and no backend `PUT`/`DELETE` on recipes either.
- **`ingredients.category` is `VARCHAR(80)` free text** (`V1__inventory_schema.sql:42`) — never rendered,
  filtered or edited. No category table anywhere in either service.
- **`vendor_catalogues` exists in SQL (`purchasing V1:30`) and is 100% dead code** — no entity, no
  repository, no controller, no frontend. `grep -rn "VendorCatalogue" services/purchasing-service/src/main/java`
  returns zero hits.
- **`MockIngredientCategoryResolver`** loads a static classpath `spend-category-map.yml` and defaults to
  `"Uncategorized"`. The Analytics "spend by category" table is fabricated — it never calls inventory-service.
- **PO lines require a hand-typed ingredient UUID** with a free-text unit
  (`PurchaseOrderFormDialog.tsx:29-32, 201-221`) — the Zod message literally says
  *"no ingredient picker exists yet"*. `useIngredients()`/`useUoms()` already exist and are unused here.
- **No read model for on-hand stock.** `ingredient_branch_stock` has no controller; receipts/transfers/
  counts/opening-balance are write-only. A stock-levels screen has nothing to call.
- **No DELETE endpoint anywhere in inventory-service**, and `is_active` exists on `ingredients` in the DB
  but is not in `UpdateIngredientRequest`.

## Locked decisions

- **D-01 (vendor↔items): BOTH, catalog authoritative.** Build `vendor_item` (vendor SKU, pack size,
  purchase UOM, MOQ, lead time) → one ingredient, plus **append-only effective-dated `vendor_item_price`**.
  Vendor↔category tags exist *only* to filter pickers and suggest vendors and must **never** authorize a
  purchase. Rationale: category-only cannot price a PO line, cannot 3-way match, cannot compare vendors
  or do pack-size math. Never update a price in place — append-only gives price-change history, contract
  compliance, invoice validation and correct historical costing for free.
- **D-02 (categories): self-referencing tree hard-capped at 3 levels**, enforced by a DB trigger, not
  convention. Exactly **one required primary category** per ingredient (GL derivation needs a
  deterministic answer); use tags/custom fields for the long tail. Nodes carry default GL accounts,
  inherited most-specific-wins.
- **D-03 (item model): evolve `ingredients` in place.** Additive Flyway migrations only, so existing
  stock lots, inventory movements and MAC history stay intact. Add `item_type`
  (`PURCHASED|PREPARED|BOTH`) + `produced_by_recipe_id` so prep/sub-recipes are *modelled* now even if
  full prep-recipe authoring defers.
- **D-04 (deletion): archive, never hard-delete.** FK `RESTRICT` + `archived_at`, never `CASCADE`, on any
  master data with transaction history. Archiving a category still in use is refused.
- **D-05 (recipe editing): revision, not mutation.** "Edit" authors a new version pre-filled from the
  current one. The depletion loop resolves the version effective at `closedAt` (08.1 D-01) — destructive
  edits would silently rewrite history and are forbidden.
- **D-06 (UOM): three roles + typed dimensions.** purchase / stock / recipe UOM, with **purchase UOM
  belonging to the `vendor_item`, not the ingredient**. Cross-dimension conversions (density, each-weight,
  pack size) live in a per-item conversion table. Lock the measure type once transactions exist.
- **D-07 (yield): three distinct numbers**, not one — item-level AP→EP trim yield, per-recipe-line waste %,
  and recipe-header net yield for cooking reduction. `recipe_lines.yield_pct` already exists; do not
  overload it to mean all three.
- **D-08 (rounding): convert source→base→target in exactly two `DECIMAL` multiplications**, rounding only
  at display/GL. Chained float conversions accumulate error across millions of depletions.
- **D-09 (analytics): delete the mock, don't bypass it.** `MockIngredientCategoryResolver` and
  `spend-category-map.yml` are removed and replaced by a real inventory-service call.

## Constraints / conventions to honor

- **Additive migrations only.** Confirm the next free Flyway version per service (inventory is at V4 after
  `V4__menu_item_catalog.sql`; purchasing is at V2). The 08.1 depletion proof must stay green throughout —
  treat `DepletionService`, `MacCalculator` and `stock_lots` as load-bearing.
- inventory FORCE-RLS convention: new tenant-scoped tables get `ENABLE` + `FORCE ROW LEVEL SECURITY` with
  NULLIF-guarded `tenant_isolation`. Note the consumer-path RLS GUC ordering fix from 08.1-06 lives in
  shared-lib `TenantAwareMessageProcessor` — don't regress it.
- **No new permission codes needed** — reuse `inventory.item.view` / `inventory.item.manage` and the
  `vendor.*` set. (`vendor.manage` / `vendor.po.approve` were repaired by auth changeset 045; a fresh DB
  gets them from 030.)
- Frontend: enforced four-layer pattern (Zod schema → adapter → repository → TanStack hook → component),
  ESLint `no-restricted-imports` blocks `components/**` importing `lib/api-client`. shadcn/ui
  (`radix-nova`, neutral) — note there is **no `select`, `table`, `tabs`, `checkbox` or `textarea`
  primitive**; existing code uses styled raw elements. Forms are react-hook-form + zod via
  `createZodResolver`; toasts are `sonner`; gating is `<PermissionGuard>` + `<FeatureGuard>`.
- Move inventory + purchasing hooks onto the central `lib/hooks/query-keys.ts` registry (they currently
  use local const arrays, unlike pos/finance).
- **Exemplars to model on:** `components/purchasing/VendorFormDialog.tsx` (create-or-edit in one dialog),
  `app/(tenant)/app/finance/house-accounts/page.tsx` (list + row actions), and
  `components/inventory/RecipeFormDialog.tsx` (`useFieldArray` dynamic lines).

## Carried-over defects to fix in this phase (found 2026-07-22, deliberately not hot-fixed)

- **Gateway resilience config is incomplete.** `resilience4j.circuitbreaker.instances` defines only
  auth/authorization/platformAdmin/user/file/finance; `purchasing`, `pos`, `kitchen`, `inventory` fall back
  to defaults. Reproduced twice: after any upstream restart the gateway 503s **persistently** — upstream
  healthy on its own port, one UP instance in Eureka, no `No servers available` in the gateway log — until
  the gateway itself is restarted.
- **`SERVICE_UNAVAILABLE` has no entry in `USER_FACING_BY_CODE`** (`frontend/lib/api-client/errors.ts:146-160`),
  so users see raw gateway text with no indication of which module is down.
- **`scripts/start-dev.sh:189`** still excludes `services/purchasing-service` from its `mvn -pl` list
  (the `.ps1` side was fixed).
- **Regression guards for the coverage bug:** a unit test on the local-midnight date conversion, and a
  backend IT seeding a **future** `effectiveFrom` asserting it is reported as *not* covered.
  `RecipeCoverageIT.java:97` only ever seeds `Instant.now().minus(1, DAYS)`, and the MSW mock
  (`frontend/mocks/inventory.handlers.ts:132-146`) uses `r.current` while the real backend uses
  `effective_from <= now()` — the two definitions disagree exactly on the future-dated case, which is why
  no test caught it. **Align the mock with the backend.**
- **`RecipeService.getCoverage()` is N+1** (`RecipeService.java:85-89`) — one query per active menu item.
- **`RecipeRepository.findByMenuItemIdOrderByVersionDesc`** has no tenant predicate, relying purely on RLS
  + the Hibernate tenant filter.
