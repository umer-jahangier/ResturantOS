<objective>
Fix Bug 2 — expense creation fails 100% of the time — by ISOLATING ITS ACTUAL ROOT CAUSE, not by papering
over the symptom.

The symptom is a contradiction: `GET /api/v1/finance/accounts?type=EXPENSE&active=true` returns `6300` as an
active account for the tenant, but `POST /api/v1/finance/expenses` with `expenseAccountCode:"6300"` returns
`400 INVALID_ACCOUNT_CODE`. Codes 6000, 6100, 6300 all fail. The read path and the write path disagree about
what accounts exist for this tenant. **One of them is lying, and finding out which is the entire job.**

Do NOT "fix" this by relaxing `validateAccountCode`. If the write path is right and the read path is leaking
another tenant's accounts, then relaxing validation would let a user post a journal entry against a foreign
tenant's account — turning a 400 into a data-integrity and cross-tenant breach.

Purpose: a finance user can actually record an expense; and whichever path was lying gets fixed at the root.
Output: a written, evidence-backed root cause; the fix; a real-Postgres cross-tenant negative control; and a
real-browser expense creation.
</objective>

This task produces EVIDENCE and a written root-cause statement. Write no fix in this task. The previous
gap-closure wave shipped 12 green plans onto a broken module by fixing symptoms; do not repeat it.

Prior investigation has already narrowed the field. Read these facts before touching anything:

  FACT A — the two paths use DIFFERENT tenant scoping mechanisms.
    READ  (`CoaServiceImpl.listAccounts`) → `coaRepo.findByAccountTypeAndActive(type, active, pageable)`.
          This JPQL has **no tenant predicate at all**. It is tenant-scoped ONLY by (i) the Hibernate
          `@Filter(name="tenantFilter", condition="tenant_id = :tenantId")` on `TenantAuditableEntity` and
          (ii) Postgres RLS.
    WRITE (`ExpenseService.validateAccountCode`) → `coaRepository.findByTenantIdAndCode(tenantId, code)`.
          This has an **explicit `tenant_id = ?` predicate** bound from `tenantContext.requireTenantId()`.
    So the write path is the STRICTER one. If they disagree, the most likely story is that the READ path is
    under-filtering, not that the write path is over-filtering.

  FACT B — the Hibernate filter may be a no-op on the HTTP path. `TenantFilterInterceptor.preHandle` calls
    `entityManager.unwrap(Session.class).enableFilter(...)` OUTSIDE any transaction. Spring's shared
    EntityManager proxy may hand back a temporary Session there, distinct from the Session later opened for
    the `@Transactional` service method — in which case the filter is enabled on a Session nobody uses.

  FACT C — the RLS GUC may never survive. `TenantAwareDataSource.configureTenant` runs
    `SELECT set_config('app.current_tenant_id', ?, true)` at CONNECTION CHECKOUT. The third arg `true` means
    **is_local = transaction-scoped**. At checkout there is typically no open transaction, so the setting is
    discarded when that statement's implicit transaction commits.

  FACT D — RLS is `ENABLE ROW LEVEL SECURITY` (V1__finance_schema.sql) but **never `FORCE`**. A table's OWNER
    bypasses RLS entirely. If the app connects as the owner, RLS is decoration.

  FACT E — `CoaServiceImpl` calls `ensureTenantGuc()` / `TenantGucHelper.apply(...)` at the top of its
    methods. `ExpenseService` does NOT. Someone already hit this wrinkle on the read side and patched it
    there only.

  FACT F — COA rows are seeded PER TENANT at provisioning time
    (`ProvisioningService` → `CoaServiceImpl.seedForTenant` → `PakistanRestaurantCoaTemplate.build(tenantId)`,
    reachable via `POST /internal/finance/tenants/{tenantId}/seed-coa`). The demo tenant
    (`a0000001-0000-4000-8000-000000000001`) was hand-seeded via Liquibase into auth/platform-admin. Nothing
    obviously ever called finance's seed-coa for it.

THE DECISIVE EXPERIMENT — run this FIRST, against the live dev DB (stack up per 10-19). One query settles it:

    SELECT tenant_id, code, name, active FROM chart_of_accounts WHERE code IN ('6000','6100','6300')
    ORDER BY tenant_id, code;
    SELECT tenant_id, count(*) FROM chart_of_accounts GROUP BY tenant_id;

Then compare against the tenant_id actually carried by the JWT used in the UAT (decode the manager1 /
finance_demo access token, or log `tenantContext.requireTenantId()` on the create path).

  → If chart_of_accounts holds rows ONLY under a tenant_id that is NOT the demo tenant, the root cause is:
    **the demo tenant was never COA-provisioned, AND the list endpoint is leaking another tenant's accounts**
    (because per FACT B the Hibernate filter is a no-op and per FACT C/D RLS is not enforcing). The write path
    was CORRECT all along. This is a cross-tenant read leak — a P0 security bug hiding inside a 400.
  → If rows DO exist under the demo tenant_id, the write path's tenant resolution or its SQL is at fault:
    log the bound parameters and the emitted SQL for `findByTenantIdAndCode`
    (`logging.level.org.hibernate.SQL=DEBUG`, `...orm.jdbc.bind=TRACE`) and diff the tenant_id it binds
    against the one in the row. Also `SELECT current_setting('app.current_tenant_id', true)` from inside the
    same transaction to see whether the GUC survived (FACT C).

ALSO CHECK, while you have the evidence in hand: the 3 known-failing finance ITs
(`JournalEntryImmutabilityIT`, `JournalEntryBalanceTriggerIT`, `InternalAutoPostIT` — "Branch context
required") were flagged as possibly the SAME tenant/branch-context wrinkle. State in the SUMMARY whether they
are, with a one-line reason. If they are, fixing the root cause here should fix them — check.

hypothesis, say so and keep digging — do not proceed to Task 2 on a guess.
