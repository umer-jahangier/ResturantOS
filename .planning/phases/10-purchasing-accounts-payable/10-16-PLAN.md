<objective>
Make a missing encryption key fail loudly instead of silently discarding the vendor's bank account.

Purpose: PUR-01 and ROADMAP SC#1 specifically require the vendor bank account to be stored FIELD-ENCRYPTED. `VendorService.apply` currently sets `bankAccountNo` and `bankAccountLast4` to **null** when the `EncryptionService` bean is absent (i.e. `restaurantos.encryption.key` unset — the bean ships via an opt-in `EncryptionAutoConfiguration`, decision 02-02). That is silent data loss on the exact field the requirement calls out, and the failure mode is invisible: the vendor saves fine, the UI shows a blank bank account, and nobody learns why until an AP payment needs it. A security control that degrades quietly is not a security control.

Output: fail at startup if the key is missing; never null-out a submitted bank account.
</objective>

Key facts:
- Decision 02-02: `EncryptionService` lives in shared-lib behind an opt-in `EncryptionAutoConfiguration`
  keyed on `restaurantos.encryption.key`. It is NOT part of `SharedAutoConfiguration`, so a service that
  forgets the property gets NO bean — which is exactly how `VendorService` ended up with an
  `Optional<EncryptionService>`-shaped null-out branch.
- `application.yml` currently sets a dev default for `restaurantos.encryption.key`, which is why the
  encryption IT passes. The hole opens in any environment where the env var is unset and the default is
  overridden to empty.
- Run `impact({target: "VendorService", direction: "upstream"})` before editing, per CLAUDE.md.
</context>

  <files>

  <action>
1. `VendorService`: change the `EncryptionService` dependency from optional/nullable to a REQUIRED
   constructor parameter, and delete the branch that nulls out `bankAccountNo`/`bankAccountLast4`.
   `apply()` should encrypt unconditionally when a bank account number is supplied. If the caller supplies
   no bank account at all, leaving the fields null is correct and must stay — only the
   "key-missing-so-drop-the-data" path is removed.
   Do NOT swallow an encryption exception: let it propagate (it will surface as a 500 with a logged stack
   trace + traceId now that `GlobalExceptionHandler` logs — see the working-tree fix from this UAT).

2. `EncryptionRequiredConfig` — a `@Configuration` in purchasing-service that fails FAST at startup if the
   `EncryptionService` bean is absent. Simplest correct form: since the constructor injection in step 1
   already makes it a required bean, Spring will fail context startup with `NoSuchBeanDefinitionException`
   on its own. Prefer that over adding a config class IF the resulting error message names the missing
   property. If it does not, add the config class with an `@PostConstruct` (or a `BeanFactoryPostProcessor`)
   that throws an `IllegalStateException` with an ACTIONABLE message:
   "purchasing-service stores field-encrypted vendor bank accounts (PUR-01) and requires
   restaurantos.encryption.key to be set. Refusing to start."
   Pick one approach, not both, and justify it in the SUMMARY.

3. Add a comment on the `restaurantos.encryption.key` entry in `application.yml` stating the service will
   not start without it (do not remove the dev default — that would break every existing IT).
  </action>
  <verify>
`grep -n "bankAccountNo\|setBankAccountLast4" services/purchasing-service/src/main/java/io/restaurantos/purchasing/service/VendorService.java` — no branch assigns null because a bean is missing.
`mvn -q -pl services/purchasing-service -am compile` succeeds.
  </verify>
  <done>VendorService cannot run without a real EncryptionService; the silent null-out branch is gone.</done>
