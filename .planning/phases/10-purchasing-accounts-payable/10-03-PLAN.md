<critical_codebase_facts>
Read these before coding — they differ from naive assumptions:
1. `VendorInvoiceLine` has `poLineId` and money fields but NO `ingredientId`. To resolve a
   category you MUST go: invoiceLine.poLineId → PurchaseOrderLine (PurchaseOrderLineRepository has
   `findByPurchaseOrderId`; also fetch by id via findById) → PurchaseOrderLine.ingredientId →
   IngredientCategoryResolver.resolve(ingredientId).
2. `InvoiceStatus` enum values are: PENDING_MATCH, MATCHED, MISMATCHED, APPROVED_FOR_PAYMENT, PAID.
   Spend includes MATCHED and PAID only (exclude PENDING_MATCH/MISMATCHED/APPROVED_FOR_PAYMENT).
3. `VendorInvoice` fields: vendorId, branchId, purchaseOrder ref, invoiceDate (LocalDate),
   status (InvoiceStatus), totalPaisa, `lines` (List<VendorInvoiceLine>, lazy).
4. `VendorAnalyticsService` currently exposes only `scorecard(...)` — do NOT change it. Add a new
   `spendReport(...)` method. `VendorAnalyticsController` maps `/api/v1/purchasing/analytics` with
   only `/scorecard`; add a new `/spend` GET, keep `/scorecard` untouched.
5. There is NO `src/test/resources` dir in purchasing-service and tests migrate Flyway from
   `classpath:db/migration` only. Seed test data via JPA repositories inside a TenantContext (mirror
   how ThreeWayMatchIT / PurchasingMockE2EIT set tenant and persist entities) — do NOT rely on a SQL
   fixture file unless you also create the test resources dir and wire it.
6. Controller responses use `ApiResponse.ok(...)` envelope. Feature gate: `@RequiresFeature("FEATURE_VENDOR")`
   already on the controller class.
7. Money is BIGINT paisa. Compute deltaPct as ((current - prior) / prior * 100), guarding prior==0
   (return null or a sentinel like 100.0 for new spend — pick one and document it).
8. **PUR-05 price variance (verified unbuilt):** `VendorScorecardDto` is a record with exactly
   `(UUID vendorId, UUID branchId, double onTimeDeliveryPct, double fillRatePct, long totalSpendPaisa,
   int purchaseOrderCount)` — no variance field. `VendorAnalyticsService.scorecard()` never computes one.
   `grep -rni "pricevariance|price_variance" services/ frontend/ .planning/` returns ZERO hits.
9. **Do NOT invent the variance math** — `ThreeWayMatchService.matchLine()` already computes exactly the
   per-line quantity you need:
   `BigDecimal priceRatio = BigDecimal.valueOf(invPrice).divide(BigDecimal.valueOf(poPrice), 6, RoundingMode.HALF_UP)`
   where `poPrice = poLine.getUnitPricePaisa()` and `invPrice = invoiceLine.getUnitPricePaisa()`.
   Per-line variance is `priceRatio - 1`. Aggregate that across the vendor's lines. Reuse the shape;
   do not re-derive tolerances (variance is a *metric*, not a tolerance check — no LineMatchStatus involved).
10. `VendorInvoiceLine` again has NO ingredientId and NO vendorId — reach the vendor via the parent
   `VendorInvoice.getVendorId()`, and the PO price via `poLineId` → PurchaseOrderLine (same hop as fact #1).
</critical_codebase_facts>
