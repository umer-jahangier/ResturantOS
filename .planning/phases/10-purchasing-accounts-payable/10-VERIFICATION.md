## Gaps found later (2026-07-12 replan audit) — HISTORICAL RECORD, preserved from the previous verification pass

**Root cause:** the *original* (2026-07-01) verification report scored the 11 must-haves it was given, not
the requirement TEXT. The must-haves were narrower than the requirements, so any clause they omitted was
never checked and silently passed. The "11/11 passed" score measured the wrong thing — it never checked
PUR-02's CLOSED terminal state, PUR-05's third scorecard metric, or FIN-05's expense-approval clause, because
none of those were in the must-haves it was handed. THREE requirements shipped incomplete this way:

- **PUR-02 — PO `CLOSED` transition missing.** ROADMAP SC#2 requires DRAFT→…→CLOSED. `PoStatus.CLOSED`
  was declared in the enum but was dead code — no service, controller, or IT ever set it; the lifecycle
  stopped at FULLY_RECEIVED (`GrnReceiptSimulator.updatePoReceiveStatus`). The "PO lifecycle + OPA
  approval ✓" row in the original report verified DRAFT→SENT only.
  **Closed by 10-04**: `PurchaseOrderService.close(id, reason)` — free close from FULLY_RECEIVED,
  OPA-gated (`vendor.po.close`) mandatory-reason short-close from PARTIALLY_RECEIVED, `V3__po_close.sql`,
  `POST /{id}/close`, `PurchaseOrderCloseIT` (5 tests), Close PO UI on the PO detail page.
  **Re-confirmed in this pass**: `grep -rn "PoStatus.CLOSED" services/purchasing-service/src/main/java/`
  still returns the real assignment at `PurchaseOrderService.java:120`; `PurchaseOrderCloseIT` still green.
- **PUR-05 — price variance never built.** PUR-05 requires THREE metrics: lead-time adherence, fill
  rate, and **price variance per vendor**. `VendorScorecardDto` shipped with only on-time %, fill rate,
  total spend, and PO count; `VendorAnalyticsService.scorecard()` computed no variance. A repo-wide grep
  for `priceVariance|price_variance` returned ZERO hits before 10-03 — it wasn't even a known deferred gap.
  **Closed by 10-03 Task 4**: `VendorScorecardDto.priceVariancePct` +
  `VendorAnalyticsService.computePriceVariancePct()` — a spend-weighted mean of per-line
  `(invoiceUnitPricePaisa/poUnitPricePaisa - 1)*100`, reusing `ThreeWayMatchService`'s exact priceRatio
  math. **Re-confirmed in this pass**: `VendorScorecardIT` has a real weighted-mean assertion (0.052% vs a
  naive 2.5% average — proving the weighting logic actually runs, not a hardcoded return), still green.
- **FIN-05 — OPA-limited expense approval never built.** FIN-05 is "AP/AR tracked; **expense approval
  respects OPA approval limits**." Only the AP-aging half shipped originally. There was no Expense entity,
  no ExpenseService/Controller anywhere, and no code path called OPA with an expense action — the
  `finance.expense.approve` permission sat in auth-service's seed with ZERO consumers.
  **Closed by 10-05**: `Expense` entity + `V5__expenses.sql`, `finance.feign.AuthorizationClient`
  (finance-service's first OPA/authorization-service consumer), `ExpenseService.approve()` double-gated
  on RBAC (`finance.expense.approve` authority) + OPA approval-limit check (deny → 403
  `EXPENSE_APPROVAL_LIMIT_EXCEEDED`), balanced JE auto-post on approval via the existing idempotent
  `autoPostInternal`. **Re-confirmed in this pass**: `ExpenseApprovalIT` (4 tests, fresh green failsafe

  reject-requires-reason; the OPA rego policy (`policies/restaurantos/finance.rego`) has a real
  `approval_limit_paisa` comparison backed by its own rego test suite.

## Lesson for future phase verification (standing, do not remove)

Score against the requirement/ROADMAP-success-criterion TEXT, decomposed clause-by-clause, not against a
plan's self-declared must-haves — a plan's must-haves can be narrower than the requirement it claims to
satisfy, and that gap is invisible unless the verifier goes back to the original requirement wording for
every single clause. This phase escaped three requirements to "Complete" once via must-have-scoring; the
fix is verifying against source with an adversarial mindset, decomposing every ROADMAP criterion into
individually-checkable sub-clauses, and preferring live grep/test-report evidence over any prior SUMMARY or
VERIFICATION.md prose (including this one, for the next phase's verifier).
