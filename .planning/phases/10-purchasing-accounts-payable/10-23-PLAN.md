<objective>
Fix Bug 5 — vendor create is not idempotent.

During UAT round 2 a genuine transient 503 (which had in fact SUCCEEDED server-side) was retried once, and
produced two distinct vendors both named `UAT Produce Co` (`0de48c39…`, `089ba094…`). `POST
/api/v1/purchasing/vendors` has no idempotency-key support.

This is not a new capability. shared-lib already ships `IdempotencyService` and an `idempotency_keys` table
from Phase 1, and `ApPaymentController` in this very service already accepts an `Idempotency-Key` header and
threads it into `ApPaymentService.create(req, idempotencyKey)`. Vendor create simply never used the seam.

Purpose: a retry after a transient failure cannot silently corrupt the vendor master with duplicates.
Output: vendor create wired to the existing idempotency seam, end to end, browser-proven by double-submit.
</objective>

  <action>
The backend seam is useless if the browser sends a fresh key on every retry — the UAT's duplicate came from a
BROWSER retry, so the key must be stable across retries of the SAME user submit.

In the repository layer (`frontend/lib/repositories/vendor-repository.ts` — respect the strict four-layer
contract: schema → adapter → repository → hook → component; `no-restricted-imports` forbids reaching around
it from `components/**`), have `createVendor` accept an `idempotencyKey` and send it as the `Idempotency-Key`
header.

Generate the key ONCE PER SUBMIT ATTEMPT, not per HTTP request:
  - Mint `crypto.randomUUID()` when the user submits the form.
  - Reuse that same key for any automatic TanStack Query retry AND for a user-initiated "retry" of that same
    submission.
  - Mint a NEW key only when the user starts a genuinely new create.
Mirror how the AP payment create mutation does this if it already sends a key; if it does not, this is the
reference implementation and note that in the SUMMARY.

The failure mode to design against is precisely the UAT's: request succeeds server-side, the response is lost
(503 at the gateway), the client retries → must NOT create a second vendor.
