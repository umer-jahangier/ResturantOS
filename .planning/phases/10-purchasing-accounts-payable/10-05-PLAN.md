<critical_codebase_facts>
Verified against live source — several of these will bite you if you assume otherwise:
1. **finance-service has NO authorization/OPA client today.** Its `feign/` package contains only
   InventoryInternalClient, PosInternalClient, PurchasingInternalClient (+ fallbacks). You must add one.
2. **Do NOT reach for shared-lib's `AuthorizationService`/`OpaClient` here.** shared-lib DOES ship
   `io.restaurantos.shared.authz.{OpaClient, DefaultOpaClient, AuthorizationService, OpaInput, OpaDecision}`,
   BUT `SharedAutoConfiguration` registers the `OpaClient` bean under
   `@ConditionalOnProperty(name = "restaurantos.opa.url", matchIfMissing = false)`, and NEITHER
   finance-service NOR purchasing-service sets `restaurantos.opa.url` in any application*.yml. Wiring
   shared-lib's AuthorizationService into finance would fail with NoSuchBeanDefinitionException at runtime.
   That is precisely why 10-01 used a Feign client to authorization-service instead. **Mirror
   purchasing:** copy `services/purchasing-service/.../feign/AuthorizationClient.java` verbatim into
   `io.restaurantos.finance.feign` — `@FeignClient(name = "authorization-service", path = "/internal")`,
   `ApiResponse<AuthorizeResult> authorize(@RequestBody AuthorizePayload)`, with records
   `AuthorizePayload(String module, String action, Resource resource)`,
   `Resource(String type, UUID id, UUID tenantId, UUID branchId, UUID createdBy, String status, Long amountPaisa)`,
   `AuthorizeResult(boolean allow, String reason)`.
3. **Deny-check shape to copy** is `PoApprovalService.assertOpaAllows(po)`:
   `if (response.data() == null || !response.data().allow()) throw new <...>Exception();`
   Fail-closed on a null body — keep that.
4. **Auto-post API:** `JournalEntryService.autoPostInternal(InternalAutoPostJeRequest)` where the record is
   `(UUID branchId, LocalDate entryDate, String description, String sourceType, UUID sourceId,
   List<CreateJeLineRequest> lines)`. `autoPostInternal` is ALREADY idempotent — it looks up
   `jeRepo.findByTenantIdAndSourceTypeAndSourceId(...)` and returns the existing JE rather than
   double-posting. So a second approve cannot create a second JE; your test should assert this rather
   than you hand-rolling dedup.
5. **Migrations:** finance has V1–V4, so the new one is `V5__expenses.sql`. Follow V3's style — the new
   table needs `GRANT SELECT, INSERT, UPDATE, DELETE ON expenses TO finance_user;`, an RLS policy on
   `tenant_id = current_setting('app.current_tenant_id')::uuid`, and `ALTER TABLE expenses FORCE ROW LEVEL
   SECURITY;`. Omitting the GRANT/FORCE RLS lines is the classic failure here.
6. **Exception → HTTP mapping** is done in `FinanceGlobalExceptionHandler` (a `@RestControllerAdvice`) with
   explicit `@ExceptionHandler` methods returning `errorBody(CODE, message)`; e.g. TotpRequiredException →
   403. Add an `@ExceptionHandler(ExpenseApprovalLimitExceededException.class)` → status 403,
   code `EXPENSE_APPROVAL_LIMIT_EXCEEDED`. There is no generic 403 fallback — if you skip this the
   exception surfaces as a 500.
7. **Controller conventions:** `@RestController`, `@RequestMapping("/api/v1/finance/...")`,
   `@RequiresFeature("FEATURE_FINANCE")` on the class, `@PreAuthorize("hasAuthority('...')")` per method,
   returning `ApiResponse.ok(...)` (see ApArController).
8. **`FinanceTestBase` does NOT mock an AuthorizationClient** (it predates one) — it only @MockitoBeans
   StringRedisTemplate, RabbitTemplate, IdempotencyKeyRepository, OutboxRepository. Your IT must declare
   its own `@MockitoBean AuthorizationClient` and stub
   `when(authorizationClient.authorize(any())).thenReturn(ApiResponse.ok(new AuthorizationClient.AuthorizeResult(true|false, reason)))`.
9. Existing finance exceptions to reuse where apt: `InvalidAccountCodeException`, `AccountNotFoundException`,
   `PeriodLockedException` (auto-post into a LOCKED period already returns 423 — do not re-implement).
</critical_codebase_facts>

<design_decisions priority="high">
DECIDED — implement as written:
- **Expense lifecycle:** DRAFT is unnecessary. Create lands directly in `PENDING_APPROVAL`.
  approve → `APPROVED`; reject(reason, mandatory) → `REJECTED`. Approving/rejecting from any state other
  than PENDING_APPROVAL is an error.
- **OPA is the approval-limit authority, RBAC is the coarse gate.** Both apply on approve:
  `@PreAuthorize("hasAuthority('finance.expense.approve')")` (can this role approve expenses at all?) AND
  the OPA call (is THIS amount within THIS user's approval limit?). The RBAC permission already exists in
  auth-service with no consumer — this plan is what consumes it.
- **JE on approve only.** DR the requested expense account, CR bank `1110` (the same bank code 10-02 uses
  for AP payments). `sourceType="EXPENSE"`, `sourceId=expenseId`. Validate the expense account code exists
  via the existing COA lookup and throw `InvalidAccountCodeException` if not — do not invent accounts.
- **No frontend in this plan.** FIN-05's requirement text is about the API respecting OPA limits; there is
  no expense UI in the Phase 10 roadmap or §8.4 route list. Keep scope tight. (If a UI is wanted later it
  is a separate, cheap plan.)
- **Out of scope:** expense categories/attachments/recurring expenses, AR (FIN-05's AR half is a read-only
  balance already covered by the finance GL endpoints), multi-tier expense approval (single OPA check).
</design_decisions>
