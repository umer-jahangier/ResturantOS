<objective>
Make this class of escape impossible to repeat.

Phase 10 shipped 18 plans. Every one was green — unit tests, real-Postgres ITs, even real-OPA container ITs
with watched negative controls. And a human opening a browser found that **no PO could be approved by anyone**,
**expense creation failed 100% of the time**, **PO detail pages hung on "Loading…" forever**, and **a cashier
could see the entire Purchasing module**. Backend ITs verify the callee. Nothing verified the caller, the
browser, or the persona.

Plans 10-19..10-25 fixed those bugs and each proved itself in a browser by hand. Hand-driven proof does not
survive contact with the next phase. This plan converts the 10 UAT journeys into a committed Playwright suite
wired to CI, so the next regression is caught by a machine in minutes rather than by a human three phases
later.

A Playwright scaffold already exists from plan 04-03 (`frontend/playwright.config.ts`, `frontend/e2e/` with
`smoke.spec.ts` and `pos-offline.spec.ts`) — extend it, do not build a second harness.

Purpose: institutionalise the standing lesson.
Output: the 10 UAT journeys as executable specs, running as real seeded personas against the real stack, in CI.
</objective>

This is the assertion Phase 10 exists to make true, and the one it has been failing: **a manager runs a PO
from creation through approval, receipt, matching, and payment.** Encode the whole chain so it can never
silently break again.

`purchasing-lifecycle.spec.ts` — one end-to-end journey, real personas, real stack:
  1. (J3, PUR-02) manager1 creates a PO (vendor + lines) → DRAFT. Opens the detail page **by hard navigation**
     (not just client-side routing — Bug 4 only reproduced on hard reload; a test that only clicks through
     from the list would have missed it entirely). Asserts the PO renders and the Submit button is present.
     Submit → PENDING_APPROVAL.
  2. (J3) manager2 approves → **APPROVED**. Then Send → **SENT**. This is the Bug 1 assertion.
  3. (J4, 10-07) distinct-approver: manager1 submits a second PO and attempts to approve it themselves →
     rejected (409 surfaced in the UI); PO stays PENDING_APPROVAL.
  4. (J5, 10-12) per-line partial receipt against the SENT PO → PO reaches PARTIALLY_RECEIVED; per-line
     received quantities are correct.
  5. (J6, PUR-03/04) create a vendor invoice against the PO and run the 3-way match → MATCHED.
  6. (J7) record an AP payment against the matched invoice → payment recorded, AP balance reduced.
  7. (J9, PUR-05/06) open /app/purchasing/analytics; now that a PO has actually completed, assert the data is
     non-empty and the period/vendor selectors filter it. (Journey 9 "passed" in UAT only because the report
     was honestly empty — an empty report is not a working report.)

Money is BIGINT paisa — assert displayed rupee amounts are the correct rendering, not off by 100×.
Make each numbered journey its own `test()` with a shared serial fixture so a failure names the exact stage
that regressed. Keep them ordered/serial: this is a genuine state machine, not independent cases.
  </action>
