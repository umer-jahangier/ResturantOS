<objective>
Replace the mocked authorization seam in the PO-approval and expense-approval integration tests with a REAL OPA container evaluating the REAL `policies/` bundle, so the action-string contract fixed in 10-07 can never silently regress.

Purpose: standing lesson 10-06-A. `PurchaseOrderApprovalIT` and `ExpenseApprovalIT` both `@MockitoBean` the Feign `AuthorizationClient` and stub `allow=true`. That is exactly why 18 green ITs coexisted with a policy that denied 100% of real requests. A test that mocks the thing that broke proves nothing. This plan builds the missing test: service action string -> OpaInput -> real OPA -> real rego -> allow/deny.

Output: A reusable `OpaBackedAuthorizationClient` test double in each service (real HTTP to a real OPA container, no mocking) plus two ITs that assert allow AND deny through the real policy.
</objective>

Do NOT stand up authorization-service in-process. Instead reproduce, in the test, exactly what
`AuthorizeService.authorize()` does in production — then let a real OPA decide.

1. `OpaBackedAuthorizationClient implements AuthorizationClient` (test scope, package
   `io.restaurantos.purchasing.opa`):
   - Constructor takes the OPA base URL, plus the caller's identity (userId, tenantId, branchId,
     permissions, attributes) held in a mutable `TestPrincipal` so a test can switch approver identity
     between calls.
   - `authorize(AuthorizePayload payload)` builds the OPA input EXACTLY as `AuthorizeService` does:
     `{"input": {"user": {"id","tenant_id","branch_id","permissions","attributes"},
       "resource": {"type","id","tenant_id","branch_id","created_by","status","amount_paisa"},
       "action": <payload.action() VERBATIM>, "environment": {...}}}`
     — snake_case keys, amount in paisa, action passed through with ZERO rewriting. The whole point is
     that the production action string reaches the real policy untouched.
   - POSTs to `{opaUrl}/v1/data/restaurantos/{payload.module()}` and returns
     `ApiResponse.ok(new AuthorizeResult(result.allow == true))`, defaulting to `allow=false` when the
     document is undefined/absent (fail-closed, matching `AuthorizeService`'s catch).
   - Use `RestClient`/`JdkClientHttpRequestFactory` + Jackson, same as authorization-service's IT.

2. `RealOpaTestConfig` (`@TestConfiguration`): starts the OPA `GenericContainer` (image
   `openpolicyagent/opa:1.17.1`, command `run --server --addr=0.0.0.0:8181 /policies`, filesystem-bind the
   repo `policies/` dir read-only, `Wait.forHttp("/health")`) — copy `policiesDir()` and the container
   declaration from `authz/integration/BaseIntegrationTest.java` — and exposes
   `@Bean @Primary AuthorizationClient` returning the `OpaBackedAuthorizationClient`. No `@MockitoBean`
   anywhere in this IT.

3. `PurchasingOpaPolicyIT extends PurchasingTestBase` with `@Import(RealOpaTestConfig.class)`. Build POs
   through the existing service beans (reuse the fixture helpers in `PurchaseOrderApprovalIT`; extract
   them to a shared helper if that is cleaner than duplicating). Tests:
   - `approve_allowedByRealPolicy_whenWithinApprovalLimit` — approver has permission `vendor.po.approve`
     and `attributes.approval_limit_paisa` above `po.totalPaisa` -> PO reaches APPROVED. **This test MUST
     fail if the action string is reverted to `vendor.po.approve`** — assert that explicitly in a comment.
   - `approve_deniedByRealPolicy_whenOverApprovalLimit` — `approval_limit_paisa` below `po.totalPaisa` ->
     `ApprovalLimitExceededException`, PO stays PENDING_APPROVAL, no PO_APPROVED event published.
   - `approve_deniedByRealPolicy_whenPermissionMissing` — permissions list without `vendor.po.approve`.
   - `approve_deniedByRealPolicy_whenCrossTenant` — resource tenant != user tenant.
   - `close_allowedByRealPolicy_forShortClose` — PARTIALLY_RECEIVED PO + reason + permission
     `vendor.po.close` -> status CLOSED. (This exercises the `close_po` rule 10-07 added; before 10-07 the
     rule did not exist and this would deny.)
   - `close_deniedByRealPolicy_whenPermissionMissing`.

4. Add the Testcontainers modules needed (`org.testcontainers:testcontainers` for `GenericContainer`) to
   `services/purchasing-service/pom.xml` test scope if not already present.
