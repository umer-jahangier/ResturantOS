<objective>
Build Accounts Receivable: the house/corporate customer-account entity, the AR sub-ledger, AR balances, AR aging, and the internal seam that Phase 7's POS "charge to account" tender will call on order close.

Purpose: FIN-05 says "AP/AR tracked". Only AP was ever built. 10-17 recorded the decision (10-17-A) that AR is **in scope**, sourced from **corporate/house accounts** — restaurants bill corporate clients and regulars on account, and the customer settles later. This plan builds that.

**The thing that makes this NOT speculative infrastructure:** AR gets a real writer *in this phase*. POS does not exist yet (Phase 7 is 0/4 plans), so the POS tender cannot be the writer today. Instead this plan ships (a) a manual "post a charge to a house account" path a human can drive from the Finance UI right now — which is itself a real restaurant workflow (catering invoice, phone order on account, month-end corporate billing) — and (b) the internal seam, integration-tested against a real Postgres, that Phase 7 will call. An AR ledger with no writer is the always-empty sub-ledger the audit warned about; this one has two.

Output: a house-account you can create, charge, settle, and age — with every movement landing as a balanced journal entry against the already-seeded account `1200 Accounts Receivable`.
</objective>

## Ground truth, already verified against source — build on these, don't re-derive

**The CoA accounts you need are ALREADY SEEDED** (`PakistanRestaurantCoaTemplate`, 55 accounts). Use them; do
not invent accounts:
- `1200 Accounts Receivable` — ASSET, **postable**, tag `AR`. (`1210`/`1220` are non-postable children — post to `1200`.)
- `1110 Bank - Main Account` — ASSET, postable, tag `BANK`. (Same account `ExpenseService` credits.)
- `4100 Food Sales` — REVENUE, postable, tag `REVENUE`. `4400 Catering Revenue` also exists but is NOT postable —
  so `4100` is the default revenue account for a charge. Let the caller override the revenue account code, and
  validate it exactly the way `ExpenseService.validateAccountCode` does (`ChartOfAccountRepository
  .findByTenantIdAndCode` + `isActive`, else `InvalidAccountCodeException`).

**The JE posting seam already exists.** `JournalEntryService.autoPostInternal(InternalAutoPostJeRequest(
branchId, entryDate, memo, sourceType, sourceId, List<CreateJeLineRequest>))` posts a JE and is **already
idempotent on (sourceType, sourceId)**. `ExpenseService.approve()` is your worked example. Use it. Do not
hand-roll JE creation, and do not write to `1200` from a side table that bypasses the ledger.

**The internal seam already has an auth mechanism.** `FinanceInternalServiceFilter` guards `/internal/**` with
a constant-time `X-Internal-Service` secret check (403 `INTERNAL_AUTH_REQUIRED` on miss).
`InternalFinanceController` takes `@RequestHeader("X-Tenant-Id") UUID tenantId` and brackets the call with
`tenantHelper.activate(tenantId)` / `tenantHelper.clear()`. Your AR seam is a new method on that SAME
controller, following that SAME shape.

**Double-entry is enforced by the database.** Every JE must balance or a DEFERRABLE trigger rejects it at
COMMIT (06-01-B/C). Posted JEs are IMMUTABLE — corrections are reversal-only (06-01-E). Both the charge and
the settlement post balanced JEs; neither ever mutates a posted one.

**Period locking**: posting into a LOCKED period returns 423 `PERIOD_LOCKED` (Phase 6). You get this for free
via `autoPostInternal` — but you must prove it, and prove no `ar_transaction` row survives the rollback.

## Constraints — violating any of these breaks a build guard someone else just installed

- **Flyway, not Liquibase, for finance-service** (06-01-A). One new file: `V6__accounts_receivable.sql`.
  auth-service is the opposite — it is Liquibase, so the permission seed is an XML changeset there.
- **Money is BIGINT paisa everywhere** (`MoneyUtils`, shared-lib). No decimals, no floats, no BigDecimal on
  the wire.
- **Every tenant-scoped table needs an RLS policy in the same migration** or the build check fails. Copy
  `V5__expenses.sql`'s policy + `FORCE ROW LEVEL SECURITY` + `GRANT ... TO finance_user` exactly.
- **10-09 installed a reflection guard** that fails the build on any request-mapped public controller method
  with no `@PreAuthorize` — in purchasing-service. finance-service has no such guard yet; **this plan adds
  one** (`FinanceEndpointAuthorizationIT`), so every AR endpoint MUST carry `@PreAuthorize` and its permission
  MUST be seeded in auth-service. If the new guard flags a *pre-existing* ungated finance endpoint, gate it
  with the correct existing permission and say so in the SUMMARY — do NOT allowlist it away.
- **10-11 installed a nav drift test** that reads backend flag definitions off disk (`TierFeatureDefaults
  .java`, `RouteFeatureMap.java`) and fails on any nav item whose feature string does not exist in the
  backend. AR lives inside the **existing Finance module** under `FEATURE_FINANCE` — which is already in both
  files, and `RouteFeatureMap` already maps `/api/v1/finance/` -> `FEATURE_FINANCE`. **Do NOT invent
  `FEATURE_AR` or `FEATURE_RECEIVABLES`.** AR ships as tabs inside Finance, so no new sidebar item is needed
  and no drift is introduced.
- **AR is NOT OPA-gated** (10-17-A). A credit limit is a domain invariant on the account, not an approval
  workflow. Do not add an OPA action verb. If you find yourself editing a `.rego` file, you are in the wrong
  plan. (RBAC via `@PreAuthorize` still applies — that is a different gate.)
- **finance-service consumes OPA via its own Feign `AuthorizationClient`**, not shared-lib's `OpaClient`
  (10-05-A) — noted only so you do not wire one in "for consistency". You need neither here.
- **Frontend is strict four-layer** (Zod schema -> adapter -> repository -> TanStack hook -> component),
  ESLint-enforced by `no-restricted-imports` on `components/**`.
- **PII**: house accounts carry a contact name/phone/email. `EncryptionService` is opt-in via
  `EncryptionAutoConfiguration` (02-02), and 10-16 just made it fail-fast on a missing/blank
  `restaurantos.encryption.key` rather than silently nulling. Contact name/phone/email are **not** encrypted
  here (they are business-contact fields, same class as a vendor's contact, which is not encrypted either —
  only the vendor *bank account* is). Do **not** add a bank-account field to `customer_accounts` in this
  plan; if one is ever needed it must go through `EncryptionService` like `VendorService` does. State this
  explicitly in the SUMMARY so a future reader knows it was considered, not overlooked.
