<objective>
Close the OPA action-string mismatch that silently DENIES every real-OPA authorization in this phase, and make the PO approval policy actually enforce the approval limit + distinct approvers.

Purpose: ROADMAP Phase 10 SC#2 ("tiered approval enforced by OPA") and SC#4 ("expense approvals respect OPA approval limits") are both currently FALSE against a real OPA. The services send permission-code-shaped action strings (`vendor.po.approve`, `vendor.po.close`, `finance.expense.approve`) while every rego module in `policies/` keys on short verbs (`approve_po`, `approve`, `post_journal`, ...). `AuthorizeService` passes `action` through verbatim to OPA, and every module has `default allow := false` — so every one of these calls is denied in production. It was never caught because `PurchaseOrderApprovalIT` and `ExpenseApprovalIT` both `@MockitoBean` the `AuthorizationClient` (standing lesson 10-06-A: never assert against a mocked seam).

Additionally `vendor.rego`'s `approve_po` rule never compares `resource.amount_paisa` to `user.attributes.approval_limit_paisa` (finance.rego DOES), and `vendor.po.close` has NO rego rule whatsoever — so PO close would be denied even after the action string is fixed.

Output: One canonical action vocabulary (rego's short verbs), a hardened `vendor.rego`, rego unit tests, and a distinct-approver constraint in `PoApprovalService`.
</objective>
