<objective>
Fix Bug 4 — and fix it as the CLASS of bug it is, not as two one-off edits.

`purchase-orders/[id]/page.tsx` and `invoices/[id]/page.tsx` both destructure `params.id` synchronously
(`{ params }: { params: { id: string } }`), the pre-Next-15 pattern. In Next 15 `params` is a Promise, so
`params.id` is `undefined` → `usePurchaseOrder(undefined)` never fires (`enabled: Boolean(id)`) → the page
is stuck on "Loading…" forever → **no Submit/Approve/Reject/Send/Receive/Close button ever renders**. This
independently blocks the entire PO and invoice lifecycle from the UI, on top of Bug 1.

Finance's detail pages already do it right (`params: Promise<...>` + `use(params)`). The audit is already
done: of the 4 dynamic routes in the app, exactly these 2 are broken and finance's 2 are correct. So the fix
is small — but nothing stops the next one. This plan adds a guard so it cannot recur.

Purpose: unblock every PO/invoice lifecycle action in the browser, and make this escape class impossible.
Output: two corrected pages + a build-failing guard against sync params anywhere in the app.
</objective>
