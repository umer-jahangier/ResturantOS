<objective>
Fix Bug 1 — THE critical bug. No purchase order can be approved by anyone, through the UI or the API.

`InternalServiceFilter` validates the `X-Internal-Service` shared secret correctly (constant-time compare)
and then calls `chain.doFilter(...)` — but it **never calls
`SecurityContextHolder.getContext().setAuthentication(...)`**. `SecurityConfig` requires
`.requestMatchers("/internal/**").authenticated()`, so the request falls through to `JwtAuthenticationFilter`
(there is no JWT on an internal call, so it no-ops), fails the `.authenticated()` check, and returns **401**.
purchasing-service's `GlobalExceptionHandler` then masks that as a client-facing **503**.

Reproduced directly with curl after a clean restart:
`curl -H "X-Internal-Service: <secret>" POST /internal/authorize` → `401`.

Blast radius: PO approve, PO short-close, and expense approve/reject (ExpenseService uses the same client).
Transitively blocks UAT journeys 4, 5, 6 and 7. No PO in this tenant has EVER reached APPROVED or SENT.

Why 12 green plans missed it: 10-08's real-OPA Testcontainer ITs prove the POLICY is correct. They never
exercise the service-to-service HTTP CALL PATH. The policy is right; **the call never arrives**. This plan
tests the caller.

Purpose: unblock the phase's core promise — a manager runs a PO through approval.
Output: an authenticating internal filter, an IT that exercises the real call path, honest error mapping, and
a browser-driven approval.
</objective>

  <files>services/authorization-service/src/main/java/io/restaurantos/authz/config/InternalServiceFilter.java, services/purchasing-service/src/main/java/io/restaurantos/purchasing/exception/GlobalExceptionHandler.java</files>
  <action>
FIX (authorization-service) — in `InternalServiceFilter.doFilterInternal`, on a SUCCESSFUL constant-time
secret match, populate the SecurityContext BEFORE `chain.doFilter(...)`:

    Authentication auth = new UsernamePasswordAuthenticationToken(
        "internal-service", null, List.of(new SimpleGrantedAuthority("ROLE_INTERNAL_SERVICE")));
    SecurityContextHolder.getContext().setAuthentication(auth);
    try {
        chain.doFilter(request, response);
    } finally {
        SecurityContextHolder.clearContext();   // MUST clear — a leaked principal on a pooled request
                                                // thread would authenticate an unrelated later request
    }

Keep the existing failure branch exactly as it is (wrong/absent secret → 403, constant-time compare
preserved). Do NOT weaken `SecurityConfig`'s `.requestMatchers("/internal/**").authenticated()` — the config
is correct; the filter was the thing not holding up its end. "Fixing" this by permitting `/internal/**`
would delete the shared-secret gate entirely and turn a 401 into an open internal API.

FIX (purchasing-service) — `GlobalExceptionHandler` turned a 401 from authorization-service into a
client-facing 503 "service unavailable". That single mistranslation is a large part of why this took a full
UAT round to find: it said "the authz service is down" when the authz service was up and answering.
Map Feign failures honestly:
  - authorization-service unreachable / connect timeout → 503 is correct.
  - authorization-service ANSWERED with 4xx (401/403) → that is a WIRING or CONFIG fault, not an outage. It
    must NOT be reported as 503. Surface a distinct 500-class error whose message/code names the real
    condition (e.g. authorization call rejected: 401), and LOG the upstream status and body at ERROR.
An error handler that lies about which service is broken costs more than the bug it hides.
  </action>
  <verify>
  Rebuild + restart authorization-service into the running stack (10-19), then curl the real endpoint:
    - Valid secret → `curl -X POST -H "X-Internal-Service: <secret>" ... /internal/authorize` → **200** with
      an allow/deny decision body. (This is the exact command that returned 401 in the UAT.)
    - NEGATIVE CONTROL, watched going red: WRONG secret → **403**. Absent header → **403**. If a bad secret
      now returns 200, the fix has opened the internal API and is worse than the bug.
  Paste all three curl outputs into the SUMMARY.
  </verify>
  <done>/internal/authorize returns 200 for a valid secret and 403 for an invalid/absent one; purchasing no longer reports authz 4xx as 503.</done>
