**Date:** 2026-07-14

This UAT ran *after* all 12 gap-closure plans (10-07..10-18) reported success. Every plan's own
verification was green. The module still does not work for a human. The gap-closure wave fixed the

## Bugs

### Bug 1 — CRITICAL — internal authorize endpoint always 401s; no PO can be approved
**File:** `services/authorization-service/src/main/java/io/restaurantos/authorization/config/InternalServiceFilter.java`

Validates the `X-Internal-Service` shared secret correctly but **never calls
`SecurityContextHolder.getContext().setAuthentication(...)`**. `SecurityConfig` requires
`.requestMatchers("/internal/**").authenticated()`, so the request falls through to
`JwtAuthenticationFilter` (no JWT on internal calls → no-op) and fails the authenticated check → `401`.
purchasing-service's `GlobalExceptionHandler` masks it as a client-facing **503**.

Reproduced directly: `curl -H "X-Internal-Service: <secret>" POST /internal/authorize` → `401`.
Confirmed after a clean restart — not a stale-connection artifact.

**Blast radius:** PO approve, PO short-close, expense approve/reject (`ExpenseService` uses the same
client). No PO in this tenant has EVER reached APPROVED/SENT. Transitively blocks journeys 4–7.

**Why the tests missed it:** 10-08's real-OPA Testcontainer ITs prove the *policy* is correct. They do
not exercise the *service-to-service HTTP call path* into authorization-service. The policy is right;
the call never arrives.

### Bug 2 — CRITICAL — expense creation always fails account validation
**File:** `services/finance-service/src/main/java/io/restaurantos/finance/service/ExpenseService.java` (`validateAccountCode()`, ~L149-155)

`coaRepository.findByTenantIdAndCode(tenantId, accountCode)` returns empty and throws
`InvalidAccountCodeException`, for accounts the API itself lists as active for that same tenant.

Reproduced: `GET /api/v1/finance/accounts?type=EXPENSE&active=true` → returns `6300` (+11 others).
`POST /api/v1/finance/expenses` with `expenseAccountCode:"6300"` → `400 INVALID_ACCOUNT_CODE`.
Tried 6000, 6100, 6300 — **100% failure**, not code-specific.

Likely a tenant-context / RLS mismatch between the read path and the write path. Expense creation is
completely non-functional via UI and API.

### Bug 3 — HIGH — Purchasing frontend has no permission guard; cashier sees the whole module
**Files:** `frontend/app/(tenant)/app/purchasing/layout.tsx`, `frontend/components/shared/sidebar-nav-items.ts` (L78-84, L199-206)

The layout wraps content in `<FeatureGuard feature="FEATURE_VENDOR">` only — **no `PermissionGuard`**.
Finance correctly uses `PermissionGuard require="finance.journal.view"` + `FeatureGuard`.

The nav entries carry the comment `// RBAC: vendor.view (see 10-09)` but **never set a `permission:`
field** — every other nav entry (POS, Finance, Inventory, RBAC admin) does.

Result: a cashier sees the Purchasing sidebar link, the landing page, all 5 tabs, and on Vendors an
empty state reading *"No vendors yet — use Add vendor to create your first vendor"* with a live,
clickable Add button. The backend correctly `403`s every call, so nothing leaks — but the UI
advertises a module the user cannot use, which is precisely the class of defect this phase was
reopened to eliminate.

Plan 10-09 gated the backend; the frontend was never brought along.
Screenshot: `.playwright-mcp/10-cashier-purchasing-leak.png`

### Bug 4 — CRITICAL — PO + invoice detail pages permanently stuck on "Loading…"
**Files:** `frontend/app/(tenant)/app/purchasing/purchase-orders/[id]/page.tsx`, `frontend/app/(tenant)/app/purchasing/invoices/[id]/page.tsx`

Both destructure `params.id` synchronously (`{ params }: { params: { id: string } }`) — the pre-Next-15
pattern. Finance's detail pages correctly use `params: Promise<...>` + `use(params)`.

`params.id` resolves to `undefined` → `usePurchaseOrder(undefined)` never fires (`enabled: Boolean(id)`)
→ the page never loads → **no Submit/Approve/Reject/Send/Receive/Close button ever renders**, for any PO
or invoice. Reproducible on hard reload.

Independently blocks the entire PO/invoice lifecycle from the UI, on top of Bug 1.

### Bug 5 — MEDIUM — vendor create is not idempotent
`POST /api/v1/purchasing/vendors` has no idempotency-key support. A genuine transient 503 (which had
in fact succeeded server-side), retried once, produced two distinct vendors with the same name
(`UAT Produce Co`: `0de48c39...`, `089ba094...`).

## Standing blocker (out of Phase 10 scope, needs its own item)

**OWNER/ACCOUNTANT accounts are permanently bricked.** `AuthServiceImpl.requiresTotpStepUp()` forces TOTP
step-up for any holder of `rbac.manage` or `finance.period.close`, but neither seeded user has an enrolled
`totp_secret`, and enrolment requires an authenticated session. A real tenant promoting a user to those
permissions before they enrol 2FA locks them out identically. Needs an enrolment-gated token state.

---

## Bottom line

Only **vendor management with bank-account masking** and **nav/shell/analytics wiring** are demonstrably
true for a real user. The phase's core promise — *a manager can run a PO from creation through approval,
receipt, matching, and payment* — is **false**.

Seven of ten journeys are blocked by **four precise, reproducible root causes** (2 backend, 2 frontend) —
not flaky infra. Each is a small fix. The phase does not close until they are fixed and re-verified in a
browser.

**Lesson (extends 10-06-A):** green tests at every layer — unit, real-DB IT, real-OPA container IT — still
did not catch a service-to-service wiring bug (Bug 1), a Next.js API-version bug (Bug 4), or a missing
frontend guard (Bug 3). Backend ITs verify the callee. Nothing verified the caller, the browser, or the
persona. **Every future phase needs a real-browser journey pass before it can be called done.**
