<task type="auto">
  <name>Task 1: Fix RabbitMQ zero-users at the root — declarative users in definitions.json</name>
  <files>deploy/init/rabbitmq-definitions.json, deploy/init/rabbitmq.conf, deploy/docker-compose.yml, scripts/dev-stack-up.sh</files>
  <action>
ROOT CAUSE (already isolated — do not re-investigate from scratch, but DO confirm it before fixing):
`deploy/init/rabbitmq.conf` contains `load_definitions = /etc/rabbitmq/definitions.json`, and
`deploy/init/rabbitmq-definitions.json` declares `vhosts`, `exchanges`, `queues`, `bindings` — but has
**no `users` key and no `permissions` key**. When RabbitMQ loads a definitions file at boot it SKIPS the
`RABBITMQ_DEFAULT_USER` / `RABBITMQ_DEFAULT_PASS` bootstrap entirely. The env vars in docker-compose.yml
(L65-66) are therefore dead, and the broker comes up with zero users — exactly the reported symptom.

CONFIRM FIRST (one command, cheap):
  docker compose -f deploy/docker-compose.yml down -v
  docker compose -f deploy/docker-compose.yml up -d rabbitmq
  docker exec restaurantos-rabbitmq rabbitmqctl list_users
Expect an EMPTY user list. Record that output in the SUMMARY as the proof of root cause. If the list is
NOT empty, stop and re-diagnose — the hypothesis is wrong and the fix below is wrong too.

FIX — make the users declarative so they are created at boot with no human step:
Add `users`, `permissions`, and `topic_permissions` blocks to `deploy/init/rabbitmq-definitions.json`.
RabbitMQ definitions store a `password_hash`, not a plaintext password, so the hash must be produced from
`${RABBITMQ_PASSWORD}` in `deploy/.env`. Two acceptable approaches — pick ONE and justify it in the SUMMARY:

  (A) PREFERRED — generate at bring-up. Add a small step to `scripts/dev-stack-up.sh` that renders
      `deploy/init/rabbitmq-definitions.json` from a committed template
      (`deploy/init/rabbitmq-definitions.template.json`) by injecting a computed password_hash.
      RabbitMQ's default hashing (`rabbit_password_hashing_sha256`) is:
        salt = 4 random bytes; hash = base64( salt || sha256( salt || utf8(password) ) )
      Set `"hashing_algorithm": "rabbit_password_hashing_sha256"` on the user entry.
      Grant the app user `configure/write/read = ".*"` on vhost `/`.
      Keep the rendered definitions.json gitignored; commit the template.

  (B) FALLBACK — if the hash generation proves fiddly, REMOVE `load_definitions` from rabbitmq.conf so the
      `RABBITMQ_DEFAULT_USER` bootstrap works normally, and instead import the topology after boot via the
      management HTTP API (`POST /api/definitions`) from `scripts/dev-stack-up.sh`, authenticating with the
      now-existing default user.

Do NOT "fix" this by documenting a manual `rabbitmqctl add_user` step. The whole point of this task is that
the manual step disappears.
  </action>
  <verify>
From a genuinely cold state:
  docker compose -f deploy/docker-compose.yml down -v     # wipes the rabbit volume
  bash scripts/dev-stack-up.sh
  docker exec restaurantos-rabbitmq rabbitmqctl list_users
MUST list the application user WITHOUT any human having run rabbitmqctl.
Then: `docker exec restaurantos-rabbitmq rabbitmqctl list_permissions` shows the user with `.*` on `/`.
NEGATIVE CONTROL (watch it go red): temporarily re-remove the `users` block from the definitions,
`down -v`, bring up, and confirm `list_users` is empty again and the health gate FAILS. Restore the block.
Record both outputs in the SUMMARY.
  </verify>
  <done>A `down -v` + single bring-up command yields a RabbitMQ with the app user present and all services able to connect, with zero manual intervention.</done>

<task type="auto">
  <name>Task 2: Repair `make dev-up` and add a health gate that fails loudly</name>
  <files>deploy/Makefile, Makefile, scripts/dev-stack-up.sh, scripts/DEV-STACK-RUNBOOK.md</files>
  <action>
PROBLEM 1 — `make dev-up` is broken. `deploy/Makefile` L29-31 runs `docker compose up -d --build`, which
force-rebuilds the eureka/config images; the in-Docker Maven build FAILS (and on this 8GB host it is also
liable to OOM). The images already exist. `dev-up-fast` (L33-35, `up -d` without `--build`) works.

Fix: make the WORKING path the default.
  - `dev-up` becomes `$(COMPOSE) up -d` (no `--build`) + `_dev-up-post` + the new health gate.
  - Add an explicit `dev-rebuild` target for when images genuinely must be rebuilt. If the in-Docker Maven
    build is broken, `dev-rebuild` must build the JARs on the HOST first (`mvn -q -DskipTests package`) and
    have the Dockerfiles COPY the prebuilt jar — do not leave a target that is known to fail.
  - Keep `dev-up-fast` as an alias of `dev-up` so the runbook's existing muscle memory still works.

PROBLEM 2 — nothing verifies the stack actually came up. 10-UAT-2 spent real time discovering services were
DOWN. Create `scripts/dev-stack-up.sh` as the single entry point:
  1. `export PATH="/Applications/Docker.app/Contents/Resources/bin:$PATH"` (Docker's credential helper is
     not on the default PATH on this host — without it compose fails obscurely).
  2. Pre-flight: fail with a clear message if something native is squatting :5432
     (`lsof -nP -iTCP:5432 -sTCP:LISTEN`) — a native EDB PostgreSQL 18 blocks the postgres container.
  3. Bring up infra + services (no `--build`).
  4. HEALTH GATE: poll each service's `/actuator/health` until UP, with a hard timeout (services take
     60-90s to boot on this host). On timeout, print the offending service's last 50 log lines and
     `exit 1`. This is the part that must never be silent.
  5. Print a ready banner with the frontend URL and the working personas.
Keep ClickHouse + pgAdmin stopped by default (this host is memory-constrained and the stack was OOM-killed
once mid-session); expose them behind an opt-in flag/profile.

Then rewrite `scripts/DEV-STACK-RUNBOOK.md` to reflect reality: one command (`bash scripts/dev-stack-up.sh`
or `make dev-up`), the host pitfalls above (Docker PATH, :5432 squatter, memory pressure), and the working
seeded personas (manager1@demo.local / Manager1#2026, manager2@demo.local / Manager2#2026,
cashier@demo.local / Cashier#2026, finance_demo@demo.local / Finance#2026).
  </action>
