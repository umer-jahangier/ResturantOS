<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → /api/v1/inventory/** | User JWT crosses into read/write of inventory master + stock data; OPA must decide view vs manage |
| finance/service → /internal/** | Service-to-service call, JWT-exempt, must present the X-Internal-Service secret |
| service → OPA (data.restaurantos.inventory) | Authorization decision is delegated to the policy; fail-closed on any error |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-8-AC | Elevation of Privilege | all /api/v1/inventory/** endpoints | high | mitigate | Real ASVS V4 control authored here: inventory.rego grants only on inventory.item.view (read) / inventory.item.manage (write) with same_tenant_and_branch (proven by opa test policies/ at 100% coverage), and InventoryAuthorizationService is the seam each controller (08-03/04/06/07/08) calls so a JWT without inventory.item.manage is denied (403) at every write endpoint — the per-controller 403 ITs are the runtime proof. Severity raised to high: this is the phase's primary access-control mitigation, previously claimed but unbuilt. |
| T-8-SPOOF | Spoofing | /internal/** (e.g. GET /internal/grn/pending-count in 08-06) | high | mitigate | InventoryInternalServiceFilter created here guards /internal/** with the X-Internal-Service constant-time secret, mirroring every /internal/* endpoint in the repo; 08-06's GrnPendingCountIT exercises the contract. |
| T-8-SC | Tampering (supply chain) | dependencies | low | accept | No new packages — OPA/rego is an existing repo tool; the Java classes reuse shared-lib AuthorizationService/OpaClient and Spring Security already on the parent POM. |
</threat_model>
