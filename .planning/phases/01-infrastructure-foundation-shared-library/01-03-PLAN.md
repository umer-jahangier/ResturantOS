  <name>Task 2: generate-keys.sh (RS256 keypair + AES-256 key → deploy/.env)</name>
  <files>deploy/generate-keys.sh</files>
  <action>
    Create an idempotent `deploy/generate-keys.sh` (`#!/usr/bin/env bash`, `set -euo pipefail`) that produces the dev secrets that agent-spec 05 §5.1 marks as `JWT_PRIVATE_KEY` (RS256 PEM, auth-service signing key) and `FIELD_ENCRYPTION_KEY` (base64 AES-256-GCM key, used by agent-spec 09 §9.5):
      1. Generate an RSA-2048 private key: `openssl genpkey -algorithm RSA -pkeyopt rsa_keygen_bits:2048 -out <tmp>/jwt-private.pem`; derive the public key: `openssl rsa -in jwt-private.pem -pubout -out jwt-public.pem`. (RS256 per agent-spec 09 §9.2.)
      2. Generate a 32-byte AES-256 key base64-encoded: `openssl rand -base64 32`.
      3. Write/refresh `deploy/.env` from `deploy/.env.example` if `.env` is absent (`cp` once), then set the three keys. CRITICAL (RESEARCH "multiline PEM in .env" gotcha): a raw multiline PEM breaks `KEY=VALUE` parsing. Encode the PEMs as single-line values — either base64 the whole PEM (`base64 -w0`, macOS `base64` has no `-w`, so pipe through `tr -d '\n'`) OR replace newlines with `\n` — and document in the script comment which encoding consumers must reverse. Use base64-of-PEM for robustness and note that `JwksKeyProvider`/`JwtProperties` (shipped in 01-04) decode it.
      4. Use idempotent in-place replacement (delete existing `JWT_PRIVATE_KEY=`/`JWT_PUBLIC_KEY=`/`FIELD_ENCRYPTION_KEY=` lines then append the fresh ones) so re-running rotates keys without duplicating lines. Never echo secrets to stdout; print only "wrote N keys to deploy/.env".
      5. `chmod +x` the script and ensure `deploy/.env` and `*.pem` are gitignored (already added in 01-01's .gitignore).
    Cross-platform: prefer `openssl` (present on macOS + Linux); avoid GNU-only flags.
  </action>
  <verify>`bash deploy/generate-keys.sh` exits 0 and prints a non-secret confirmation; `grep -c '^JWT_PRIVATE_KEY=' deploy/.env` == 1, same for `JWT_PUBLIC_KEY` and `FIELD_ENCRYPTION_KEY`; decoding `FIELD_ENCRYPTION_KEY` yields 32 bytes (`echo $VAL | base64 -d | wc -c` == 32); running the script twice does not duplicate lines.</verify>
  <done>Running the script writes a valid RS256 keypair and a 32-byte AES key into deploy/.env as single-line (base64) values, is idempotent, and never commits secrets.</done>
