DECISIONS RESOLVED IN THIS PLAN (from 04-RESEARCH D1/D2/D3 + on-disk verification of auth-service):
- **D1 — layout:** `frontend/` root layout, NO `src/`. The ESLint `no-restricted-imports` override targets `components/**` (not the spec's stale `src/components/**`), and schemas live at `frontend/lib/api-client/schemas/**` so the §P5.2.5 path + the 04-03 schema-sync job match verbatim.
- **D2 — route protection:** `proxy.ts` (Next 16 rename of `middleware.ts`, exported fn `proxy`, Node runtime) does an OPTIMISTIC redirect on refresh-cookie presence only; real validation is a `server-only` DAL (`lib/auth/dal.ts`, `await cookies()`). Document the `middleware.ts → proxy.ts` deviation against FE-03's wording in the SUMMARY. The proxy is NOT a security boundary (CVE-2025-29927).
- **D3 — auth contract (CONFIRMED against `services/auth-service`, supersedes the research's speculative `ros_refresh`/`/me`):** refresh cookie is **`refresh_token`** (HttpOnly, Secure, SameSite=Strict, Path=`/api/v1/auth`); login body is **`{email, password, tenantSlug, totpCode?}`** (camelCase); success is `ApiResponse<{accessToken, expiresInSeconds, userId, tenantId, branchId}>`; `ApiResponse<T>` = `{data:T, meta, warnings[]}`; `ApiError` = `{error:{code, message, details[], traceId}}`. There is **no `/api/v1/me` endpoint** — current-user roles/permissions/attributes come from DECODING the in-memory access JWT (`lib/auth/jwt.ts`), not an HTTP call. The access token rides in the response body (held in Zustand memory); only `refresh_token` is a cookie.
