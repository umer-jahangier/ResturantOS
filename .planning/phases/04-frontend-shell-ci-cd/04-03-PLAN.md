<objective>
Stand up the GitHub Actions quality-gated CI/CD pipeline that runs lint → test → build → schema-sync with no manual intervention in the core path, enforcing the project's coverage gates and producing cosign-signed multi-arch images — the last piece of the verified Sprint-1 "GO" set.

Purpose: Delivers INFRA-05 and covers SC4 (CI runs lint→test→build→schema-sync automatically; gates finance/inventory ≥75%, others ≥60%, OPA 100%; signed images). The lint/test steps for the frontend depend on the `package.json` scripts + Dockerfile created in 04-01 (hence depends_on 04-01); the Java/OPA steps are independent of the frontend, so this plan runs in wave 2 in PARALLEL with 04-02 (no shared files).

Output: `.github/workflows/ci.yml`, a data-driven `coverage-gates.json`, the Playwright config + one smoke spec (scaffold only), and the `package.json` script additions CI invokes.

DECISIONS RESOLVED IN THIS PLAN:
- **D5 — schema-sync tooling:** implement schema-sync as (a) `tsc --noEmit` over `frontend/lib/api-client/schemas/**` (always works, satisfies FE-08's "schemas typecheck") PLUS (b) a best-effort OpenAPI↔Zod drift check ONLY IF a maintained tool is verified to exist at execution time (`openapi-to-zod-check` is unverified — DO NOT assume it exists; verify on npm first). If no maintained tool exists, ship (a) + a clearly-documented placeholder step that logs "OpenAPI↔Zod cross-repo diff deferred — backend SpringDoc OpenAPI is a Phase-3+ dependency." Schemas stay at `frontend/lib/api-client/schemas/**` (D1) so the §P5.2.5 path matches verbatim.
- **D6 — Playwright scope:** Phase 4 only scaffolds the Playwright config + the `e2e` job wiring + ONE smoke journey (login redirect). The full ~50-journey suite runs against staging and spans every later phase — explicitly NOT a Phase-4 deliverable. INFRA-05's "no manual intervention" is satisfied by the lint→test→build→schema-sync core; the promote-to-prod gate is a manual GitHub environment approval BY DESIGN (§D1.4), not a pipeline failure.
- **Pitfall 9/10:** coverage gates are PER-AREA and data-driven; cosign uses keyless GitHub OIDC (`id-token: write`) against GHCR; multi-arch via `docker buildx`.
</objective>
