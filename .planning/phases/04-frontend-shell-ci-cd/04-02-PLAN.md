DECISIONS RESOLVED IN THIS PLAN:
- **D4 — feature-flag source for `FeatureGuard`:** feature flags are NOT in the JWT (the gateway enforces them server-side against `tenant_features` and returns 403 `FEATURE_DISABLED`; the JWT carries only roles/permissions/attributes). The frontend therefore proactively hides UI via `useFeatureFlags()` → `FeatureRepository.getFlags()` (the endpoint shape frozen in 04-01, MSW-mocked here). Flag the live endpoint as a Phase-3 contract to confirm with the gateway team. `PermissionGuard` reads `permissions[]` from the decoded JWT (no extra call).
- **Login error codes (RE-VERIFIED on disk in `AuthExceptionHandler`/`AuthServiceImpl`, corrects the earlier §7.4 assumptions — Blocker 3):** bad credentials / unknown user / non-ACTIVE (suspended) tenant ALL return `UNAUTHENTICATED` **401** (there is deliberately NO `INVALID_CREDENTIALS` and NO `TENANT_SUSPENDED` code — suspended tenants are masked as generic `UNAUTHENTICATED` so the UI must NOT claim to distinguish "suspended" vs "bad password"); account locked → `ACCOUNT_LOCKED` **423 LOCKED** (NOT 401). The form maps exactly these via the 04-01 `parseApiError` guards `isUnauthenticated()`/`isAccountLocked()`/`isTotpRequired()`.
- **Conditional TOTP (CONFIRMED against 02-02 + on-disk):** the auth-service returns **401 `TOTP_REQUIRED`** (via `AuthExceptionHandler`) when step-up is required (privileged permission OR `totp_enabled`) and `totpCode` is missing/invalid. The login form reveals the TOTP field on that error and resubmits with `totpCode` (FD-2 O→P→Q→R).
- **Branch switch (CONFIRMED against 02-02 + on-disk):** `POST /api/v1/auth/switch-branch {branchId}` reissues the access JWT (new branch_id/permissions/attributes); the refresh cookie/session is unchanged. The client must therefore `queryClient.clear()` since every server-state key is branch-scoped. A denied switch returns `BranchSwitchDeniedException` → **403 `BRANCH_ACCESS_DENIED`** (W3) — the client surfaces it and leaves the current session/branch intact.
- **D7 scope guard:** no offline queue / charts / heavy tables here — Sidebar is permission/feature-driven nav only.
</objective>
