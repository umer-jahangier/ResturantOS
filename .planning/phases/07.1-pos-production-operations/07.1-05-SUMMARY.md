# Phase 07.1 Plan 05: POS/KDS Four-Layer Data Extension + StatusBadge/Revision-Chip Summary

**Extended the frontend four-layer pipeline (schema/model/adapter/repository/hook) end-to-end for the new backend contract (7-value item status, revisions, derived order status, table detail, order summaries, instructions, KDS ticket detail) and encoded the UI-SPEC Status System once in a shared StatusBadge + new revision-chip primitive.**

## Performance

- **Duration:** ~55 min
- **Started:** 2026-07-11 (following 07.1-04)
- **Completed:** 2026-07-11
- **Tasks:** 3
- **Files modified:** 22 (5 created, 17 modified)

## Accomplishments

- **Schema/model/adapter extension (Task 1, POS-11/POS-12):** `apiOrderItemSchema` widened `kdsStatus` to the 7-value `OrderItemStatus` enum + `revisionNo`/`firedAt`; `apiOrderSchema` gained a distinct `derivedStatus` field (never overloading the existing 9-value settlement `status`). New `apiOrderSummarySchema`, `apiTableDetailSchema`, `apiUpdateInstructionsSchema` mirror the backend DTOs exactly. `pos.model.ts` renames the domain field to `itemStatus` at the adapter boundary and adds a `getOrderDisplayStatus()` single seam that merges settlement + derived status into the UI-SPEC's 7-state order-status value. `apiDiningTableSchema`/`TableStatus` widened to 3 values (`NEEDS_BUSSING`). `kds.schema.ts` widened the ticket-item status to kitchen-service's real 5-value `TicketItemStatus` and added `revisionNo`/`firedAt`; a forward-declared `orderNotes` field was added (currently always null — backend gap, documented).
- **Repository + hook layer (Task 2, POS-10/12/13):** `PosRepository.getActiveOrderForTable`, `listOrderSummaries` (replaces the now-broken `listOrders`), `updateInstructions`, `markItemServed`; `sendToKds` now takes a `clientFireId` sent as `Idempotency-Key` (mirrors `closeOrder`/`voidOrder`'s header pattern exactly). `KdsRepository.getTicketDetail` added (ApiResponse-enveloped, unlike the bare-JSON list/bump/recall endpoints on the same repository). New hooks: `useTableDetail` (new `use-tables.ts`), `useOrderSummaries`, `useUpdateInstructions` (offline-safe — a new `UPDATE_INSTRUCTIONS` outbox op type + sync-engine replay branch), `useMarkServed`, `useKdsTicketDetail`. `useSendToKds` now generates a fresh `clientFireId` internally per call with zero call-site changes.
- **StatusBadge + revision-chip (Task 3, POS-11/12):** Extended the existing shared `components/ui/status-badge.tsx` (the actual DS-04 primitive location — not `components/shared/status-badge.tsx` as the plan named it) with an icon+label+semantic-token descriptor for all 7 line-item and 7 order-level statuses from the UI-SPEC Status System table; legacy Finance-only variants (`active`/`inactive`/etc.) stay fully backward compatible, label-only. New `components/pos/revision-chip.tsx`: `RevisionBadge` (Rev 1 unbadged, Rev n>1 shows an accent `REV n` pill) + `RevisionCountChip` (tappable "{n} revisions" that expands to a "Rev {n} · {time} · {count} item(s)" log). 25 new vitest assertions across both new test files, all passing.
