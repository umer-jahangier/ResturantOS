# Phase 07.1 Plan 02: Kitchen-Service Order Revisions + Ticket Detail Summary

**Converts `TicketRoutingService.route()` from skip-if-exists to append-to-existing-ticket (the single structural fix that unblocks order revisions on the KDS side), adds `revisionNo`/`firedAt` to `KdsTicketItem`, mirrors the additive `ORDER_SENT_TO_KDS` payload fields, and ships a permission-guarded ticket-detail read endpoint.**

- **Tasks:** 3

## Accomplishments
- `KdsTicketItem` gains `revisionNo` (default 1) and `firedAt` (nullable Instant); `TicketItemStatus` extends to the kitchen-owned lifecycle subset (`PENDING, ACCEPTED, PREPARING, READY`), retaining `COOKING` as a legacy value so existing rows never fail deserialization — the current `bumpItem` flow (`PENDING -> COOKING -> READY`) is unchanged this plan.
- Additive `V3__ticket_item_revision.sql`: adds `revision_no`/`fired_at` columns, an idempotent `COOKING -> PREPARING` data-fix, and a `(ticket_id, revision_no)` index — no DROP/RENAME.
- `KitchenEventPayloads.OrderSentToKdsPayload` gains `revisionNo` (int) + `orderNotes` (String, nullable) as fields APPENDED after `items`, mirroring the pos-service publisher's future field names exactly (Pitfall 4 / field-name parity is the only contract enforcement Jackson gives us).
- **Core fix:** `TicketRoutingService.route()` now appends new items to an existing `(orderId, stationCode)` ticket instead of silently skipping the whole event — reopening a `READY` ticket to `PENDING` (clearing `readyAt`) and notifying WebSocket subscribers. True event-redelivery dedup stays exclusively with `ProcessedEventService.tryProcess` one layer up in `OrderSentToKdsConsumer` (untouched).
- New `GET /api/v1/kitchen/kds/tickets/{ticketId}` (KDS-03 "open a ticket for full order detail"), backed by `TicketService.getTicketDetail` + a dedicated `@EntityGraph(items)` repository finder to avoid the Phase-7 UAT #6 `LazyInitializationException` class of bug; `ApiResponse`-wrapped per the phase's shared-pattern convention.
- New `TicketRevisionRoutingIT`: proves append-not-skip, revision stamping, no duplicate ticket, redelivery no-op via `ProcessedEventService`, and ticket-detail returning all items with their `revisionNo`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Widened TicketServiceImpl switch expressions for enum exhaustiveness**
- **Found during:** Task 1
- **Issue:** Extending `TicketItemStatus` with `ACCEPTED`/`PREPARING` made the two existing Java switch *expressions* in `TicketServiceImpl` (`markItemStatus`'s next-state selection, `validateTransition`) non-exhaustive — a compile error, since switch expressions (not statements) require every enum constant to be covered.
- **Fix:** Added `ACCEPTED`/`PREPARING` cases mapping forward toward `READY`, without changing the existing `PENDING -> COOKING -> READY` default path any caller currently exercises. Also widened the "ticket moves to COOKING at the ticket level" condition to also trigger on `ACCEPTED`/`PREPARING` for forward-compatibility.
- **Files modified:** `TicketServiceImpl.java` (already in scope for Task 3's `toDto` work; pulled forward to Task 1 since Task 1's own acceptance criterion is "kitchen-service compiles").
- **Verification:** `mvn -q -o compile` green.
- **Committed in:** `e89a896`

**2. [Rule 3 - Blocking] Mechanically updated 3 existing IT files' `OrderSentToKdsPayload` constructor calls**
- **Found during:** Task 1
- **Issue:** Extending the `OrderSentToKdsPayload` record with 2 new trailing fields (`revisionNo`, `orderNotes`) changed its canonical-constructor arity from 5 to 7 args, breaking every positional-arg construction site in `TicketRoutingIT`, `TicketLifecycleIT`, and `OrderVoidedCancelsTicketsIT` (6 call sites total).
- **Fix:** Added `1, null` (Rev 1, no order notes) as trailing args to each existing construction, preserving each test's original intent.
