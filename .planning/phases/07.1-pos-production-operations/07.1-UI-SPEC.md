## Status System (POS-11) â€” the foundational cross-cutting contract

This is the single design decision every other surface in this phase depends on. It is deliberately concrete because the checker and executor must apply it identically on the cashier terminal, Order Management, Table Detail, and the KDS board.

### Line-item status (7 states)

| Status | Hue | Icon (lucide) | Label | Set by |
|--------|-----|----------------|-------|--------|
| `PENDING` | neutral/muted | `Clock` (outline) | "Pending" | Cashier adds item, not yet fired |
| `SENT` | info (blue) | `Send` | "Sent" | Automatic, the instant "Send to Kitchen"/"Send New Items" fires the line |
| `ACCEPTED` | info (blue) | `CheckCircle2` (outline) | "Accepted" | **Automatic** the instant the ticket/item renders on the KDS board â€” "seen = accepted." No manual kitchen tap for this transition (avoids a wasted tap during rush, consistent with industry-standard POS behavior per the phase's research directive). |
| `PREPARING` | info (blue) | `Flame` (filled, subtle pulse) | "Preparing" | Kitchen taps "START" |
| `READY` | success (green) | `Bell` | "Ready" | Kitchen taps "DONE" |
| `SERVED` | success (green) | `CheckCheck` | "Served" | **Cashier/server**, from POS/Order-Detail/Table-Detail â€” NOT the kitchen. Kitchen has no visibility once food leaves the pass. Action: "Mark Served" per item, or "Mark All Served" per order. |
| `CANCELLED` | destructive (red) | `Ban`, item name gets `line-through` | "Cancelled" | Cashier cancels a line from Order Detail/OrderPanel (not from KDS). If the line was already `SENT`+, it stays visible on the ticket with the cancelled treatment for a grace period rather than disappearing, so kitchen sees it was pulled ("86'd" per research directive). |

Same hue (info/blue) spans SENTâ†’ACCEPTEDâ†’PREPARING deliberately â€” the family communicates "in the kitchen pipeline"; the icon + label disambiguate the exact stage. This keeps the hue count small (4 hues + neutral) while still WCAG-compliant (never color-alone).

### Derived order status (aggregate, computed â€” never hand-set)

| Status | Hue | Icon | Label | Derivation rule |
|--------|-----|------|-------|------------------|
| `DRAFT` | neutral | `FileEdit` | "Draft" | No line has been sent (all `PENDING`) |
| `IN_PROGRESS` | info (blue) | `Flame` | "In Progress" | â‰¥1 line `SENT`/`ACCEPTED`/`PREPARING`, none/some `READY`, none `SERVED` yet |
| `PARTIALLY_SERVED` | warning (amber) | `CircleDashed` (half-fill) | "Partially Served" | â‰¥1 non-cancelled line `SERVED`, â‰¥1 non-cancelled line not yet `SERVED` |
| `SERVED` | success (green) | `CheckCheck` | "Served" | All non-cancelled lines `SERVED` |
| `CLOSED` | slate/neutral | `Lock` | "Closed" | Settlement complete (existing `OrderStatus.CLOSED`) |
| `VOIDED` | destructive (red) | `Ban` | "Voided" | Existing `OrderStatus.VOIDED` |
| `REFUNDED` | warning (amber) | `Undo2` | "Refunded" | Existing `OrderStatus.REFUNDED` â€” reuses the amber hue from `PARTIALLY_SERVED` but the two never co-occur (pre- vs. post-settlement), so no ambiguity |

`PARTIALLY_SERVED` is the one case that intentionally breaks the "hue = pipeline family" pattern â€” it needs to visually stand out as "not uniform, needs attention" even though nothing is wrong, which is exactly what amber communicates elsewhere in the system (till variance, low stock, etc.).

### Revision marker (POS-12 / KDS-03)

- Every line carries a `revisionNo` (1, 2, 3â€¦) set at fire time. **Rev 1 items get no badge** (default treatment â€” reduces visual noise for the common case). Items in Rev 2+ get a small pill badge: `"REV {n}"` for n>1, using `--accent` background / `--accent-foreground` text, positioned top-right of the item row.
- This accent-colored revision pill is a **different visual channel** from the aging border color on the KDS card (green/amber/red, ticket-level urgency, unchanged from Phase 7) and from the line-status hue above (pipeline stage) â€” the three channels must remain visually distinguishable (border color = card-level urgency; item badge hue = pipeline stage; small accent pill = "this item is a later addition"). Never conflate them into one color.
- Ticket header (KDS) and Order Detail header (POS) both show a compact revision-count chip, e.g. "3 revisions", tappable to expand a revision log: each revision listed as "Rev {n} Â· {time}" with its item count.

## Assumptions / Open Decisions (Claude's Discretion, recorded per CONTEXT.md)

1. **Order Management is a third tab, not a new route/sidebar item.** Chosen to reuse the existing permission/feature-guard wrapper and avoid IA fragmentation. If the planner finds routing reasons to prefer a dedicated `/app/pos/orders` route, the visual contract (DataTable + drawer) still applies unchanged.
