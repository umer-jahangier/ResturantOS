# Phase 07.1 Plan 03: sendToKds Revisions + Item-Status Transitions + Instructions Summary

**Repeatable, per-fire-idempotent sendToKds that fires only newly-added order lines as an incrementing revision, wires OrderStatusDerivationService into every item-status mutation path (sendToKds, markItemServed, cancelItem, item-granular ORDER_READY sync), and adds order/item instruction editing that reaches the kitchen payload.**

## Performance

- **Duration:** 45 min
- **Started:** 2026-07-11T14:05:00Z (approx.)
- **Completed:** 2026-07-11T14:50:00Z (approx.)
- **Tasks:** 3
- **Files modified:** 16 (4 created, 12 modified)

## Accomplishments
- **State machine + addItem guard loosened symmetrically** (Task 1): additive self-loop transitions (SENT_TO_KDS/PARTIAL_READY/READY → self) let `sendToKds` be called repeatedly; `addItem`'s guard widened from DRAFT/OPEN-only to any non-terminal status via a shared `isTerminal()` predicate, so items can always be added right up until settlement — never a dead end where an item is added but can't be fired.
- **`sendToKds(orderId, clientFireId)` rewritten as the single fire-only-PENDING seam** (Task 2): filters to `itemStatus==PENDING`, computes `nextRevision = max(revisionNo)+1`, stamps `SENT`/`revisionNo`/`firedAt` on the fired lines only, builds the KDS payload from those lines exclusively, and recomputes `Order.derivedStatus` via the pure derivation service — the settlement `Order.status` field is still hand-set to `SENT_TO_KDS` per-fire for event-contract compatibility, but no longer carries kitchen-progress meaning.
- **Per-fire idempotency**: wraps the fire in the existing `IdempotencyService.checkAndLock`/`markComplete` shape (the exact pattern already proven for `voidOrder`/`closeOrder`), keyed `sendToKds:{orderId}:{clientFireId}` — a NEW namespace distinct from `clientOrderId`. A replayed offline fire with the same key returns the original response without a second publish.
- **`PosEventPayloads.OrderSentToKdsPayload`** gains additive `revisionNo`/`orderNotes` fields, field-name-identical to kitchen-service's `KitchenEventPayloads.OrderSentToKdsPayload` from plan 02.
- **`markItemServed`/`cancelItem`** (Task 3): per-line transition mutations with guards (reject serving a never-fired PENDING line, reject cancelling an already-SERVED line), each recomputing `derivedStatus` via the single derivation seam.
- **`updateInstructions`**: new PATCH endpoint + service method for order-level (240 char) and per-item (140 char) notes, with server-side length enforcement independent of the MVC `@Valid` layer (mapped to 422 via a new `IllegalArgumentException` handler).
- **`OrderReadyConsumer` rewritten to item-granular sync**: on `ORDER_READY`, only lines at the readied station still in `PENDING/SENT/ACCEPTED/PREPARING` advance to `READY` — `SERVED`/`CANCELLED`/already-`READY` lines are explicitly excluded, so a redelivered or late event can never downgrade a line. `derivedStatus` recomputed afterward; no more order-level hand-set.
- **`OrderDto`** extended with per-item `revisionNo`/`firedAt` and top-level `derivedStatus` (kept the existing `kdsStatus` wire field name — see Decisions).
- Four new/extended test files: `OrderRevisionIT`, `SendToKdsIdempotencyIT`, `OrderInstructionsIT`, extended `OrderStateMachineUnitTest`; mechanically/behaviorally updated the pre-existing `OrderLifecycleIT` (a second `sendToKds` is now legal — see Deviations).

## Decisions Made
- Kept `OrderDto.OrderItemDto.kdsStatus` field name (added `revisionNo`/`firedAt` alongside it) rather than renaming to `itemStatus` — mirrors plan 07.1-01's identical decision for the same field, avoiding a second JSON-contract break this cycle; the rename is still deferred per PATTERNS.md.
- `Order.status` (settlement `OrderStatus` enum) keeps its DRAFT/OPEN/SENT_TO_KDS/VOIDED/CLOSED hand-sets exactly as before — the plan's Task 2 explicitly instructs keeping the state-machine transition "for event-contract compatibility." `Order.derivedStatus` is the sole kitchen-progress aggregate and is verifiably never hand-set (see coverage D4's grep).
- `OrderRevisionIT`/`SendToKdsIdempotencyIT` assert item-level `revisionNo`/`itemStatus`/`firedAt` via direct `OrderRepository` entity reads rather than `OrderDto`, since Task 2 (which wrote these tests) precedes Task 3 (which extends `OrderDto` with those fields) — avoids a forward reference across tasks without weakening the assertions.
- `markItemServed`/`cancelItem` treat re-invocation in the same terminal state (serve-an-already-SERVED-item, cancel-an-already-CANCELLED-item) as an idempotent no-op rather than an error — only the two illegal cross-transitions the plan specifies are rejected.
- `updateInstructions` added a terminal-order guard (`isTerminal()`, same predicate as `addItem`) — not explicitly requested by the plan text but a reasonable data-integrity addition (Rule 2) consistent with the rest of the mutation surface.
- Char-limit validation (240/140) is enforced explicitly in the service layer (`IllegalArgumentException` -> new 422 handler), not solely via the DTO's `@Size` annotation, because `OrderInstructionsIT` calls the service directly (bypassing the MVC `@Valid` pipeline) — this matches the codebase's existing direct-service-call IT convention (`VoidRefundOpaIT`).
- `OrderReadyConsumer.markOrderReady` visibility widened from package-private to `public` solely so `OrderInstructionsIT` can simulate an `ORDER_READY` event directly (a full RabbitMQ round-trip is unavailable in this session's environment regardless of test approach) — no behavioral change.
