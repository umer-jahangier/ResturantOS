<objective>
Implement the pos-service side of order revisions (POS-12), the item-status transition methods + derivation wiring (POS-11), and instruction editing (POS-13). This is the plan that makes `sendToKds` callable repeatedly and fire-only-new-items, loosens the state machine + addItem guards symmetrically, adds per-fire idempotency via a new `clientFireId`, wires `OrderStatusDerivationService` into every item-status mutation path (removing every hand-set of the kitchen-progress status range per RESEARCH.md Pitfall 3), and adds mark-served / cancel-item / edit-instructions.

Every "hard" problem here already has a proven primitive in this codebase (RESEARCH.md "Don't Hand-Roll"): idempotency = `IdempotencyService.checkAndLock/markComplete/getCompletedResponse` (as in voidOrder/closeOrder), derivation = the plan-01 pure service, redelivery dedup = existing consumer layer. The work is wiring these correctly, not inventing infrastructure.

Purpose: POS-12 (revisions) + POS-11 (derived status live) + POS-13 (instruction edit) — the operational heart of the phase.
Output: Loosened state machine, rewritten sendToKds, new item-status + instruction methods/endpoints, additive payload fields, and three integration tests + a state-machine unit test.
</objective>
