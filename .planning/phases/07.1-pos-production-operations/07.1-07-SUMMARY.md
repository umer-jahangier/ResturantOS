# Phase 07.1 Plan 07: KDS Board Redesign — Stable Sort, Per-Item Status, Ticket Detail Summary

**Fixed the KDS "cards bounce" UAT complaint with a deterministic once-per-batch sort key, replaced the 3-value item-status dict with the shared 7-state StatusBadge + revision pill, and added a tap-to-open ticket-detail view with revisions grouped by Rev/time and an order-level Kitchen Notes callout — always-dark board and aging-border urgency channel both preserved exactly.**

## Accomplishments

- **Task 1 (KDS-03/POS-13) — 7-state per-item status + revision pill:** Replaced `kds-ticket-card.tsx`'s hardcoded 3-value `statusColors` dict with the shared `StatusBadge` (icon+label+semantic-token), via a `toLineItemStatusVariant()` seam that normalizes kitchen-service's legacy `COOKING` value to `PREPARING`. Composed the shared `RevisionBadge` for the Rev-n pill instead of redefining revision logic. Preserved the un-recolored `text-2xl` item name (too noisy at 2m to recolor per-status), the aging-border urgency channel unchanged, and restricted bump buttons to the two kitchen-actionable transitions (START for PENDING, DONE for COOKING/PREPARING) — no button for ACCEPTED (automatic) or READY.
- **Task 2 (KDS-03) — deterministic stable board sort + fully-resolved auto-remove:** Extracted and exported `sortKdsTickets()` from `kds-board.tsx` — a pure function sorting by `receivedAt` ascending with `ticket.id` tiebreak, computed once per batch via `useMemo` keyed on the tickets query's array reference (never on unrelated re-renders). Because the sort key only reads immutable `receivedAt`/`id`, a card's position provably cannot change when only its items' statuses update. Extended the 60s auto-remove rule in `kds-ticket-card.tsx` from ticket-level (`ticket.status === "READY"`) to fully-resolved-item-level (`ticket.items.every(i => i.status === "READY")`). New-ticket fade-in uses the existing `animate-fade-in` utility applied unconditionally, relying on React's keyed-mount semantics rather than extra tracking state (see Deviations — the first two implementation attempts violated this repo's strict `react-hooks` lint rules).
- **Task 3 (POS-13) — ticket-detail view:** New `kds-ticket-detail.tsx` consumes the plan-05 `useKdsTicketDetail` hook, groups items by `revisionNo` into "Rev {n} · {time}" sections (each item showing its `StatusBadge` + italic notes), and renders the order-level "Kitchen Notes" callout (`bg-warning/10`, `MessageSquare` icon) when `orderNotes` is present. Wired via a shadcn `Dialog` triggered by tapping the ticket card header; `DialogContent` explicitly forces `dark` + dark-token classes since Radix's Portal renders to `document.body`, outside the board's `.dark`-scoped container.

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Fully-resolved-item auto-remove implemented in kds-ticket-card.tsx, not kds-board.tsx**

- **Verification:** `tsc --noEmit` clean; existing `kds-board.test.tsx` still passes unchanged.
- **Committed in:** `94aab83` (Task 2 commit).

**2. [Rule 1 - Bug] New-ticket fade-in redesigned twice to satisfy this repo's react-hooks lint rules**
- **Found during:** Task 2 (implementing the UI-SPEC's "new tickets insert at their sorted position, fade-in 200ms" requirement)
- **Issue:** The first implementation (a `useRef<Set<string>>` of seen ticket ids, read inside a `useMemo` to compute "new" ids) failed `eslint`'s `react-hooks/refs` rule ("Cannot access refs during render" — `useMemo` runs during the render phase). The second attempt (moved the seen-id set into `useState`, updated via `setState` inside a `useEffect`) failed `react-hooks/set-state-in-effect` ("Calling setState synchronously within an effect can trigger cascading renders").
- **Fix:** Dropped all new-ticket-tracking state entirely. Apply `animate-fade-in` unconditionally to every ticket card's root div. Because each card is keyed by `ticket.id`, React's reconciliation only creates a fresh DOM node — and therefore only replays the CSS mount-time animation — for a genuinely new ticket; an existing ticket that re-renders (e.g. an item-status update) reuses its DOM node and the already-applied class does not replay the animation. This achieves the spec'd behavior with zero extra hooks.
- **Files modified:** `kds-board.tsx`, `kds-ticket-card.tsx`.
- **Verification:** `npx eslint components/kds --max-warnings=0` clean; `npx tsc --noEmit` clean; existing + new vitest suites pass.
- **Committed in:** `9952b4f` (Task 3 commit — the fade-in simplification was folded into the same commit as the ticket-detail wiring since both touched the same files in the same working session).

**3. [Rule 3 - Blocking] Test fixtures needed valid-UUID-format ids for zod-schema-validated MSW responses**
- **Found during:** Task 3 (writing `kds-ticket-card.test.tsx`'s open-detail test)
- **Issue:** The existing `kds-board.test.tsx` fixture convention uses ids like `"t0000001-0000-4000-8000-000000000001"` (leading `"t"` — not valid hex) because those objects are passed directly as component props and never zod-validated. The new ticket-detail test's MSW response payload IS parsed through `apiKdsTicketSchema` (`z.string().uuid()` on `id`/`orderId`/item `id`/`orderItemId`), so non-hex-leading ids threw `ZodError: Invalid UUID`, which the component's `isLoading || !ticket` guard silently rendered as a perpetual "Loading ticket…" (masking the real error).
- **Fix:** Used valid hex-leading UUIDs (`"d..."`) for every id in the MSW response fixtures.
- **Files modified:** `frontend/__tests__/kds/kds-ticket-card.test.tsx`.
- **Verification:** `npx vitest run __tests__/kds/kds-ticket-card.test.tsx` — 6/6 pass.
- **Committed in:** `9952b4f` (Task 3 commit).
