# Phase 07.1 Plan 01: Item-Status + Derived-Status Data Model Summary

**7-value OrderItemStatus lifecycle + pure, unit-tested OrderStatusDerivationService replacing the shallow 3-value KdsItemStatus, laying the POS-11 foundation for revisions, order management, and table-centric flow.**

## Accomplishments
- Replaced the 3-value `KdsItemStatus` (PENDING/COOKING/READY) with the full 7-value `OrderItemStatus` lifecycle (PENDING/SENT/ACCEPTED/PREPARING/READY/SERVED/CANCELLED) as `OrderItem`'s per-line status type, keeping the existing `kds_status` DB column mapping.
- Added `DerivedOrderStatus` (DRAFT/IN_PROGRESS/PARTIALLY_SERVED/SERVED) as a new aggregate enum, persisted on `Order.derivedStatus` (default DRAFT), explicitly separate from the settlement `OrderStatus` lifecycle (CLOSED/VOIDED/REFUNDED stay untouched).
- Added `TableStatus.NEEDS_BUSSING` and `OrderItem.revisionNo`/`OrderItem.firedAt` fields (0/null = not yet fired) in preparation for the revision-fire work in plan 03.
- Created the pure, side-effect-free `OrderStatusDerivationService.derive(List<OrderItem>)` — the single seam allowed to compute derived order status, proven by a 14-test unit suite covering every boundary case plus a 500-trial randomized property-based backstop.
- Additive Flyway `V4__item_status_revision.sql` (no DROP/RENAME) adding `order_items.revision_no`, `order_items.fired_at`, `orders.derived_status`, and a `(order_id, revision_no)` index.
- Mechanically reconciled the last remaining `KdsItemStatus`/`getKdsStatus()` references in `OrderServiceImpl` and `OrderDto` to `OrderItemStatus`/`getItemStatus()` — behavior-preserving, no derivation wiring yet (that's plan 03).
