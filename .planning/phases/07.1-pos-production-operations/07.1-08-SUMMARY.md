# Phase 07.1 Plan 08: POS Terminal — Menu Search, Item-Cap Investigation/Fix, Table Binding Summary

**Debounced client-side menu search; a real item-cap root-cause investigation that found the "cap" is a rapid-tap order-creation race (not a numeric limit), fixed by ref-based single-flight order creation + a mutate-time `useAddItem` redesign; and `tableId` now threads from the page's selected table into `createOrder`.**

## Accomplishments

- **Menu search (Task 1, POS-15):** `menu-grid.tsx` gained a full-width 44px search input above the category pills, 150ms debounced (reusing the existing `useDebouncedValue` hook — no new debounce implementation), clearable via an X button, `Search` (lucide) icon left-aligned. Filters the currently-loaded category's `activeItems` client-side, case-insensitive substring match. Distinct "No items match your search" empty state added alongside the pre-existing "No items available" state. No list virtualization added (UI-SPEC defers it past ~60 items; current seed is ~10).
- **Item-cap investigation + fix (Task 2, POS-15):** GitNexus MCP was unavailable this session; investigated via Grep/Read across the entire add-item surface (frontend: `pos-terminal.tsx`, `use-orders.ts`, `order-panel.tsx`, `pos.repository.ts`, `pos.schema.ts`, `outbox.ts`/`db.ts`, `query-provider.tsx` defaults; backend: `OrderServiceImpl.addItem`, `OrderMapper.toDto`) and found **no hardcoded cap anywhere**. The real bug is a stale-closure race: rapid taps landing inside the same async window as an in-flight `createOrder` call each independently see a null `activeOrderId` and create a duplicate order, orphaning earlier items. Fixed via a ref-based `ensureOrderId` that dedupes concurrent order-creation onto one in-flight promise, plus a `useAddItem` redesign (`orderId` moved from a hook argument to a mutate-time variable) that removes the stale-binding hazard the original "first item" workaround existed to route around. `tableId` now threads through: `PosTerminal` accepts a `tableId` prop, included in `createOrder`'s payload; `page.tsx` passes its existing `selectedTableId` state through (the single authorized pass-through line per this plan's file-scope note).
- **Side effect — layer-boundary fix:** the pre-existing (07-UAT bug #4 era) direct `PosRepository`/`enqueue` import in `pos-terminal.tsx` was a components->hooks-only ESLint boundary violation. The `useAddItem` redesign removed the need for that direct import entirely — `pos-terminal.tsx` now goes exclusively through Layer-3 hooks, and `npx eslint components/pos/menu-grid.tsx components/pos/pos-terminal.tsx lib/hooks/pos/use-orders.ts --max-warnings=0` is clean.
