<objective>
Extend the frontend four-layer data pipeline (schema → model → adapter → repository → hook) and build the two shared render primitives (`StatusBadge` icon-per-status extension, `revision-chip.tsx`) that every POS/KDS surface in this phase depends on. This is the single seam that carries the new backend contract (7-value item status, revisionNo/firedAt, derived order status, table detail, order summary, instructions, KDS ticket detail) into the UI, and the single place the status-color/icon/label system from the UI-SPEC Status System is encoded — so plans 06-10 only compose these, never redefine them.

The four-layer boundary is mandatory (ESLint-enforced): components never import `lib/api-client`/`lib/repositories`; every response is `.parse()`d before adaptation. `MoneyDisplay` renders all paisa. This plan touches only lib/* and the two shared primitives — zero page/terminal/board surface files (those are plans 06-10), so it has no file conflict with them.

Purpose: shared data + status-render foundation for POS-09/10/11/12/13 and KDS-03 UI.
Output: extended schemas/models/adapters/repositories/hooks + StatusBadge + revision-chip + their vitest coverage.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| API response → frontend | untrusted JSON parsed by Zod before adaptation |
| free-text notes → React render | instruction/notes strings rendered in JSX |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-07.1e-01 | Tampering | Zod parse of extended envelope | medium | mitigate | Every response `.parse()`d (throwing variant) before adaptation; malformed/oversized fields rejected at the boundary; four-layer ESLint rule keeps parsing centralized |
| T-07.1e-02 | Tampering (XSS) | instruction/notes render | medium | mitigate | Notes rendered as plain text via JSX auto-escaping; no dangerouslySetInnerHTML introduced; server also enforces the char limits (plan 03) |
| T-07.1e-03 | Tampering / Repudiation | clientFireId idempotency header | high | mitigate | clientFireId = crypto.randomUUID() generated per Send action, sent as Idempotency-Key; the server (plan 03) scopes it via IdempotencyService — no double-fire, not replayable across sessions |
| T-07.1e-SC | Tampering | npm installs | high | mitigate | No new packages (RESEARCH.md audit: none; @tanstack/react-table, framer-motion, lucide-react already installed) |
</threat_model>
