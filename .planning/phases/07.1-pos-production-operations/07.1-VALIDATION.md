## Critical Invariants To Cover (from RESEARCH ## Validation Architecture)

These are the highest-value automated checks for this phase ‚Äî the planner MUST route tasks to them:

1. **Derived order status** ‚Äî given a set of line statuses, the computed order status is correct across
   the boundary cases: no line SENT ‚Üí `DRAFT`; ‚â•1 SENT, none READY ‚Üí `IN_PROGRESS`; some SERVED + some not
   ‚Üí `PARTIALLY_SERVED`; all non-cancelled SERVED ‚Üí `SERVED`. Pure unit test (no DB). **Non-inferable
   backstop:** property-based over random line-status vectors.
2. **Revision fires only PENDING items** ‚Äî after Rev 1 (Burger/Fries/Coke) is SENT, adding Cheesecake and
   firing again sends ONLY Cheesecake; prior lines are never re-fired and keep their status. Integration
   test through `OrderServiceImpl.sendToKds` asserting the outbox `ORDER_SENT_TO_KDS`/ticket payload
   contains only the new line. **Backstop:** kitchen-service `TicketRoutingService` appends to the existing
   `(orderId, stationCode)` ticket rather than skipping or duplicating.
3. **State-machine allows re-fire** ‚Äî `SENT_TO_KDS ‚Üí SENT_TO_KDS` transition and `addItem()` on a SENT
   order are permitted (the two guards RESEARCH flagged as currently blocking POS-12).
4. **Event-contract additivity** ‚Äî `ORDER_CLOSED`, `ORDER_SENT_TO_KDS`, `ORDER_READY`, `TILL_*` payloads
   remain backward-compatible (new fields optional/defaulted); existing pos/kitchen IT suites stay green.
5. **Void succeeds through the UI path** ‚Äî once the void-403 root cause (JWT staleness per RESEARCH) is
   fixed, `VoidRefundOpaIT` + a browser/E2E void pass.
