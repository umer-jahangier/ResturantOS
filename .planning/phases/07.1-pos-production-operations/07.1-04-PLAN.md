<objective>
Deliver the pos-service backend for the Order Management screen (POS-09), table-centric dine-in (POS-10), and close the Phase-7 void 403 (POS-14 backend half). This provides: a `listOrders` default that never filters out non-terminal orders + an `OrderSummaryDto` for the list, permission-gated own-vs-all-branch scoping (a hard tenant/branch isolation requirement), a table→active-order lookup endpoint + `TableDetailDto` with a live bill summary, `createOrder` table binding + table-status lifecycle derivation, and the void-403 diagnosis/fix per RESEARCH.md's diagnostic recipe.

RESEARCH.md establishes the void-403 is most likely JWT staleness (the "different permission string in the error message" is a confirmed red herring), but the executor MUST run the full diagnostic recipe (fresh-JWT permission check → branch match → OPA bundle) via an IT before concluding, and fix the genuine root cause if it is a code bug (branch/created_by/OPA-input mismatch).

Purpose: POS-09 + POS-10 need list + table-detail backends; POS-14 needs the void to actually work server-side.
Output: New DTOs + repository method + table lifecycle + list scoping + void fix, with two new integration tests.
</objective>
