## Summary

Phase 7 shipped a real order/kitchen pipeline, but three structural facts block Phase 7.1 from being a
pure "add a field" exercise:

1. **`sendToKds` cannot be called twice, at all, today.** `OrderStateMachine`'s transition table has no
   `SENT_TO_KDS → SENT_TO_KDS` entry, so a second `sendToKds()` call on an already-sent order throws a 409
   `StateInvalidException` before any item-filtering logic even runs. `OrderServiceImpl.addItem()` also
   explicitly rejects adding items to any order not in `DRAFT`/`OPEN`. **Revisions are blocked by the state
   machine, not just missing "which items to send" logic** — this is the first fact the plan must address.
2. **`TicketRoutingService.route()` is idempotent-by-skip, not idempotent-by-merge.** If a ticket already
   exists for `(orderId, stationCode)`, a second `ORDER_SENT_TO_KDS` event is silently dropped in full —
   it does not append the new items. This was written as a dedup guard against duplicate event delivery,
   but it also happens to be the exact mechanism that would need to change for revisions: the fix is to
   make it **append new items to the existing ticket** (keyed by a new item-level `revisionNo`) rather than
   skip, using the payload's own item list as the source of truth for what's new (every item in a
   revision's payload is new by construction, once `sendToKds` is fixed to only include `PENDING` items).
3. **Order-level `OrderStatus` and item-level `kdsStatus`/`TicketItemStatus` are three separate, currently
   unsynchronized state machines** (pos-service order enum, pos-service item enum, kitchen-service ticket
   item enum). POS-11's "derive order status from items" requirement, done correctly, collapses these
   into one authoritative item-status timeline with two read-projections (pos-service's copy, updated via
   the existing `ORDER_READY` consumer path, and a derived aggregate on `Order`).

Industry research (Toast, Square, Oracle Simphony/NCR Aloha docs, and the South-Asian "KOT" ticketing
convention — directly relevant given this product's Pakistan-first market) confirms the shape the PRD and
UI-SPEC already assumed: **fire only unfired/PENDING items, never re-fire sent items; coursing and ad-hoc
add-ons are the same underlying mechanism with only the UI trigger differing; "served" is never
kitchen-tracked in any researched vendor; and the modern (post-2020) KDS convention shows revisions as
in-place item highlights on ONE ticket card, not as separate ticket cards per fire** — which matches the
already-approved UI-SPEC exactly and requires no UI-SPEC rework.

**Primary recommendation:** Keep **one `KdsTicket` per `(orderId, stationCode)`** (do not create a new
ticket per revision). Add `revisionNo` (int) + `firedAt` (Instant) to `KdsTicketItem` (not `KdsTicket`).
Change `TicketRoutingService.route()` from skip-if-exists to append-items-to-existing-ticket, reopening the
ticket's status if it had reached `READY`/terminal. Change `OrderServiceImpl.sendToKds()` to (a) allow being
called from any non-terminal order state, (b) build the KDS payload from **only `PENDING` items**, (c) stamp
those items `revisionNo = currentMaxRevisionOnOrder + 1`, `firedAt = now`, and flip them to `SENT`. Loosen
`OrderStateMachine`/`addItem` guards accordingly. This is the option that best matches the industry evidence
(fire-only-new-items, one visual ticket per station) while requiring the fewest structural changes to the
already-correct `(orderId, stationCode)` idempotency key kitchen-service already uses.

### Pattern 1: Fire-only-unfired-items (industry-standard, POS-12 core)

**What:** A "send"/"fire" action operates on the CURRENT set of not-yet-sent items only. It is never
parameterized by "which items" from the UI — it is always "all items currently `PENDING`." This mirrors
Toast's Server Item Firing model where every item has an implicit Send/Hold state, and firing acts on the
unfired subset.
**When to use:** Every `sendToKds` call, whether it's the very first fire (Rev 1) or the Nth (Rev N).
**Example:**
```java
// Source: derived from Toast/Square/coursing research — no proprietary code available;
// this is an original implementation following the confirmed industry semantics.
@Override
public OrderDto sendToKds(UUID orderId) {
    UUID tenantId = tenantContext.requireTenantId();
    Order order = findOrderForTenant(orderId, tenantId);

    // Loosened guard: allow from any state where items can still be added
    // (i.e. not CLOSED/VOIDED/REFUNDED) rather than requiring status==OPEN.
    List<OrderItem> newItems = order.getItems().stream()
            .filter(i -> i.getItemStatus() == OrderItemStatus.PENDING)
            .toList();

    if (newItems.isEmpty()) {
        throw new PosExceptions.ZeroValueOrderException("Nothing new to send to kitchen");
    }

    int nextRevision = order.getItems().stream()
            .mapToInt(OrderItem::getRevisionNo).max().orElse(0) + 1;

    Instant firedAt = Instant.now();
    for (OrderItem item : newItems) {
        item.setItemStatus(OrderItemStatus.SENT);
        item.setRevisionNo(nextRevision);
        item.setFiredAt(firedAt);
    }

    // Build KDS payload from ONLY newItems — never order.getItems() wholesale.
    var kdsItems = newItems.stream().map(this::toKdsItemPayload).toList();
    // ... publish ORDER_SENT_TO_KDS with kdsItems, revisionNo=nextRevision
}
```

### Pattern 2: Append-to-existing-ticket routing (KDS-03 / POS-12)

**What:** `TicketRoutingService` must distinguish "duplicate event redelivery" (same `eventId` already
processed — handled by `processed_events`/`ProcessedEventService` at the consumer layer, NOT by ticket
existence) from "a new revision for an order that already has a ticket at this station" (a legitimate
second call with NEW items). The current code conflates these by using ticket-existence as its dedup
signal, which is wrong once revisions exist.
**When to use:** Every `route()` call.
**Example:**
```java
// Source: original implementation, extending the existing TicketRoutingService pattern
Optional<KdsTicket> existing = ticketRepository.findByOrderIdAndStationCode(orderId, stationCode);
if (existing.isPresent()) {
    KdsTicket ticket = existing.get();
    List<KdsTicketItem> appended = buildItems(stationItems, ticket);
    ticket.getItems().addAll(appended);
    // Reopen: new PENDING items mean the ticket is no longer fully done.
    if (ticket.getStatus() == TicketStatus.READY) {
        ticket.setStatus(TicketStatus.PENDING);
        ticket.setReadyAt(null);
    }
    ticketRepository.save(ticket);
    webSocketHandler.notifySubscribers(ticket.getBranchId(), ticket.getStationCode(), toDto(ticket));
} else {
    // unchanged: create new ticket (Rev 1 path)
}
```
Note: true event-redelivery dedup (the actual "already processed this exact `eventId`") is already handled
one layer up by `ProcessedEventService.tryProcess(CONSUMER_NAME, envelope.eventId(), ...)` in
`OrderSentToKdsConsumer` — so removing the ticket-existence skip does NOT reintroduce a duplicate-processing
bug; it only removes an *incorrect* dedup signal that was blocking legitimate revisions.

### Pattern 3: Derived status as a pure function, computed at every item-status write

**What:** A single function `deriveOrderStatus(List<OrderItem>) -> DerivedStatus` that never has
side effects and is called from every mutation path that touches item status (send-to-kds, kitchen-ready
consumer, mark-served, cancel-item). Never let two code paths independently decide the order's aggregate
status.
**When to use:** `sendToKds`, the `ORDER_READY`-triggered per-item sync, `markServed`, `cancelItem`.
**Example:**
```java
// Source: original — this is the core POS-11 unit-testable seam
public DerivedOrderStatus derive(List<OrderItem> items) {
    List<OrderItem> active = items.stream()
            .filter(i -> i.getItemStatus() != OrderItemStatus.CANCELLED)
            .toList();
    if (active.isEmpty() || active.stream().allMatch(i -> i.getItemStatus() == OrderItemStatus.PENDING)) {
        return DerivedOrderStatus.DRAFT;
    }
    if (active.stream().allMatch(i -> i.getItemStatus() == OrderItemStatus.SERVED)) {
        return DerivedOrderStatus.SERVED;
    }
    boolean anyServed = active.stream().anyMatch(i -> i.getItemStatus() == OrderItemStatus.SERVED);
    return anyServed ? DerivedOrderStatus.PARTIALLY_SERVED : DerivedOrderStatus.IN_PROGRESS;
}
```

### Anti-Patterns to Avoid
- **Re-firing the whole order on every `sendToKds` call:** the exact bug present in today's code
  (`order.getItems().stream()` with no status filter in `sendToKds`). Never build the KDS payload from
  the full item list once revisions exist.
- **Using ticket existence as event-redelivery dedup:** conflates "duplicate delivery of the same event"
  with "legitimate second fire" — these must be two different mechanisms (`processed_events` for the
  former, item-status filtering for the latter).
- **Creating a new `KdsTicket` per revision:** fragments the KDS board contrary to both the modern
  cross-vendor convention (single ticket, in-place highlighting) and the already-approved UI-SPEC.
- **Setting derived order status by hand anywhere:** any `order.setStatus(...)` call outside the
  derivation function for the kitchen-progress states (IN_PROGRESS/PARTIALLY_SERVED/SERVED) is a bug by
  definition per POS-11's "computed, never hand-set" requirement.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Revision-fire idempotency | A bespoke dedup table for "fire batches" | `IdempotencyService` (`checkAndLock`/`markComplete`/`getCompletedResponse`) already proven for order-create/close/void | Same proven pattern (checkAndLock + TTL), just a new key namespace (`sendToKds:{orderId}:{clientFireId}`) — don't reinvent |
| Event-redelivery dedup for the revised `ORDER_SENT_TO_KDS` payload | Ticket-existence checks (current bug) | `ProcessedEventService.tryProcess(CONSUMER_NAME, eventId, ...)` — already wraps `TicketRoutingService.route()` | It's already there; the bug is an *extra*, wrong, second dedup signal layered on top, not a missing one |
| Derived-status computation duplicated in frontend | Recomputing DRAFT/IN_PROGRESS/PARTIALLY_SERVED/SERVED client-side from raw item statuses | Server computes and returns it on `OrderDto`; frontend only renders `StatusBadge` from the server value | Avoids drift between two independent implementations of the same derivation logic (a classic dual-source-of-truth bug) |
| Table→order lookup | A new denormalized "current order per table" cache | A single indexed query `findByTableIdAndStatusNotIn(tableId, {CLOSED,VOIDED,REFUNDED})` on the existing `orders` table (at most one non-terminal order per table, enforced by table status guard) | The existing schema already supports this; no new storage needed |

**Key insight:** Every "hard" problem in this phase (idempotency, event dedup, status derivation) already
has a proven primitive or an existing table in this codebase. The actual work is *wiring these correctly*
(fixing two conflated dedup mechanisms, moving status derivation into one seam), not inventing new
infrastructure.

### 1. Add-to-order / "fire" semantics

**Toast** [CITED: support.toasttab.com, central.toasttab.com — via search snippets, full page blocked by
bot-protection]: The "Stay" button sends only the currently-selected/highlighted items to the kitchen while
keeping the server on the order screen; any unselected items remain held. The opt-in "Server Item Firing"
feature generalizes this into a per-item Send/Hold state model — every item defaults to held until the
server explicitly fires it, and firing acts on the currently-unfired subset only, never re-sending
already-fired items.

**Square for Restaurants** [CITED: squareup.com/help — via search snippets]: Items already sent to the
kitchen cannot be edited (must void + re-enter to correct a mistake). To add more items to an in-progress
check, staff add new lines (or use the "Repeat"/duplicate function for a repeated item), and the system
generates a **separate add-on ticket** at the kitchen printer containing only the new items.

**Coursing vs. ad-hoc — CONFIRMED SAME MECHANISM** [CITED: multiple KDS vendor doc aggregations, `HIGH`
confidence — this directly resolves the CONTEXT.md/PRD "deferred unless research shows it's the same
mechanism" question]: "When a server fires Course 2 from a handheld at table 8, the KDS should receive
only those items — not re-show everything already plated." Coursing's "fire this course" button and an
ad-hoc "select new items → Send" action both reduce to the identical underlying operation: **send the
currently-unfired subset**. They differ only in *which UI element* triggers the fire (a course button vs.
an ad-hoc item-selection), not in the underlying data operation.

**RECOMMENDATION:** Do not build a separate coursing UI this phase (correctly deferred per CONTEXT.md) —
the "Send New Items (n)" button the UI-SPEC already specifies for POS-12 **is** the coursing mechanism,
just without formal course names. If coursing is ever added later, it slots in as a filter on top of the
same fire-unfired-items primitive with zero backend rework required. **Confidence: HIGH.**

### 2. Kitchen ticket revisions — new ticket vs. amended ticket, and numbering

Two coexisting real-world patterns were found:

- **South-Asian "KOT" convention** [CITED: gofrugal.com, ivepos.com, petpooja.com blog — `MEDIUM`
  confidence, directly relevant given this product's Pakistan-first market]: the first fire produces a
  "Fresh KOT" (a numbered ticket **document**); every subsequent fire produces a "Repeat KOT"/"Running
  KOT" — a **new, separately-numbered ticket**, still linked to the same table/order, with one final bill
  aggregating all KOTs for that table.
- **Modern (post-2020) Western KDS convention** [CITED: multiple KDS vendor blogs, `MEDIUM` confidence]:
  the KDS shows **one card per order** (or per order-station), and newly-fired items are shown via
  **in-place visual differentiation** — "new items get a green label, voided items are highlighted in
  red," with a flash/pulse animation on the card when new items arrive — rather than spawning a second
  visual ticket.

**RECOMMENDATION (decision-ready, locks POS-12/KDS-03 backend design):** Model revisions as **item-level
metadata on ONE ticket per (order, station)** — i.e., adopt the modern-KDS *display* convention (matches
UI-SPEC exactly: single ticket card, `REV n` item badges, expandable revision-log chip) while adopting the
KOT convention's *data* semantics one level down (every fire is a distinct, numbered, timestamped batch —
just tracked as `revisionNo`+`firedAt` on `KdsTicketItem` rows appended to the existing ticket, not as a
new `KdsTicket` row). This requires the **smallest structural change** to the current codebase: extend
`TicketRoutingService.route()` from "skip if ticket exists" to "append items to existing ticket, reopening
its status if needed" (see Pattern 2 above), rather than adding a new entity or foreign-key relationship.
Revision numbering: **`revisionNo` starts at 1** for the first fire and increments per subsequent
`sendToKds` call **for that order** (not per station — so Rev 2 at the Grill station and Rev 2 at the
Drinks station both correspond to the same "the server added a coffee" moment, even if Drinks has no new
items yet and Grill does — the UI-SPEC's example "Rev 1 = Burger/Fries/Coke; Rev 2 = Cheesecake; Rev 3 =
Coffee" implies order-wide, not per-station, revision counters). **Confidence: HIGH** on the recommendation
(reconciles UI-SPEC + minimal-change + industry data-semantics); **MEDIUM** on whether real vendors track
revision numbers per-order vs. per-station internally (undocumented, proprietary).

### 3. Item-level status lifecycle & derived order status

**Cross-vendor KDS lifecycle** [CITED: Toast platform guide, Oracle Simphony KDS docs, NCR Aloha Cloud
docs — `HIGH` confidence, consistent across all three independently-built systems]:
`Sent` (arrives at KDS) → `Fired` (released to prep display) → `Started`/`In Progress` (cook
acknowledges/begins) → `Bumped` (cook taps done) → `Ready` (expo flashes ready for pickup).

Mapping onto this codebase's target lifecycle:

| Target status | Industry equivalent | Set by |
|---|---|---|
| `PENDING` | (pre-fire; item entered but held) | Cashier adds item, not yet fired |
| `SENT` | "Sent"/"Fired" | Automatic, the instant `sendToKds` fires the line |
| `ACCEPTED` | Implicit in all three researched vendors — none document a manual "acknowledge" tap between Sent/Fired and Started; the ticket simply **rendering** on the KDS is the closest documented equivalent | **Automatic** on ticket/item render — confirms the UI-SPEC's own decision #2 (no manual kitchen tap) |
| `PREPARING` | "Started"/"In Progress" | Kitchen taps START |
| `READY` | "Bumped" | Kitchen taps DONE/bump |
| `SERVED` | **Not tracked by any researched KDS.** Every vendor's KDS loses visibility once an item is bumped/expo'd; delivery-to-guest is exclusively front-of-house | Cashier/server, POS-side — confirms UI-SPEC decision #3 |
| `CANCELLED` | "Voided" — shown in red, remains visible rather than disappearing | Cashier, from POS-side |

This gives **HIGH confidence, cross-vendor validation** for the two UI-SPEC assumptions flagged as
"Claude's Discretion, recorded per CONTEXT.md" (`ACCEPTED` automatic; `SERVED` POS-side-only) — they are
not just internally consistent design choices, they match how every major KDS vendor researched actually
behaves.

**Derived order status:** Oracle Simphony/NCR Aloha model orders with "suborders" (per fire-round/course
groupings), and documented behavior gates order-level operations on suborder state ("if any suborder has
been bumped, you cannot unclaim the order") [CITED: docs.oracle.com, docs.ncrvoyix.com — `MEDIUM`
confidence]. This is architecturally the same shape as this phase's derived-order-status requirement:
order-level state is a **read-only rollup**, never an independently-settable field. **Confidence: HIGH**
that "derived, never hand-set" is the correct, industry-validated design (this was already locked in
CONTEXT.md; research corroborates rather than changes it).

### 4. Concurrency / idempotency of firing revisions

**No vendor publishes exact internal idempotency mechanics** for a "fire additional items while offline"
operation [ASSUMED — no authoritative source found across Toast/Square/Simphony/Aloha public docs; this is
proprietary implementation detail, not a documented best practice]. Generic offline-POS marketing material
confirms the *pattern* (local queue, unique reference id per transaction, sync-and-flag-conflicts on
reconnect) but not the specifics.

**RECOMMENDATION (ASSUMED, needs no external validation — grounded entirely in this codebase's own proven
pattern, ARCHITECTURALLY SOUND either way):** Reuse the exact same primitive already proven for order
creation (`client_order_id` as `Idempotency-Key`), but with a **new, per-fire key** — do not reuse
`clientOrderId` itself (it's one-per-order; a revision-fire needs one-per-fire-action). Client generates a
`clientFireId` (UUID v4) at the moment the cashier taps "Send New Items"; this is sent as the
`Idempotency-Key` header on the `sendToKds` (or a renamed `POST /orders/{id}/fire`) call and enqueued
identically to `APPEND_ITEMS` in the existing IndexedDB outbox pattern. Server-side: `IdempotencyService`
already provides `checkAndLock`/`markComplete`/`getCompletedResponse` — apply it to `sendToKds` exactly as
already done for `voidOrder`/`closeOrder`. A replayed offline fire after reconnect returns the original
response without re-publishing `ORDER_SENT_TO_KDS` a second time (no double-fire). **Confidence: HIGH**
that this is architecturally correct (it's the same pattern the codebase already proves at 07-01/07-02);
**flagging the specific key name/shape as a planner decision, not industry-mandated**.

## POS-14 — Void 403 Root-Cause Hypothesis (diagnostic, not fully solved)

Direct code read of the full authorization chain:

- `OrderServiceImpl.voidOrder` → `PosAuthorizationService.authorizeVoid(...)` → calls
  `authorizationService.authorize("pos", "void", resource)`.
- `AuthorizationService.authorize` builds `OpaInput` from **`SecurityContextHolder`'s current JWT
  claims** (`claims.permissions()`), evaluates OPA, and on deny throws
  `PermissionDeniedException("Not permitted: " + module + "." + action)` — i.e. **the error message
  "Not permitted: pos.void" is just `module+"."+action` string concatenation, not a signal about which
  rego rule or permission code actually failed.** The UAT's observation that "the error message references
  a different permission string (`pos.void`) than what's granted (`pos.order.void.own`)" is **a red
  herring** — that mismatch is expected and not the bug; `pos.rego`'s void rules do not gate on
  `input.action` at all (unlike the refund/discount/split rules, which do check `input.action ==`), so
  the action string passed is cosmetic for this specific rule.
- The real failure must be in one of: `common.has_permission(input, "pos.order.void.own")` (permission not
  actually present in `input.user.permissions`), `input.resource.created_by == input.user.id` (mismatch),
  `input.resource.status == "OPEN"` (order status mismatch), or `common.same_tenant_and_branch(input)`
  (tenant/branch mismatch).
- **Most likely root cause [ASSUMED, HIGH-PLAUSIBILITY, needs verification]:** JWTs are minted at
  login/branch-switch time (`PermissionResolver.resolve`) and the `043-cashier-void-own-permission`
  changeset was applied to the DB **after** the test session's JWT was issued. Permission grants added to
  `role_permissions` do not retroactively update already-issued tokens — the cashier must **log out and
  back in** (or the token must expire/refresh, 15-min TTL) to receive a JWT whose `permissions` claim
  includes `pos.order.void.own`. If the UAT re-test reused an old session/token, this alone fully explains
  a continued 403 even though the DB grant is correct.
- **Diagnostic recipe for the planner/executor (must run before assuming a code bug):**
  1. Force a fresh login for the CASHIER test user; decode the resulting JWT and confirm
     `pos.order.void.own` is present in the `permissions` claim.
  2. If present and still 403: verify `order.getBranchId()` (passed as `resource.branch_id`) matches
     `claims.branchId()` (passed as `user.branch_id`) for the specific test order — a branch mismatch
     (e.g., order created against a different branch than the cashier's active JWT branch) would fail
     `same_branch` silently.
  3. If both check out: verify the OPA server actually has the current `pos.rego` bundle loaded (policy
     bundle reload/restart timing) — a stale bundle predating the void.own rule addition would deny
     regardless of DB/JWT state.
  4. Only if all three check out is this a genuine, uninvestigated code bug requiring further debugging.

**Confidence: MEDIUM** on the JWT-staleness hypothesis specifically (plausible, common failure mode in
JWT-permission systems, matches the evidence of "the changeset ran but it still failed"); **HIGH** that the
"different permission string in the error message" is NOT the bug (verified by direct rego/Java read).

## Common Pitfalls

### Pitfall 1: Re-firing the entire order on every `sendToKds` call
**What goes wrong:** `OrderServiceImpl.sendToKds` currently builds the KDS payload from
`order.getItems()` unconditionally — every item, every time. Once the state-machine guard is loosened to
allow repeated calls, this would re-send already-served items to the kitchen on every revision.
**Why it happens:** The original Phase-7 implementation only needed to support exactly one fire per order
(the state machine enforced this by rejecting a second call outright); the payload-building code was never
designed to filter.
**How to avoid:** Filter to `status == PENDING` before building the payload (Pattern 1 above); make this the
single seam that constructs `kdsItems` — no other code path should build this list.
**Warning signs:** Any test that sends a Rev 2 and finds Rev 1's items reappearing on the KDS board.

### Pitfall 2: Ticket-existence-as-dedup blocking legitimate revisions
**What goes wrong:** `TicketRoutingService.route()` currently returns immediately (does nothing) if a
ticket already exists for `(orderId, stationCode)` — this was written to guard against duplicate event
delivery, but it also silently discards every legitimate revision's items.
**Why it happens:** True event-redelivery dedup is a *different, already-solved* problem
(`ProcessedEventService.tryProcess` one layer up), but the routing service additionally implemented its own
(now-wrong) dedup signal.
**How to avoid:** Change to append-if-exists (Pattern 2); trust `ProcessedEventService` for redelivery
dedup exclusively.
**Warning signs:** A Rev 2 fire produces no visible change on the KDS board at all (not even the new items).

### Pitfall 3: Order status hand-set from a second code path
**What goes wrong:** POS-11 requires order status to be a pure derivation, but Phase-7 code sets
`order.setStatus(...)` directly in five places (`createOrder`, `addItem`, `sendToKds`, `voidOrder`,
`closeOrder`). If the derivation function is added but a manual `setStatus` call remains for the
"kitchen-progress" range of statuses, the two will drift and the unit test the PRD requires
("marking one line SERVED leaves PARTIALLY_SERVED") will be flaky or wrong under concurrent writes.
**Why it happens:** Incremental extension of existing code without auditing every `setStatus` call site.
**How to avoid:** Split `OrderStatus` conceptually into "settlement lifecycle" (DRAFT/OPEN/CLOSED/
VOIDED/REFUNDED — still hand-set by the state machine) vs. "kitchen-progress" (SENT_TO_KDS/PARTIAL_READY/
READY/SERVED-equivalent to IN_PROGRESS/PARTIALLY_SERVED/SERVED — always derived). Audit and remove any
hand-set of the latter range.
**Warning signs:** Two different derived-status values computed by two different code paths in the same
request.

### Pitfall 4: Missing revision context in the `ORDER_SENT_TO_KDS` payload breaks the frozen-contract rule
**What goes wrong:** The PRD requires `ORDER_SENT_TO_KDS`/`ORDER_CLOSED`/etc. to remain valid for
downstream consumers (Phase 8/9). Adding `revisionNo`/order-level notes to the payload is safe (additive
field), but if the planner instead changes the **shape** of the existing `items` array (e.g. renaming
fields) to accommodate revision data, that breaks Phase 8/9's Feign/consumer contracts silently (no schema
registry enforces this).
**Why it happens:** Payload records are POJOs (Java records) with no versioned schema; a rename compiles
fine on both sides until a real message flows.
**How to avoid:** Only ADD new fields (`revisionNo`, `orderNotes`) to `OrderSentToKdsItem`/
`OrderSentToKdsPayload`; never rename/remove existing fields. Extend
`KitchenEventPayloads.OrderSentToKdsPayload`/`Item` records identically on the consumer side (mirroring the
Phase-7 lesson already documented as [Phase 07 cold-start bug #4] where a payload field mismatch silently
dropped every message).
**Warning signs:** kitchen-service logs "could not deserialize message — skipping" after this phase ships
(exact symptom already seen once in Phase 7 for a related reason).

### Pitfall 5: `TableStatus` enum has no `NEEDS_BUSSING` value yet
**What goes wrong:** UI-SPEC (§2, Table floor view) requires an `AVAILABLE → OCCUPIED → NEEDS_BUSSING →
AVAILABLE` table lifecycle, but the current `TableStatus` enum only has `AVAILABLE, OCCUPIED`.
**Why it happens:** Phase 7 only needed two states (no derived-order-status-driven table state existed
yet).
**How to avoid:** Add `NEEDS_BUSSING` to `TableStatus`, and drive the transition to it from the derived
order status reaching `SERVED` (not `CLOSED`) — i.e., table state is itself partially derived from order
state, a second application of the same derivation pattern as POS-11.
**Warning signs:** UI-SPEC's table-tile three-state contract cannot be implemented without this enum
change; flag it explicitly in the plan's migration list.

### Pitfall 6: `addItem`'s state guard and `sendToKds`'s state guard must be loosened consistently
**What goes wrong:** If `addItem` is loosened to allow adding to `SENT_TO_KDS`/`PARTIAL_READY`/`READY`
orders but `sendToKds`'s state-machine transition table (`OrderStateMachine.TRANSITIONS`) is not
symmetrically updated to allow `SENT_TO_KDS → SENT_TO_KDS` (self-loop) and `PARTIAL_READY → PARTIAL_READY`,
etc., items can be added but never fired — a silent dead end.
**Why it happens:** Two separate guard checks (`addItem`'s explicit if-check, `sendToKds`'s
`stateMachine.assertTransition`) must both be updated; missing one leaves an inconsistent half-fix.
**How to avoid:** Treat "can add items" and "can fire" as the same predicate (order not in a terminal
settlement state), not as enum-transition-table entries, OR add explicit self-loop transitions for every
non-terminal `OrderStatus` value in the transition table.
**Warning signs:** Integration test: add item to a `SENT_TO_KDS` order succeeds, but the subsequent
`sendToKds` call 409s.

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | no (unchanged from Phase 7) | JWT/JWKS, already implemented |
| V3 Session Management | no (unchanged) | — |
| V4 Access Control | **yes** | `OpaClient` fail-closed evaluation via `AuthorizationService`; every new endpoint (table→order lookup, order/item instructions PATCH, revision-fire) MUST call through the existing `PosAuthorizationService`/OPA pattern, never a bespoke check |
| V5 Input Validation | **yes** | Bean Validation (`@NotNull`/`@Min`) on new request DTOs (e.g. instructions PATCH body, char-limit per UI-SPEC's 240/140-char maxes — enforce server-side too, not just client-side) |
| V6 Cryptography | no (no new secrets/PII introduced) | — |

### Known Threat Patterns for this stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Cross-tenant/cross-branch order access via the new table→order lookup endpoint | Information Disclosure / Elevation of Privilege | Same pattern as every existing endpoint: filter by `tenantId` (JWT) + `branchId` (query param, validated against JWT branch or `pos.tables.manage` all-branch permission) — never trust a client-supplied `tenantId` |
| Idempotency-key reuse/prediction allowing a malicious client to replay another user's fire action | Tampering / Repudiation | `clientFireId` is a client-generated UUID scoped by the existing `IdempotencyService` keyed per-request, not guessable/shared across sessions — same protection already proven for `clientOrderId` |
| Instructions field used as an XSS/injection vector (free-text order/item notes rendered on KDS board) | Tampering | Notes are rendered as plain text in React (auto-escaped by JSX) on both POS and KDS frontends — no `dangerouslySetInnerHTML` should be introduced; enforce char-limits server-side (not just client `maxLength`) to prevent oversized payload abuse |
| A demoted/downgraded permission still valid via a stale JWT allowing a void/refund the current permission set would deny | Elevation of Privilege | This is the INVERSE of the POS-14 void-403 hypothesis (A2) — same underlying JWT-staleness mechanism can cut both ways (a revoked permission remains valid until token expiry, 15-min TTL per existing AUTH-02). Not a new risk introduced by this phase, but worth flagging: no new mitigation needed beyond the existing 15-min JWT TTL, already an accepted tradeoff project-wide |
