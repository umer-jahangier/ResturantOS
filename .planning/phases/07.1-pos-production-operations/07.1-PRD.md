## 1. Background & Problem Statement

Phase 7 shipped a **working POS/KDS MVP**: a cashier can create an order, add items, fire it at the
kitchen, and (via the API) take split-tender payment; the KDS routes tickets to stations and progresses
them PENDING‚ÜíCOOKING‚ÜíREADY. Phase-7 verification passed and the phase is closed.

A fresh **Playwright UAT pass (2026-07-11)** ‚Äî driving the real browser as both `cashier@demo.local`
and `chef@demo.local` ‚Äî confirmed the core write-path works but found the product is **not usable for
real restaurant service**:

- **The settlement UI is not wired.** `payment-panel.tsx`, `till-session-bar.tsx`, and
  `void-refund-dialog.tsx` exist in `frontend/components/pos/` but are **never rendered by any page** ‚Äî
  a cashier has no way to charge, open/close a till, or void/refund through the actual UI.
- **Order status is order-level only** (a single `OrderStatus` on the whole order), so a half-served
  order can't be represented; the kitchen can't tell a server "the fries are up but the steak isn't."
- **There is no way to add items to an order after it's been sent** ‚Äî the single most common real-world
  restaurant workflow ("and a coffee to finish"). Re-firing would resend the whole order to the kitchen.
- **No active-order management surface.** You can create-and-fire, but you can't list active orders,
  reopen one, edit it, or drive it to payment from a dedicated screen.
- **The table view is largely non-functional** and is not the primary dine-in navigation it should be.
- **KDS cards jump/bounce**, show no full order detail, and surface no instructions.
- Assorted state bugs (first-item add, an item-cap after N items) and two carried-over UAT gaps
  (void 403; offline sync-badge never updates until reconnect).

Phase 7.1 turns the MVP into a **production-ready restaurant operations surface**. It is explicitly a
hardening/expansion phase, not a rewrite: it must **reconcile with and extend the Phase-7 data model and
event contracts**, never duplicate or break them (Phase 8 Inventory and Phase 9 auto-posting/loyalty
consume `ORDER_CLOSED`/`TILL_*` and MUST keep working unchanged).

## 2. Goals / Non-Goals

**Goals**
1. A dine-in cashier/server can run a full table's lifecycle through the UI: seat a table ‚Üí open order ‚Üí
   add/fire items ‚Üí add more items later (new revision) ‚Üí track item-level status ‚Üí take payment ‚Üí close.
2. Item-level kitchen status is first-class; the order's status is **derived** from its items.
3. Add-to-existing-order kitchen ticket **revisions** work exactly as a real POS does (researched).
4. The already-built payment/till/void UI is wired and reachable; the Phase-7 UAT gaps are closed.
5. The cashier terminal and KDS are fast, stable, and legible for busy service.

**Non-Goals (this phase)**
- No new payment-gateway integration (v1 records tenders only ‚Äî unchanged from Phase 7).
- No change to the multi-tenant / money / event-envelope / OPA / four-layer-frontend invariants.
- No inventory depletion, auto-posting, or loyalty work (Phases 8‚Äì9 own those).
- No change to the `KITCHEN_STAFF` role-isolation guarantees proven in Phase 7.

## 4. Requirements & Acceptance Criteria

### POS-11 ‚Äî Item-level status lifecycle (order status derived) ‚Äî *foundational; do first*
Extend the per-line status from the shallow `KdsItemStatus` to a full lifecycle and **derive** the order
status from lines instead of setting it independently.
- Line status: `PENDING ‚Üí SENT ‚Üí ACCEPTED ‚Üí PREPARING ‚Üí READY ‚Üí SERVED`, plus `CANCELLED` (terminal).
- Order status **derived** from its lines: `DRAFT` (no line sent), `IN_PROGRESS` (‚â•1 line sent, none/some
  ready), `PARTIALLY_SERVED` (some served, some not), `SERVED` (all non-cancelled lines served), then
  `CLOSED`/`VOIDED`/`REFUNDED` on settlement ‚Äî reconciled with the existing `OrderStatus` enum and the
  Phase-7 state machine.
- **Acceptance:** given a 3-line order, marking one line SERVED leaves the order `PARTIALLY_SERVED`;
  marking all SERVED ‚Üí `SERVED`; the derivation is computed (never hand-set) and covered by a unit test.
- **Constraint:** the `ORDER_CLOSED` payload/timing and the DRAFT‚ÜíOPEN‚Üí‚Ä¶‚ÜíCLOSED transitions downstream
  consumers rely on MUST remain valid; this is an extension, not a replacement.

### POS-12 ‚Äî Order revisions / add-to-existing (research-backed)
Add items to an already-sent active order; **send only the newly-added items** to the kitchen as a new
revision; never resend previously-sent/served lines; keep a per-order revision history.
- **Research first** (see ¬ß5): how Toast / Square / Lightspeed / TouchBistro / Aloha model "add to order",
  "fire course", "add-on / order-revision ticket", and how the kitchen distinguishes revisions. Decide,
  with rationale, between (a) a new `KdsTicket` per revision linked to the order vs. (b) one ticket with
  revision-tagged items ‚Äî and how revision numbers are assigned.
- Line gains a revision marker (e.g. `revisionNo` / `firedAt`); `sendToKds` fires only `PENDING` lines.
- **Acceptance:** create order with Burger+Fries+Coke ‚Üí send (Rev 1); add Cheesecake ‚Üí send ‚Üí only
  Cheesecake goes to the kitchen as Rev 2; add Coffee ‚Üí send ‚Üí only Coffee as Rev 3; the KDS shows Rev 2
  and Rev 3 as clearly distinct add-ons, and the earlier lines are not re-fired (verified end-to-end).

### POS-09 ‚Äî Order Management screen
A dedicated screen listing active orders (own or all-branch per permission) with derived status; open,
edit, reopen, and take payment on any active order; an order stays active (OPEN) until paid and closed.
- **Acceptance:** the screen lists active orders with table, derived status, total, and server; clicking
  one opens it for edit/settlement; a non-closed order never disappears from the list; permissions gate
  own-vs-all-branch visibility.

### POS-10 ‚Äî Table-centric dine-in
The table floor view becomes the primary dine-in entry point; selecting a table shows its active order,
status, assigned server/cashier, and live bill summary; every dine-in order is linked to a table.
- **Acceptance:** selecting an occupied table shows its active order + bill summary + server; selecting an
  available table starts a new order bound to that table; table status reflects order lifecycle
  (available ‚Üí occupied ‚Üí needs-bussing/available on close).

### POS-13 ‚Äî Order & item instructions
Order-level special-instructions field + optional per-item instructions (reuse `OrderItem.notes`),
captured at create/edit, surfaced to the kitchen on the ticket + order-detail view.
- **Acceptance:** entering "no onions" on a line and "birthday ‚Äî bring cake last" on the order shows both
  on the KDS ticket/detail; instructions round-trip through create and edit.

### KDS-03 ‚Äî KDS revision & detail
Stable (non-jumping) cards; open a ticket for full order detail; visually distinguish newly-added revision
items from earlier ones; show instructions/kitchen notes; show per-item status (not just ticket status).
- **Acceptance:** cards keep a stable position/order (no bounce on refresh/WebSocket update ‚Äî a real UAT
  complaint); a revision's items render with a distinct "NEW"/"Rev N" treatment; opening a ticket shows the
  full order with per-item status and all instructions.

### POS-14 ‚Äî Wire cashier settlement actions + close Phase-7 UAT gaps
Render `PaymentPanel`, `TillSessionBar`, `VoidRefundDialog` into the live POS/order flow so a cashier can
charge, open/close a till, and void/refund through the UI; fix the void 403 and the offline sync-badge.
- **Acceptance:** a cashier opens a till, takes a split-tender payment that closes the order, and voids an
  open order ‚Äî all through the rendered UI, no dev tools; the void succeeds (403 root-caused and fixed);
  the sync badge updates as the offline queue drains (not only after reconnect).
- **Known UAT context:** void currently 403s despite `pos.order.void.own` being seeded to CASHIER
  (reconfirmed unchanged) ‚Äî diagnose (OPA input? branch/created_by mismatch?) as part of this.

### POS-15 ‚Äî Cashier experience
Fast order creation, quick item search, easy editing of active orders, clear order/item status indicators,
efficient service navigation; fix the terminal state bugs.
- **Acceptance:** first menu tap adds on the first click (regression-test the fixed first-item bug);
  no cap on items added to an order (fix the "can't select after N items" bug); item search filters the
  menu quickly; active-order status is visible at a glance.

## 5. Research Directive (mandatory, feeds POS-11 + POS-12)

Before planning the revision/item-status design, research **how industry-standard restaurant POS systems
handle order revisions and item-level kitchen status** (Toast, Square for Restaurants, Lightspeed,
TouchBistro, Aloha/NCR, Oracle Simphony). Produce concrete, decision-ready findings on:
- Add-to-order / "fire" semantics; coursing; how add-ons appear as new kitchen tickets vs. amended tickets.
- Revision numbering & history; how the KDS visually flags new/added items and voids/86'd items.
- The canonical item-status lifecycle names/transitions (map onto our `PENDING‚ÜíSENT‚ÜíACCEPTED‚ÜíPREPARING‚Üí
  READY‚ÜíSERVED/CANCELLED`) and how order status is derived from item status.
- Concurrency/idempotency for firing revisions (don't double-fire; offline-add then sync).

## 6. Constraints (inherited invariants ‚Äî non-negotiable)

- **Tenancy:** every entity `TenantAuditableEntity`; `tenant_id` from JWT only; RLS on tenant-scoped tables.
- **Money:** BIGINT paisa end-to-end via `MoneyUtils`; per-line floored tax, HALF_UP; discount floor keeps
  line ‚â• 0 (POS-05, preserve).
- **Events:** standard envelope + transactional outbox + idempotent consumers; **do not change** the
  `ORDER_CLOSED`, `ORDER_SENT_TO_KDS`, `ORDER_READY`, `TILL_OPENED`/`TILL_CLOSED` contracts (extend
  additively if needed, with downstream-safe defaults).
- **Authz:** OPA fail-closed; preserve `KITCHEN_STAFF` isolation (kds.view/update only; blocked from POS/finance).
- **Frontend:** four-layer abstraction, Zod-parse every response, `ApiResponse` envelope, components never
  import api-client/repositories (ESLint), skeleton-first loading, `MoneyDisplay`, reduced-motion safe.
- **Offline:** preserve the Service Worker + IndexedDB outbox + `client_order_id` idempotent sync; new
  mutations (add-item revision, instructions) must be offline-safe.
- **Migrations:** Flyway for pos-service (per `[07-01-A]`); additive, RLS-guarded.
