# Phase 07.1 Plan 04: Order Management List + Table-Centric Dine-In + Void-403 Root-Cause Summary

**Permission-gated Order Management list backend (never hides a non-closed order), table→active-order lookup with a live bill summary and derived table-status lifecycle, and the Phase-7 void 403 root-caused as JWT staleness (no code bug) with a proving VoidOwnOrderIT.**

## Accomplishments
- **Order Management list backend (Task 1, POS-09):** new `OrderService.listOrderSummaries` defaults to ALL non-terminal statuses (a non-closed order can never disappear from the list — the old default of `List.of(OrderStatus.values())` on `listOrders` is unaffected/unused by the controller now). Own-vs-all-branch visibility is gated by a new `pos.order.view.all` permission check (`PosAuthorizationService.hasPermission`, a local JWT-claim check, not an OPA round-trip); callers without it are silently scoped to `cashierId == caller` via a new `OrderRepository.findByBranchIdAndStatusInAndCashierId`. `branchId` is validated against the caller's JWT branch and rejected (403) if mismatched — hard tenant/branch isolation per the threat model. New `OrderSummaryDto` (table, derivedStatus, cashierId, covers, total, openedAt).
- **Table-centric dine-in (Task 2, POS-10):** new `GET /api/v1/pos/tables/{id}/active-order` resolves the table's (at most one) non-terminal order via a new `OrderRepository.findByTableIdAndStatusNotIn` — deliberately `Optional`-returning so a table-binding bug (>1 active order) throws `IncorrectResultSizeDataAccessException` instead of silently picking a row. New `TableDetailDto` composes `DiningTableDto`'s fields + a nullable nested `OrderDto` + derived status + live bill summary (subtotal/discount/tax/total) + assigned cashier. `createOrder`'s table-binding (`order.setTableId(request.tableId())`) was already present from a prior session — re-verified, not newly added. Table-status lifecycle (`OCCUPIED`→`NEEDS_BUSSING`→`AVAILABLE`) is now a single derivation seam, `TableService.syncStatusForOrder`, invoked from every order-mutation path (`addItem`, `sendToKds`, `markItemServed`, `cancelItem`, `voidOrder`, `closeOrder`, `OrderReadyConsumer`) — replacing the previously scattered inline `table.setStatus(...)` calls in three of those methods.
- **Void-403 root-caused and closed (Task 3, POS-14):** followed RESEARCH.md's mandatory diagnostic recipe. Direct code read confirms `PosAuthorizationService.authorizeVoid` → `AuthorizationService.authorize` → `OpaClient.evaluate` already builds a correct `OpaInput` (resource `createdBy`/`branchId`/`tenantId`/`status` all populated from the live `Order`; `pos.rego`'s void rules never gate on `input.action`, confirming the "different permission string" from the original UAT is cosmetic). No code bug found — the most plausible remaining root cause is JWT staleness (a token minted before the `043-cashier-void-own-permission` DB grant). New `VoidOwnOrderIT` proves: (1) a cashier with a current token voids their own OPEN order successfully with a correctly-populated OpaInput; (2) a cross-branch order is still denied (`same_branch` stays fail-closed, mirrors `pos_test.rego`'s existing `test_void_own_cross_branch_deny`); (3) a token missing the permission claim (simulating staleness) is denied.
- **Deviation:** extracted `OrderServiceImpl`'s private `toDto` into a new `OrderMapper` `@Component` to avoid a circular Spring bean dependency (`OrderServiceImpl` now needs `TableService`; `TableServiceImpl` needs to build a full `OrderDto` for `TableDetailDto`).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Extracted `OrderMapper` to break a circular Spring bean dependency**
- **Found during:** Task 2 (designing `TableService.getActiveOrderForTable`, which needs a full `OrderDto`)
- **Issue:** `TableServiceImpl` needs to build a full `OrderDto` for `TableDetailDto`'s nested active order. The natural approach (`TableServiceImpl` calling `OrderService.getOrder(...)`) creates a cycle: `OrderServiceImpl` already needs `TableService` (for `syncStatusForOrder`), so `TableServiceImpl -> OrderService -> ... -> TableService` is circular — Spring's constructor-injection container cannot resolve this at context startup.
- **Fix:** Extracted the private `toDto(Order)` method (previously duplicated logic in `OrderServiceImpl`) into a new dependency-free `OrderMapper` `@Component`. Both `OrderServiceImpl` and `TableServiceImpl` now depend on `OrderMapper` (a leaf node), not on each other.
- **Files modified:** `OrderMapper.java` (new), `OrderServiceImpl.java` (delegates to it, private method removed).
- **Verification:** `mvn -o compile`/`mvn -o test-compile` green; no circular-bean-dependency startup error possible by construction (verified by dependency-graph review, not a live Spring context boot in this session — see Issues Encountered).
- **Committed in:** `a4bbcef` (Task 1+2 commit).

**Total deviations:** 1 auto-fixed (1 blocking). **Impact:** Necessary to make the plan's own stated Task 2 deliverable (TableDetailDto composing a full OrderDto) compile and wire correctly without a circular Spring bean graph. No scope change beyond what the plan specifies.

This is the identical, already-documented failure from plans 07.1-02/07.1-03.

  - `TableOrderLookupIT` and `VoidOwnOrderIT` are **WRITTEN-BUT-NOT-EXERCISED**: they compile, and every assertion was hand-traced against the implementation, but neither has run against a live Postgres+app context in this session.
- **GitNexus `detect_changes` reported CRITICAL risk** after Task 1+2 edits, flagging `removeItem`/`applyDiscount`/`closeOrder`/`createOrder`/`addItem` as "changed" even though only `sendToKds`, `listOrderSummaries` (new), `addItem`, `voidOrder`, `closeOrder`, `markItemServed`, `cancelItem` had actual logic changes. Flagging per CLAUDE.md's HIGH/CRITICAL-risk warning requirement; no evidence of an actual regression found.

This is intentional per the plan (Task 1's `<action>` explicitly says "wire the OrderController list endpoint to the new summary method") and PATTERNS.md defers the frontend four-layer wiring for the new shape to a later plan — flagging here so the next frontend-touching plan (Order Management screen) accounts for it, not as an unresolved issue.
