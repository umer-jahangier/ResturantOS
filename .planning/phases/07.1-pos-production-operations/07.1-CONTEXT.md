# Phase 07.1: POS Production Operations & Item-Level Kitchen Tracking - Context

**Gathered:** 2026-07-11
**Status:** Ready for planning
**Source:** PRD Express Path (`07.1-PRD.md`)

<domain>
## Phase Boundary

Upgrade the Phase-7 POS/KDS MVP into a production-ready dine-in operations surface. In scope: an
active-order management screen, table-centric dine-in navigation, item-level kitchen status with the
order status **derived** from its items, industry-standard "add items to an existing order" kitchen
ticket **revisions**, order/item special instructions, wiring the already-built-but-unrendered
payment/till/void UI, closing the Phase-7 UAT gaps (void 403, offline sync badge), a redesigned fast
cashier terminal, and KDS UX fixes (stable cards, ticket detail, per-item status, revision highlighting).

**Hard boundary — extend, do not break.** Phase 8 (Inventory) and Phase 9 (auto-posting/loyalty) consume
`ORDER_CLOSED`/`TILL_*`. These event contracts and the DRAFT→OPEN→…→CLOSED order state machine are frozen;
this phase extends them additively with downstream-safe defaults. `KITCHEN_STAFF` role isolation from
Phase 7 is preserved. No payment-gateway processing, no inventory/finance/loyalty work.
</domain>

<decisions>
## Implementation Decisions

### Item-level status lifecycle (POS-11) — foundational, do first
- Replace/extend the shallow per-line `KdsItemStatus` (`PENDING, COOKING, READY` on `OrderItem.kdsStatus`)
  with the full lifecycle: `PENDING → SENT → ACCEPTED → PREPARING → READY → SERVED`, plus `CANCELLED`.
- Order status is **derived** from line statuses (`DRAFT`, `IN_PROGRESS`, `PARTIALLY_SERVED`, `SERVED`,
  then `CLOSED`/`VOIDED`/`REFUNDED` on settlement), reconciled with the existing `OrderStatus` enum
  (`DRAFT, OPEN, SENT_TO_KDS, PARTIAL_READY, READY, SERVED, CLOSED, VOIDED, REFUNDED`). Derivation is
  computed, never hand-set; unit-tested.
- Reconcile pos-service `OrderItem.kdsStatus` with kitchen-service `TicketItemStatus` so item status stays
  consistent across the two services via the existing `ORDER_READY`/ticket event path.

### Order revisions / add-to-existing (POS-12) — research-backed
- Add items to an already-sent active order; `sendToKds` fires **only** newly-added (`PENDING`) lines as a
  new revision; previously-sent/served lines are never re-fired.
- Maintain a per-order revision history (Rev 1, Rev 2, …); line carries a revision marker (`revisionNo`/
  `firedAt`). The kitchen distinguishes revision/add-on items visually.
- **Research decides** (before implementing): new `KdsTicket` per revision (linked to order) vs. one ticket
  with revision-tagged items; revision numbering; offline-add-then-sync idempotency (no double-fire).

### Order Management screen (POS-09)
- Dedicated screen listing active orders (own or all-branch per permission) with derived status; open,
  edit, reopen, take payment. Orders stay OPEN/active until paid and closed (never auto-disappear).

### Table-centric dine-in (POS-10)
- Table floor view is the primary dine-in entry point. Selecting a table shows its active order, order
  status, assigned server/cashier, and live bill summary. Every dine-in order links to a table; table
  status tracks the order lifecycle (available → occupied → available on close).

### Order & item instructions (POS-13)
- Order-level special-instructions field + optional per-item instructions (reuse `OrderItem.notes`),
  captured at create/edit, surfaced to the kitchen on ticket + order-detail. Offline-safe.

### Wire settlement UI + close UAT gaps (POS-14)
- Render `payment-panel.tsx`, `till-session-bar.tsx`, `void-refund-dialog.tsx` into the live POS/order
  flow (they exist but are unrendered). A cashier can charge (split-tender), open/close a till, and
  void/refund entirely through the UI.
- Diagnose + fix the void **403** (seeded `pos.order.void.own` on CASHIER — suspect OPA input or
  created_by/branch mismatch) and the offline **sync badge** not updating until reconnect.

### Cashier experience + bug fixes (POS-15)
- Fast order creation, quick item search, easy active-order editing, at-a-glance status indicators,
  efficient service navigation. Fix: first-item add (already patched in working tree — regression-test it)
  and the "can't select items after N" cap.

### KDS revision & detail (KDS-03)
- Stable, non-jumping cards (deterministic ordering; no bounce on WebSocket/poll refresh). Open a ticket
  for full order detail + instructions. Distinct treatment for new/revision items. Per-item status shown,
  not just ticket status. Board stays always-dark (kitchen readability), preserved from Phase 7.

### Claude's Discretion
- Exact schema/migration shape for line status + revision fields (Flyway, additive, RLS-guarded).
- Whether derived order status is a computed getter vs. a persisted-and-recomputed column.
- Frontend component decomposition, routing, and state-management details (within the four-layer boundary).
- KDS card ordering/stability algorithm; revision visual treatment specifics (subject to UI-SPEC).
- New internal endpoints/DTOs needed for the Order Management screen and table detail.
</decisions>

<specifics>
## Specific Ideas

- Revision example the KDS must express: Rev 1 = Burger, Fries, Coke; Rev 2 = Cheesecake; Rev 3 = Coffee.
- Item-instruction examples: "no onions", "medium rare", "extra spicy".
- Inherited invariants (non-negotiable): BIGINT-paisa via `MoneyUtils` (per-line floored tax HALF_UP,
  discount floor ≥ 0); tenant_id from JWT + RLS; event envelope + outbox + idempotent consumers; OPA
  fail-closed; offline Service Worker + IndexedDB outbox keyed on `client_order_id`.
</specifics>

</deferred>
