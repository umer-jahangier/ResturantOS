<objective>
Implement the kitchen-service half of order revisions (POS-12) and the KDS revision/detail backend (KDS-03): add `revisionNo`/`firedAt` to `KdsTicketItem`, extend `TicketItemStatus` to the kitchen-owned lifecycle subset, mirror the additive `ORDER_SENT_TO_KDS` payload fields (revisionNo, orderNotes) EXACTLY as pos-service will publish them, and — the core change — convert `TicketRoutingService.route()` from skip-if-exists to append-items-to-the-existing `(orderId, stationCode)` ticket (RESEARCH.md Pattern 2), reopening a READY ticket when new items arrive. Add a ticket-detail read endpoint for KDS-03's "open a ticket for full order detail."

This is the single structural fix that unblocks every revision on the KDS side (RESEARCH.md Summary fact 2: the current skip-if-exists silently drops every legitimate revision). True event-redelivery dedup already lives one layer up in `ProcessedEventService.tryProcess` inside `OrderSentToKdsConsumer` — this plan removes only the *incorrect* second dedup signal (ticket existence), it does NOT weaken redelivery safety.

Purpose: POS-12 and KDS-03 require the kitchen board to accept and visually track add-on revisions on ONE ticket per station.
Output: Extended ticket-item model + additive V3 migration + append-routing + ticket-detail endpoint + a new integration test.
</objective>
