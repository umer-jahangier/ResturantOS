# Phase 07.1 Plan 09: Order Management Screen + Shared Order/Table Detail Drawer Summary

**Built the ONE shared Order/Table Detail drawer (resolves via orderId or tableId) and the POS-09 Order Management screen — a DataTable of active orders with status filters, a permission-gated My/All-Branch toggle, and a 200ms fade-out-on-close instead of an abrupt row disappearance — plus fixed a pre-existing query-invalidation gap that would otherwise have made the new screen show stale data after any settlement action.**

- **Completed:** 2026-07-11

## Accomplishments

- **Shared Order/Table Detail drawer (Task 1, POS-09/POS-10):** `order-table-detail-drawer.tsx` resolves either an `orderId` or a `tableId` via `useOrder`/`useTableDetail` and renders the full contract: header (order #, derived `StatusBadge`, `RevisionCountChip`), per-item lines (`StatusBadge`, `RevisionBadge`, "Mark Served" via `useMarkServed`), editable order-level Special Instructions (`useUpdateInstructions`), an inline Quick Add search (`useMenuItems` + `useAddItem`, 150ms debounced) with a "Full Menu →" escape hatch, and the shared `SettlementActions` footer (rendered exactly once, per UI-SPEC §7's "used identically in exactly 3 places" rule). `revision-chip.tsx` gained an exported `deriveRevisionLog()` helper so this new component reuses order-panel.tsx's exact grouping logic rather than re-implementing it.
- **Order Management screen (Task 2, POS-09):** `order-management.tsx` renders the DS-04 `DataTable` over `useOrderSummaries()` with Order/Table, Status, Server/Cashier, Covers, Total, Age, and an "Open" action column; status filter chips (All/Draft/In Progress/Partially Served/Served) and a `PermissionGuard`-gated My Orders/All Branch toggle are pure client-side filters over one already-fetched list (server already scopes by `pos.order.view.all`). A new `useFadeOutList` hook diffs successive fetches by `orderId`: a row that drops out of the SAME query's refetch stays rendered for 200ms (via `DataTable`'s new optional `rowClassName` prop) instead of vanishing instantly.
- **Bug fix required for the screen's own acceptance criteria:** `useSendToKds`/`useMarkServed`/`useCancelItem`/`useCreateOrder`/`useAddItem` (use-orders.ts) and `useCloseOrder`/`useVoidOrder`/`useRefundOrder` (use-payments.ts) only ever invalidated the legacy `["pos", branchId, "orders"]` query key — never `queryKeys.pos.orderSummaries`'s `"order-summaries"` segment, which this plan's `useOrderSummaries` is the first real consumer of. Without the fix, closing/voiding an order from this drawer's own settlement footer would never refresh the Order Management list.
- **Wired into the real POS UI:** `page.tsx`'s 3rd tab now renders `<OrderManagement />` instead of plan 06's "coming soon" placeholder; its `onFullMenu` callback reuses the page's pre-existing `selectedTableId`/`view` state (no new page-level state introduced).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Critical] Fixed order-summaries query-invalidation gap across 8 mutations**
- **Found during:** Task 2 (implementing `useFadeOutList`/the fade-out acceptance criterion, then tracing why a settlement action wouldn't refresh the list)
- **Issue:** `queryKeys.pos.orderSummaries(branchId, statuses)` produces a query key under the `"order-summaries"` segment, distinct from the pre-existing `["pos", branchId, "orders"]` key that `useSendToKds`, `useMarkServed`, `useCancelItem`, `useCreateOrder`, `useAddItem` (use-orders.ts) and `useCloseOrder`, `useVoidOrder`, `useRefundOrder` (use-payments.ts) already invalidated on success. Since `useOrderSummaries` (07.1-05, unused by any UI until this plan) was never invalidated by any of these, the new Order Management screen would silently show stale derived-status/total/membership data after any order-mutating action taken anywhere in the app — including from this drawer's own settlement footer, directly breaking this plan's own "closing/voiding removes it" acceptance criterion.
- **Fix:** Added `queryClient.invalidateQueries({ queryKey: ["pos", branchId, "order-summaries"] })` alongside each existing `orders`-key invalidation (prefix-match invalidates every statuses-filter cache variant), and added it to `useMarkServed`/`useCancelItem` which previously invalidated neither key.
- **Files modified:** `frontend/lib/hooks/pos/use-orders.ts`, `frontend/lib/hooks/pos/use-payments.ts` (neither in this plan's stated file list).
- **Verification:** `tsc --noEmit` clean; `order-management.test.tsx`'s fade-out test proves the cache-update -> re-render -> fade path works; full mutation -> auto-invalidate -> refetch loop is exercised at the query-key level (unit) but not against a live backend this session (dev stack stale/503, per this plan's environment note).
- **Committed in:** `5bb55ab` (Task 2 commit).

**2. [Rule 2 - Missing Critical] Extended `DataTable` with an additive `rowClassName` prop**
- **Found during:** Task 2 (implementing the fade-out-on-close acceptance criterion)
- **Issue:** The DS-04 `DataTable` primitive (UI-SPEC "reused DS-04 primitives, do not reimplement") has no way to apply a per-row transient class, but "closing/voiding removes it via a smooth fade-out, not an abrupt reflow" is a hard `must_haves.truths` requirement that needs exactly that. Re-implementing the table markup outside `DataTable` would violate the "do not reimplement" rule; leaving it unfaded would violate this plan's own acceptance criteria.
- **Fix:** Added an optional `rowClassName?: (row: TData) => string | undefined` prop, applied via `cn()` on the `<tr>`. Confirmed via grep that `DataTable` has zero existing consumers anywhere in the codebase (Finance's `AccountTable` uses its own table, not this primitive) — so this is a zero-risk, purely additive extension.
- **Files modified:** `frontend/components/ui/data-table.tsx` (not in this plan's stated file list).
- **Verification:** `tsc --noEmit` clean; `order-management.test.tsx`'s fade-out test exercises the new prop end-to-end.
- **Committed in:** `5bb55ab` (Task 2 commit).

**3. [Rule 2 - Missing Critical] Extracted `deriveRevisionLog()` as a shared export from `revision-chip.tsx`**
- **Found during:** Task 1 (building the drawer's revision-count chip)
- **Issue:** `order-panel.tsx` (plan 06, not touched here) has this exact grouping logic inlined as a private function. Duplicating it verbatim in the new drawer would violate the phase's own established pattern ("Revision markers live in one shared component... never reimplemented per-surface" — 07.1-05 SUMMARY).
- **Fix:** Added an exported `deriveRevisionLog()` to `revision-chip.tsx` (the file that already owns the shared revision-marker primitives) with the identical implementation; `order-panel.tsx` itself was left untouched (owned by plan 06, out of this plan's scope) rather than refactored to call the new export.
- **Files modified:** `frontend/components/pos/revision-chip.tsx`.
- **Verification:** `tsc --noEmit` clean; `order-table-detail-drawer.test.tsx`'s revision-chip assertions pass.
- **Committed in:** `493f2bf` (Task 1 commit).

**4. [File-path/plan-text correction] `settlement-actions` referenced once (footer), not also in the header**
- **Found during:** Task 1 (reading the task's own action text against UI-SPEC §7)
- **Issue:** The plan's Task 1 action text parenthetically listed "Void/Refund via settlement-actions" as part of the header composition, in addition to the stated footer placement — which would render `SettlementActions` twice in one drawer, violating UI-SPEC §7's explicit "used identically in exactly 3 places" hard requirement (OrderPanel footer / Order Detail drawer footer / Table Detail drawer footer — this drawer counts as ONE of those three, not two).
- **Fix:** Rendered `SettlementActions` once, in the footer only, per the UI-SPEC (authoritative design contract) rather than the plan prose's parenthetical.
- **Files modified:** none (interpretation choice, not a code deviation).
- **Verification:** n/a (design-contract compliance, not a runtime behavior).
- **Committed in:** `493f2bf` (Task 1 commit).

## Known Stubs

- **Cashier/server name display:** the Order Management "Server/Cashier" column and the drawer's implicit cashier context show a truncated `cashierId` (first 8 hex chars), not a resolved human name — no staff/user-name lookup hook or API exists in the codebase yet. Not a stub that blocks this plan's own goal (the column renders real, correct data — an id — just not a friendly name); flagged for whichever future plan adds a staff-directory lookup.
- **"Full Menu →" / "Go to POS" order-centric preloading:** these only ever bind the Terminal tab to a `tableId` (via the pre-existing `page.tsx` state); there is no mechanism yet to preload the Terminal tab with an EXISTING order (only `PosTerminal`'s own `createOrder` binding by `tableId` exists, from plan 08). For a TAKEAWAY/DELIVERY order (no `tableId`) the escape hatch still switches tabs correctly, just without any order/table binding. Not fixed here — would require a `PosTerminal` change (owned by plan 08, out of this plan's file scope) to accept and load an existing `orderId`.
