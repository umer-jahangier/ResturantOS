<objective>
Close two auth-service seed-data gaps blocking UAT Tests 4, 7, and 10:

1. Test 7: void order is 403-denied for CASHIER on their own OPEN order. Root cause (confirmed by reading `030-create-roles-permissions.xml` in full, not just `041-pos-permissions.xml` as the UAT tester did): the permission code `pos.order.void.own` already EXISTS in the `permissions` table (seeded by changeset `auth-1.0.0-034-seed-permissions`) and is already granted to MANAGER (`auth-1.0.0-036-seed-role-permissions`) plus blanket-granted to OWNER/TENANT_ADMIN — but CASHIER was deliberately excluded there ("CASHIER: NON-privileged — NO void permissions"), a decision made before Phase 7's actual `pos.rego` design existed (`void.own: cashier can void their own OPEN order`). This plan adds the missing grant only — it does NOT touch `void.any` (unchanged, still MANAGER+ only).

2. Tests 4/10: no KITCHEN_STAFF or MANAGER demo user exists in `900-seed-auth-dev-data.xml`, despite `Docs/agent-specs/11-seed-data-specification.md` already documenting `chef@demo.local`/`manager@demo.local`. This blocks all Kitchen Display testing and the reverse-direction role-isolation check (KITCHEN_STAFF denied POS API).

Note: seeding `pos.order.void.own` to CASHIER alone is necessary but not sufficient for Test 7 to actually pass — see the companion pos-service gap-closure plan (07-06), which fixes `OrderServiceImpl` never setting `Order.cashierId`, the other half of the same root cause (OPA's `created_by == user.id` check cannot match a null `created_by`).

Purpose: every permission and role the codebase's own OPA policies and design docs already assume must actually exist in the seed data that provisions them.
Output: CASHIER granted pos.order.void.own; chef@demo.local (KITCHEN_STAFF) and manager@demo.local (MANAGER) demo users seeded with known dev passwords; regression tests proving both.
</objective>
