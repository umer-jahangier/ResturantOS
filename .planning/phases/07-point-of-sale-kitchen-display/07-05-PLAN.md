<objective>
Close UAT Test 5's blocker (`423 PERIOD_LOCKED` on every `POST /orders/{id}/close` attempt on a fresh tenant, even with an exact split-tender payment sum).

Root cause (confirmed by direct source reading, not the UAT tester's grep-only investigation, which missed `InternalProvisioningController` because it is a different file from `PeriodController`): finance-service DOES already seed accounting periods during tenant provisioning (`ProvisioningService.provision` -> `AccountingPeriodServiceImpl.seedForTenant`), but `InternalProvisioningController` â€” the only controller platform-admin-service's saga actually calls (`FinanceInternalClient.seedCoa` -> `POST /internal/finance/tenants/{tenantId}/seed-coa`) â€” computes the fiscal year as `java.time.Year.now().getValue()` (plain calendar year) instead of `PakistanFiscalYear.current()` (Pakistan FY: period 1 = July of `fiscalYear-1`, period 12 = June of `fiscalYear`). Any tenant provisioned from July through December seeds periods for the WRONG, already-elapsed fiscal year, leaving zero period rows covering "today" for the rest of that calendar year. `AccountingPeriodServiceImpl.getPeriodStatus` then throws `PeriodNotFoundException` (404), which pos-service's fail-closed `FinancePeriodClient.assertPeriodOpen` [decision 07-02-A] converts into a permanent `423 PERIOD_LOCKED` â€” exactly the UAT symptom.

Purpose: closing a POS order must never be permanently blocked by a missing accounting-period row, on any tenant, at any point in the Pakistan fiscal calendar.
Output: fixed fiscal-year computation at provisioning time, PLUS a self-healing fallback in the period-status lookup itself (defense in depth â€” a tenant provisioned in one fiscal-year window will otherwise still go blank the moment the calendar crosses into the next fiscal year, since no annual re-seed job exists).
</objective>
