<objective>
Close two UAT gaps that share a single root cause pattern in `OrderServiceImpl.createOrder` ‚Äî it never populates `Order.cashierId` or `Order.tillSessionId` at creation time:

1. UAT Test 6: till-close's "reject if orders still open" gate is a permanent no-op because `TillServiceImpl.closeTill`'s query against `tillSessionId` always returns empty (nothing ever sets it), and the close response's `variancePaisa` is silently null even when both inputs are populated (a separate Hibernate `@Generated` staleness bug).
2. UAT Test 7 (root-cause half owned by this plan ‚Äî see the companion auth-service gap-closure plan for the permission-seeding half): even after CASHIER is granted `pos.order.void.own`, `pos.rego`'s `void.own` rule requires `input.resource.created_by == input.user.id`, but `PosAuthorizationService.authorizeVoid` passes `order.getCashierId()`, which is always null ‚Äî so void.own could never match for ANY cashier, on ANY order, independent of the permission grant.

Both defects were invisible to the existing test suite because `VoidRefundOpaIT` mocks `opaClient.evaluate(eq("pos"), any())` with an unconditional `any()` matcher on the input ‚Äî it never checks that `created_by` is actually correct, so a permanently-null `cashierId` could never fail that test.

Purpose: an order must always know who created it and which till session it belongs to, so till reconciliation and void.own authorization are both reachable in production, not just theoretically implemented.
Output: `Order.cashierId`/`Order.tillSessionId` populated at creation; `TillSession.variancePaisa` correctly refreshed after close; regression tests that would have caught both bugs.
</objective>

  <files>services/pos-service/src/main/java/io/restaurantos/pos/service/OrderServiceImpl.java, services/pos-service/src/main/java/io/restaurantos/pos/domain/model/TillSession.java</files>

When no till is open, tillSessionId stays null (a legitimate state, e.g. a MANAGER creating an order without a personal till) and must not throw.
    - TillSession.variancePaisa: Hibernate must re-fetch this DB-computed (GENERATED ALWAYS AS ... STORED) column after BOTH INSERT and UPDATE, not just INSERT ‚Äî closeTill performs an UPDATE, and PostgreSQL already recomputes the column correctly on UPDATE; only the Java-side refresh was missing.
  </behavior>
  <action>
    Root cause 1 (confirmed by reading OrderServiceImpl.java in full): `createOrder()` never calls `order.setCashierId(...)` or `order.setTillSessionId(...)` anywhere in the file ‚Äî every order is created with both columns permanently null. This breaks two independent things: (a) `TillServiceImpl.closeTill`'s open-orders gate (`orderRepository.findByTillSessionId(tillId)`) can never find ANY order, so it is a permanent no-op no matter how many orders are open under that till; (b) `PosAuthorizationService.authorizeVoid` passes `order.getCashierId()` as `created_by` to OPA, and `pos.rego`'s `void.own` rule requires `input.resource.created_by == input.user.id` ‚Äî with `created_by` always null this can never match for any cashier on any order, entirely independent of whether `pos.order.void.own` is granted.

    Inject `TillSessionRepository` into `OrderServiceImpl` ‚Äî add it as a constructor parameter and a `private final` field, following the exact same constructor-injection style already used for the other 13 dependencies in this class ‚Äî and add the import `io.restaurantos.pos.domain.enums.TillStatus`. In `createOrder()`, immediately after the `if (request.tableId() != null) { order.setTableId(request.tableId()); }` block and before `order = orderRepository.save(order);`, resolve `tenantContext.getUserId()`; if present, call `order.setCashierId(userId)`, then look up `tillSessionRepository.findByCashierIdAndStatus(userId, TillStatus.OPEN)` and, if present, call `order.setTillSessionId(till.getId())`. Follow the file's established `Optional.ifPresent(...)` lambda style (see the `tableRepository.findByIdAndBranchId(...).ifPresent(...)` call a few lines above in `addItem` for the established pattern).

    Root cause 2 (confirmed by reading TillSession.java and TillServiceImpl.closeTill together): `variancePaisa` is annotated `@Generated(event = EventType.INSERT)`, which tells Hibernate to re-SELECT this column only after an INSERT statement. `closeTill` performs an UPDATE (the session row already exists from `openTill`) via `saveAndFlush`, so Hibernate never refreshes `variancePaisa` on the managed entity, and the subsequent `tillSessionRepository.findById(tillId)` call returns the SAME managed instance from the first-level persistence-context cache (no new SELECT is actually issued), so the stale null value flows all the way into the response DTO. Change `@Generated(event = EventType.INSERT)` to `@Generated(event = {EventType.INSERT, EventType.UPDATE})` so Hibernate re-fetches the column after both INSERT and UPDATE.
  </action>
