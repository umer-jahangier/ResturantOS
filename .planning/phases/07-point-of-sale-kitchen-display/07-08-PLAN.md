<objective>
Close the two remaining, still-unfixed items from UAT Test 1's cold-start gap: (1) `docker compose up --build` aborts entirely because the only two application-service Dockerfiles actually wired into `deploy/docker-compose.yml`'s `build:` stanzas (eureka-server, config-server) — and, on inspection, seven other service Dockerfiles plus pos-service's own — stage an incomplete list of module `pom.xml` files, so Maven's reactor validation (which checks EVERY `<module>` declared in the root `pom.xml`, even with `-pl`/`-am`) fails with "Child module ... does not exist"; (2) `scripts/start-dev.ps1` and `scripts/restart-service.ps1` have no entries for pos-service or kitchen-service, so there is no single-command way to cold-start a phase-7-inclusive dev stack — they must be started manually.

Purpose: the tooling that stands up this codebase must know about every service the codebase actually contains.
Output: every Dockerfile that follows the shared multi-stage build pattern COPYs all 14 module pom.xml files; pos-service and kitchen-service are first-class citizens of the dev start/restart scripts.
</objective>

<task type="auto">
  <name>Task 1: Fix module pom.xml COPY lists in every Dockerfile missing pos-service/kitchen-service</name>
  <files>eureka-server/Dockerfile, config-server/Dockerfile, gateway/Dockerfile, services/audit-service/Dockerfile, services/auth-service/Dockerfile, services/authorization-service/Dockerfile, services/file-service/Dockerfile, services/finance-service/Dockerfile, services/user-service/Dockerfile, services/pos-service/Dockerfile</files>
  <action>
    Root cause (confirmed by reading all 13 backend-service Dockerfiles): the root `pom.xml` declares 14 `<module>` entries (`shared-lib`, `eureka-server`, `config-server`, `gateway`, 8 services including `auth-service`/`authorization-service`/`user-service`/`platform-admin-service`/`notification-service`/`audit-service`/`file-service`/`finance-service`, plus `services/pos-service` and `services/kitchen-service` added in Phase 7). Every multi-stage Dockerfile in this repo stages a fixed COPY list of module `pom.xml` files before running `mvn -pl <module> -am ...`. Maven's reactor validates EVERY declared `<module>` even with `-pl`/`-am`, so a single missing `pom.xml` anywhere aborts the build with "Child module ... does not exist" — regardless of which service is actually being packaged. `services/kitchen-service/Dockerfile` already stages all 14 module pom.xml files (already correct — do not touch it). Every other Dockerfile in the `<files>` list above ends its COPY list at `services/finance-service/pom.xml` (added before Phase 7 introduced pos-service/kitchen-service) and is missing both. This is exactly what breaks `docker compose up --build` — `deploy/docker-compose.yml` only has `build:` stanzas for `eureka` and `config-server` (every other application service is deliberately host-run via `scripts/start-dev.ps1`, per the comment block at the bottom of that compose file — do not add pos-service/kitchen-service AS docker-compose services, that would contradict the established host-run architecture) — and would equally break a standalone `docker build` of any of the other Dockerfiles listed here (e.g. in a future CI image-build step).

    For 9 of the 10 files listed — `eureka-server/Dockerfile`, `config-server/Dockerfile`, `gateway/Dockerfile`, `services/audit-service/Dockerfile`, `services/auth-service/Dockerfile`, `services/authorization-service/Dockerfile`, `services/file-service/Dockerfile`, `services/finance-service/Dockerfile`, `services/user-service/Dockerfile` — insert these two lines immediately after each file's existing `COPY services/finance-service/pom.xml services/finance-service/pom.xml` line (the last module-pom COPY line in every one of them) and before the following blank line / `RUN mvn ... dependency:go-offline` line: `COPY services/pos-service/pom.xml services/pos-service/pom.xml` then `COPY services/kitchen-service/pom.xml services/kitchen-service/pom.xml`.

    For `services/pos-service/Dockerfile` specifically: it already has its own `COPY services/pos-service/pom.xml services/pos-service/pom.xml` line but is missing kitchen-service's — insert only `COPY services/kitchen-service/pom.xml services/kitchen-service/pom.xml` immediately after that existing line.

    Do NOT modify `services/kitchen-service/Dockerfile` (already correct) or `services/platform-admin-service/Dockerfile` (uses a different, pre-existing, undiagnosed `COPY src ./src`-style build pattern that does not stage per-module pom.xml files at all — out of scope for this gap; rewriting it would be inventing scope beyond what UAT diagnosed).
  </action>
  <verify>
    <automated>for f in eureka-server/Dockerfile config-server/Dockerfile gateway/Dockerfile services/audit-service/Dockerfile services/auth-service/Dockerfile services/authorization-service/Dockerfile services/file-service/Dockerfile services/finance-service/Dockerfile services/user-service/Dockerfile; do grep -q "services/pos-service/pom.xml" "$f" && grep -q "services/kitchen-service/pom.xml" "$f" || echo "MISSING:$f"; done | grep . && exit 1 || echo ALL_OK; grep -q "services/kitchen-service/pom.xml" services/pos-service/Dockerfile && echo POS_OK || exit 1</automated>
  </verify>
  <done>All 9 general Dockerfiles plus pos-service's Dockerfile now COPY every one of the 14 module pom.xml files declared in the root pom.xml's &lt;modules&gt; block; the verification script prints ALL_OK and POS_OK with no MISSING lines.</done>
