## Shared Patterns

### `@PreAuthorize` authority gate on `/api/v1/finance/**` controllers
**Source:** `services/finance-service/.../web/PeriodController.java` (every `@*Mapping` method has a `@PreAuthorize("hasAuthority('...')")` immediately above it, e.g. lines 33, 49, 55, 61)
**Apply to:** the new `POST /api/v1/finance/periods/provision` endpoint — use `@PreAuthorize("hasAuthority('finance.period.open')")`.

### `ApiResponse.ok(...)` response envelope
**Source:** every method in `PeriodController.java` returns `ResponseEntity.ok(ApiResponse.ok(...))`
**Apply to:** the new provision endpoint's success path — same wrapper, no ad-hoc `Map`/DTO returns.

### Exception → HTTP status mapping via `FinanceGlobalExceptionHandler`
**Source:** `services/finance-service/.../exception/FinanceGlobalExceptionHandler.java` (per RESEARCH.md, `@RestControllerAdvice` maps exceptions to `{code, message, timestamp}` bodies) — `TotpRequiredException` → 403 is an existing example already read by research.
**Apply to:** no new exception type is needed for this phase's endpoint (idempotent, no new failure mode beyond what `@PreAuthorize`'s existing 403 and standard validation already cover) — but if `fiscalYear` validation (V5, plausible-year check) throws, it should route through this same handler rather than a bespoke try/catch in the controller.
