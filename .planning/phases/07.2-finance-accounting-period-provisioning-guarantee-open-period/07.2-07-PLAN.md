<objective>
Give the Finance module a real front door for period provisioning. Today `/app/finance/periods` only ever shows the hardcoded "current" fiscal year (`const fiscalYear = currentPakistanFiscalYear();` — no way to view any other year) and, when it's empty, dead-ends with "System Admin need to run the script to load COA and periods" — copy that is now simply wrong once 07.2-05's self-service endpoint exists, and there is no button anywhere that calls it. This plan closes that gap with an industry-standard, calendar-based provisioning UI: a fiscal-year navigator (so any year — past, current, or a year that hasn't arrived yet — can be viewed, never just "current"), and a `finance.period.open`-gated "Provision Periods" dialog that previews the 12 monthly periods a fiscal year will produce — computed dynamically from the exact same Jul-Jun formula the backend uses (`AccountingPeriodServiceImpl.seedForTenant`), never a hardcoded year or date — before the user confirms.

Purpose: FIN-10 — a tenant admin should never need platform-ops or a raw HTTP call to open a period, this year or any future one. The design mirrors the period/fiscal-year setup screens of QuickBooks Online, Xero, and NetSuite: pick a fiscal year, see exactly which periods will be created (with already-open ones visually distinguished), confirm.
Output: A `getFiscalYearPeriods` utility (TDD, unit-tested across multiple fiscal years including leap-affected ones), four-layer API wiring for the provision endpoint, a `FiscalYearNav` + `ProvisionPeriodDialog` component pair, their wiring into the existing Periods page, and a Playwright spec proving the flow (live execution deferred to 07.2-06's restart-and-verify gate, since this plan's wave runs before that restart).
</objective>
