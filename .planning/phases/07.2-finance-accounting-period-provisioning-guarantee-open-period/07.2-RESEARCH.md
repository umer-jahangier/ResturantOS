## Summary

This phase fixes a **live, currently-reproducible** production blocker: a fresh tenant provisioned through the platform has **zero accounting periods**, so `POST /orders/{id}/close` permanently 423s. Research confirms the blocker is real today (2026-07-11) — the three tenants seeded in the local dev Postgres (`finance_db.accounting_periods`) only have Pakistan FY2026 periods (Jul 2025–Jun 2026); nothing covers "today," which now falls in FY2027 (period 1 = July 2026).

The investigation described in the phase brief (silent try/catch swallow in `platform-admin-service`'s onboarding saga) is real but is **not the primary root cause**. The actual primary cause, verified via `git blame` and the live `application.yml`, is that **`provisioning.seed-coa.enabled` defaults to `false`** (`PROVISIONING_SEED_COA_ENABLED:false`), a flag introduced 2026-06-25 when `platform-admin-service` was first scaffolded — *before finance-service existed*. The Feign client's own Javadoc says explicitly: *"finance-service is a LATER phase… In production (enabled=true) a finance failure becomes a saga compensation step."* Finance-service has been live since Phase 6 (completed weeks ago) and the flag was never flipped. Under the current default, **step 5 of the onboarding saga never even attempts to call finance-service** — the try/catch swallow is a secondary bug that only matters once the flag is turned on.

A second major finding: the auto-seed-on-miss fallback added in commit `5babd73` (07-05, 2026-07-10 22:06) is correct in source and **is** wired into the live call path used by `pos-service` (`GET /internal/finance/periods/status` → `AccountingPeriodServiceImpl.getPeriodStatus`). But the **running finance-service process was stale** — it was started at 02:57 AM on 2026-07-10, hours before the fix was committed at 22:06 that same day — so Phase 07.1's E2E run genuinely hit the pre-fix code despite the fix being present in the git history and even in the rebuilt jar (built 22:08, two minutes after the commit). This research restarted finance-service against the current jar as part of verifying this hypothesis; the planner's phase-completion checklist should include a `restart-service.ps1 finance-service` (and any other stale service) step as standard hygiene, not just for this phase.

A third finding materially simplifies the endpoint work: `finance-service` already has an internal, fully-idempotent `ProvisioningService.provision(tenantId, fiscalYear)` (seeds CoA via `CoaServiceImpl.seedForTenant` + periods via `AccountingPeriodServiceImpl.seedForTenant`, both dedupe on existing rows) exposed only internally at `POST /internal/tenants/{tenantId}/provision`. The new public self-service endpoint is a thin wrapper around this exact same idempotent method — no new business logic needed, just a `@PreAuthorize`-gated controller method that resolves `tenantId` from `TenantContext` instead of a path variable/internal-secret header. A ready-made "is finance set up?" primitive (`GET /api/v1/finance/accounts/setup/status` → `FinanceSetupStatusDto(accountCount, periodCount, isComplete)`) already exists and is unused by any frontend — this is the natural signal for a "finance setup pending" UX, no new backend endpoint or tenant-status enum value required.

**Primary recommendation:** (1) Flip `provisioning.seed-coa.enabled` default to `true` (or remove the flag's gating of the *call*, keeping it only as an emergency kill-switch) and make finance-seeding failure abort the saga (mark `PROVISIONING_FAILED`, not swallow) — but do **not** rely on the existing `retry()` method to recover a finance-only failure, it has a pre-existing bug (see Pitfalls) that would corrupt a partial retry. (2) Add `POST /api/v1/finance/periods/provision` in finance-service as a thin, `@PreAuthorize("hasAuthority('finance.period.open')")`-gated wrapper around the existing internal `ProvisioningService.provision()`. (3) Gate the `getPeriodStatus` auto-seed-on-miss branch behind a new `@Value`-driven property (default `true` in dev/staging profiles, `false` in prod) with a `log.warn` when it fires. (4) Restart `finance-service` (and confirm `platform-admin-service`, also currently running a stale June-27 jar though not directly implicated) as part of phase verification — a code fix that isn't running is not a fix.

### Pattern 1: Idempotent internal service reused as a public self-service action

**What:** `finance-service.ProvisioningService.provision(tenantId, fiscalYear)` already exists, is already idempotent (both `CoaServiceImpl.seedForTenant` and `AccountingPeriodServiceImpl.seedForTenant` dedupe against existing rows before inserting), and is already exposed once internally (`POST /internal/tenants/{tenantId}/provision`, called with a static internal secret). The new public endpoint should be a **thin controller method**, not a new service method — resolve `tenantId` via `tenantContext.requireTenantId()` (standard JWT-derived context, same as every other `/api/v1/finance/**` endpoint) instead of a path variable, and delegate straight through.

**When to use:** Any time a phase needs to expose internal-only orchestration to authenticated tenant users without duplicating logic.

**Example (structure to follow, based on `PeriodController.closePeriod`):**
```java
// Source: services/finance-service/.../web/PeriodController.java (existing pattern, read 2026-07-11)
@PostMapping("/api/v1/finance/periods/provision")
@PreAuthorize("hasAuthority('finance.period.open')")
public ResponseEntity<ApiResponse<ProvisioningResult>> provisionPeriods(
        @RequestBody(required = false) ProvisionRequest request) {
    UUID tenantId = tenantContext.requireTenantId();
    int fiscalYear = (request != null && request.fiscalYear() != null)
            ? request.fiscalYear()
            : PakistanFiscalYear.current();
    return ResponseEntity.ok(ApiResponse.ok(provisioningService.provision(tenantId, fiscalYear)));
}
```
Note: `provisioningService.provision()` already calls `TenantGucHelper.apply(entityManager, tenantContext)` internally — no `tenantHelper.activate()`/`clear()` dance needed here (that pattern is only for `/internal/**` endpoints that don't have a JWT-derived tenant context).

### Pattern 2: Liquibase incremental permission changeset (never edit the base seed changeset)

**What:** New permission codes and role grants are added via a brand-new numbered Liquibase changeset file — never by editing `030-create-roles-permissions.xml`'s `runOnChange="false"` seed changesets (which will not re-run on existing databases anyway).

**When to use:** Adding any new `permissions`/`role_permissions` row.

**Example:**
```xml
<!-- Source: services/auth-service/.../db/changelog/v1.0.0/043-cashier-void-own-permission.xml (existing pattern) -->
<!-- New file: 044-finance-period-open-permission.xml -->
<changeSet id="auth-1.0.0-044-finance-period-open-permission" author="restaurantos-agent" runOnChange="false">
    <insert tableName="permissions">
        <column name="code" value="finance.period.open"/>
        <column name="module" value="finance"/>
        <column name="description" value="Provision/open accounting periods"/>
    </insert>
    <!-- OWNER and TENANT_ADMIN get this automatically via 036's wildcard SELECT rules —
         no insert needed for them. ACCOUNTANT must be granted explicitly, matching how
         finance.period.close was granted to ACCOUNTANT in 036. -->
    <insert tableName="role_permissions">
        <column name="role_code" value="ACCOUNTANT"/>
        <column name="permission_code" value="finance.period.open"/>
    </insert>
</changeSet>
```
**Important:** `OWNER` gets every permission via `SELECT 'OWNER', code FROM permissions` and `TENANT_ADMIN` gets every permission except `rbac.manage` via a similar wildcard `SELECT` — **both of those wildcard `SELECT` statements live in changeset `036`, which is `runOnChange="false"` and will NOT re-run on an existing database.** A brand-new permission code inserted by changeset `044` will **not** automatically propagate to `OWNER`/`TENANT_ADMIN` on any environment that already ran `036` (i.e. every existing dev/staging/prod DB). The new changeset **must explicitly grant** the new permission to `OWNER`, `TENANT_ADMIN`, and `ACCOUNTANT` — do not assume the wildcard grants retroactively apply.

### Pattern 3: `@Value` property with environment-specific default (config toggle convention)

**What:** This codebase's convention for a config toggle is a plain Spring `@Value("${property.path:default}")` field, with the *actual* effective default supplied in `application.yml` via an env-var-backed placeholder (`${ENV_VAR_NAME:actualDefault}`) — **not** `application-{profile}.yml` files (none exist in this codebase's services; profiles are not used for this kind of toggle). See `provisioning.seed-coa.enabled` in `platform-admin-service/application.yml` as the exact existing precedent for the pattern to replicate for the new auto-seed-on-miss toggle.

**When to use:** Any environment-specific behavioral toggle in this codebase.

**Example:**
```java
// Source: services/platform-admin-service/.../service/ProvisioningService.java (existing pattern)
@Value("${provisioning.seed-coa.enabled:true}")
private boolean seedCoaEnabled;
```
```yaml
# Source: services/platform-admin-service/.../application.yml (existing pattern)
provisioning:
  seed-coa:
    enabled: ${PROVISIONING_SEED_COA_ENABLED:false}   # <-- this line is what actually took effect; the code-level :true default was masked
```
**Pitfall this reveals (see below):** the `@Value` annotation's inline default and the YAML's env-var default can silently diverge, and the YAML wins. Any new toggle for this phase must have its YAML-level default reviewed as carefully as its code-level default — do not assume the `@Value(...:true)` inline default is what actually ships.

### Anti-Patterns to Avoid

- **Swallow-and-continue on a required dependency:** `catch (Exception ex) { log.warn(...); }` around a step that later code assumes succeeded (Step 6 marks ACTIVE unconditionally) silently produces an inconsistent tenant state. If a step is required for the tenant to be usable, its failure must propagate to the saga's existing `catch` block (which already correctly sets `PROVISIONING_FAILED` and runs compensation) — do not add a second, more lenient catch inside the step itself.
- **Reusing a generic `retry()` for a partial/single-step failure:** see Common Pitfalls — the existing `retry()` unconditionally re-runs every step from scratch, including steps that already succeeded.
- **Treating OPA rego as required for every new `@PreAuthorize` endpoint:** this codebase's `finance.rego` already has a `close_period` rule that is defined and unit-tested (`policies/tests/finance_test.rego`) but is **not invoked by any live Java code** — `PeriodController.closePeriod` uses only `@PreAuthorize`. Do not assume a new rego rule is required for the new endpoint; follow the established (if slightly inconsistent) precedent of `@PreAuthorize`-only enforcement for finance-service's period endpoints, to avoid unnecessarily touching the `opa test` 100%-coverage gate.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| CoA + period seeding logic | A new seeding method for the public endpoint | `ProvisioningService.provision(tenantId, fiscalYear)` (already idempotent, already tested by `AccountingPeriodIT`/`CoaProvisioningIT`) | Duplicating this logic risks the two paths (internal vs. public) drifting out of sync |

| Permission/role plumbing | Hand-written SQL against `permissions`/`role_permissions` | Liquibase incremental changeset (Pattern 2 above) | Liquibase changelog is the only mechanism that runs consistently across fresh Testcontainers DBs, dev, and any future staging/prod DB |

The work is almost entirely *wiring and config correction*, not new logic — which also means the risk surface is mostly in getting the config defaults and the saga failure-path right, not in algorithmic complexity.

## Common Pitfalls

### Pitfall 1: `retry()` is unsafe for a finance-only saga failure

**What goes wrong:** `ProvisioningService.retry(tenantId)` in platform-admin-service unconditionally re-runs the **entire** saga from Step 2 (feature flags) through Step 6, including re-creating the branch (`userClient.createBranch(...)`) and re-provisioning the admin user (`authClient.provisionAdmin(...)`) — even if those steps already succeeded the first time and only the finance step failed. It also passes a **literal string `"(retry)"`** as the `adminEmail` argument (not the tenant's real admin email, which isn't persisted anywhere retrievable by `retry()`), so re-provisioning the admin user would create a nonsensical account.
**Why it happens:** `retry()` was written as a generic "start over" mechanism, presumably before any single-step (rather than whole-saga) failure mode existed. It predates this phase's finance-seeding-failure scenario.
**How to avoid:** If the planner chooses to mark the tenant `PROVISIONING_FAILED` on a finance-seeding failure, do **not** route recovery through the existing `retry()` endpoint. Either (a) make Step 5's failure NOT flip the tenant out of `ACTIVE` at all — instead track a separate "finance incomplete" signal and let the tenant owner self-heal via the new public `POST /api/v1/finance/periods/provision` endpoint (which only touches finance-service, is safe to call on a tenant with an already-existing branch/admin), or (b) fix `retry()` to skip already-completed steps and use the real admin email before repurposing it. Option (a) is far lower-risk and reuses the phase's own new endpoint — recommended.
**Warning signs:** Any plan step that says "call `retry()` to fix a tenant with a finance-only provisioning gap" should be treated as a red flag during plan review.

### Pitfall 2: The `@Value(...)` inline default is not necessarily the effective default

**What goes wrong:** `ProvisioningService.java`'s `@Value("${provisioning.seed-coa.enabled:true}")` reads as "defaults to true," but `application.yml` supplies `${PROVISIONING_SEED_COA_ENABLED:false}` for that same property key — and Spring resolves the YAML value first, so the *actual* live default is `false`. A reviewer reading only the Java file would reasonably (and incorrectly) conclude the flag defaults on.
**Why it happens:** The YAML placeholder's own inline default was set once, deliberately, when finance-service didn't exist (commit `3319226`, 2026-06-25) and never revisited when finance-service went live.
**How to avoid:** When changing any `@Value`-backed toggle in this phase, grep for the property key across `application.yml` (and any profile-specific files) in the same service to confirm what actually ships, not just what the `@Value` annotation says. Update both if changing the default.
**Warning signs:** A "fix" that only edits the `@Value(...)` inline default without touching `application.yml` will have no effect on the running service.

### Pitfall 3: A correct code fix in git does not mean the running service has it

**What goes wrong:** As demonstrated live in this research session, `finance-service`'s auto-seed-on-miss fix (`5babd73`) was correctly committed and even correctly rebuilt into the jar — but the actual running JVM process had been up since before either event. Any live/E2E verification against a long-running dev stack can silently test stale code.
**Why it happens:** This project's dev workflow runs services as long-lived background `java -jar` processes (`scripts/restart-service.ps1`); there is no auto-reload, and nothing prompts a restart after a `git pull`/rebuild.
**How to avoid:** Any phase's plan/verification steps that involve live/E2E testing against the dev stack should explicitly restart every touched service first and confirm via `/actuator/health` (or an equivalent liveness signal) before drawing conclusions from a failed manual test.
**Warning signs:** A bug that "should be fixed based on the code" but still reproduces live — check process start time vs. fix commit time before assuming the fix is wrong.

### Pitfall 4: New permission codes do not retroactively reach `OWNER`/`TENANT_ADMIN` wildcard grants

**What goes wrong:** Changeset `036`'s `INSERT ... SELECT 'OWNER', code FROM permissions` and the equivalent `TENANT_ADMIN` statement are `runOnChange="false"` and only ever execute once, at initial DB creation. A new permission code inserted by a later changeset (e.g. `044`) will exist in the `permissions` table but will have **no** corresponding `role_permissions` row for `OWNER`/`TENANT_ADMIN` on any database that already ran `036` — which is every existing environment.
**Why it happens:** Liquibase's `runOnChange="false"` guarantees the SQL text runs exactly once per changeset ID; it does not re-evaluate a `SELECT ... FROM permissions` against newly inserted rows.
**How to avoid:** The new changeset must explicitly `INSERT INTO role_permissions` rows for `OWNER`, `TENANT_ADMIN` (and any other role that should have the new permission), not rely on the old wildcard `SELECT` statements. See Pattern 2's code example above.
**Warning signs:** An IT/manual test where `OWNER` or `TENANT_ADMIN` unexpectedly gets `403` on the new endpoint despite "having all permissions" is this exact bug.

### Pitfall 5: `businessDate` in `closeOrder` is derived from `order.openedAt`, not `LocalDate.now()`

**What goes wrong:** `OrderServiceImpl.closeOrder`'s period-lock check uses `order.getOpenedAt().atOffset(UTC).minusHours(4).toLocalDate()` (a business-day-boundary formula) when the order has an `openedAt` timestamp, falling back to `LocalDate.now()` only if it's null. A period-provisioning fix that only considers "today" (`PakistanFiscalYear.current()`) could still miss an edge case for an order opened just before a fiscal-year boundary (e.g. an order opened June 30 late at night, closed after the July 1 rollover) if the seeded period set doesn't yet cover the *order's* business date.
**Why it happens:** Business-day boundary (`-4h` offset) plus fiscal-year boundary (`month >= 7`) are two independent date-shifting rules that can interact at year edges.
**How to avoid:** This is already handled correctly by the auto-seed-on-miss fallback (it seeds whatever fiscal year the *requested* date falls into, not just "today"), so no code change is needed here — but the planner should be aware of this when writing verification tests, and should test a date near a fiscal-year boundary, not just "today."
**Warning signs:** A test that only exercises `LocalDate.now()` would not catch a boundary regression.

## Open Questions

1. **Should `provisioning.seed-coa.enabled` be removed entirely (call always fires) or just flipped to default `true`?**
   - What we know: The flag's original purpose (avoid calling finance-service before it existed) no longer applies; finance-service has been live since Phase 6.
   - What's unclear: Whether the flag should be kept as an emergency kill-switch (e.g. for a future finance-service outage during onboarding) or removed as dead config now that the swallow-on-failure anti-pattern is being fixed anyway.
   - Recommendation: Keep the flag (cheap insurance, already has a name/precedent) but flip its **YAML** default to `true`, and make failure-when-enabled abort the saga rather than swallow. This satisfies both "guarantee open period" and "don't remove a legitimate escape hatch."

2. **Reconcile Phase 6's `Pending` status in REQUIREMENTS.md traceability vs. ROADMAP.md showing it complete.**
   - What we know: ROADMAP.md phase list shows `[ ] Phase 6` unchecked and its plans list as `- [ ]` (not done), while STATE.md's Performance Metrics table lists "Phase 6: 2/2 plans executed (COMPLETE)" and REQUIREMENTS.md's traceability table lists FIN-01/02/04/06 as `Pending`. The actual finance-service code (CoA, JE engine, periods, close/lock) is unambiguously built and working (verified by reading it directly).
   - What's unclear: Whether this is a stale-checkbox bookkeeping bug in ROADMAP.md/REQUIREMENTS.md or signals something not actually done.
   - Recommendation: The planner should flag/fix this status mismatch as part of phase setup (likely a simple checkbox-sync bug, given the code is clearly live and this phase itself is explicitly named "guarantee open period" implying periods already exist) — not block on it, but don't let it silently persist either.

3. **Confirm `.env`'s actual live value for `PROVISIONING_SEED_COA_ENABLED`** (see Assumption A1) — a human with `.env` access should check this before the plan finalizes its config-default strategy, since this research session's file access was sandboxed for that path.

### Known Threat Patterns for this stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Cross-tenant period provisioning (attacker-supplied `tenantId`) | Spoofing / Elevation of Privilege | The new public endpoint must resolve `tenantId` from `tenantContext.requireTenantId()` (JWT-derived), **never** from a path variable or request body — matches the existing pattern for every other `/api/v1/finance/**` endpoint; do not mirror the internal endpoint's `@PathVariable UUID tenantId` shape for the public one |
| Privilege escalation via new permission not restricted enough | Elevation of Privilege | Only grant `finance.period.open` to `OWNER`/`TENANT_ADMIN`/`ACCOUNTANT` in the new Liquibase changeset — do not grant to `MANAGER`, `CASHIER`, `INVENTORY_MANAGER`, or `KITCHEN_STAFF` (none of whom hold `finance.period.close` today either) |
| Repeated/abuse calls to the provisioning endpoint (DoS-adjacent, resource exhaustion via repeated fiscal-year seeding) | Denial of Service | Low risk — the operation is idempotent and bounded (12 periods max per call, deduped), but the planner may still want to note there's no explicit rate limit beyond the gateway's general 600/min/IP (GW-04); no additional mitigation required given the idempotency |
