<objective>
Make the silent auto-seed-on-miss fallback in `AccountingPeriodServiceImpl.getPeriodStatus` configurable and audited (FIN-09). Today the method unconditionally seeds a whole fiscal year of periods when a lookup misses — convenient in dev, but in production it silently masks a genuine provisioning gap. This plan gates that behavior behind a Spring property (default on for dev/staging, intended off for prod via env var) and adds a WARN audit line when it fires. Behavior with the toggle ON is unchanged, so pos-service's fail-closed 423 path is untouched (the auto-seed either succeeds and returns OPEN, or the toggle is off and the miss surfaces as PeriodNotFoundException → 423).

Purpose: Preserve the dev-friendly self-heal while giving production an explicit, auditable opt-out — without changing any pos-service behavior (out of scope per RESEARCH.md locked decision 5).
Output: A `@Value`-gated auto-seed branch with a WARN log, the matching env-var-backed default in `application.yml`, and a toggle-off IT.
</objective>

<artifacts_this_phase_produces>
This plan (04) produces:
- New `@Value("${finance.period.auto-seed-on-miss:true}")` boolean field **`autoSeedOnMiss`** on `AccountingPeriodServiceImpl`
- A gated auto-seed branch in **`getPeriodStatus`** + a `log.warn(...)` audit line (new `Logger` on the class)
- New config property **`finance.period.auto-seed-on-miss`** in `application.yml`, backed by env var **`FINANCE_PERIOD_AUTO_SEED_ON_MISS`** (default `true`)
- New integration test class **`AccountingPeriodAutoSeedToggleIT`** (toggle-off → `PeriodNotFoundException` on miss)
