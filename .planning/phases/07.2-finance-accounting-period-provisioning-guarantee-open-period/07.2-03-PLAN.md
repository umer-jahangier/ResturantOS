<objective>
Stop the onboarding saga from silently swallowing a finance-seeding failure, so every tenant that reaches ACTIVE is guaranteed to have had its CoA + accounting periods seeded (FIN-07). Today, `platform-admin-service.ProvisioningService.provision()` Step 5 wraps the finance call in an inner `try/catch` that logs a WARN and continues — Step 6 then marks the tenant ACTIVE regardless, producing an ACTIVE tenant with zero accounting periods (the root of the 423 PERIOD_LOCKED blocker). Compounding this, the flag that even ENABLES the call (`provisioning.seed-coa.enabled`) defaults to `false` in `application.yml`, so Step 5 currently never fires at all.

Purpose: Make finance seeding a required, fail-fast saga step — its failure must propagate to the saga's existing outer catch (which already sets `PROVISIONING_FAILED` and runs compensation). Recovery is via the new self-service endpoint (plan 05), NOT the existing `retry()` (which is unsafe — see threat model / RESEARCH.md Pitfall 1).
Output: Hardened Step 5, corrected flag default (both `@Value` and YAML), a new `stubFinanceSeedCoaFail()` test helper, a per-class-overridable seed-coa flag in the test base, and an integration test proving finance-failure → PROVISIONING_FAILED.
</objective>

<artifacts_this_phase_produces>

- Corrected `provisioning.seed-coa.enabled` default → `true` in `application.yml` (`PROVISIONING_SEED_COA_ENABLED:true`) AND confirmed-consistent `@Value` inline default

| saga recovery path | `retry()` re-runs branch/admin creation with a literal `"(retry)"` admin email — an unsafe recovery route for a finance-only failure |
