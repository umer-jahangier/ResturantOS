# Phase 07.3 Plan 10: KDS Station-Isolated Redesign Summary

**Station-isolated New/Started/Preparing/Ready item-status board (picker -> board -> dedicated
detail page, no Dialog) with a slim collapsed card and subtle escalation-threshold aging driven
by a single shared clock, replacing the old multi-station-columns board and its
animate-bounce/full-red aging (KDS-04/KDS-05)**

## Performance

- **Duration:** ~23 min
- **Started:** 2026-07-12T21:01:10+05:00 (first commit)
- **Completed:** 2026-07-12T21:23:34+05:00 (last commit)
- **Tasks:** 4 (Task 2 TDD: RED + GREEN)
- **Files:** 21 changed (9 created, 10 modified, 2 deleted)

## Accomplishments
- `kitchen/` is now a station picker (`station-picker.tsx`) that auto-navigates when exactly one
  active station exists (the common seeded case, per 07.3-05's DEFAULT-station auto-seed-on-miss)
  and otherwise lists all active stations; the old "No active stations configured" empty-state
  copy is preserved.
- `kitchen/[stationCode]` renders `station-board.tsx` with four `kds-item-column.tsx` columns
  (New=PENDING, Started=ACCEPTED, Preparing=PREPARING+legacy COOKING, Ready=READY), item-centric
  so a mixed-status order (e.g. Burger PREPARING + Fries ACCEPTED + Coke READY) shows a fragment
  in each relevant column. Each fragment exposes a per-item "Move to {next column}" action calling
  the new `useUpdateItemStatus` hook (Task 1), which posts to 07.3-05's explicit item-status
  endpoint and invalidates the `["kds", branchId]` query keys.
- `kds-ticket-card.tsx` slimmed to ONLY order number, table (`tableNumber`, newly added to the
  model/schema/adapter), age, and menu-item names — StatusBadge and bump buttons removed from the
  collapsed view (they live in the column's move action and the detail page now).
- `kitchen/[stationCode]/orders/[ticketId]` is a dedicated detail PAGE (kills the pre-07.3-10
  tap-to-open `Dialog`) rendering `kds-station-detail.tsx`, which reuses the existing
  `kds-ticket-detail.tsx` (extended with an optional `canUpdate` prop for per-item transition
  controls) for the revision-grouped item list + Kitchen Notes callout.
- `useKdsClock`/`KdsClockProvider` (Task 1) is a single shared ticking context — ONE
  `setInterval` for the whole board instead of one per card (KDS-05/D-13, threat T-07.3-31). The
  old `getAgingClasses`' `animate-bounce`/`bg-red-950` full-red treatment is replaced with a
  subtle colored left border + timer chip that turns amber at 0.66x and red at 1.0x the station's
  `escalationThresholdSeconds` (falls back to the DB default of 900s/15min when the station
  hasn't loaded).
- New Wave-0 Playwright spec `frontend/e2e/kds-stations.spec.ts` proves KDS-04 (seeded board, all
  4 columns render, card selection routes to the detail page with no dialog) and the KDS-05
  aging-subtlety backstop (no card ever carries `animate-bounce`/`bg-red-950`; screenshot of the
  oldest visible ticket written to `e2e/__screenshots__/kds05-aged-ticket.png`) — ran live twice
  against the dev stack, both PASS.

## Decisions Made

- `use-kds-clock.ts` renamed to `.tsx` — the `KdsClockProvider` component needs JSX, which a
  `.ts` file cannot parse (`tsc` errored immediately on the arrow-function JSX return).
- Deleted `kds-board.tsx` (the old multi-station-side-by-side `KdsBoard`/`StationColumn`) and its
  test rather than leaving it as unreferenced dead code — once `kitchen/page.tsx` became the
  station picker, nothing in the app rendered `KdsBoard` anymore, and its test exercised the
  aging/bump/Dialog behavior this plan explicitly removes. `sortKdsTickets` was moved into
  `station-board.tsx` unchanged; `kds-board-sort.test.ts`'s import path was updated accordingly.
- `kds-ticket-detail.tsx` (not in the plan's declared file list) was extended with an optional
  `canUpdate` prop instead of duplicating its revision-grouping logic inside
  `kds-station-detail.tsx` — DRY, and the plan's own read_first pointed at reusing this file's
  content.
- The final slim `kds-ticket-card.tsx` design (order#/table/age/item-names + aging border/timer
  chip) was built across Tasks 2 and 3 on the same file rather than adding a placeholder in Task
  2 and reworking it in Task 3 — Task 2 already needed a working age display from the Task-1
  `useKdsClock`, so Task 3 only added the escalation-threshold color computation on top.
