# Phase 07.3 Plan 05: Kitchen-Service Backend — Item-Status Endpoint, Table Number, Station Seeding Summary

**Explicit item-status REST endpoint wrapping markItemStatus, order->event->ticket->DTO table-number propagation with cross-service field-name parity, and runtime DEFAULT-station auto-seed-on-miss so the KDS board is never empty (KDS-04)**

## Performance

- **Duration:** ~20 min
- **Started:** 2026-07-12T14:11:23Z (first commit)
- **Completed:** 2026-07-12T14:28:33Z (last commit)
- **Tasks:** 3 (2 TDD)
- **Files modified:** 13 (3 created, 10 modified)

## Accomplishments
- V5 Flyway migration adds nullable `table_number` to `kds_tickets`; `KdsTicket.tableNumber`,
  `KdsTicketDto.tableNumber`, and `KitchenEventPayloads.OrderSentToKdsPayload.tableNumber`
  (additive last field, exact name parity with pos-service's 07.3-04 producer field) carry it
  end-to-end; `TicketRoutingService` sets it on ticket creation and refreshes it on a later
  revision append if not yet set.
- New `POST /api/v1/kitchen/kds/tickets/{ticketId}/items/{itemId}/status` endpoint wraps
  `TicketService.markItemStatus` (no duplicated transition logic), gated by
  `authz.authorizeUpdate` (`pos.kds.update`), so it inherits the `KITCHEN_ITEM_STATUS_CHANGED`
  pos-sync event emit from 07.3-02 on every transition.
- `TicketRoutingService.ensureStation` idempotently upserts a station row before a ticket is
  created at that station; `KdsController.getStations` auto-seeds a DEFAULT station when a
  branch has zero rows — the board is never empty, closing the live KDS-04 blocker.

## Decisions Made
- `validateTransition` already supported the full PENDING→ACCEPTED→PREPARING/COOKING→READY
  path (landed in 07.3-02) — no change needed there; only the missing REST endpoint was added.
- `KdsStationRepository.findByBranchIdAndCode` already existed — reused directly by
  `ensureStation`, no new repository method needed.
- `ensureStation` is called for every station code a ticket routes to (GRILL, DRINKS, DEFAULT,
  etc.), not only the DEFAULT fallback — any station a ticket lands on becomes visible via
  `GET /stations` immediately, which is a strict superset of the plan's "DEFAULT-only" framing
  and still satisfies every acceptance criterion.
- GitNexus MCP tools were unavailable in this session (not present in the tool list) — fell
  back to Grep/Glob for pre-edit impact analysis per the `mcp_tools` instruction's documented
  fallback; `detect_changes()` was not run before commit for the same reason.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Fixed 4 existing IT files' `OrderSentToKdsPayload` positional constructor calls**
- **Found during:** Task 1 (adding `tableNumber` as the new last field of the record)
- **Issue:** `TicketRoutingIT`, `TicketRevisionRoutingIT`, `OrderVoidedCancelsTicketsIT`, and
  `TicketLifecycleIT` all construct `OrderSentToKdsPayload` positionally; adding a field broke
  compilation (`test-compile` failed) for all 4 files.
- **Fix:** Appended a trailing `null` (no table number) to each positional constructor call —
  no behavior change to those tests.
- **Files modified:** the 4 IT files listed above.
- **Verification:** `mvn -q -pl services/kitchen-service -am test-compile` passes; all 4 files'
  tests re-verified green afterward.
- **Committed in:** `d8e5831` (Task 1 commit).

**2. [Rule 3 - Blocking] Stubbed mocked `StringRedisTemplate.opsForValue()` in both new ITs**
- **Found during:** Task 2 (first test in this service to call `KdsController` directly rather
  than its underlying service beans)
- **Issue:** `KdsController` is class-level `@RequiresFeature("FEATURE_KDS")`. Calling it
  directly through its Spring proxy triggers `RedisFeatureFlagService.isEnabled`, which NPEs
  on the unstubbed `@MockitoBean StringRedisTemplate` (`opsForValue()` returns null by
  default) before authz/business logic ever runs — every prior IT calls services directly and
  never exercised this aspect.
- **Fix:** Added a `mock(ValueOperations.class)` stub in `setUp()` returning `"true"` from
  `get(any())`, in both `ItemStatusEndpointIT` and `KdsStationSeedTableNumberIT`.
- **Files modified:** `ItemStatusEndpointIT.java`, `KdsStationSeedTableNumberIT.java`.
- **Verification:** both IT classes pass (3/3 and 3/3 respectively).
- **Committed in:** `3783eae` (Task 2 GREEN commit); pattern reused directly in Task 3's test.
