# Phase 07.3: POS & Kitchen Production Bug-Fixes & UX Revamp - Context

**Gathered:** 2026-07-12
**Status:** Ready for planning
**Source:** User testing feedback (`bugs.md`) + parallel code investigation + user decision session

<domain>
## Phase Boundary

Production hardening of the **already-shipped Phase-7.1 POS/KDS**, driven by 16 issues found
during testing (`bugs.md`). Scope spans the Next.js `frontend`, `pos-service`,
`kitchen-service`, and (for durable payments) `finance-service`.

**In scope:** order-taking lifecycle (no-draft lazy creation, quantity merge, optional table +
order type, terminal reset & charge-gating), real-time kitchen→POS item-status sync, add-to-
existing-order revision firing from Order Management, full-page settlement with decoupled
payment/close semantics (Paid AND Served ⇒ CLOSED), Order Management completeness (closed/paid
visibility, search, payment-status, item-quantity column, assign-table), the POS operational
modal→page revamp, the connectivity-404 fix, and the KDS station-isolated status-column redesign
with subtle prioritization.

**Out of scope (this phase):** a real reservations feature (Reserved table status deferred),
converting the command-palette or finance modals, any new business module, and any change to the
Phase-7.1 revision/derivation model beyond what these fixes require.
</domain>

<decisions>
## Implementation Decisions

### Order lifecycle
- **D-01** (POS-16): The order-taking terminal holds a **client-only cart** in local React
  state. No DB order is created on menu-item tap. Persistence happens lazily, only on the first
  explicit **Send to Kitchen** or **Charge** action. `DRAFT` is removed from all user-visible
  flows; no empty/abandoned orders may appear in any list.
- **D-02** (POS-17): Repeated taps of the same menu item **merge into one line (×N)** with ±
  controls, keyed on `menuItemId` + modifier set + notes. Differing modifiers/notes ⇒ separate
  line. Re-ordering an already-fired item creates a **new PENDING revision line**, never mutates
  a fired line.
- **D-03** (POS-18): Order creation supports **Dine-in / Takeaway / Pickup**. Table is
  **optional** (takeaway/pickup need none). A **searchable table selector** shows status;
  occupied tables are not selectable. **Reserved is NOT implemented this phase** — show only
  Available/Occupied (no reservations feature exists yet).
- **D-04** (POS-19): After **Send to Kitchen**, the terminal does NOT auto-reset. It shows an
  explicit **Clear / New Order** action so the cashier can start fresh (the fired order stays
  editable in Order Management). **Charge Now is disabled until the order has been sent to the
  kitchen** (guarantees the order is properly persisted before money is taken).

### Kitchen ↔ POS sync
- **D-05** (POS-20): Add a fine-grained kitchen→POS event so **per-item** status changes
  propagate to `pos-service`; the POS reflects live item status without a manual reopen. The
  existing aggregate `ORDER_READY` path is preserved. Cross-service payload **field-name parity
  is mandatory** (a mismatch silently drops messages).
- **D-06** (POS-21): Order Management's detail view gets a revision-aware **"Send New Items (N)"**
  action that fires ONLY newly-added items as a new revision (backend flow already correct — this
  is wiring). Added items appear instantly (seed cache from the mutation response). Order
  Management gets a **manual Refresh** action.

### Settlement
- **D-07** (POS-22 + POS-25): **Charge Now becomes a dedicated full-page / large component**
  (not a small modal) showing order no., table, customer, cashier, order time, item list
  (qty/status/notes), tax/discount/charge breakdown, payment history, payment methods, and
  remaining balance.
- **D-08** (POS-23): **Persist `OrderPayment` rows on payment** and add a GET-payments endpoint.
  Recording a payment sets a **derived payment status** (Unpaid/Partially Paid/Paid) and does
  **NOT** close the order. An order transitions to **CLOSED only when it is BOTH fully Paid AND
  fully Served**, enforced by a single `maybeCloseOrder` seam invoked from **both** the payment
  flow and the mark-served flow. A paid-but-unserved order shows **Paid** with its live status.
- **D-09** (POS-24): Order Management shows **closed/paid orders** (filters + search), a
  **payment-status badge** per row (Paid/Unpaid/Partially Paid/Refunded), replaces the **Cover**
  column with **item quantity** (e.g. "8 Items", optional "5 Items / 8 Qty"), and adds an
  **Assign Table** action (available tables only; blocks occupied; updates order + table status
  immediately). Already-paid orders block duplicate payment but stay accessible for history.

### Revamp & cleanup
- **D-10** (POS-25): The **POS + KDS operational surfaces only** are converted from modals/
  sliders to dedicated full-page or large in-place components: payment, order/table detail
  drawer, void/refund, till open/close, and the KDS ticket detail. **Command palette and finance
  modals are explicitly left as-is.**
- **D-11** (POS-26): **Remove** the mis-targeted `HEAD /api/v1/pos/menu/categories` connectivity
  ping (it hits the Next.js origin, not the gateway, and has no auth). Keep the `navigator.onLine`
  event path. Do NOT add a HEAD handler.

### KDS
- **D-12** (KDS-04): KDS is redesigned into **station-isolated views** with **New / Started /
  Preparing / Ready** columns mapped to `TicketItemStatus` (New=PENDING, Started=ACCEPTED,
  Preparing=PREPARING/legacy COOKING, Ready=READY), item-centric so **mixed item statuses within
  one order** are shown. Collapsed cards show **only order#, table, time, menu items**. Selecting
  a card opens a **dedicated detail page (not a modal)**. Backend additions: an **explicit
  item-status endpoint** (wraps existing `markItemStatus`), **table-number propagation**
  (order→event→ticket→DTO), and **station seeding** (≥ a DEFAULT station per branch — current
  live blocker: the board renders empty because no station rows are ever created).
- **D-13** (KDS-05): Prioritization uses **subtle** indicators — a colored left border + a timer
  chip driven by the station's existing `escalationThresholdSeconds` — not `animate-bounce` /
  full-screen effects. A single shared clock replaces per-card timers so it scales; new arrivals
  get a subtle highlight.

### Claude's Discretion
- Exact route paths for the new full-page surfaces (charge page, KDS station/detail routes),
  component decomposition, and shared-component extraction.
- Whether no-draft persistence uses a new atomic `create-with-items` endpoint vs. sequential
  create+addItem — planner/executor may choose based on transaction/idempotency cleanliness.
- Naming of the new kitchen→POS event and the exact `paymentStatus` enum values/derivation site.
- Test structure and the specific Playwright flows, within the Validation Architecture below.
</decisions>

<specifics>
## Specific Ideas
- Item-status column mapping: New=PENDING, Started=ACCEPTED, Preparing=PREPARING(+COOKING), Ready=READY.
- Payment-status derivation: `sum(OrderPayment.amountPaisa)` vs `Order.totalPaisa` ⇒
  UNPAID / PARTIALLY_PAID / PAID; REFUNDED via settlement status.
- Close condition: `paymentStatus == PAID && derivedStatus == SERVED`.
- Quantity merge key: `menuItemId` + sorted `modifierIds` + `notes`; fired lines excluded from merge.
</specifics>

<deferred>
## Deferred Ideas
- **Reserved table status** and a real reservations feature (D-03) — not this phase.
- Converting the **command palette** and **finance** modals to pages (D-10) — out of scope.
- Separate **"server" vs "cashier"** identity on the order — no such concept exists today; Charge
  page shows cashier only until a server concept is introduced.
- Customer/cashier **name resolution** (currently bare UUIDs) — nice-to-have; may show short IDs
  if name lookup is not wired this phase.
</deferred>
