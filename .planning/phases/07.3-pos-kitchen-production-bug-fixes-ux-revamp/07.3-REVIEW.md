# Code Review — Phase 07.3 (POS & Kitchen Production Bug-Fixes & UX Revamp)

Standard-depth review of production source changed during Phase 07.3, run as two
domain-scoped passes. Full per-finding detail (locations, repro, suggested fixes) lives in
the two detail reports:

- **Backend:** [07.3-REVIEW-backend.md](07.3-REVIEW-backend.md) — 23 files · 4 Critical, 4 Warning, 3 Info
- **Frontend:** [07.3-REVIEW-frontend.md](07.3-REVIEW-frontend.md) — 37 files · 2 Critical, 6 Warning, 2 Info

**Combined: 6 Critical · 10 Warning · 5 Info (21 total).** Advisory — not a completion blocker.

## Critical findings (6)

### Backend
- **BE-CR-01** — `OrderServiceImpl.closeOrder` legacy tender path (`POST /orders/{id}/close`)
  validates `request.payments()` but never persists them as `OrderPayment` rows (only
  `PaymentServiceImpl.recordPayment` saves). Orders closed via the legacy path get empty
  `paymentStatus`/`amountPaidPaisa` and an empty `GET /orders/{id}/payments` — i.e. it
  silently breaks THIS phase's own new settlement read model for that path.
- **BE-CR-02** — `OrderServiceImpl.assignTable` releases the previous table via
  `TableServiceImpl.syncStatusForOrder`, which derives status from the order's own live
  (non-terminal) state, so the vacated table stays stuck OCCUPIED/NEEDS_BUSSING instead of
  AVAILABLE.
- **BE-CR-03** — `PaymentController.RecordPaymentRequest.amountPaisa` has no positivity
  constraint and `recordPayment` has no defense-in-depth check → zero/negative payments can
  be recorded with no refund audit trail.
- **BE-CR-04** — Cross-branch IDOR on KDS ticket-mutation endpoints (new `setItemStatus`,
  plus `bumpItem`/`recallTicket`/`getTicketDetail`): OPA compares a caller-supplied
  `branchId` param to the JWT, but `TicketServiceImpl` loads the ticket by id with only
  tenant-level scoping — never checking the ticket's actual branch.

### Frontend
- **FE-CR-01** — `pos-terminal.tsx` `handleSendToKitchen` has no try/catch and `addItem`
  sends no idempotency key. A mid-flow failure leaves the cart un-cleared / `orderId` null
  with no feedback; a natural cashier retry re-runs the whole add loop → duplicated lines,
  double-billing, duplicate kitchen tickets.
- **FE-CR-02** — `charge-summary.tsx` split-tender `handleRecordPayment` only resets `rows`
  after ALL tenders succeed; if tender 2/2 fails, the already-recorded tender 1 stays and a
  retry records a duplicate payment.

## Warning findings (10)
- **BE-WR** (4): station auto-seed race can 500 unhandled; inconsistent branch-scoping on new
  order-mutation endpoints vs. the explicit check added to `listOrderSummaries`; missing
  `@Valid`/`@NotNull` on `ItemStatusRequest`; unvalidated negative split-tender entries.
- **FE-WR** (6): overpay guard sums typed rows (incl. keyboard-entered negatives) not what is
  submitted; Charge page ignores `VOIDED` orders (only handles CLOSED/PAID/REFUNDED);
  `till-session-bar` open-till `paisa < 0` guard misses `NaN`, close-till has no guard;
  strict `kds.schema` `datetime({offset:true})` throws inside `.map()` and fails the entire
  station-board fetch on one bad record; `queryKeys.pos.orders()` defined but bypassed by
  ~10 hand-rolled literal arrays.

## Info findings (5)
- Backend: misleading `@Transactional` self-invocation; duplicated idempotency boilerplate;
  duplicated magic number.
- Frontend: unused `status` param on KDS query-key builder; Order-Management "Age" column
  doesn't tick like the KDS shared-clock board.
