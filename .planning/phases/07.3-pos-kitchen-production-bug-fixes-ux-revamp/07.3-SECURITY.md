Verification method: L1 (ASVS Level 1) — grep/read-confirmed presence of each declared
mitigation in the implementation files cited by the plan's mitigation text, not documentation
or intent. `block_on: high` — only OPEN threats with severity `high` (or unranked, fail-closed)
block phase advancement; `medium`/`low` opens are tracked but non-blocking.

This is a plan-time-authored register (`register_authored_at_plan_time: true`): all 11 plans
(07.3-01 through 07.3-11) carry `<threat_model>` blocks. Plan 07.3-04 and plan 07.3-11 both

distinct IDs (`T-07.3-11` for the assign-table IDOR threat vs. `T-07.3-11-01/02/03` for the
legacy-close-retirement threats) — treated as distinct threats below, disambiguated by component.

## Recommendation

No BLOCKER. One non-blocking hardening follow-up recommended: add `@Version` (or an explicit
`SELECT ... FOR UPDATE` / conditional `UPDATE ... WHERE status = 'AVAILABLE'`) to `DiningTable`
so `OrderServiceImpl.assignTable`'s AVAILABLE re-check is genuinely race-proof under concurrent
assignment, matching the plan's own stated mitigation text for T-07.3-12.
