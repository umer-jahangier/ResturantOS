<objective>
Add a fine-grained kitchen→POS event so per-item kitchen status changes propagate to pos-service in
real time (POS-20 / D-05). The existing aggregate `ORDER_READY` path is preserved; this adds an
intermediate-state channel: every `markItemStatus` transition in kitchen-service publishes
`KITCHEN_ITEM_STATUS_CHANGED`, and a new pos-service consumer applies it to the matching
`OrderItem`, recomputing the derived order status.

Purpose: Today the POS shows stale item statuses because kitchen-service only tells pos-service the
whole order is done (`ORDER_READY` fires only when nothing is PENDING/COOKING). Intermediate states
(ACCEPTED/PREPARING) and per-item "Done" never reach the POS (RESEARCH POS-20).

Output: the new event record on both sides (field-name parity mandatory), the emit call, the
consumer, the queue topology, and an end-to-end IT proving the message is consumed (not dropped).
</objective>
