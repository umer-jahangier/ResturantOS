# Phase 07.3 Plan 07: Full-Page Charge Route + Payment-Status Badge Summary

**Replaced the `sm:max-w-md` PaymentPanel modal with a dedicated `/app/pos/orders/[orderId]/charge` page showing full order + payment analytics, payment history, and a split-tender "Take Payment" form that records via the decoupled `POST .../payments` seam (never closes directly) — plus a reusable 4-state `PaymentStatusBadge`.**

## Accomplishments
- `payment-status-badge.tsx` — 4-state (PAID/PARTIALLY_PAID/UNPAID/REFUNDED) hue+icon+label
  chip reusing the DS `StatusBadge` visual system, with a non-throwing neutral fallback
  for unrecognized values (registry safety) — TDD RED→GREEN (6 unit tests).
- `derivePaymentStatus()` in `pos.model.ts` — a pure client-side mirror of backend
  `PaymentStatusDerivationService.derive()`, since `GET /orders/{id}` doesn't expose a
  `paymentStatus` field (only the Order Management list row does).
- `useOrderPayments`/`PosRepository.getPayments` (read history) and
  `useRecordPayment`/`PosRepository.recordPayment` (record ONE tender, backend returns
  the running total, not an updated order) — both Zod-parsed at the repository boundary.
- Full-page Charge route (`/app/pos/orders/[orderId]/charge`) + `charge-summary.tsx`:
  header (order no./table/type/customer/cashier/time/status/payment chips), items list,
  money breakdown (subtotal/discounts/service charge/taxes/total/paid/remaining), payment
  history ("No payments yet" empty state), and a split-tender Take Payment form (multiple
  rows submitted sequentially, "Full amount" quick-fill, disabled once the order is
  fully paid — duplicate-payment guard) that renders a 423 PERIOD_LOCKED response as a
  user-facing message instead of crashing.
- `settlement-actions.tsx` rewired: CHARGE NOW now `router.push()`s to the charge route
  instead of opening a `Dialog`+`PaymentPanel`; every existing caller (OrderPanel footer,
  Order/Table Detail drawer footer) preserved unchanged.
- `pos-settlement.spec.ts` S5 reworked into S5 (charge-page navigation, full payment,
  Paid≠Closed-while-unserved) + a new S5b (mark Served → transitions to Closed) — proven
  live against the dev stack this session up to a pre-existing gateway/backend gap (see
  Deviations).

## Decisions Made
- `derivePaymentStatus()` pure-mirrors the backend derivation exactly (REFUNDED settlement
  status wins; paid<=0 -> UNPAID; paid<total -> PARTIALLY_PAID; else PAID) rather than
  extending `OrderDto` server-side, keeping this plan strictly frontend-only per its own
  file list and context note.
- Split-tender payments submit sequentially via `useRecordPayment.mutateAsync` in a loop
  (awaiting each), matching the backend's one-tender-per-call `POST .../payments`
  contract — there is no multi-payment array endpoint outside the legacy `closeOrder`
  exact-tender path.
- The Charge page never calls `closeOrder` directly; it relies entirely on the backend's
  `maybeCloseOrder` seam (07.3-01) firing inside `recordPayment` to auto-close once Paid
  AND Served, matching the plan's "Paid ≠ Closed until Served" truth.
- `payment-panel.tsx` was left in place (unused by `settlement-actions.tsx` after the
  reroute) rather than deleted — not in this plan's file list, and its own dedicated
  test file (`payment-panel.test.tsx`) still exercises it directly and passes unchanged.
