# Phase 07.3 Plan 01: Settlement Backend — Decoupled Payment/Close Summary

**Payments now persist independently of order close; an order transitions to CLOSED only through one `maybeCloseOrder` seam, fired from both the payment path and the mark-served path, when the order is fully Paid AND fully Served — plus a new `GET /orders/{id}/payments` history endpoint.**

## Accomplishments
- `PaymentStatus` enum (UNPAID/PARTIALLY_PAID/PAID/REFUNDED) and a pure, unit-tested
  `PaymentStatusDerivationService.derive(paidPaisa, totalPaisa, settlementStatus)` — REFUNDED
  wins over the sum, overpay clamps to PAID.
- `OrderService.maybeCloseOrder(UUID)` — the single seam that closes an order iff
  `paymentStatus==PAID && derivedStatus==SERVED`, invoked from `PaymentServiceImpl.recordPayment`
  and `OrderServiceImpl.markItemServed`. A no-op when either condition is unmet or the order is
  already terminal.
- Extracted `closeOrder`'s period-lock check / CLOSED transition / table sync / `ORDER_CLOSED`
  publish into a single private `performClose(Order, paymentEntries)` shared by both the legacy
  exact-tender `closeOrder` endpoint and `maybeCloseOrder` — exactly one
  `eventPublisher.publish(POS_EXCHANGE, ORDER_CLOSED_KEY, ...)` call site in the class.
- `GET /api/v1/pos/orders/{id}/payments` (`PaymentService.listPayments`, tenant-scoped exactly
  as `recordPayment`) returning persisted `OrderPayment` history as `OrderPaymentDto`.
- `SettlementSemanticsIT` proves the (paid × served) close matrix: CLOSED only in the
  (paid, served) cell — partial payment on a served order stays open, full payment on an
  unserved order stays open, serving the last line of an already-paid order closes it.

## Decisions Made
- `maybeCloseOrder` no-ops (returns the order unchanged, throws nothing) rather than raising when
  conditions aren't met — needed so it's safe to call unconditionally from both `recordPayment`
  and `markItemServed` on every invocation without an extra pre-check at each call site.
- Shared `performClose(Order, paymentEntries)` private helper is the single close mechanics seam;
  `closeOrder` and `maybeCloseOrder` differ only in how `paymentEntries` is sourced (request
  payload vs. persisted `OrderPayment` rows) — satisfies the plan's "exactly ONE ORDER_CLOSED
  publish" acceptance criterion via `grep`.
- `closeOrder`'s legacy exact-tender path was intentionally left NOT persisting `OrderPayment`
  rows — out of this plan's scope (plan explicitly frames it as "remain callable for the legacy
  exact-tender path"); only `recordPayment` persists payments.
