# Phase 07.3: Code Review Report — Frontend

**Reviewed:** 2026-07-12
**Depth:** standard
**Files Reviewed:** 37
**Status:** issues_found

## Summary

The client-only cart, full-page Charge surface, Order Management settlement columns,
and station-isolated KDS board are all functionally coherent and the vast majority of
the money/status derivation logic (`cart-reducer.ts`, `derivePaymentStatus`,
`getOrderDisplayStatus`, the fade-out list in `order-management.tsx`) is correct,
immutable, and well-guarded against nulls.

However, two retry-safety defects were found in the money-handling paths that this
phase's own domain notes call out as HIGH risk ("race conditions on lazy order
persist + fire-to-kitchen"): the first-Send-to-Kitchen persist flow in
`pos-terminal.tsx` and the multi-tender payment loop in `charge-summary.tsx` are both
non-idempotent on partial failure — a retry after a mid-flow error resubmits work
that already succeeded, producing duplicate order lines or a duplicate payment. Both
are classified Critical because they cause real financial/data-integrity damage
(double-billing / double-added kitchen items) under an ordinary network hiccup, not
just a maliciously-crafted input. A cluster of smaller Warnings covers input
validation gaps (NaN/negative amounts bypassing HTML `min`), a terminal-state gap on
the Charge page for VOIDED orders, an overly strict Zod datetime schema that can fail
the whole KDS board fetch on one bad record, and a query-key registry entry that's
never actually used (every invalidation call site hand-rolls the same literal array
instead).

## Critical Issues

### CR-01: Lazy order persist duplicates cart lines when a mid-flow request fails and the cashier retries

**File:** `frontend/components/pos/pos-terminal.tsx:94-129`

**Issue:** `handleSendToKitchen` has no `try/catch` at all — only a `finally` that
resets `sendInFlightRef`. If `addItem.mutateAsync` throws partway through the
sequential per-line loop (network blip, validation error on one line, etc.):

- The thrown error becomes an **unhandled promise rejection** — the caller invokes
  this via `onClick={() => void onSendToKitchen()}` in `order-panel.tsx`, so nothing
  catches it and the cashier gets **zero feedback** (no toast, no inline error —
  `createOrder`/`addItem`'s `isError` state is never rendered anywhere in
  `PreSendCart`).
- `setOrderId` and `setCart(clearCart())` are only reached *after* the loop
  completes, so on a mid-loop failure the cart is left fully populated and
  `orderId` stays `null` — the UI looks exactly like nothing happened, and
  `canSend` becomes `true` again once `isPersisting` clears, inviting a retry.
- On retry, `createOrder.mutateAsync` is called again with the **same**
  `clientOrderIdRef.current` — that part is correctly idempotent via the
  `Idempotency-Key` header (`pos.repository.ts` `createOrder`), so the server
  returns the *same* order rather than creating a duplicate.
- But the `for (const line of cart) { await addItem.mutateAsync(...) }` loop then
  replays **every line in the (unchanged, still-full) cart** — including the lines
  that were already successfully added on the first attempt. `PosRepository.addItem`
  sends **no idempotency key at all** (unlike `createOrder`/`sendToKds`/
  `closeOrder`/`voidOrder`/`refundOrder`, which all set `Idempotency-Key`). The
  result is duplicate `OrderItem` rows on the server order — the guest is billed
  twice for the already-added items and the kitchen ticket shows duplicated
  quantities.

This is exactly the "retry duplicates already-successful sub-steps" failure mode the
file's own comments claim is solved ("ONE clientOrderId per cart lifetime —
preserves idempotency across retries of the SAME persist attempt") — that claim is
only true for the `createOrder` step, not for the `addItem` loop that follows it.

**Fix:** Wrap the whole flow in `try/catch` with user-facing error feedback, and make
the retry path resumable instead of full-replay. At minimum: track which cart lines
have already been confirmed added (e.g. an index/cursor into `cart`, only advanced
after each successful `addItem`), and set `orderId` as soon as `createOrder` resolves
so a retry never re-runs `createOrder` for lines that are already committed:

```tsx
const handleSendToKitchen = useCallback(async () => {
  if (cart.length === 0 || sendInFlightRef.current) return;
  sendInFlightRef.current = true;
  try {
    const newOrder = orderId
      ? { id: orderId } // already created by a previous attempt
      : await createOrder.mutateAsync({ branchId, clientOrderId: clientOrderIdRef.current, ... });
    if (!orderId) setOrderId(newOrder.id);

    // Only add lines not yet confirmed on the server — never replay the whole cart.
    const alreadyAdded = addedLineKeysRef.current;
    for (const line of cart) {
      const key = cartLineKey(line.menuItemId, line.modifierIds, line.notes);
      if (alreadyAdded.has(key)) continue;
      await addItem.mutateAsync({ orderId: newOrder.id, payload: { ... } });
      alreadyAdded.add(key);
    }

    setCart(clearCart());
    await fireToKitchen.mutateAsync({ orderId: newOrder.id });
  } catch {
    toast.error("Failed to send order to kitchen. Please try again.");
  } finally {
    sendInFlightRef.current = false;
  }
}, [cart, orderId, createOrder, addItem, fireToKitchen, branchId, orderType, selectedTableId]);
```

(A fuller fix should also add an `Idempotency-Key` to `PosRepository.addItem` server
-side so a duplicate call is safe regardless of client-side bookkeeping — the client
-side cursor above is the minimum fix for the reported symptom.)

### CR-02: Split-tender payment recording resubmits already-recorded tenders on retry

**File:** `frontend/components/pos/charge-summary.tsx:138-160`

**Issue:** `handleRecordPayment` submits every tender row sequentially:

```tsx
for (const row of toSubmit) {
  await recordPayment.mutateAsync({ ... });
}
toast.success(...);
setRows([newTenderRow()]);
```

`setRows([newTenderRow()])` (the only place `rows` is reset) is only reached **after
every row in the loop has succeeded**. If tender row 2 of 2 fails (e.g. the
423 `PERIOD_LOCKED` case this file's own `getRecordPaymentErrorMessage` explicitly
handles, or any transient network error), the `catch` block sets `recordError` but
**never removes or marks row 1**, which already recorded a real payment against the
order server-side. The `rows` state — and therefore each row's `amountPaisa` —
is left completely unchanged.

When the cashier sees the error and clicks "Record Payment" again (a natural
recovery action), `toSubmit = rows.filter((r) => r.amountPaisa > 0)` includes **row
1 again**, and it gets submitted to `POST /orders/{id}/payments` a second time —
recording a duplicate tender and overstating the guest's amount paid (and, per the
`maybeCloseOrder` seam described in this file's own comments, potentially closing
the order on a phantom "fully paid" balance).

**Fix:** Make the loop resumable — drop each row from state as soon as it is
confirmed recorded, so a retry only resubmits the tenders that actually failed:

```tsx
const handleRecordPayment = async () => {
  setRecordError(null);
  const toSubmit = rows.filter((r) => r.amountPaisa > 0);
  if (toSubmit.length === 0) return;

  try {
    for (const row of toSubmit) {
      await recordPayment.mutateAsync({
        method: row.method,
        amountPaisa: row.amountPaisa,
        referenceNo: row.referenceNo || null,
      });
      // Remove the just-recorded row immediately — a retry after a later
      // row's failure must never resubmit an already-successful tender.
      setRows((prev) => prev.filter((r) => r.id !== row.id));
    }
    toast.success(toSubmit.length > 1 ? "Payments recorded" : "Payment recorded");
    setRows((prev) => (prev.length === 0 ? [newTenderRow()] : prev));
  } catch (err) {
    const shaped = err as { status?: number; message?: string } | undefined;
    setRecordError(getRecordPaymentErrorMessage(shaped));
  }
};
```

## Warnings

### WR-01: Overpay guard can pass while the amount actually submitted exceeds the remaining balance

**File:** `frontend/components/pos/charge-summary.tsx:124-127, 337-347`

**Issue:** The `<input type="number" min={0} ...>` for a tender's amount is
"min-guarded" only via the HTML attribute, which does **not** block keyboard entry
of a negative value (`min` only affects stepper buttons / constraint-validation
state, not what can be typed). The `onChange` handler does no clamping:

```tsx
onChange={(e) => updateRow(row.id, { amountPaisa: parseInt(e.target.value || "0", 10) || 0 })}
```

`canRecord`'s overpay guard uses `tenderTotalPaisa`, the sum of **all** rows
(`rows.reduce((acc, r) => acc + r.amountPaisa, 0)`), including any negative-amount
row. But `handleRecordPayment` only submits rows where `amountPaisa > 0`
(`rows.filter((r) => r.amountPaisa > 0)`). So a row with a negative amount lowers
`tenderTotalPaisa` (making the `tenderTotalPaisa <= remainingPaisa` guard pass) while
being silently excluded from what's actually POSTed — the real submitted total can
exceed `remainingPaisa` even though the UI's own guard says it's within bounds.

**Fix:** Clamp negative input in `updateRow`/`onChange`, and compute the overpay
guard from the same set that's actually submitted:

```tsx
onChange={(e) =>
  updateRow(row.id, { amountPaisa: Math.max(0, parseInt(e.target.value || "0", 10) || 0) })
}
```
```tsx
const submittableTotalPaisa = rows
  .filter((r) => r.amountPaisa > 0)
  .reduce((acc, r) => acc + r.amountPaisa, 0);
const canRecord =
  !blocksNewTenders && !isClosed && hasValidTenders && remainingPaisa > 0 &&
  submittableTotalPaisa <= remainingPaisa;
```

### WR-02: Charge page doesn't guard against a VOIDED order — only CLOSED/PAID/REFUNDED are blocked

**File:** `frontend/components/pos/charge-summary.tsx:119-122, 313-320`

**Issue:** `blocksNewTenders` only checks `paymentStatus === "PAID" || paymentStatus
=== "REFUNDED"`, and the only settlement-status check anywhere on the page is
`isClosed = order?.status === "CLOSED"`. `order.status === "VOIDED"` is never
checked, so a voided order (which typically has `paidPaisa` still `0`, so
`paymentStatus` derives to `"UNPAID"`) falls straight through to the live "Take
Payment" form — the cashier can fill in a tender and press Record Payment. The page
relies entirely on `PaymentServiceImpl.recordPayment` rejecting the call server-side
(per this file's own doc-comment), so no money is actually lost, but the UX is
wrong: the cashier gets a generic/confusing failure instead of a
"this order was voided" message, and every other terminal state (`PAID`,
`REFUNDED`, `CLOSED`) does get a friendly, dedicated message.

**Fix:** Fold `VOIDED` into the terminal-state check alongside `isClosed`:

```tsx
const isTerminal = isClosed || order.status === "VOIDED";
// ...
{blocksNewTenders ? (
  ...
) : isTerminal ? (
  <p className="text-sm text-muted-foreground">
    This order {order.status === "VOIDED" ? "was voided" : "is closed"} — no payment can be taken.
  </p>
) : ( ... )}
```

### WR-03: `handleOpenTill` doesn't guard against non-numeric input (NaN bypasses the negative check)

**File:** `frontend/components/pos/till-session-bar.tsx:50-56`

**Issue:**

```tsx
const paisa = Math.round(parseFloat(openingFloat || "0") * 100);
if (paisa < 0) return;
await openTillMutation.mutateAsync({ branchId, openingFloatPaisa: paisa });
```

If `openingFloat` parses to `NaN` (any non-numeric leftover state), `paisa` is `NaN`.
`NaN < 0` evaluates to `false` in JavaScript, so the guard does **not** stop the
submission — `NaN` is sent as `openingFloatPaisa`, which `JSON.stringify` serializes
to `null`, silently sending a missing/invalid required field to the backend instead
of failing fast with a client-side message.

**Fix:** Use `Number.isFinite` (or `Number.isNaN`) explicitly:

```tsx
if (!Number.isFinite(paisa) || paisa < 0) return;
```

### WR-04: `handleCloseTill` has no negative/NaN amount guard at all (inconsistent with `handleOpenTill`)

**File:** `frontend/components/pos/till-session-bar.tsx:58-69`

**Issue:** `handleOpenTill` (even with the NaN gap above) at least attempts a
`paisa < 0` guard before submitting. `handleCloseTill` computes the same way
(`Math.round(parseFloat(declaredCash || "0") * 100)`) but has **no guard at all** —
a negative or NaN declared-cash amount is submitted straight to
`closeTillMutation.mutateAsync` with no client-side check.

**Fix:** Mirror the (corrected) guard from `handleOpenTill`:

```tsx
const handleCloseTill = async () => {
  if (!activeTill) return;
  const paisa = Math.round(parseFloat(declaredCash || "0") * 100);
  if (!Number.isFinite(paisa) || paisa < 0) return;
  ...
};
```

### WR-05: Strict Zod `datetime({ offset: true })` on KDS timestamps can fail the entire board fetch on one bad record

**File:** `frontend/lib/api-client/schemas/kds.schema.ts:37-39`

**Issue:** `apiKdsTicketSchema.receivedAt/startedAt/readyAt` all use
`z.string().datetime({ offset: true })`, which **requires** a timezone offset in the
string and throws a `ZodError` on parse otherwise. `KdsRepository.getTickets` maps
the whole response array through `apiKdsTicketSchema.parse(item)`:

```ts
return rawList.map((item) => adaptKdsTicket(apiKdsTicketSchema.parse(item)));
```

Since `.parse()` throws synchronously, a single malformed/legacy record anywhere in
the page (e.g. a timestamp emitted without an offset by an older kitchen-service
build, or a data-migration artifact) fails the whole `.map()` call — the entire
station board's ticket list query goes to `isError`, and **no tickets render for any
ticket**, not just the offending one. This is a much higher blast radius than the
per-item defensive-fallback pattern used elsewhere in this same file
(`kdsStatus`/`orderNotes`/`tableNumber` are all `.optional()`/`.nullable()`
specifically to avoid exactly this failure mode).

**Fix:** Either relax to `z.string()` (adapter already does `new Date(raw.receivedAt)`,
which tolerates offset-less ISO strings) or catch per-item parse failures and skip/
report the one bad ticket instead of failing the whole page:

```ts
const rawList = response.data.content ?? [];
const tickets: KdsTicket[] = [];
for (const item of rawList) {
  const parsed = apiKdsTicketSchema.safeParse(item);
  if (parsed.success) tickets.push(adaptKdsTicket(parsed.data));
  // else: log/report and skip, rather than failing the whole board
}
return tickets;
```

### WR-06: `queryKeys.pos.orders()` registry entry is defined but never used — every call site duplicates the literal instead

**File:** `frontend/lib/hooks/query-keys.ts:25-26`, `frontend/lib/hooks/pos/use-orders.ts` (7+ sites), `frontend/lib/hooks/pos/use-payments.ts` (4+ sites)

**Issue:** `query-keys.ts` defines `pos.orders(branchId, statuses?)` explicitly as the
registry entry for this key, with a module comment stating the registry exists so
"a branch switch can invalidate cleanly." In practice, **no hook anywhere in
`use-orders.ts`/`use-payments.ts` calls `queryKeys.pos.orders(...)`** — every
`invalidateQueries` call for this key hand-writes the literal array instead:

```ts
queryClient.invalidateQueries({ queryKey: ["pos", branchId, "orders"] });
```

This literal is duplicated across at least 10 call sites (`useCreateOrder`,
`useAddItem`, `useSendToKds`, `useAssignTable`, `useCloseOrder`, `useVoidOrder`,
`useRefundOrder`, `useRecordPayment`, ...). Bypassing the typed helper defeats its
entire purpose: a typo in any one of these literals (e.g. `"order"` vs `"orders"`,
or a missing `branchId` segment) would silently break cache invalidation for that
one call site with no type error to catch it, and the registry itself becomes
misleading — a maintainer reading `query-keys.ts` would reasonably assume this key
is actually produced by the function next to it.

**Fix:** Either call `queryKeys.pos.orders(branchId)` at every invalidation site (and
drop the unused `statuses` param since it's never passed either), or delete the
unused registry entry and document that this key is a fixed 3-segment prefix by
convention. Prefer the former for type-safety:

```ts
queryClient.invalidateQueries({ queryKey: queryKeys.pos.orders(branchId) });
```

## Info

### IN-01: `queryKeys.kds.tickets`'s third `status` parameter is declared but never passed by any caller

**File:** `frontend/lib/hooks/query-keys.ts:56-57`, `frontend/lib/hooks/kds/use-kds-tickets.ts:15-16`

**Issue:** `kds.tickets: (branchId, stationCode?, status?) => [...]` accepts a
`status` segment, but `useKdsTickets` only ever calls
`queryKeys.kds.tickets(branchId, stationCode)` — `status` is always `undefined` in
the cache key even though `KdsRepository.getTickets` internally defaults to
`"PENDING,COOKING"` and could in principle be called with a different status filter
in the future. Not currently a correctness bug (there's only one call site and it's
always the same effective status), but it's a latent trap: if a future caller adds a
second `useKdsTickets` variant with a different status filter without also updating
the query key, both variants will collide on the same cache entry.

**Fix:** Either pass the actual status string through to the key today, or drop the
unused parameter until it's needed.

### IN-02: Order Management's "Age" column doesn't tick — it only updates on refetch/re-render

**File:** `frontend/components/pos/order-management.tsx:50-60`

**Issue:** `formatAge(openedAt)` computes `Date.now() - openedMs` at render time only.
Unlike the KDS board's shared `useKdsClock` (a 10s ticking context specifically built
so age displays stay live), `useOrderSummaries()` has no `refetchInterval`, so the
displayed order age visibly goes stale between the initial load and the next manual
"Refresh" click or filter change — a 2-minute-old order can sit on screen reading
"2m" for an arbitrarily long session. This is likely acceptable given POS-21's
explicit manual-refresh design, but is worth confirming intentional given the KDS
board in the same phase solved the identical problem with a shared clock.

**Fix (optional):** Reuse a lightweight ticking hook (or the existing
`KdsClockProvider` pattern) if live-updating age is actually desired here; otherwise
no action needed if manual refresh is the intended UX.
