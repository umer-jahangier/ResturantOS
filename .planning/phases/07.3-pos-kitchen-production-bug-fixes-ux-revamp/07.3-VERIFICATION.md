# Phase 07.3: POS & Kitchen Production Bug-Fixes & UX Revamp Verification Report

**Phase Goal:** Turn the Phase-7.1 POS/KDS into a production-grade surface by fixing the 16
issues from testing (`bugs.md`): eliminate draft-order persistence, merge item quantities,
make table optional with an order-type selector, reset the terminal after send with charge
gated on send, propagate per-item kitchen status back to POS in real time, fire only
newly-added items from Order Management as a revision, replace modal-heavy flows with
dedicated full-page/large views, decouple payment from close so an order closes only when
Paid AND Served, surface closed/paid orders with search + payment status + item-quantity +
assign-table, remove the stray connectivity 404, and redesign the KDS into
station-isolated New/Started/Preparing/Ready item-status columns with subtle
prioritization — WITHOUT regressing the Phase-7.1 revision/derivation model or the
cross-service messaging contracts.

### Gaps Summary

One BLOCKER-tier gap: the legacy `POST /orders/{id}/close` endpoint in pos-service still
transitions orders to CLOSED without checking the Served precondition and without
persisting `OrderPayment` rows, directly contradicting the phase's own stated,
unconditional cross-cutting truth ("an order closes only when Paid AND Served") and
corrupting this same phase's new settlement read model for any order closed via that path.
The phase's own frontend never calls this endpoint anymore (its sole caller,
`PaymentPanel`, is now orphaned dead code, superseded by the new Charge page), so the
delivered UX is not observably broken — but the backend contract itself remains open and
is a genuine, not merely cosmetic, goal gap given how explicitly the roadmap frames this
invariant. Recommend either gating `closeOrder` through the same `maybeCloseOrder`
Paid-AND-Served semantics (persisting `OrderPayment` rows along the way) or formally
deprecating/removing the endpoint now that it has been fully superseded.

Five additional Critical findings from the advisory review (BE-CR-02/03/04, FE-CR-01/02)
were assessed individually and classified as production-hardening beyond this phase's
literal stated must-haves rather than goal gaps — each is either UI-unreachable in its
buggy branch (BE-CR-02), an input-validation/security gap orthogonal to the phase's stated
truths (BE-CR-03/04), or a retry-safety concern not covered by any stated truth (FE-CR-01/
FE-CR-02, though FE-CR-01 has corroborating live-E2E evidence and is worth prioritizing in
a fast-follow). All are recommended for a dedicated hardening pass
(`/gsd-code-review 07.3 --fix` or a follow-up gap-closure phase) but do not block this
phase's goal achievement on their own.
