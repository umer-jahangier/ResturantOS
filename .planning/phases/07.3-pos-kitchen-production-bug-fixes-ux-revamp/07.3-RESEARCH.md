# Phase 07.3 Research — POS & Kitchen Production Bug-Fixes & UX Revamp

**Researched:** 2026-07-12
**Method:** 4 parallel code-reading passes over `frontend`, `pos-service`, `kitchen-service`,
`finance-service`. Full detail (every file:line) in
`.planning/research/pos-kitchen-bugfix-investigation.md` — this file is the planner-facing
summary and is authoritative for scope.

## Key finding (shapes the whole plan)

The backend is **far more capable than the UI uses**. Item-level status, kitchen-ticket
revisions, optional tables, per-item transitions (`markItemStatus`), and station routing already
exist server-side. Most fixes are **frontend wiring + one missing cross-service event + one
settlement-flow change**. Three threads tie the 16 bugs together:
1. The cart is **server-authoritative from the first tap** → POS-16, POS-17, POS-19.
2. Kitchen→POS emits only a **coarse aggregate `ORDER_READY`** → POS-20; the Order-Management
   drawer has **no fire CTA** → POS-21.
3. `OrderSummaryDto` is **deliberately thin** + everything is a **modal** → POS-22, POS-24, POS-25.

## Per-requirement findings (root cause → approach → files → risk)

### POS-16 — No-draft lazy order creation
- **Root cause:** `frontend/components/pos/pos-terminal.tsx:52-82` `ensureOrderId()` fires
  `useCreateOrder` on first tap → `POST /pos/orders` → `OrderServiceImpl.java:112-145` sets
  `status=DRAFT` (:127) and saves (:143). `listOrderSummaries` default filter **includes DRAFT**
  (`OrderServiceImpl.java:427-431`) so empty orders leak.
- **Approach:** local React-state cart in `pos-terminal.tsx`; persist lazily on Send/Charge
  (createOrder + addItem, or new atomic `create-with-items`). Backend: exclude DRAFT from
  summaries default; ideally retire DRAFT as a persisted state.
- **Risk: HIGH** — preserve `clientOrderId` idempotency (`OrderServiceImpl.java:115-118`, unique
  constraint `Order.java:92`), table occupancy `syncStatusForOrder` (:220), `ORDER_CREATED`
  event (:228-243), offline stub path (`use-orders.ts:66-79`).

### POS-17 — Quantity merge + ± controls
- **Root cause:** every tap = `addItem` qty 1 (`pos-terminal.tsx:80-89`); backend always
  `new OrderItem()` (`OrderServiceImpl.java:169,208`); no quantity endpoint; static `×{qty}`
  render (`order-panel.tsx:309-313`).
- **Approach:** merge in local cart on `menuItemId` + sorted `modifierIds` + `notes`; ± controls.
  Fired lines excluded from merge (re-order ⇒ new PENDING line; `sendToKds` fires only PENDING,
  `OrderServiceImpl.java:326-344`). Modifier equality on `modifierIds` (snapshots are weak, :185-191).
- **Risk: MEDIUM.**

### POS-18 — Optional table + order type
- **Root cause:** backend `tableId` already optional (`CreateOrderRequest.java:12`, bound only if
  non-null :132-134). Mandatoriness is UX: only entry is AVAILABLE floor-tile tap
  (`table-floor-view.tsx:37-43`), `pos-terminal.tsx:61` hardcodes `DINE_IN`. No RESERVED status.
- **Approach:** order-type toggle + searchable selector (Available/Occupied; occupied disabled).
  Reserved deferred (D-03).
- **Risk: LOW–MED** — `page.tsx:73` remounts terminal via `key={selectedTableId}`.

### POS-19 — Terminal reset & charge gating
- **Root cause:** `activeOrderId` never cleared after send/charge (`pos-terminal.tsx:22,41,45`).
- **Approach:** explicit **Clear / New Order** action; reset only on confirmed success. Gate
  Charge Now on order-sent state.
- **Risk: LOW.**

### POS-20 — Real-time kitchen→POS item-status sync
- **Root cause:** frontend already refetches on open (`useOrder` staleTime:0). Data is stale
  because pos-service is never told: `TicketServiceImpl.markItemStatus` (`kitchen-service:53-88`)
  updates only kitchen state + KDS socket; publishes back only via `checkAndPublishOrderReady`
  (:158-178) which emits `ORDER_READY` **only when the whole order is done**. Per-item + intermediate
  states never propagate.
- **Approach:** emit `KITCHEN_ITEM_STATUS_CHANGED {orderId, orderItemId, newStatus, revisionNo}`
  on every transition; new pos-service consumer maps `orderItemId`→`OrderItem` (join key present:
  `TicketRoutingService.buildItems:122`), sets `itemStatus`, recomputes `derivedStatus` via
  `OrderStatusDerivationService`. Frontend: modest `refetchInterval` on `useOrder`.
- **Risk: MEDIUM** — messaging topology + field-name parity (silent-drop warning
  `KitchenEventPayloads.java:17-20`), idempotency, don't downgrade SERVED/CANCELLED (reuse
  `ELIGIBLE_FOR_READY` guard `OrderReadyConsumer.java:45-47`).

### POS-21 — Add-to-existing fires revision from Order Management
- **Root cause:** backend revision flow fully correct (`sendToKds:295-392` fires only PENDING,
  stamps revision, publishes only new items; `TicketRoutingService.route/appendToExistingTicket:53-94`
  appends). BUT the drawer renders only QuickAddSearch + Instructions + SettlementActions
  (`order-table-detail-drawer.tsx:123-136`) — **no Send-to-Kitchen CTA** (only in
  `order-panel.tsx:75,152-158`). Added items stay PENDING forever.
- **Approach (frontend-only):** wire `useSendToKds` + revision-aware "Send New Items (N)" into the
  drawer; seed cache from mutation response (`useAddItem` discards it, `use-orders.ts:127-131`;
  repo returns it `pos.repository.ts:111-114`); add manual Refresh to Order Management.
- **Risk: LOW — no backend change.**

### POS-22 — Full-page settlement surface
- **Root cause:** `settlement-actions.tsx:69-76` mounts `PaymentPanel` in `Dialog sm:max-w-md`;
  panel renders only totals + split-tender (`payment-panel.tsx:54,116-161`). Most fields already
  on `Order` but unused. Missing: payment history (no persistence/endpoint), names (bare UUIDs),
  no "server" concept.
- **Approach:** dedicated full-page charge route with full header + items + breakdown + history +
  split-tender + remaining. Backend: `GET /orders/{id}/payments`.
- **Risk: MEDIUM** — `PaymentPanel` shared by `order-panel.tsx:159`, `order-table-detail-drawer.tsx:135`.

### POS-23 — Settlement semantics (Paid AND Served ⇒ CLOSED)
- **Root cause:** `closeOrder(payments)` records payment + flips to CLOSED atomically
  (`OrderServiceImpl.java:498-592`). Payments are **write-only**: closeOrder never persists
  `OrderPayment`; only `PaymentServiceImpl.recordPayment:35` writes it (never called by FE); no
  GET endpoint; `OrderPaymentRepository.findByOrderId` unused.
- **Approach:** persist `OrderPayment` on payment; `paymentStatus` derived
  (`sum(OrderPayment)` vs `totalPaisa`); single `maybeCloseOrder(order)` closes iff
  `PAID && derivedStatus==SERVED`, invoked from payment path **and** mark-served path
  (`markItemServed`/derivation). FE: Charge shows "Paid ✓" + live status; Closed only when both.
- **Risk: HIGH** — changes settlement state machine (`stateMachine.assertTransition`); interacts
  with finance period-lock (see Dependencies). Duplicate-payment already blocked
  (`PaymentServiceImpl.java:43-48`).

### POS-24 — Order Management completeness
- **Root cause:** `useOrderSummaries` called with no `statuses` (`order-management.tsx:109`);
  backend defaults to non-terminal (`OrderServiceImpl.java:427-431,694`). `STATUS_FILTERS` only 4
  derived values (:29-35); no settlement status on `OrderSummary`; no search; `covers` column
  (:156-160); no assign-table anywhere (no route/hook/selector); `toSummaryDto` has `getItems()`
  (:698-710) so item-qty is addable; note `getTableNumber` vs `tableName` accessor mismatch (:442).
- **Approach:** extend `OrderSummaryDto` (settlement status + paymentStatus + amountPaid +
  itemQuantity + distinctItemCount); Closed/Paid filters + search + payment badges; item-qty
  column; `PATCH /orders/{id}/table` via `syncStatusForOrder` + `useAssignTable` + selector reuse.
- **Risk: MED** — `useFadeOutList` invariant assumes non-closed rows never disappear
  (`order-management.tsx:58-105`).

### POS-25 — POS operational modal→page revamp
- **Inventory (convert):** payment `Dialog` (`settlement-actions.tsx:69-76`); order/table detail
  Dialog-drawer (`order-table-detail-drawer.tsx:74-140`); void/refund hand-rolled modal
  (`void-refund-dialog.tsx:95-215`); till open/close modals (`till-session-bar.tsx:89-128,161-239`);
  KDS ticket detail Dialog (`kds-ticket-card.tsx:77-100`). **Leave:** command palette
  (`command-palette.tsx`), finance modals, transition overlays.
- **Approach:** dedicated pages / large in-place panels. Only `components/ui/dialog.tsx` exists
  (no Sheet/Drawer primitive).

### POS-26 — Connectivity 404 fix
- **Root cause:** `frontend/lib/offline/use-online-status.ts:5,38-42` pings **relative**
  `/api/v1/pos/menu/categories` (hits Next origin :3000, no `next.config.ts` rewrite) with no
  auth, HEAD. Real calls go via axios `apiClient` :8080.
- **Approach:** delete ping + interval; keep `navigator.onLine` path. Do NOT add HEAD handler.
- **Risk: VERY LOW.**

### KDS-04 — Station-isolated status-column redesign
- **Root cause / readiness:** backend ~70% ready — `KdsStation` (with
  `escalationThresholdSeconds` 900s), `KdsTicket` (one per order×station), `KdsTicketItem` with a
  persisted `status: TicketItemStatus` column (`V1__kitchen_schema.sql:62`), station routing
  (`TicketRoutingService.route:54-70`), general `markItemStatus` (`TicketServiceImpl.java:53-88`,
  NOT exposed via REST — only 2-step `bumpItem` :105-110). Current UI: stations-as-columns
  (`kds-board.tsx:71`), heavy card, **Dialog** detail (`kds-ticket-card.tsx:77-105`). **LIVE
  BLOCKER:** no station rows are ever seeded → `GET /stations` empty → board shows "No active
  stations configured." Table number is absent end-to-end.
- **Approach:** routes `kitchen/` (station picker) → `kitchen/[stationCode]` (New/Started/
  Preparing/Ready item columns) → `kitchen/[stationCode]/orders/[ticketId]` (detail page — kills
  Dialog). Columns map to `TicketItemStatus`. Slim card. Backend: explicit item-status endpoint
  (wraps `markItemStatus`), table-number propagation (pos→event→ticket+column via new Flyway→DTO),
  station seeding (≥ DEFAULT per branch).
- **Risk: HIGH** — cross-service contract + station-seed blocker.

### KDS-05 — Kitchen prioritization
- **Root cause:** sort `receivedAt` asc FIFO (`kds-board.tsx:18-26`); aging uses `animate-bounce`
  + full red (`getAgingClasses:26-35`); per-card `setInterval` (`kds-ticket-card.tsx:17-24`) won't scale.
- **Approach:** subtle left-border + timer chip driven by `escalationThresholdSeconds`; single
  shared clock; new-arrival highlight; wire existing (never-set) `priority` flag.
- **Risk: LOW.**

## Dependencies & risks
- **Finance period-lock precondition:** persisting payments / closing hits `finance-service`;
  fresh tenants 423 `PERIOD_LOCKED`. Treat "open accounting period exists" as a **precondition**
  for POS-22/23 testing. Phase 07.2 (in-flight) provides period provisioning — coordinate.
- **Station-seed blocker (KDS-04):** board renders empty without seeded stations — must ship.
- **Cross-service messaging parity (POS-20, KDS-04):** highest-risk — exact field-name match.
- **Shared surfaces:** `PaymentPanel` and the detail drawer are reused in 2–3 call sites — the
  modal→page conversion must preserve all callers.

## Validation Architecture

The phase goal is validated at four levels; every requirement has at least one **observable**
proof. Plans MUST carry these into `must_haves.truths` and `<verify>` blocks.

- **Backend unit / service tests (pos-service, kitchen-service):**
  - POS-16: `listOrderSummaries` excludes DRAFT (assert query filter); no order row exists after
    a "cart only" interaction path.
  - POS-17: cart-merge key logic (if any server-side) / `addItem` still creates distinct lines
    for differing modifiers.
  - POS-23: `paymentStatus` derivation (Unpaid/Partial/Paid) unit test; `maybeCloseOrder` closes
    **only** when `PAID && SERVED` — table-driven test over the 4 combinations
    (paid×served) asserting CLOSED only in the (paid,served) cell.
  - POS-20: pos consumer maps `orderItemId`→item and sets status; does NOT downgrade
    SERVED/CANCELLED; idempotent on duplicate delivery.
  - KDS-04: explicit item-status endpoint transitions New→Started→Preparing→Ready via
    `markItemStatus`; ticket auto-promote/`ORDER_READY` still fire.
- **Integration tests (Spring `@SpringBootTest` / Testcontainers, mirroring Phase-7.1 ITs):**
  - POS-20 end-to-end event: kitchen `markItemStatus` publishes `KITCHEN_ITEM_STATUS_CHANGED`;
    pos consumer updates `OrderItem.itemStatus` + recomputes `derivedStatus` (field-name parity
    asserted — the message is actually consumed, not dropped).
  - POS-22/23: `GET /orders/{id}/payments` returns persisted rows; close blocked until served.
  - POS-24: `OrderSummaryDto` carries settlement status + paymentStatus + itemQuantity; assign-
    table endpoint rejects OCCUPIED and flips table to OCCUPIED via `syncStatusForOrder`.
  - KDS-04: station seeding present (a DEFAULT station exists for a fresh branch); table number
    flows onto the ticket DTO.
- **Frontend component / unit tests (Vitest/RTL, per existing `__tests__`):**
  - POS-16/17/19: cart reducer — no persistence on add; merge to ×N; ± adjust; Clear resets;
    Charge disabled until sent.
  - POS-24: payment-status badge rendering; item-quantity column; filters include Closed.
  - POS-26: `use-online-status` no longer issues the HEAD ping (no fetch to the categories path).
- **E2E (Playwright — user preference; extend `frontend/e2e`):**
  - POS-16: tapping items creates NO order (assert Order Management shows none) until Send/Charge.
  - POS-20: two-context test — kitchen bumps an item → POS order detail reflects it live without reload.
  - POS-21: add "Fresh Lime" to an existing order from Order Management → fires a new revision;
    kitchen receives only the new item.
  - POS-22/23: Charge is a full page; paying an unserved order shows Paid (not Closed); serving
    then closes it.
  - KDS-04: station-isolated board renders (stations seeded), item moves across New→…→Ready
    columns, card opens a detail **page** (URL changes, no dialog role).
- **Manual/observed (documented in SUMMARY):** the subtle-aging visual (KDS-05) and the
  modal→page revamp (POS-25) verified via Playwright screenshots + console-clean check (POS-26).

Where a truth is not cheaply unit-testable (visual subtlety, cross-service timing), the backstop
is an E2E/Playwright assertion plus a screenshot artifact — no silent gaps.

## Recommended plan decomposition (waves)

The planner should organize ~7–9 plans across waves so cross-service contracts land before their
consumers. Suggested shape (planner may refine):
- **Wave 1 (independent quick wins + shared backend):** POS-26 (404); POS-24 backend DTO
  extension (paymentStatus/itemQty) — unblocks the column + badges.
- **Wave 1/2 (order-taking refactor):** POS-16 + POS-17 + POS-19 + POS-18 (local cart is the
  shared substrate — plan together or tightly sequenced).
- **Wave 2 (kitchen↔POS sync):** POS-20 (event + consumer) then POS-21 (drawer fire CTA).
- **Wave 2/3 (settlement):** POS-23 (persist payments + maybeClose) → POS-22 (full-page charge).
- **Wave 3 (order management UI):** POS-24 frontend (closed/search/badges/assign-table) + POS-25
  remaining modal→page conversions.
- **Wave 3 (KDS):** KDS-04 (backend seed/endpoint/table-prop + frontend station-column redesign)
  then KDS-05 (prioritization).

Sequence rationale: shared cart substrate first; cross-service events before their UI consumers;
settlement backend before its page; KDS seed/endpoint before its board.
