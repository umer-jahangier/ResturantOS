# Phase 07.3 Plan 02: Kitchen→POS Fine-Grained Item Status Sync Summary

**KITCHEN_ITEM_STATUS_CHANGED event on every kitchen markItemStatus transition, consumed live by a new pos-service KitchenItemStatusConsumer that recomputes derived order status via the single derivation seam.**

## Accomplishments
- kitchen-service's `TicketServiceImpl.markItemStatus` now publishes a fine-grained `KITCHEN_ITEM_STATUS_CHANGED` event (routing key `kitchen.item.status-changed`) on EVERY valid per-item transition — not just at READY — while leaving the existing aggregate `ORDER_READY` path completely untouched.
- pos-service now has a `KitchenItemStatusConsumer` (mirroring `OrderReadyConsumer`'s shape) that maps the kitchen status onto `OrderItemStatus`, applies it only when it is a genuine forward move (ordinal-based guard), recomputes `Order.derivedStatus` via `OrderStatusDerivationService`, and syncs table status — all gated by `processed_events` idempotency.
- New `pos.kitchen-item-status.queue` RabbitMQ topology binding in `PosKitchenTopologyConfig`, matching the existing `pos.order-ready.queue` DLQ pattern.
- `KitchenItemStatusSyncIT` proves field-name parity end-to-end by constructing the exact wire-shape envelope kitchen-service would emit and feeding it through the real `onMessage(Message)` deserialization path (a stronger proof than the existing `OrderReadyConsumer` test pattern, which calls the business method directly).

## Decisions Made
- Used an ordinal-comparison guard (`OrderItemStatus.ordinal()`) instead of a fixed `ELIGIBLE` set in `KitchenItemStatusConsumer`, since the incoming target status varies per message (unlike `OrderReadyConsumer`'s single fixed READY target) — this generalizes the "never move backward" guarantee to every possible kitchen-driven transition.
- Kitchen `PENDING` maps to pos `SENT` (not `PENDING`) so a fired line can never revert to "never sent."
- The IT constructs a locally-mirrored payload record (field-name-identical to kitchen's real record, since pos-service has no Maven dependency on kitchen-service) and feeds it through the actual `onMessage(Message)` path — genuinely proving the deserialization contract, not just the business logic.
- Discovered and used the dev-stack RabbitMQ credentials (`RABBITMQ_USERNAME=restaurantos`, `RABBITMQ_PASSWORD=dev_rabbit_2026` from `deploy/.env`) to unblock the `@RabbitListener` context-startup `ACCESS_REFUSED` error that STATE.md had previously flagged as an environmental blocker for kitchen-service ITs — this also applies to pos-service and let both full test suites run green locally.
