# Phase 07.3 — UI Design Contract

**Status:** draft (authored from investigation + locked CONTEXT decisions)
**Inherits:** `.planning/phases/07.1-pos-production-operations/07.1-UI-SPEC.md` (design system,
spacing, typography, color, Status System, copywriting) — unchanged. This file specifies only the
07.3 deltas.

## Design System

**Reused DS primitives (mandatory, do not reimplement):** `StatusBadge`, `MoneyDisplay`,
`DataTable`, `EmptyState`, `Skeleton`, plus the Phase-7.1 `revision-chip`, `settlement-actions`,
`StatusBadge` icon system.

**New cross-cutting pattern — dedicated surface over modal (POS-25):** operational POS/KDS
surfaces move from `Dialog`/hand-rolled overlays to either (a) a **routed full page** or (b) a
**large in-place panel** that occupies the primary content column (not a centered `sm:max-w-md`
dialog). The `components/ui/dialog.tsx` primitive stays for genuinely transient confirmations
only (e.g. inline "cancel item" confirm). Command-palette and finance modals are untouched.

**Surfaces converted this phase:** payment/Charge (→ full page), order/table detail drawer (→
large panel/page), void/refund (→ panel), till open/close (→ panel), KDS ticket detail (→ page).

## Status System deltas

- **Item-status → KDS column mapping (KDS-04):** New = `PENDING`, Started = `ACCEPTED`,
  Preparing = `PREPARING` (+ legacy `COOKING`), Ready = `READY`. Reuse the 07.1 per-item
  `StatusBadge` hues/icons; a column is just a grouping of item rows by status.
- **Payment status badge (POS-23/24) — NEW:** four states, each hue+icon+label (never color-only):
  - `PAID` — success hue, check icon, "Paid"
  - `PARTIALLY_PAID` — warning hue, half-circle icon, "Partial"
  - `UNPAID` — neutral/muted, dot icon, "Unpaid"
  - `REFUNDED` — danger/muted, undo icon, "Refunded"
  Distinct from the derived order status and settlement status — rendered as a separate chip.

## Screens & Interaction Contracts

### 1. Order-taking terminal — client cart (POS-16, POS-17, POS-18, POS-19)
- **Cart is local** until Send/Charge; no persistence indicator until first persist. Empty cart =
  `EmptyState` ("Add items to start an order"), never a DRAFT row.
- **Line row:** name, ×qty with **− / +** stepper (qty 0 removes), line total via `MoneyDisplay`,
  modifiers/notes shown beneath. Same item+modifiers+notes merges; differing ⇒ separate row.
- **Order-type toggle:** segmented control Dine-in / Takeaway / Pickup (default Dine-in). Takeaway/
  Pickup hide the table requirement.
- **Table selector:** searchable combobox (not the floor grid) listing tables with a status chip;
  `OCCUPIED` options disabled; table optional. (Reserved not shown — deferred.)
- **Actions:** `Send to Kitchen` (primary) always available when cart non-empty; **`Charge Now`
  disabled until the order has been sent** (tooltip: "Send to kitchen first"). After Send, a
  **`Clear / New Order`** action resets the terminal to an empty cart. No previous selection leaks.

### 2. Charge — dedicated full page (POS-22, POS-23, POS-25)
- **Route:** a dedicated page (e.g. `/app/pos/orders/[orderId]/charge`), not a dialog. Responsive;
  no fixed narrow width; content scrolls within regions, page body never overflows horizontally.
- **Regions:**
  - **Header:** order no., table (or "Takeaway"/"Pickup"), customer (or "—"), cashier, order time,
    live order/derived status chip + **payment-status chip**.
  - **Items:** list with qty, per-item status badge, notes, line totals.
  - **Money breakdown:** subtotal, discounts, service charge, taxes, **total**, **amount paid**,
    **remaining balance** — all `MoneyDisplay`.
  - **Payment history:** rows of prior `OrderPayment` (method, amount, ref, time). Empty ⇒ "No
    payments yet".
  - **Take payment:** split-tender entry (method + amount [+ ref]); "Add tender"; charge button
    disabled when remaining ≤ 0 or amount invalid.
- **States:** recording payment updates amount-paid/remaining and the payment chip **without
  closing**. When `PAID && SERVED` the order shows **Closed** (read-only, "Payment completed");
  when `PAID && !SERVED` it shows **Paid** with the live kitchen status and remains open. A paid
  order blocks new tenders (duplicate-payment guard) but stays viewable.

### 3. Order Management additions (POS-24)
- **Filters:** existing derived-status chips PLUS Closed / Paid (settlement) filters. **Search
  box** over order no. / table name.
- **Columns:** replace **Cover** with **Items** — total item quantity (e.g. "8 Items"), optional
  "5 Items / 8 Qty" secondary text. Add a **payment-status badge** column.
- **Row actions:** open (read-only when closed); **Assign Table** action for tableless orders →
  the same searchable selector (available tables only; occupied blocked); on assign, order + table
  status update immediately across the UI.
- **Refresh:** a manual Refresh control (POS-21) re-fetches the list.

### 4. KDS station board + detail (KDS-04, KDS-05)
- **Station picker:** `kitchen/` lists active (seeded) stations; a single station auto-enters.
  Empty stations ⇒ `EmptyState` ("No active stations configured") only when truly none seeded.
- **Station board (`kitchen/[stationCode]`):** four columns **New / Started / Preparing / Ready**.
  Item-centric — an order with mixed item statuses appears in each relevant column, each card
  fragment showing the order header + only that column's items. Moving an item calls the item-
  status endpoint; the card migrates columns on the next socket frame.
- **Collapsed card:** ONLY order number, table, time (age), menu-item names. No inline badges/bump
  buttons on the collapsed card.
- **Detail (`kitchen/[stationCode]/orders/[ticketId]`):** a **dedicated page** (kills the Dialog)
  with full items grouped by revision, instructions/kitchen notes, and per-item transition
  controls.
- **Prioritization (KDS-05):** subtle only — a colored **left border** + a **timer chip** that
  turns amber then red based on the station's `escalationThresholdSeconds`. No `animate-bounce`,
  no full-red backgrounds. New arrivals get a brief subtle highlight. One shared clock (not one
  timer per card) so it scales to many tickets.

### 5. Modal→page conversion contract (POS-25)
Each converted surface must: (a) render in the primary content area or a routed page, (b) be
responsive with no horizontal body overflow, (c) carry the full analytic information for its
purpose (not a stripped popup), and (d) preserve every existing caller (payment + detail drawer
are reused in 2–3 places — all must route to the new surface).
