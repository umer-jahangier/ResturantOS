## Decisions Made
- Menu taps are ALWAYS cart-only, even once the order has been sent — adding more items to an already-fired order is Order Management's revision-fire flow (POS-21/D-06), a separate plan's wiring, not the terminal's. This kept `handleItemSelect` grep-clean of any persist call, per Task 3's literal acceptance criterion.
- Added `frontend/lib/hooks/pos/use-fire-to-kitchen.ts` (mutate-time-orderId `sendToKds` sibling) instead of editing `use-orders.ts`, which the plan explicitly reserves for 07.3-06. `useSendToKds(orderId)` binds its target at hook-creation time — unusable for firing an order whose id is only known inside the SAME click handler that both persists and fires it (the identical stale-closure class already fixed once for `useAddItem`, 07.1-08).
- `OrderPanel`'s post-send rendering reuses the ORIGINAL revision-aware Send-to-Kitchen/settlement code verbatim rather than re-deriving it — `order.id` is stable and already known by the time `sentOrder` is non-null, so the hook-argument-binding pattern that's unsafe for the lazy first-send is perfectly safe here.
- `pos-no-draft.spec.ts`'s final assertion tracks the SPECIFIC created order's id (from the create-order response) rather than a bare Order-Management total-row-count diff — the live shared dev stack accumulates orders across repeated test runs / concurrent sessions, which made a naive count-diff flaky (observed 2 false failures before switching to per-order-id assertion).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Added a new mutate-time-orderId send hook instead of extending useSendToKds**
- **Found during:** Task 3 (terminal rewrite)
- **Issue:** `useSendToKds(orderId)` binds its `orderId` at hook-CREATION time. The lazy-persist first Send needs to fire the JUST-created server order id inside the SAME click handler that discovers it — a hook bound to the previous render's null/"" id would target the wrong (or no) order, and the plan explicitly forbids editing `use-orders.ts` this phase (07.3-06 owns it).
- **Fix:** Added `frontend/lib/hooks/pos/use-fire-to-kitchen.ts`, a `useFireToKitchen()` hook mirroring `useSendToKds`'s invalidation but with `orderId` as a mutate-time variable (identical shape to the existing `useAddItem` fix for the same bug class).
- **Files modified:** `frontend/lib/hooks/pos/use-fire-to-kitchen.ts` (new)
- **Verification:** `pos-terminal.test.tsx#Send to Kitchen persists the cart...` asserts `sendToKdsCallCount === 1` after one click; live E2E confirms the order transitions to `SENT_TO_KDS` and Charge Now enables.
- **Committed in:** `71fd904` (Task 3 commit)

**2. [Rule 1 - Bug] Rewrote pos-terminal.test.tsx — old suite asserted the removed behavior**
- **Found during:** Task 3
- **Issue:** The pre-existing test suite asserted "creates a DRAFT order on the first tap" — the EXACT behavior D-01 requires removing. Running it post-rewrite failed 5/5 relevant tests.
- **Fix:** Rewrote the suite around the new contract: no network call on tap, quantity merge to 2, lazy persist + fire on Send (with a `send-to-kds` mock endpoint added), Charge Now gating, table-binding-at-Send (not at-tap), and a fast-double-click-Send guard test.
- **Files modified:** `frontend/__tests__/pos/pos-terminal.test.tsx`
- **Verification:** 8/8 tests green.
- **Committed in:** `71fd904` (Task 3 commit)

**3. [Rule 1 - Bug] Added a sendInFlightRef guard against double-firing lazy persist on a rapid double-click**
- **Found during:** Task 3, while writing the double-click regression test
- **Issue:** `handleSendToKitchen`'s `if (orderId) return` guard reads React state, which does not update synchronously between two `fireEvent.click` calls in the same tick — a fast double-click would fire two concurrent `createOrder` attempts (the same class of race the codebase already fixed once for menu-item taps via `creatingOrderRef`).
- **Fix:** Added `sendInFlightRef` (plain ref, read/written synchronously) guarding the handler's entry, cleared in a `finally` block.
- **Files modified:** `frontend/components/pos/pos-terminal.tsx`
- **Verification:** `pos-terminal.test.tsx#does not double-fire createOrder on a fast double-click...` — `createOrderCallCount === 1` after two same-tick clicks.
- **Committed in:** `71fd904` (Task 3 commit)

**4. [Rule 1 - Bug] Fixed pos-no-draft.spec.ts's Order Management row-count assertions for DataTable pagination and live-stack concurrency**
- **Found during:** Task 4, running the spec live
- **Issue:** (a) `DataTable` paginates at 10 rows/page — counting visible `open-order-*` DOM rows undercounted the true total once the demo branch had more than a page of active orders. (b) Even after reading the table's "Showing X–Y of Z" footer for the true total, a bare `baseline+1` diff proved flaky against the shared live dev stack's own concurrent order churn between reads.
- **Fix:** (a) Parse the DataTable footer's `of (\d+)` text for the true total instead of counting DOM rows. (b) Replaced the final `baseline+1` diff with a precise assertion that the SPECIFIC just-created order (id captured from the create-order network response) appears in Order Management.
- **Files modified:** `frontend/e2e/pos-no-draft.spec.ts`
- **Verification:** 2 clean runs against the live dev stack after the fix (1 unrelated transient login-navigation timeout, mitigated with a login retry).
- **Committed in:** `cbd1ffc` (Task 4 commit)
