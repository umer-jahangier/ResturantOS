## Decisions Made

- `useOrder`'s live-refetch fix is a flat 5s `refetchInterval`, not a WebSocket — matches
  the existing KDS board's own HTTP-poll fallback pattern and needed no new infrastructure
- The detail surface is built directly on `DialogPrimitive.{Root,Portal,Overlay,Content,Title,Close}`
  from `radix-ui`, not the shared `DialogContent` component, specifically to drop that
  component's `sm:max-w-sm` default and become a large (`inset-4 sm:inset-6 lg:inset-10`)
  near-full-viewport panel while keeping Radix's focus-trap/ESC/backdrop-click behavior
- The CTA always reads "Send New Items (N)" on this surface (never "Send to Kitchen") —
  unlike `order-panel.tsx`'s dual-label logic — since every order reaching Order
  Management/Table Detail already has at least one fired line by definition
- **Playwright `locator.isVisible({ timeout })` does not auto-retry or wait** — it is an
  immediate DOM snapshot; only the locator-resolution step respects `timeout`, not the
  visibility condition itself. This was root-caused live mid-session after several
  false-negative `BLOCKED` verdicts on `pos-add-existing-revision.spec.ts` (the row was
  actually rendering correctly, just a beat after the check ran) and both new E2E specs
  were corrected to use `expect(locator).toBeVisible({ timeout })` or
  `locator.waitFor({ state: "visible", timeout })` wherever a genuine multi-second wait
  for an async-rendered element is intended. Retained the codebase's existing
  `isVisible({timeout}).catch(() => false)` pattern only for true "is this optional
  pre-existing chrome already on screen" branches (matches `pos-settlement.spec.ts`
  convention), not for "wait for X to eventually appear."

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Fixed Playwright `isVisible({timeout})`-as-wait anti-pattern discovered live while authoring Task 4's E2E specs**
- **Found during:** Task 4, while debugging `pos-add-existing-revision.spec.ts`'s
  "order never appeared in Order Management" false-Blocked result
- **Issue:** `locator.isVisible({ timeout })` does not retry — it's an immediate DOM
  check, so a check written to "wait up to 15s for an async row to render" was actually
  failing after ~0ms every time, well before React/TanStack Query had a chance to render
  the fetched data
- **Fix:** Replaced with `expect(locator).toBeVisible({ timeout })` (retrying assertion)
  or `locator.waitFor({ state: "visible", timeout })` at every genuine wait-point in both
  new spec files
- **Files modified:** frontend/e2e/pos-add-existing-revision.spec.ts, frontend/e2e/pos-kitchen-live-sync.spec.ts
- **Verification:** `pos-add-existing-revision.spec.ts` now passes green, repeatably,
  against the live dev stack
- **Committed in:** fcdfabb (Task 4 commit)

**2. [Rule 1 - Bug] Fixed a name-based revision-line matching bug in `pos-add-existing-revision.spec.ts`'s own assertion**
- **Found during:** Task 4, first real (non-Blocked) run of the spec
- **Issue:** The test matched the "already-fired" vs "newly-added" line in the
  `send-to-kds` response by `itemNameSnapshot`, which is ambiguous when the same menu
  item is picked twice (realistic — the demo menu's first item was picked both times)
- **Fix:** Partitioned the response's `items[]` by `revisionNo` instead (exactly 1 line
  at `revisionNo === 1`, exactly 1 line at `revisionNo > 1`) — robust regardless of
  item-name collisions, and more directly tests the actual POS-21 behavior under test
- **Files modified:** frontend/e2e/pos-add-existing-revision.spec.ts
- **Verification:** spec passes green
- **Committed in:** fcdfabb (Task 4 commit)

## Issues Encountered

**Out-of-scope kitchen-service defect blocks `pos-kitchen-live-sync.spec.ts` (POS-20) from a live PASS on this dev branch** — root-caused via direct live investigation (not guessed):

1. `KdsController.getTickets` (kitchen-service, `KdsController.java:64`) calls
   `ticketRepository.findAll(pageable).map(ticketService::toDto)` directly from the
   controller, outside any `@Transactional` boundary. When called with no `stationCode`
   (the unscoped branch), this throws `org.hibernate.LazyInitializationException` on
   `KdsTicket.items`, surfaced as HTTP 500/503. The station-scoped path the live board
   actually calls per-column does NOT hit this.
2. Independent of (1): `GET /kitchen/kds/tickets` has no explicit sort and Spring's
   default `Pageable` is unsorted + `size=20`. This dev branch's GRILL station has
   accumulated 29 PENDING tickets from many prior test/UAT runs (kds-board.tsx never
   expires stale PENDING tickets), so a freshly-fired order's ticket — confirmed present
   in the data via a direct authenticated `GET ...?size=100` — lands at position 29/29,
   beyond the frontend's default page-1/size-20 fetch, and never renders on the board.

Both are pre-existing (`KdsController.java` last touched in 07.1-02, well before this
07.3 phase) and entirely in kitchen-service Java code, out of scope for this
frontend-only plan ("no backend change" per this plan's own context). Restarted
kitchen-service and pos-service mid-investigation (`scripts/restart-service.ps1`,
`-SkipBuild`) to rule out stale-process artifacts before concluding this — the defect
reproduces identically on a fresh process. Logged in full to `deferred-items.md` with
the exact fix shape (wrap `getTickets`'s unscoped branch in `@Transactional(readOnly =
true)`, mirroring `getTicketDetail`'s already-correct pattern two lines below;
separately, add stale-ticket cleanup or a sort+larger-page default). The E2E spec
correctly self-diagnoses this live (inspects the `GET .../tickets` response status) and
reports `BLOCKED`, not `FAIL` — Playwright's process exit code for the run is 0 (skipped,
not failed).
