# Phase 07.3: Code Review Report (Backend — pos-service + kitchen-service)

**Reviewed:** 2026-07-12T16:47:53Z
**Depth:** standard
**Files Reviewed:** 23
**Status:** issues_found

## Summary

Reviewed the pos-service and kitchen-service backend changes for 07.3 (payment/close decoupling, kitchen→POS item-status sync, order read-model/OrderSummaryDto, KDS explicit item-status endpoint, station auto-seed, table-number/pickup migrations). Event-payload field parity between the two services was checked field-by-field and is correct (`OrderSentToKdsPayload`/`ItemStatusChangedPayload` on both sides). The `KitchenItemStatusConsumer` ordinal no-downgrade guard is correctly implemented and tenant-scoped via `TenantAwareMessageProcessor`.

Four Critical issues were found, all of which corrupt persisted state or bypass authorization in ways a user/attacker can trigger through normal API usage:

1. Orders closed through the legacy `POST /orders/{id}/close` tender path never get their payments persisted as `OrderPayment` rows, so the brand-new `OrderSummaryDto.paymentStatus`/`amountPaidPaisa` and `GET /orders/{id}/payments` built in this same phase silently show UNPAID/$0/empty for those orders.
2. `assignTable`'s "release the previous table" call computes the previous table's new status from the *order's own live state* rather than "no longer bound," so the vacated table is left stuck OCCUPIED/NEEDS_BUSSING instead of AVAILABLE.
3. `POST /orders/{id}/payments` accepts zero/negative `amountPaisa` with no validation, letting a client silently reduce the recorded paid sum without any refund audit trail.
4. The KDS item-status endpoints (new `setItemStatus`, plus `bumpItem`/`recallTicket`/`getTicketDetail`) authorize against a caller-supplied `branchId` query param that is never cross-checked against the branch the fetched ticket actually belongs to — verified against the OPA `kds.rego` policy, this is a genuine cross-branch IDOR within the same tenant.

## Critical Issues

### CR-01: Orders closed via the legacy tender path never persist `OrderPayment` rows — new paymentStatus/payment-history features silently show wrong data

**File:** `services/pos-service/src/main/java/io/restaurantos/pos/service/OrderServiceImpl.java:558-574` (closeOrder) and `620-670` (performClose)

**Issue:** `closeOrder(orderId, request, idempotencyKey)` validates `request.payments()` sum against `order.getTotalPaisa()` (`splitTenderCalculator.validateExact`) and forwards them into the `ORDER_CLOSED` outbox event as `PosClosePayloads.PaymentEntry`, but **never calls `orderPaymentRepository.save(...)`** for any of them. The only place in the entire pos-service that persists an `OrderPayment` row is `PaymentServiceImpl.recordPayment` (`services/pos-service/src/main/java/io/restaurantos/pos/service/PaymentServiceImpl.java:56-63`).

This phase (07.3-01/07.3-04) added two features that read `OrderPayment` rows exclusively:
- `PaymentStatusDerivationService` used by `OrderSummaryDto.paymentStatus`/`amountPaidPaisa` (`toSummaryDto`, `listOrderSummaries` batched via `orderPaymentRepository.sumAmountByOrderIds`)
- `PaymentService.listPayments` / `GET /orders/{id}/payments` (`OrderPaymentDto`)

Any order closed through `POST /orders/{id}/close` (the still-live, still-tested "legacy exact-tender" endpoint in `PaymentController`) will therefore show up in Order Management as `paymentStatus=UNPAID`, `amountPaidPaisa=0`, and its payment history endpoint will return an empty list — despite the order being fully paid and CLOSED. This is a data-integrity bug that directly undermines the POS-22/23/24 features built in this same phase.

**Fix:** Persist an `OrderPayment` row for each entry in `request.payments()` inside `closeOrder` (or inside the shared `performClose` seam) before/while publishing the `ORDER_CLOSED` event, e.g.:

```java
// in closeOrder, before performClose(order, paymentEntries):
for (SplitTenderCalculator.PaymentEntry p : request.payments()) {
    OrderPayment payment = new OrderPayment();
    payment.setTenantId(tenantId);
    payment.setOrderId(order.getId());
    payment.setMethod(PaymentMethod.valueOf(p.method()));
    payment.setAmountPaisa(p.amountPaisa());
    payment.setReferenceNo(p.referenceNo());
    payment.setRecordedAt(Instant.now());
    orderPaymentRepository.save(payment);
}
```

### CR-02: `assignTable` never releases the previous table — it stays stuck OCCUPIED/NEEDS_BUSSING

**File:** `services/pos-service/src/main/java/io/restaurantos/pos/service/OrderServiceImpl.java:778-786` and `services/pos-service/src/main/java/io/restaurantos/pos/service/TableServiceImpl.java:74-91`

**Issue:** `assignTable` calls `tableService.syncStatusForOrder(previousTableId, order.getBranchId(), order.getStatus(), order.getDerivedStatus())` intending to free the vacated table. But `TableServiceImpl.syncStatusForOrder` computes the target table's new status purely from the *order's own current state*:

```java
if (TERMINAL_ORDER_STATUSES.contains(orderStatus)) {
    newStatus = TableStatus.AVAILABLE;
} else if (derivedStatus == DerivedOrderStatus.SERVED) {
    newStatus = TableStatus.NEEDS_BUSSING;
} else {
    newStatus = TableStatus.OCCUPIED;
}
```

Since `assignTable` only runs on non-terminal orders (terminal orders are rejected earlier in the method), `orderStatus` is never in `TERMINAL_ORDER_STATUSES` at this call site, so the branch that would produce `AVAILABLE` can never be taken for the previous table. The previous table is instead re-marked `OCCUPIED` (or `NEEDS_BUSSING`) — i.e. it is left exactly as "occupied" as the order that just moved away from it, permanently blocking that table from being reused for a new party until someone manually resets it.

This defeats the entire purpose of the "release the previous binding" call documented in the method's own comment ("route BOTH the previous binding... through syncStatusForOrder... never an inline table.setStatus(...)").

**Fix:** `syncStatusForOrder` needs a way to express "this table is no longer bound to this order" distinctly from "this order's live progress state." Simplest fix: always force `AVAILABLE` for the previous table in `assignTable`, bypassing the order-status-derived helper:

```java
if (previousTableId != null) {
    tableService.updateStatus(previousTableId, order.getBranchId(), TableStatus.AVAILABLE);
}
tableService.syncStatusForOrder(tableId, order.getBranchId(), order.getStatus(), order.getDerivedStatus());
```

(`TableServiceImpl.updateStatus` already exists and does an unconditional status set.)

### CR-03: `POST /orders/{id}/payments` accepts zero/negative `amountPaisa` with no validation

**File:** `services/pos-service/src/main/java/io/restaurantos/pos/web/PaymentController.java:37-41` (`RecordPaymentRequest`) and `services/pos-service/src/main/java/io/restaurantos/pos/service/PaymentServiceImpl.java:41-71` (`recordPayment`)

**Issue:** `RecordPaymentRequest` only enforces `@NotNull` on `amountPaisa`:

```java
record RecordPaymentRequest(
        @NotNull PaymentMethod method,
        @NotNull Long amountPaisa,
        String referenceNo
) {}
```

`PaymentServiceImpl.recordPayment` performs no additional check that `amountPaisa > 0` before persisting the `OrderPayment` row and re-deriving `paymentStatus`/`amountPaidPaisa` (which now feed `OrderSummaryDto` and `maybeCloseOrder`'s Paid-AND-Served gate, both new this phase). A caller can submit `amountPaisa = 0` or a negative value:
- A negative payment silently reduces `orderPaymentRepository.sumAmountByOrderId`, effectively "unpaying" an order with no refund record, no `RefundService` audit trail, and no reason/approval gate.
- It can also flip a previously-PAID order back to PARTIALLY_PAID/UNPAID, undoing an auto-close decision retroactively without any corresponding compensating event.

**Fix:** Add `@Positive` to `amountPaisa` in `RecordPaymentRequest` (Bean Validation already runs via `@Valid` on the controller method) and/or a defense-in-depth check in `recordPayment`:

```java
record RecordPaymentRequest(
        @NotNull PaymentMethod method,
        @NotNull @Positive Long amountPaisa,
        String referenceNo
) {}
```

```java
if (amountPaisa <= 0) {
    throw new StateInvalidException("Payment amount must be positive: " + amountPaisa);
}
```

### CR-04: KDS ticket-mutation endpoints authorize on a caller-supplied `branchId` that is never checked against the actual ticket's branch — cross-branch IDOR

**File:** `services/kitchen-service/src/main/java/io/restaurantos/kitchen/web/KdsController.java:93-103` (`setItemStatus`, new this phase), also `bumpItem` (74-83), `recallTicket` (111-119), `getTicketDetail` (126-134); `services/kitchen-service/src/main/java/io/restaurantos/kitchen/service/TicketServiceImpl.java:57-105` (`markItemStatus`), `108-130` (`bumpItem`), `132-153` (`recallTicket`), `167-173` (`getTicketDetail`)

**Issue:** Every ticket-mutation/detail endpoint takes `ticketId` (path) and `branchId` (query param) separately:

```java
@PostMapping("/tickets/{ticketId}/items/{itemId}/status")
public ResponseEntity<KdsTicketDto> setItemStatus(
        @PathVariable UUID ticketId,
        @PathVariable UUID itemId,
        @RequestParam UUID branchId,
        @RequestBody ItemStatusRequest request,
        @AuthenticationPrincipal JwtClaims claims) {

    authz.authorizeUpdate(claims.tenantId(), branchId);
    return ResponseEntity.ok(ticketService.markItemStatus(ticketId, itemId, request.status()));
}
```

`authz.authorizeUpdate(tenantId, branchId)` evaluates the OPA `kds.rego` policy `same_tenant_and_branch`, which compares `input.resource.branch_id` (== the request-param `branchId`) against `input.user.branch_id` (from the JWT) — confirmed by reading `policies/restaurantos/kds.rego` and `policies/restaurantos/common.rego`. It never inspects the entity actually being mutated.

`TicketServiceImpl.markItemStatus`/`bumpItem`/`recallTicket`/`getTicketDetail` then load the ticket with `ticketRepository.findById(ticketId)` (or `findDetailById`), which is scoped only by the Hibernate `tenantFilter` (tenant-level), **not** by branch.

Net effect: a user holding `pos.kds.update` for their own branch A can supply `branchId=A` (passes OPA) together with a `ticketId` that actually belongs to branch B of the *same tenant*, and successfully view/mutate branch B's ticket — a cross-branch broken-access-control gap. `setItemStatus` is the newly-introduced endpoint in this phase that inherits this pattern; the same root cause also applies to the three sibling endpoints.

**Fix:** After loading the ticket, assert `ticket.getBranchId().equals(branchId)` (throw the same 403/404 the OPA check would produce) before performing the mutation/returning detail — e.g. in `TicketServiceImpl`:

```java
KdsTicket ticket = ticketRepository.findById(ticketId)
        .filter(t -> t.getBranchId().equals(branchId))
        .orElseThrow(() -> new ResourceNotFoundException("Ticket not found: " + ticketId));
```

(requires threading `branchId` through from the controller into the service calls, which it currently receives but discards).

## Warnings

### WR-01: Station auto-seed-on-miss can throw an unhandled 500 on a first-request race

**File:** `services/kitchen-service/src/main/java/io/restaurantos/kitchen/web/KdsController.java:143-162` (`getStations`) and `services/kitchen-service/src/main/java/io/restaurantos/kitchen/service/TicketRoutingService.java:138-151` (`ensureStation`)

**Issue:** Both auto-seed paths do a find-then-create with no transactional retry/catch. The `uq_station_tenant_branch_code` unique constraint is explicitly called out in comments as the race "backstop," but neither call site catches the resulting `DataIntegrityViolationException`. Two concurrent first-ever `GET /stations` calls for a brand-new branch (or a `GET /stations` racing a `TicketRoutingService.ensureStation` for the same code) will have one caller receive an unhandled 500 instead of the seeded/existing station.

**Fix:** Wrap the save in a catch for the unique-constraint violation and re-fetch on conflict, e.g.:

```java
try {
    stations = List.of(stationRepository.save(defaultStation));
} catch (DataIntegrityViolationException e) {
    stations = stationRepository.findByBranchIdAndActiveTrue(branchId);
}
```

### WR-02: `OrderSummaryDto`/order-detail endpoints don't enforce branch ownership the same way `listOrderSummaries` explicitly does

**File:** `services/pos-service/src/main/java/io/restaurantos/pos/service/OrderServiceImpl.java` (`assignTable`, `addItem`, `removeItem`, `applyDiscount`, `sendToKds`, `markItemServed`, `cancelItem`, `updateInstructions` — all use `findOrderForTenant`, tenant-only scoping)

**Issue:** `listOrderSummaries` was hardened this phase with an explicit JWT-branch-vs-param check (T-07.1d-01: "a client-supplied branchId could otherwise leak cross-branch orders"). The new `assignTable` endpoint (and every other per-order mutation endpoint) has no equivalent check — any caller within the tenant who can reach these endpoints and knows/guesses an `orderId` can mutate an order belonging to a different branch, since `findOrderForTenant` filters by tenant only. This may be intentional (coarser tenant+role RBAC elsewhere), but it is inconsistent with the security rationale just written for `listOrderSummaries` in the same commit series, and `assignTable` is new code that could have closed this gap for at least the table-reassignment surface.

**Fix:** If branch-level isolation is intended for these mutation endpoints (consistent with `listOrderSummaries`'s stated threat model), add the same JWT-branch check used there, or scope `findOrderForTenant` by `(tenantId, branchId)` derived from `TenantContext` rather than tenant alone.

### WR-03: `ItemStatusRequest.status` has no `@NotNull`/`@Valid`, producing an opaque error for malformed requests

**File:** `services/kitchen-service/src/main/java/io/restaurantos/kitchen/web/KdsController.java:93-105`

**Issue:** `record ItemStatusRequest(TicketItemStatus status) {}` has no validation annotation, and the controller method isn't annotated `@Valid` on the `@RequestBody`. A request body of `{}` (missing `status`) reaches `TicketServiceImpl.validateTransition` with `next == null`; every branch of the transition `switch` compares `next == SomeEnum`, which is null-safe, so it doesn't NPE — but it surfaces as a generic `StateInvalidException("Invalid item status transition: X -> null")` rather than a clean 400 validation error, which is confusing for API consumers debugging a malformed request.

**Fix:** Add `@NotNull` to the `status` field and `@Valid` on the controller parameter:

```java
public record ItemStatusRequest(@NotNull TicketItemStatus status) {}
...
@RequestBody @Valid ItemStatusRequest request,
```

### WR-04: `SplitTenderCalculator.validateExact`/`PaymentEntry` allow negative entries that net out to the exact total

**File:** `services/pos-service/src/main/java/io/restaurantos/pos/service/SplitTenderCalculator.java:19-30` (referenced from `CloseOrderRequest`, consumed by `OrderServiceImpl.closeOrder`)

**Issue:** `validateExact` only checks `sum(payments) == totalPaisa`; it never checks that each individual `PaymentEntry.amountPaisa` is non-negative. A close request like `[{CASH, +total+1000}, {CARD, -1000}]` still passes validation and is forwarded verbatim into the `ORDER_CLOSED` event's per-method payment breakdown, which downstream finance/reporting consumers may treat as ground truth for tender-method reconciliation. Combined with CR-01 (these entries are never persisted as `OrderPayment` rows either), this is a soft spot in the legacy close path's money integrity.

**Fix:** Reject any `PaymentEntry` with `amountPaisa <= 0` in `validateExact` (or via a `@Positive` constraint on the DTO used in `CloseOrderRequest.payments`).

## Info

### IN-01: `KitchenItemStatusConsumer.applyItemStatus`'s `@Transactional` is a no-op due to self-invocation

**File:** `services/pos-service/src/main/java/io/restaurantos/pos/consumer/KitchenItemStatusConsumer.java:75-133`

**Issue:** `applyItemStatus` is called via `env -> applyItemStatus(...)`, a lambda that resolves to a plain `this.applyItemStatus(...)` call — this bypasses the Spring AOP proxy, so its own `@Transactional` annotation has no effect. It currently works correctly only because the call chain (`processedEventService.tryProcess` → `tenantAwareMessageProcessor.process` → the lambda) is already running inside a transaction opened by `PosProcessedEventService.tryProcess` (a genuine cross-bean call). The annotation is misleading dead weight today, and would silently stop providing transactional/lazy-loading safety if `applyItemStatus` were ever invoked directly from elsewhere without that surrounding call chain.

**Fix:** Remove the misleading `@Transactional` from `applyItemStatus` (it's redundant), or add a comment noting it relies on the caller's transaction, to avoid a future refactor assuming it's self-sufficient.

### IN-02: Duplicated idempotency-check-and-reload boilerplate across `closeOrder`/`voidOrder`

**File:** `services/pos-service/src/main/java/io/restaurantos/pos/service/OrderServiceImpl.java:475-522` (`voidOrder`), `524-574` (`closeOrder`)

**Issue:** Both methods repeat the same three-step pattern (`idempotencyService.getCompletedResponse` → reload-and-return, `checkAndLock` → reload-and-return-on-not-claimed, then the real fetch) almost verbatim, each with its own `orderRepository.findById(...).filter(tenant).orElseThrow(...)` inline. This is a maintainability/duplication smell — a fix to one idempotency-reload path (e.g. adding branch scoping per WR-02) is easy to apply to one copy and forget the other.

**Fix:** Extract a private helper, e.g. `private Order idempotentReload(UUID orderId, UUID tenantId)`, and a small `withIdempotency(key, requestHash, () -> ...)` wrapper.

### IN-03: `escalationThresholdSeconds` magic number `900` duplicated across two station-seeding call sites

**File:** `services/kitchen-service/src/main/java/io/restaurantos/kitchen/service/TicketRoutingService.java:148` and `services/kitchen-service/src/main/java/io/restaurantos/kitchen/web/KdsController.java:158`

**Issue:** The default station escalation threshold (900 seconds / 15 minutes) is hardcoded identically in two places with no shared constant, so a future change to the default only updates one of the two auto-seed code paths unless both are remembered.

**Fix:** Extract a shared constant (e.g. `KdsStation.DEFAULT_ESCALATION_THRESHOLD_SECONDS = 900`) and reference it from both call sites.
