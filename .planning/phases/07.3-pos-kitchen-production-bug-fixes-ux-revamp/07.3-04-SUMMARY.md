# Phase 07.3 Plan 04: POS Order-Management Backend Extensions Summary

**Order Management backend: rich summary DTO (payment status + item quantity), assign-table endpoint with occupied-table rejection, and KDS table-number event parity — all three sharing OrderServiceImpl in one plan to avoid a same-wave file conflict.**

## Performance

- **Duration:** ~40 min
- **Completed:** 2026-07-12T17:01:10+05:00
- **Tasks:** 3
- **Files modified:** 6 modified, 3 created

## Accomplishments
- `OrderSummaryDto` extended with `settlementStatus`, `paymentStatus`, `amountPaidPaisa`, `itemQuantity`, `distinctItemCount` — computed server-side via `PaymentStatusDerivationService` and a batched (N+1-safe) payment-sum lookup across the whole page
- `listOrderSummaries` default (empty status filter) now excludes DRAFT in addition to terminal statuses; an explicit `statuses=[CLOSED]` request still works
- New `PATCH /api/v1/pos/orders/{id}/table` assign-table endpoint: rejects terminal orders, re-checks the target table's AVAILABLE status inside the transaction, and routes all table-status mutation through the single `TableService.syncStatusForOrder` seam
- `PosEventPayloads.OrderSentToKdsPayload` gains an additive `tableNumber` last field (KDS-04 pos-produce side); `sendToKds` resolves it from the order's bound `DiningTable` (null for takeaway/pickup)

## Decisions Made
- Batched `OrderPaymentRepository.sumAmountByOrderIds` (interface-projection JPQL `GROUP BY`) instead of calling `sumAmountByOrderId` per row — satisfies the plan's explicit N+1-avoidance acceptance criterion.
- `assignTable`'s table-release semantics: only the tested/specified scenario (assigning to a previously tableless order) is fully correct; a true table-to-table reassignment does not force-release the old table (see key-decisions in frontmatter). Flagged for awareness, not fixed — out of this plan's tested scope.
- GitNexus `impact` run on `toSummaryDto`/`listOrderSummaries`/`sendToKds` (all LOW/max-2-impacted risk) and `syncStatusForOrder` (MEDIUM risk, pre-existing widely-used seam — adding one more caller following the established pattern) before editing, per CLAUDE.md.
