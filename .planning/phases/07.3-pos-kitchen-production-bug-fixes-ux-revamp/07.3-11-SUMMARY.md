# Phase 07.3 Plan 11: Retire Legacy Close-Path Paid-AND-Served Bypass Summary

**Deleted the pos-service `closeOrder` tender-sum-only close bypass (endpoint retired to 410 Gone), migrated all 8 IT-fixture callers to a shared real serve+pay test helper, and removed the orphaned frontend `PaymentPanel`/`useCloseOrder` dead code — closing the BLOCKER gap (BE-CR-01/POS-23) from 07.3-VERIFICATION.md.**

- **Tasks:** 4/4
- **Files modified:** 19 (4 removed, 15 modified)

## Accomplishments

- `POST /api/v1/pos/orders/{id}/close` no longer transitions an order to CLOSED. It returns HTTP 410 Gone with `Deprecation: true`, `Sunset`, and `Link: </api/v1/pos/orders/{id}/payments>; rel="successor-version"` headers and a fixed `ENDPOINT_RETIRED` `ApiWarning` body (no internal leak) — still behind the class-level `@RequiresFeature("FEATURE_POS")` guard.
- `OrderService.closeOrder` / `OrderServiceImpl.closeOrder` (the legacy exact-tender-sum -> `performClose()` bypass that never checked `derivedStatus == SERVED` and never persisted `OrderPayment` rows) is deleted entirely. `performClose` now has exactly ONE remaining call site: the `Order closed = performClose(order, paymentEntries);` line inside `maybeCloseOrder`. `CloseOrderRequest.java` DTO deleted (zero remaining production references).
- All 8 test files that previously drove closure through `orderService.closeOrder(...)` were migrated: `AssignTableIT`, `OrderSummaryDtoIT`, `TableOrderLookupIT`, `VoidRefundOpaIT`, `OrderRevisionIT` now call the new shared `PosTestBase.closeViaServeAndPay(orderService, paymentService, order, branchId)` helper (fires PENDING lines via `sendToKds`, serves every non-terminal item, records one CASH payment for the order's freshly-reloaded total). `PeriodLockCloseIT` was rewritten to prove period-lock fail-closed on `paymentService.recordPayment(...)`. `OrderCloseIdempotencyIT` (whose entire subject was the retired endpoint's Idempotency-Key contract) was deleted; its "ORDER_CLOSED published exactly once" coverage is preserved via a new `SettlementSemanticsIT.servedAndPaidClose_publishesExactlyOneOrderClosedEvent` test.
- Frontend orphaned dead code removed: `frontend/components/pos/payment-panel.tsx`, `frontend/__tests__/pos/payment-panel.test.tsx`, and the `useCloseOrder` hook (`use-payments.ts`). `PosRepository.closeOrder` and its `CloseOrderPayload` import were grep-gated-removed once the hook's deletion left the repository method with zero callers. Stale comments referencing the deleted files/methods were reworded in `charge-summary.tsx`, `pos.repository.ts`, and `ProvisionPeriodDialog.tsx` (this last file was outside the plan's declared `<files>` but required a one-line comment reword to satisfy the plan's own repo-wide `grep -rn "payment-panel"` acceptance gate).
- Migration note for external callers: any external/undocumented client that was calling `POST /orders/{id}/close` with an exact-tender payload now receives `410 Gone` and must switch to `POST /api/v1/pos/orders/{id}/payments` — the order auto-closes once it is fully Paid AND fully Served via the `maybeCloseOrder` seam.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] closeViaServeAndPay used a stale order total, causing OrderSummaryDtoIT to fail**
- **Found during:** Task 2 (live test run)
- **Issue:** The plan's literal helper spec computed the payment amount from the caller-supplied `order` parameter's `totalPaisa()`. `OrderSummaryDtoIT.explicitClosedStatusFilter_returnsClosedOrder` captures its `order` variable BEFORE calling `addItem` and never reassigns it, so `order.totalPaisa()` was 0 at the point `closeViaServeAndPay` was called — `recordPayment` posted a $0 tender, the order never reached `PAID`, and the test failed asserting `CLOSED` but got `SENT_TO_KDS`.
- **Fix:** Changed `closeViaServeAndPay` to re-fetch the order's `totalPaisa` from the DB (`orderService.getOrder(...).totalPaisa()`) immediately before calling `recordPayment`, independent of whatever the caller passed in. Makes the shared helper robust regardless of caller freshness.
- **Files modified:** `services/pos-service/src/test/java/io/restaurantos/pos/PosTestBase.java`
- **Verification:** Full targeted IT run (25 tests across `SettlementSemanticsIT`, `PeriodLockCloseIT`, `AssignTableIT`, `OrderSummaryDtoIT`, `TableOrderLookupIT`, `VoidRefundOpaIT`) green.
- **Committed in:** `02fb0c8` (Task 2 commit)

**2. [Rule 1 - Bug] closeViaServeAndPay must fire PENDING items before serving them**
- **Found during:** Task 2 (design/read-first analysis, before first test run)
- **Issue:** The plan's literal helper spec called `markItemServed` directly on `order.items()` without first firing them via `sendToKds`. All 4 fixture callers in Task 2 (and `PeriodLockCloseIT`'s original fixtures) create orders via `createOrder` + `addItem` only — items are PENDING at that point. `markItemServed` throws `StateInvalidException` ("Cannot serve item that has not been fired to the kitchen") on a PENDING item, so the literal helper would have failed at runtime for every one of these callers.
- **Fix:** `closeViaServeAndPay` now checks for any PENDING lines and calls `orderService.sendToKds(order.id(), null)` first (a precondition of `markItemServed`) before marking items SERVED.
- **Files modified:** `services/pos-service/src/test/java/io/restaurantos/pos/PosTestBase.java`
- **Verification:** Same combined 25-test run above, plus `OrderRevisionIT` in isolation (3/4 green, unrelated deferred failure only).
- **Committed in:** `02fb0c8` (Task 2 commit)

**3. [Rule 3 - Blocking] ProvisionPeriodDialog.tsx comment fixed outside the plan's declared file list**
- **Found during:** Task 4 (running the repo-wide acceptance grep)
- **Issue:** Task 4's own acceptance criterion is `grep -rn "payment-panel" frontend/ --include=*.ts --include=*.tsx returns 0 matches` — repo-wide, not scoped to the plan's `<files>` list. `frontend/components/finance/ProvisionPeriodDialog.tsx` (a finance-phase file, not in this plan's declared scope) had a doc comment referencing `components/pos/payment-panel.tsx`'s `getChargeErrorMessage`, which would have failed the stated gate had it been left as-is.
