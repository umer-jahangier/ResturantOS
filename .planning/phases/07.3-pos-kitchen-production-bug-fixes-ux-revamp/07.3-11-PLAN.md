<objective>
Close the single BLOCKER gap in 07.3-VERIFICATION.md (BE-CR-01 / POS-23 / SC4): the legacy
`POST /api/v1/pos/orders/{id}/close` endpoint still transitions an order to CLOSED after
validating ONLY `sum(request.payments()) == order.totalPaisa` — it never checks
`derivedStatus == SERVED` and never persists `OrderPayment` rows, so any authenticated client
can close a never-served order and this phase's own new settlement read model
(`paymentStatus` / `amountPaidPaisa` / `GET /orders/{id}/payments`) then reads UNPAID/$0/empty
for a CLOSED, actually-paid order. This violates the ROADMAP-declared, unconditional
cross-cutting truth "an order closes only when Paid AND Served."

Strategy (LOCKED by the user — D-08 / POS-23; the "gate & fix in place" alternative was
rejected): **DEPRECATE & REMOVE the legacy close path.** Retire the HTTP endpoint to 410 Gone,
delete the `closeOrder` service method's tender-sum -> `performClose()` bypass entirely (so the
Paid-AND-Served `maybeCloseOrder` seam becomes the ONLY path to CLOSED), migrate the 7 existing
IT fixtures off `closeOrder`, delete the orphaned frontend dead code (`PaymentPanel` +
`useCloseOrder`), and add a regression backstop.

Purpose: make the backend contract match the phase's stated invariant and stop corrupting the
new settlement read model.
Output: a 410 retirement handler + migration note, a deleted `closeOrder` bypass, migrated ITs,
deleted FE dead code, and a green regression backstop.
</objective>
