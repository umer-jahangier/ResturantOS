<objective>
Stand up the `auth-service` module (port 8081, `auth_db`) and deliver the authentication core: email+password+tenantSlug login that issues an RS256 access JWT + HttpOnly refresh cookie, the `/.well-known/jwks.json` endpoint, refresh + logout, bcrypt-12 + lockout, and `USER_LOGIN_*` events via the shared transactional outbox — assembling the shipped (but deliberately unwired) shared-lib security beans into a Spring Security 7 SecurityFilterChain.

Purpose: Delivers SC1 (login → 15-min RS256 JWT + 7-day HttpOnly refresh cookie + JWKS + bcrypt-12 + lockout) and the SC2 refresh/logout/login-event half (branch-switch lands in 02-02). Covers AUTH-01, AUTH-02, AUTH-03, AUTH-04, AUTH-08, and freezes the JWT claim/permission shape that 02-03 (OPA input) and 02-02 (branch switch) build on. Resolves the Phase 2/3 tenant seam (Open Q1), the RLS-vs-pre-login lookup (Pitfall 3), the login-event tenant policy for invalid slugs (Open Q5), and the ABAC `approval_limit_paisa` attribute source (Open Q3).
Output: the `auth-service` Maven module (+ parent POM module line), `auth_db` Liquibase migrations (auth_tenants, users, roles/permissions/role_permissions, user_branch_roles, refresh_sessions, shared infra tables) with RLS + dev seed, the SecurityFilterChain wiring, JWT signing + JWKS, login/refresh/logout, and a Testcontainers IT suite proving SC1 + the SC2 refresh/logout/events.

BUDGET NOTE (planner→executor): this is the heaviest plan in the phase (4 tasks, ~45 files; Tasks 1–2 are scaffold+migrations, Tasks 3–4 are login service+ITs). It is deliberately kept as one plan to keep the JWT-claim/RLS/seed contract atomic for the wave-2 plans. If context pressure is felt during execution, the natural split point is BETWEEN Task 2 and Task 3 (scaffold+migrations as 02-01a, login service+IT as 02-01b) — but DO NOT split mid-task. Keep Task 4 trimmable: the three IT classes are independent, so a partial run (AuthLoginIT first) still demonstrates SC1 if the budget runs short, with the rest landing as a fast follow-up. SC2's "every attempt publishes USER_LOGIN_*" is intentionally scoped to RESOLVABLE tenants — an unknown slug writes no outbox row because `event_outbox.tenant_id` is NOT NULL (Open Q5); this is by design and reflected in the must_haves.
