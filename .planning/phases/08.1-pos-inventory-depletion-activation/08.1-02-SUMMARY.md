# Phase 08.1 Plan 02: inventory-service menu-item catalog consumer + recipe validation Summary

**inventory-service gains a FORCE-RLS `menu_item_catalog` read-model synced from pos-service's menu events via a dual-event-type consumer on one queue, a `GET /menu-items` picker endpoint, and a real 404 `MENU_ITEM_NOT_FOUND` validation gate on `RecipeService.createVersion` that closes the "any UUID accepted" gap.**

## Performance

- **Duration:** ~15 min
- **Completed:** 2026-07-19
- **Tasks:** 3
- **Files modified:** 8 modified, 11 created

## Accomplishments
- `V4__menu_item_catalog.sql` creates a tenant-scoped, FORCE-RLS `menu_item_catalog` table mirroring V1's `ingredients` table convention (NULLIF-guarded `tenant_isolation` policy, audit columns, `UNIQUE(tenant_id, menu_item_id)`, explicit grant)
- `InventoryEventPayloads` carries `MenuItemUpsertedPayload`/`MenuItemDeletedPayload` records mirroring pos-service's 08.1-01 contract field-for-field (menuItemId, name, categoryId, categoryName, active, basePricePaisa, updatedAt / menuItemId)
- `InventoryRabbitConfig` + `deploy/init/rabbitmq-definitions.json` both declare `inventory.menu-item.queue` with TWO bindings on the SAME queue (`pos.menu_item.upserted` and `pos.menu_item.deleted`, D-08) sharing the existing DLX with its own `.dlq` sibling
- `MenuItemCatalogConsumer` parses the envelope JSON once, reads `eventType`, then dispatches to the matching payload class and handler — idempotent via `processed_events` (consumer `inventory.menu-item-catalog`), tenant resolved from the envelope via `TenantAwareMessageProcessor`, poison messages and processing failures both reject-without-requeue to the DLQ
- `MenuItemCatalogService.softDelete` never removes a row (D-07) — an unknown `menuItemId` delete is silently a no-op
- `GET /api/v1/inventory/menu-items` (gated on `inventory.item.view` via `InventoryAuthorizationService`) lists active catalog entries for the future recipe-builder picker (08.1-04)
- `RecipeService.createVersion` now calls `existsByTenantIdAndMenuItemIdAndActiveTrue` as its FIRST statement and throws `MenuItemNotFoundException` (404 `MENU_ITEM_NOT_FOUND`, via a new local `InventoryExceptionHandler` advice bean) before any `Recipe` row is created
- New `MenuItemCatalogConsumerIT` (5 tests) and `RecipeMenuItemValidationIT` (3 tests) prove the dual-dispatch/idempotency/soft-delete/type-safety and the validation gate end-to-end against a live Testcontainers Postgres

## Decisions Made
- `menu_item_catalog` follows V1's FORCE-RLS convention exactly (not V3's RLS-EXEMPT registry pattern) — per CONTEXT.md's explicit per-table guidance, since it is both read under tenant context (API) and written under tenant context resolved from the envelope (consumer).
- `inventory.menu-item.queue` is a deliberate one-queue/two-event-types exception to this service's one-queue-per-event-type convention (D-08) — documented in code comments on both bindings and on the shared `Declarables` bean.
- `MenuItemNotFoundException` gets its own local `@RestControllerAdvice` (`InventoryExceptionHandler`) rather than editing shared-lib's `GlobalExceptionHandler`, whose `handleBase(RestaurantOsException)` always returns 400 for every other service sharing it.
