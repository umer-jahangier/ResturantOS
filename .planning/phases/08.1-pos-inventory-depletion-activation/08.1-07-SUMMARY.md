### Task 3: Recipe-builder UI click-through (Playwright) — checkpoint, PASS evidence gathered, awaiting human approval

Ran `npx playwright test inventory-recipe-builder` against the live stack (frontend :3000, backend via gateway :8080).

**First run: BLOCKED** (`no sidebar Inventory link visible`) — investigated live (this was NOT a regression from the catalog-sync fix; confirmed via a throwaway debug spec that the sidebar link, feature-flags, and all API calls were genuinely fine). Root cause: the spec calls `Locator.isVisible({timeout: 5000})` immediately after `waitForURL(/\/app\//)` fires; per Playwright's own docs, `isVisible()` "does not wait for the element to become visible ... and returns immediately" — it does NOT poll. The sidebar's nav items depend on an async client-side feature-flags fetch that sometimes hadn't resolved yet at that instant, producing an intermittent false BLOCKED (reproduced the race ~2 of 3 raw attempts via debug scripts). Fixed by switching to `expect(inventoryLink).toBeVisible({timeout: 8000})` (Playwright's auto-retrying assertion) — **Rule 1 bug fix**, test-file-only, no production code touched.

**After the fix: 4/4 consecutive runs PASS**, zero flakes. Real evidence captured:
- `e2e/__screenshots__/inventory-landing.png` — `/app/inventory` renders live (not comingSoon/404), Recipe Builder + Coverage cards.
- `e2e/__screenshots__/inventory-recipe-builder.png` — `/app/inventory/recipes` renders "Recipe Builder", real "Select a menu item..." dropdown (test asserted `optionCount > 1`, i.e. real synced items beyond the placeholder — PASSED).
- `e2e/__screenshots__/inventory-coverage.png` / `inventory-coverage-settled.png` — coverage dashboard renders real numbers. (Note: the first-captured screenshot froze the tile mid count-up-animation — `AnimatedNumber` has a 1.5s duration — showing 9/1/8 instead of the settled 10/1/9; re-captured after a 2.5s wait confirms the settled values exactly match the API: **10 Total Active Menu Items, 1 Covered, 9 Missing**. Purely a screenshot-timing artifact, not a data or rendering bug — the test's own assertions (`getByText` checks) don't depend on the animated value and all passed.)

- Files modified: `scripts/restart-service.ps1`.
- Commit: `2618d30`.

**2. [Rule 1 - bug] Flaky `isVisible()` race in `inventory-recipe-builder.spec.ts`**
- Found during: Task 3, first Playwright run (false BLOCKED).
- Issue: `Locator.isVisible({timeout})` does not auto-retry/poll in Playwright — it reflects DOM state at the instant called. Checking the sidebar Inventory link immediately after `waitForURL` raced the async feature-flags fetch that gates the link's render, producing an intermittent false-negative BLOCKED (confirmed via 3 raw reproduction attempts: 1 timeout, 2 false-BLOCKED, before the fix).
- Fix: replaced with `expect(inventoryLink).toBeVisible({timeout: 8000})` (Playwright's auto-retrying assertion). Verified 4/4 stable PASS afterward.
- Files modified: `frontend/e2e/inventory-recipe-builder.spec.ts`.
- Commit: `a5fd7db`.

No other deviations. All other plan steps executed exactly as written.
