# Phase 08.1 Plan 03: recipe-coverage report + DEPLETION_INCOMPLETE activation Summary

**GET /api/v1/inventory/recipes/coverage exposes real missing-recipe gaps from the menu_item_catalog + effective-recipe seam, and DepletionService.deplete now publishes DEPLETION_INCOMPLETE for every skipped line instead of silently returning when an order has zero resolvable recipes (D-03).**

## Accomplishments
- `RecipeDtos` gains `MissingMenuItemDto(menuItemId, name)` and `CoverageResponse(totalActiveMenuItems, covered, missing)`
- `RecipeService.getCoverage()` left-joins the tenant's active `menu_item_catalog` rows against the existing `resolveEffectiveRecipe(menuItemId, Instant.now())` seam — no new resolution logic
- `RecipeController` exposes `GET /api/v1/inventory/recipes/coverage`, gated on `inventory.item.view` (`InventoryAuthorizationService.authorizeView`)
- `InventoryEventPayloads` gains `DEPLETION_INCOMPLETE`/`DEPLETION_INCOMPLETE_ROUTING_KEY` and `DepletionIncompletePayload{orderId, closedAt, missingMenuItemIds}`
- `DepletionService.deplete` restructured (control-flow only — every FEFO/lock-order/MAC-COGS/movement/LOW_STOCK_ALERT line inside the per-ingredient loop is byte-identical):
  - the 08-05 all-empty early return is deleted
  - the sorted-ingredient lock-and-deplete loop and the `STOCK_DEPLETED` publish now run inside `if (!requiredByIngredient.isEmpty())`, so `STOCK_DEPLETED` never fires with zero `depletedLines`
  - `DEPLETION_INCOMPLETE` always publishes afterward when `missingMenuItemIds` is non-empty, independent of the `STOCK_DEPLETED` branch
  - net effect: fully-covered → only `STOCK_DEPLETED` (unchanged); partially-covered → both events; fully-uncovered → only `DEPLETION_INCOMPLETE`, no exception, no DLQ
- `RecipeCoverageIT` (2 tests) proves the coverage report against 3 seeded catalog rows (1 covered, 2 missing) and proves an inactive catalog row is excluded from the universe entirely
- `DepletionIncompleteIT` (2 tests) proves the partial-coverage (both events, covered line depleted) and fully-uncovered (only `DEPLETION_INCOMPLETE`, no exception) cases end-to-end via `OrderClosedConsumer.onMessage`
- Regression-verified: pre-existing `DepletionConsumerIT` (fully-covered path) still passes unchanged

## Decisions Made
- `getCoverage()` reuses `resolveEffectiveRecipe(menuItemId, atInstant)` with `Instant.now()` — exactly the seam the plan specified, no new resolution path.
- `STOCK_DEPLETED`'s publish call was relocated inside the `if (!requiredByIngredient.isEmpty())` block (immediately after the loop) rather than left at the method's end — this is the mechanical change that makes it structurally impossible to publish `STOCK_DEPLETED` with zero `depletedLines`.
- Test-only: `DepletionIncompleteIT`'s outbox assertions are scoped by `orderId` (parsed out of `envelopeJson` via the tolerant `eventObjectMapper`) rather than a bare `eventType` filter, because `InventoryTestBase`'s `@BeforeAll` schema clean/migrate runs once per test class, not per test method — the class's two test methods share `event_outbox` rows within the same run.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Test bug] Scoped DepletionIncompleteIT outbox lookups by orderId**
- **Found during:** Task 3 first verification run (`mvn verify -Dit.test=RecipeCoverageIT,DepletionIncompleteIT`)
- **Issue:** The initial test implementation filtered `outboxRepository.findAll()` by a bare `eventType` match (mirroring `DepletionConsumerIT`'s pattern). Because `InventoryTestBase`'s `@BeforeAll` migration/clean runs once per test class rather than per test method, both `DepletionIncompleteIT` test methods share the same `event_outbox` table across the run — the second test method to execute saw its own `DEPLETION_INCOMPLETE` row PLUS the row left over from the other test method, failing `assertThat(entries).hasSize(1)` with "Expected size: 1 but was: 2".
- **Fix:** Added an `extractOrderId(OutboxEntry)` helper that parses `envelopeJson` via the tolerant `eventObjectMapper` and an `entriesForOrder(orderId, eventType)` helper that filters on both eventType AND the entry's own `orderId` — every assertion now looks only at outbox rows produced by that specific test's `onMessage` call, immune to cross-test-method leakage.
- **Files modified:** `services/inventory-service/src/test/java/io/restaurantos/inventory/DepletionIncompleteIT.java` (test-only, no production code touched)
- **Verification:** Re-ran `mvn -pl services/inventory-service verify -Dit.test=RecipeCoverageIT,DepletionIncompleteIT -DfailIfNoTests=false` — 4/4 tests pass.
- **Committed in:** `141b1ba` (Task 3 commit — the fix was applied before the first commit of this file, so no separate follow-up commit was needed)
