# Phase 08.1 Plan 01: pos-service menu-item write path + event publication Summary

**pos-service gains a real menu-item write path (create/update/activate/deactivate/soft-delete) that publishes MENU_ITEM_UPSERTED/MENU_ITEM_DELETED through the transactional outbox on every mutation, plus a D-05 backfill endpoint to re-emit events for pre-existing seeded items.**

## Performance

- **Duration:** ~25 min
- **Started:** 2026-07-19T12:00:00Z
- **Completed:** 2026-07-19T12:25:00Z
- **Tasks:** 3
- **Files modified:** 6 modified, 2 created

## Accomplishments
- MenuService/MenuServiceImpl gained createItem/updateItem/setActive/deleteItem/republishAllActive, each mutation funneling through a shared `publishUpserted(MenuItem)` helper that publishes `MENU_ITEM_UPSERTED` in the same transaction as the entity save
- deleteItem is soft-delete only (`deletedAt` + `active=false`), publishing `MENU_ITEM_DELETED` — never a hard DELETE, so historical recipes/orders referencing the menuItemId stay resolvable
- MenuController exposes POST/PUT `/items`, PATCH `/items/{id}/activate|deactivate`, DELETE `/items/{id}` — all gated only by the existing class-level `@RequiresFeature("FEATURE_POS")`, no new permission code
- InternalPosController exposes `POST /internal/menu-items/republish` (X-Internal-Service guarded, X-Tenant-Id fallback, bare Long return) — the D-05 backfill so pre-existing seeded menu items enter inventory-service's future `menu_item_catalog` read-model
- New `MenuItemEventPublishingIT` — 5 integration tests proving each mutation emits exactly the right outbox event type, soft-delete non-destructiveness, and republish's exact per-active-item count

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Test bug] Test-only cleanup added to MenuItemEventPublishingIT to prevent cross-test-method contamination**
- **Found during:** Task 3 (first `mvn verify -Dit.test=MenuItemEventPublishingIT` run)
- **Issue:** `republishAllActive_emitsOneUpsertedPerActiveItem` initially failed (`expected: 2 but was: 4`). Root cause: `MenuItemRepository.findByActiveTrueOrderByName()` (as specified verbatim in the plan) relies solely on the Hibernate `tenantFilter`, which is only enabled by `TenantFilterInterceptor` on the HTTP request path. This IT calls `MenuService` directly (mirroring `SettlementSemanticsIT`'s convention), so the filter is never enabled and the query saw menu items left behind by earlier test methods in the same class (PosTestBase's Testcontainers Postgres is a static singleton shared across all test methods in a JVM run).
- **Fix:** Added `menuItemRepository.deleteAll()` + `menuCategoryRepository.deleteAll()` to the test's `@BeforeEach`, scoped only to this test class's own setup (same pattern as the existing `outboxRepository.deleteAll()`). No production code was changed — `findByActiveTrueOrderByName()` matches the plan's spec exactly and behaves correctly on the real HTTP path where the tenant filter is active.
- **Files modified:** services/pos-service/src/test/java/io/restaurantos/pos/MenuItemEventPublishingIT.java (test-only)
- **Verification:** All 5 tests pass after the fix (`mvn -pl services/pos-service verify -Dit.test=MenuItemEventPublishingIT -DfailIfNoTests=false`)
- **Committed in:** 6ed83ca (Task 3 commit)
