# Phase 08.1 — POS-Inventory Depletion Activation — CONTEXT

## Summary

Phase 8 built the inventory-service AND wired the full `ORDER_CLOSED → depletion` loop, but the loop is **inert in practice** because recipes cannot be authored against real menu items, inventory has no knowledge of the POS menu catalog, and un-recipe'd sales silently deplete nothing. This phase **activates** the loop: menu-item sync, recipe authoring + validation, coverage/observability, and a live proof. It is a vertical slice, not the full inventory frontend module.

## Verified current state — DO NOT REBUILD (confirmed by code investigation, citations in RESEARCH.md)

- POS publishes `ORDER_CLOSED` to `pos.topic`/`pos.order.closed` (type `ORDER_CLOSED`), payload carries `items[]{menuItemId, name, qty, unitPricePaisa, lineTotalPaisa}` + `closedAt`; tenant/branch on the `EventEnvelope`.
- Trigger is Paid AND Served (`OrderServiceImpl.maybeCloseOrder`), single-fire, post-void/comp resolution.
- inventory `inventory.order-closed.queue` is already bound to `pos.topic`/`pos.order.closed` (code + `deploy/init/rabbitmq-definitions.json`); `OrderClosedConsumer` uses the tolerant `eventObjectMapper`, is idempotent via `processed_events`, and DLQs on failure. Payload contract is a field-for-field match.
- `DepletionService` = FEFO + aggregate-MAC COGS + `SELECT FOR UPDATE` + `STOCK_DEPLETED` via transactional outbox. Missing-recipe = silent per-line skip today (`DepletionService.java:97-103`).
- Recipe/BOM model + API exist: `POST /api/v1/inventory/recipes` (versioned), `Recipe.menuItemId (UUID)` matched directly to pos MenuItem PK with **no validation/sync/FK**.
- No inventory frontend (`/app/inventory` = `comingSoon`). No `MENU_ITEM_*` event, no Feign, no cross-DB linkage today.

## Locked decisions

- **D-01** (INV-12): Depletion trigger stays `ORDER_CLOSED` (Paid AND Served). NOT kitchen-fire-time / served-time. Kitchen-service stays out of the inventory loop (it holds no data inventory lacks and would only add a double-count surface).
- **D-02** (INV-09): Menu-item linkage via an **event-driven read-model** — pos publishes `MENU_ITEM_UPSERTED`/`MENU_ITEM_DELETED`; inventory maintains a `menu_item_catalog` table. NOT Feign-on-demand, NOT UI-passthrough-only.
- **D-03** (INV-11): Missing-recipe is **surfaced, not silent, not hard-fail** — deplete the covered lines and publish `DEPLETION_INCOMPLETE{orderId, closedAt, missingMenuItemIds[]}`; provide a recipe-coverage report. Depletion remains non-blocking (partial depletion + signal), so a single un-recipe'd item never blocks `ORDER_CLOSED` processing.
- **D-04** (INV-10): **Vertical-slice UI first** — a minimal `/app/inventory` recipe-builder + coverage dashboard, not the full inventory module (ingredients/stock/receipts/counts/transfers UIs come later).
- **D-05** (INV-09): Backfill existing menu items via a pos republish endpoint/job (`MENU_ITEM_UPSERTED` for every active item per tenant), because events only cover go-forward changes.
- **D-06** (out of scope): Finance consuming `STOCK_DEPLETED` to post the COGS journal entry is **Phase 9** (Order-to-Ledger). This phase only emits the event.
- **D-07**: `menu_item_catalog` uses **soft-delete** (keep the row, mark inactive) so historical recipes/movements referencing a removed menu item stay resolvable; keyed `(tenant_id, menu_item_id)`.
- **D-08**: The new rabbit binding is declared in BOTH `InventoryRabbitConfig` and `deploy/init/rabbitmq-definitions.json` (this repo keeps topology in both; keep them in sync).

## Constraints / conventions to honor

- inventory FORCE-RLS convention: new tenant-scoped tables get `ENABLE`+`FORCE ROW LEVEL SECURITY` + NULLIF-guarded `tenant_isolation`; infra/read-model tables that must be read outside tenant context follow the V2 non-RLS pattern (decide per-table — the catalog is tenant-scoped and read under tenant context on the API path, but the ORDER_CLOSED consumer path already sets tenant from the envelope, so FORCE-RLS on the catalog is fine and preferred).
- Events go through the shared-lib `DomainEventPublisher` (transactional outbox); consumers use `@Qualifier("eventObjectMapper")` (tolerant) + `processed_events` idempotency + DLQ.
- Frontend uses the enforced four-layer pattern (Zod schema → adapter → repository → TanStack hook → component); ESLint `no-restricted-imports` on `components/**`.
- Inventory controllers gate on OPA (`inventory.item.view`/`inventory.item.manage`) via `InventoryAuthorizationService`.
- New DB migrations: confirm the next free Flyway version per service before adding (inventory is at V3 after the D6 registry).

## Out of scope
- COGS → finance posting (Phase 9). Full inventory frontend module. Changing the depletion trigger. Any kitchen-service change.
