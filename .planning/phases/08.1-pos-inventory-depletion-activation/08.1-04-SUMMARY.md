# Phase 08.1 Plan 04: inventory recipe-builder UI + coverage dashboard Summary

**Flipped `/app/inventory` from a `comingSoon` sidebar placeholder into a live four-layer FE vertical slice (schema/adapter/repository/hooks + MSW fixtures) with a recipe-builder page (menu-item picker + dynamic ingredient-line form), a coverage dashboard, and a Playwright smoke spec — closing INV-10 and making the whole 08.1 activated loop (catalog sync → validation → coverage → authoring) usable by a real operator.**

## Performance

- **Duration:** ~35 min
- **Completed:** 2026-07-19
- **Tasks:** 3
- **Files modified:** 3 modified, 12 created

## Accomplishments
- `inventory.schema.ts` defines Zod schemas mirroring `MenuItemCatalogDto`/`IngredientDto`/`UomDto`/`RecipeDto`/`RecipeLineDto`/`CoverageResponse`/`MissingMenuItemDto` field-for-field, reusing purchasing's `qtyField` union-string/number-to-string transform for every BigDecimal-backed field (`reorderPoint`, `toBaseFactor`, `qty`, `yieldServings`, `yieldPct`)
- `inventory.adapter.ts`/`inventory.repository.ts`/`lib/hooks/inventory/use-inventory.ts` mirror purchasing's four-layer chain exactly: identity adapters, a `InventoryRepository` static object (`listMenuItems`/`listIngredients`/`listUoms`/`createRecipe`/`listRecipeVersions`/`getCoverage`), and hooks with explicit query-key invalidation — `useCreateRecipe`'s `onSuccess` invalidates both `["inventory","coverage"]` and the broad `["inventory","recipes"]` key
- `mocks/inventory.handlers.ts` seeds 3 active menu items, 3 ingredients, 2 UOMs, and one pre-existing recipe version (Zinger Burger v1) in mutable in-memory arrays; `createRecipe` supersedes any prior current version for the same menu item and `getCoverage` recomputes covered/missing live from the same arrays — registered into `mocks/server.ts`'s `setupServer(...)` (see Deviations)
- `inventory-recipe-builder.test.ts` (4 tests): catalog/ingredient/UOM listing round-trips, `createRecipe` returns the submitted `menuItemId`/version 1/lines, `getCoverage` returns the seeded shape, and an empty `lines` array is rejected client-side before any network call
- `sidebar-nav-items.ts`'s Inventory nav entry drops `comingSoon: true` — permission/feature/href/icon untouched
- `app/(tenant)/app/inventory/layout.tsx` wraps children in `PermissionGuard(inventory.item.view)` + `FeatureGuard(FEATURE_INVENTORY)` with a Recipes/Coverage tab nav; `page.tsx` is a static landing linking to both
- `components/inventory/RecipeFormDialog.tsx` mirrors `PurchaseOrderFormDialog`'s full react-hook-form + `useFieldArray` + `createZodResolver` pattern: menu-item `<select>` (from `useMenuItemCatalog`), per-line ingredient/UOM `<select>`s (from `useIngredients`/`useUoms`), qty/yieldPct text inputs, an `effectiveFrom` date input converted to a full ISO instant on submit, toast on success/error
- `app/(tenant)/app/inventory/recipes/page.tsx`: menu-item picker + existing-recipe-versions table (version/current/effectiveFrom/yieldServings/line-count) + `EmptyState` when none exist, with the "New recipe version" trigger gated behind `PermissionGuard(inventory.item.manage)` (no fallback, matching the existing nested-guard precedent in `finance/periods/page.tsx`)
- `app/(tenant)/app/inventory/coverage/page.tsx`: three `AnimatedNumber` stat tiles (Total Active Menu Items/Covered/Missing) plus a missing-menu-item table or an `EmptyState` when fully covered
- `e2e/inventory-recipe-builder.spec.ts`: smoke test using `manager@demo.local`/`Manager#2026` (MANAGER role — both `inventory.item.view` and `inventory.item.manage`, no TOTP) proving the sidebar link is live, the recipe-builder select renders real options, and the coverage stat tiles render

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Registered inventory MSW handlers in `mocks/server.ts`, not `mocks/handlers.ts`**
- **Found during:** Task 1, while reading the `<read_first>`-referenced `frontend/mocks/purchasing.handlers.ts` + `frontend/mocks/handlers.ts` to find the registration point
- **Issue:** The plan's `files_modified` and action text describe registering the new handlers into `mocks/handlers.ts`'s exported `handlers` array. The actual current codebase (confirmed via `Grep`/`Read` of `mocks/server.ts`) registers every domain handler module (`purchasingHandlers`, `financeHandlers`) directly into `setupServer(...)` in `server.ts`; `handlers.ts` itself only exports the auth/feature-flag handlers and is never imported by the domain handler files.
- **Fix:** Created `mocks/inventory.handlers.ts` exporting `inventoryHandlers` (mirroring `purchasingHandlers`'s shape) and added it to `mocks/server.ts`'s `setupServer(...handlers, ...purchasingHandlers, ...financeHandlers, ...inventoryHandlers)` call. `mocks/handlers.ts` itself was left untouched.
- **Files modified:** `frontend/mocks/inventory.handlers.ts` (new), `frontend/mocks/server.ts` (modified)
- **Verification:** `npm --prefix frontend run test -- inventory-recipe-builder` — all 4 tests pass, proving the handlers are correctly wired into the MSW server.
- **Committed in:** `f8e32e0` (Task 1 commit)

**2. [Rule 3 - Blocking] Anchored `.gitignore`'s `coverage/` pattern so it stops matching the new route directory**
- **Found during:** Task 3, staging `app/(tenant)/app/inventory/coverage/page.tsx` for commit
- **Issue:** `.gitignore` had a bare `coverage/` entry (intended for vitest's test-coverage report output). Git's gitignore semantics match a bare directory-name pattern at any depth, so it also matched the plan's real route directory `app/(tenant)/app/inventory/coverage/` — `git status`/`git add` silently treated the new `page.tsx` as ignored, which would have made the file invisible to `git status --short` and never committed.
- **Fix:** Changed the pattern to `/frontend/coverage/` (anchored to the repo root, confirmed via `frontend/vitest.config.ts`'s default `coverage.provider` config that vitest's actual report output lands at `frontend/coverage/`, not nested under `app/`).
- **Files modified:** `.gitignore`
- **Verification:** `git check-ignore -v "frontend/app/(tenant)/app/inventory/coverage/page.tsx"` now exits 1 (not ignored); `git status --short` shows the file as untracked/staged correctly.
- **Committed in:** `94244b0` (Task 3 commit)
