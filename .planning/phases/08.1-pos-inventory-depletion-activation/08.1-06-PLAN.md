  <name>Task 2: Fix TenantAwareMessageProcessor so the RLS GUC lands on the tx connection before DML (GREEN)</name>
  <files>shared-lib/src/main/java/io/restaurantos/shared/tenant/TenantAwareMessageProcessor.java</files>
  <action>
    In `TenantAwareMessageProcessor.process(...)`, immediately AFTER `tenantContext.set(tenantId, branchId, null, null)` and BEFORE the handler runs any DML, call `TenantGucHelper.apply(entityManager, tenantContext)`. `TenantGucHelper` is in the same package (`io.restaurantos.shared.tenant`) and its static `apply(EntityManager, TenantContext)` issues `SELECT set_config('app.current_tenant_id', :tid, true)` (and the branch GUC when present) on the current session. No new import is required.

    Why this is correct: `process()` is `@Transactional`, so by the time its body runs, Spring has already `BEGIN`-ed the transaction and bound one JDBC connection. `TenantGucHelper.apply` runs `set_config(..., is_local=true)` AFTER that `BEGIN`, so — unlike the checkout-time case that `TenantAwareDataSource` explicitly warns about (a transaction-local set at checkout is discarded before the subsequent BEGIN) — this value lives for the remainder of the open transaction and is on the same connection the handler's INSERT will use. Because it is transaction-local, it auto-reverts at commit/rollback, so it cannot leak across pooled connections.

    Chosen approach vs the alternative (record the rationale in the SUMMARY): the considered alternative was to set `TenantContext` BEFORE the `@Transactional` boundary (split `process()` into a non-transactional outer method that sets context plus a `@Transactional` inner handler on a separate proxied bean) so the existing checkout-time GUC setter sees the tenant. It is rejected here because it restructures the transaction/proxy boundary of a shared-lib class that backs 10 consumers across 4 services (higher blast radius and self-invocation-proxy hazards), whereas the `TenantGucHelper` approach is a single additive statement inside the existing method that reuses a helper built for exactly this, is transaction-local (no pooled-connection leak), and is a no-op-equivalent for the already-working HTTP path (consumers only). Update the class javadoc to state that the RLS GUC is now applied explicitly at transaction start via `TenantGucHelper`, not relied upon from connection checkout (which sees an empty context for consumers).

    Then re-run the regression IT: it must now PASS. Publish the fixed artifact so downstream module-scoped builds resolve it: `mvn -f shared-lib/pom.xml clean install` (runs surefire + failsafe, including the new IT, and installs to local `.m2`).
  </action>
