<objective>
Give pos-service a real menu-item write path (create/update/activate/deactivate/delete — none exist today; MenuServiceImpl is read-only and every `menu_items` row currently only ever comes from the V1 seed migration) and wire MENU_ITEM_UPSERTED/MENU_ITEM_DELETED event publication onto every one of those mutations via the existing shared-lib DomainEventPublisher/outbox (D-02). Add the D-05 backfill endpoint `POST /internal/menu-items/republish` so pre-existing seeded items (which never went through the new write path) can be re-emitted once for inventory-service to pick up in 08.1-02.

Purpose: This is the pos-service half of the event-driven read-model (D-02) that inventory-service's `menu_item_catalog` (08.1-02) is built from — without a real mutation path and a backfill mechanism, MENU_ITEM_UPSERTED/DELETED have nothing to fire from.
Output: MenuService/MenuServiceImpl gain create/update/setActive/deleteItem/republishAllActive; MenuController exposes the corresponding HTTP endpoints; InternalPosController exposes the republish backfill; an IT proves each mutation writes the correct outbox row and republish emits the expected count.
</objective>

| HTTP client -> MenuController write endpoints | Authenticated JWT request mutates tenant-wide menu catalog data |
