## What Changed Since the Prior Verification

The prior VERIFICATION.md (2026-07-19, human_needed) proved all four Success Criteria correct at the **source/IT** level, but its one outstanding item — a live browser click-through of the recipe-builder UI — surfaced a real production bug during UAT that the prior source-level checks could not see: **every consumer write into a FORCE-RLS table was silently rejected (SQLState 42501)** because `TenantAwareMessageProcessor.process()` never applied the RLS GUC on its own transaction connection (a checkout-time-vs-transaction-time ordering bug in shared-lib, invisible to inventory-service's superuser Testcontainers ITs). This broke the menu-item catalog sync live (`GET /api/v1/inventory/menu-items` stayed `[]`) and, since `OrderClosedConsumer`→`DepletionService` shares the exact same code path, put the live depletion write at identical risk — a genuinely trustworthy gap the roadmap goal explicitly calls out ("becomes functional and trustworthy").

08.1-06 fixed the shared-lib bug (one additive `TenantGucHelper.apply` call) with a genuine RED→GREEN regression test against a non-superuser FORCE-RLS harness. 08.1-07 then re-verified BOTH affected loops live and closed the outstanding UI human-verification checkpoint with a human-approved Playwright run.
