## Accomplishments
- `LiveOrderClosedPayloadIT` (pos-service): creates a menu item via the real `MenuServiceImpl.createItem`, adds it to a real order, drives the order to CLOSED via `PosTestBase.closeViaServeAndPay` (the only real Paid-AND-Served close path, D-01), then extracts `menuItemId` from both the `MENU_ITEM_UPSERTED` and `ORDER_CLOSED` outbox envelopes and asserts they are identical — proving the catalog-sync contract (what pos syncs vs. what pos sells) does not independently drift.
- `LiveDepletionProofIT` (inventory-service): drives the full loop in production order —
  1. `MenuItemCatalogService.upsert(...)` (simulating a consumed MENU_ITEM_UPSERTED sync — the only place a catalog row is created, never a direct repository insert),
  2. `RecipeService.createVersion(...)` (the real catalog-validated authoring path — asserts it does NOT throw, proving 08.1-02's validation passes for a properly-synced item),
  3. a real `OrderClosedPayload` wrapped in an `EventEnvelope`, serialized to an AMQP `Message`, consumed through the real `OrderClosedConsumer.onMessage(...)` (tenant context cleared first so the consumer must read tenant from the envelope), then
  4. asserts: `qtyOnHand` 50 -> 42 (8 base units depleted), the single FEFO lot drained accordingly, exactly one `inventory_movements` DEPLETION row (movement_type=DEPLETION, reference_type=ORDER_CLOSED, reference_id=orderId, qty=-8, unit_cost=850), `total_cost_paisa` == `DepletionService.computeCogsPaisa(expectedBaseQty, 850)` computed INDEPENDENTLY in the test, a single `STOCK_DEPLETED` outbox row whose payload `totalCogsPaisa` equals that same value, and finally `getCoverage()` no longer lists the item as missing (loop closed back to 08.1-03).
- The lot receipt cost (1200) is deliberately set different from the branch-stock avg cost (850) so the COGS assertion structurally proves aggregate-MAC valuation, never lot cost (D-04 / Pitfall 9 / threat T-085-02).
- The manual UAT checklist for optionally re-confirming the same chain against the live dev stack lives in the plan's `<verification>` section (08.1-05-PLAN.md) and is carried forward unchanged; this plan stays autonomous/no-checkpoint, so the checklist is documentation for a human to optionally run — it is not a blocking gate.
