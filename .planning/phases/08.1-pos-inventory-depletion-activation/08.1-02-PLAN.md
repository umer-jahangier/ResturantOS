<objective>
Give inventory-service the pos-service-facing half of the event-driven read-model (D-02): a `menu_item_catalog` table populated by a new `MenuItemCatalogConsumer` that consumes 08.1-01's MENU_ITEM_UPSERTED/MENU_ITEM_DELETED events (soft-delete per D-07, dual RabbitMQ topology per D-08), a `GET /api/v1/inventory/menu-items` endpoint for the future recipe-builder picker, and — the crux of INV-09 — real validation in `RecipeService.createVersion` that a recipe can only be authored against a `menuItemId` that actually exists and is active in the tenant's synced catalog.

Output: `menu_item_catalog` table + entity + repository, `MenuItemCatalogConsumer` (idempotent, dual-event-type), `GET /menu-items`, and `RecipeService.createVersion`'s new 404 MENU_ITEM_NOT_FOUND gate.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| RabbitMQ -> MenuItemCatalogConsumer | Untrusted at-least-once delivery of TWO distinct event types on one queue |
| HTTP client -> RecipeController.createVersion | Client-supplied menuItemId now cross-checked against tenant-scoped catalog data |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-082-01 | Repudiation / Tampering | MenuItemCatalogConsumer + processed_events | high | mitigate | `tryProcess(CONSUMER_NAME='inventory.menu-item-catalog', eventId)` dedups both MENU_ITEM_UPSERTED and MENU_ITEM_DELETED — duplicate delivery of either type is a no-op; proven by MenuItemCatalogConsumerIT's duplicate-eventId case. |
| T-082-02 | Tampering | Dual-event-type dispatch on one queue | medium | mitigate | Consumer reads `envelope.eventType()` from the parsed JsonNode BEFORE choosing the payload class; an unrecognized eventType is logged and skipped rather than mis-dispatched; unparseable messages reject-without-requeue to the DLQ. |
| T-082-03 | Elevation of Privilege | menu_item_catalog table | high | mitigate | FORCE ROW LEVEL SECURITY + NULLIF-guarded tenant_isolation policy (matches every V1 domain table); MenuItemCatalogController/RecipeController both call InventoryAuthorizationService before touching data. |
| T-082-04 | Tampering (data integrity) | RecipeService.createVersion menuItemId validation | medium | mitigate | `existsByTenantIdAndMenuItemIdAndActiveTrue` rejects any menuItemId not present/active in the CALLER'S OWN tenant catalog (tenant-scoped query) before a Recipe row is ever created — closes the "no validation/sync/FK" gap confirmed in 08-CONTEXT.md. |
| T-082-SC | Tampering (supply chain) | dependencies | low | accept | No new packages. |
</threat_model>
