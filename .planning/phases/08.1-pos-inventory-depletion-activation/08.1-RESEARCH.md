**Inventory consumer + depletion**
- Binding present: `InventoryRabbitConfig.java:39-51` (`inventory.order-closed.queue` → `pos.topic`/`pos.order.closed`, DLX'd) and `deploy/init/rabbitmq-definitions.json:91`.

**Kitchen (for the record — stays OUT)**

**Topology**
- `pos.topic` is a topic exchange; `pos.order.closed` already fans out to inventory/finance/crm/reporting queues (`rabbitmq-definitions.json:90-101`). Adding a consumer = consumer-side binding only, no pos change. Topology lives in BOTH `deploy/init/rabbitmq-definitions.json` and each service `@Configuration` (idempotent no-ops).

**Gaps confirmed**

2. No inventory frontend (`sidebar-nav-items.ts:155` `comingSoon: true`; no `frontend/app/**/inventory/**`).

## B. Technical approach per wave

**Wave A — pos-service menu events**
- In `MenuServiceImpl` create/update/activate/deactivate/delete paths, publish via existing `DomainEventPublisher` (outbox). Keys `pos.menu_item.upserted` / `pos.menu_item.deleted`; types `MENU_ITEM_UPSERTED`/`MENU_ITEM_DELETED`.
- Payload `MenuItemUpsertedPayload{menuItemId, name, categoryId, categoryName, active, basePricePaisa, updatedAt}`; `MenuItemDeletedPayload{menuItemId}`. tenant/branch on envelope.
- Backfill: internal endpoint (behind the internal-service secret filter) `POST /internal/menu-items/republish` re-emitting `MENU_ITEM_UPSERTED` for all active items per tenant.
- Tests: IT asserting each mutation emits the right outbox row; republish emits N.

**Wave B — inventory catalog + validation**
- `V4__menu_item_catalog.sql`: `menu_item_catalog(tenant_id, menu_item_id UUID, name, category_name, active bool, base_price_paisa, updated_at, PK(tenant_id, menu_item_id))` + FORCE RLS + `tenant_isolation` + grant. (Confirm V4 is next free — inventory is at V3 after the D6 registry.)
- `MenuItemCatalogConsumer`: bind `inventory.menu-item.queue` → `pos.topic` keys `pos.menu_item.upserted`/`.deleted` (add to `InventoryRabbitConfig` + `rabbitmq-definitions.json`), tolerant mapper, `processed_events` idempotency, upsert / soft-delete (D-07), tenant from envelope, DLQ on poison.
- Validation: `RecipeService.createVersion` rejects `menuItemId` absent/inactive in catalog → `404 MENU_ITEM_NOT_FOUND`.
- `GET /api/v1/inventory/menu-items` (OPA `inventory.item.view`) for the UI picker.

**Wave C — coverage + observability**
- `GET /api/v1/inventory/recipes/coverage` → `{totalActiveMenuItems, covered, missing:[{menuItemId,name}]}` (left-join catalog-active vs effective recipes).
- `DepletionService`: when ≥1 line has no effective recipe, still deplete covered lines and publish `DEPLETION_INCOMPLETE{orderId, closedAt, missingMenuItemIds[]}` to `inventory.topic` (`inventory.depletion.incomplete`) via outbox; remove the all-empty silent return so the signal always fires when something was skipped.

**Wave D — recipe-builder UI (vertical slice)**
- Flip `/app/inventory` `comingSoon`; build recipe-builder (menu-item picker from `GET /inventory/menu-items`, ingredient lines qty+UOM+yield%, `effectiveFrom`, submit to `POST /inventory/recipes`) + coverage dashboard from `GET /inventory/recipes/coverage`. Four-layer pattern + vitest + a Playwright smoke.

**Wave E — live E2E proof**
- Seed ingredient + opening stock + recipe for a real menu item; ring an order (create→add→fire→pay→serve) to `ORDER_CLOSED`; assert FEFO depletion, `inventory_movements` DEPLETION row, `STOCK_DEPLETED` with correct `totalCogsPaisa`. Automated IT/E2E where feasible + manual UAT checklist.

## C. Pitfalls
- **Backfill**: without the republish step, only menu items changed after deploy enter the catalog — coverage report makes the gap visible immediately.
- **Soft-delete** the catalog (D-07): a hard delete would orphan historical recipes/movements.
- **Topology dual-source** (D-08): declare the new binding in both places or the broker JSON drifts from code.
- **Partial depletion** (D-03): deplete covered lines + signal; never all-or-nothing (one un-recipe'd item must not block real depletion or DLQ the whole order).
- **Envelope tenant**: read tenant from `EventEnvelope`, not the payload body (payloads omit it).
- **Idempotency**: reuse `processed_events` for the new catalog consumer too (distinct consumer name).
