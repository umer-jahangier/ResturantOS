3. Try each statement against the live server one at a time and keep the form that actually executes.

**What the DDL must achieve** (the behaviour is fixed even if the syntax needs adjusting):
- A settings profile (`nlq_readonly_profile`) that CONSTRAINS, not merely defaults:
  - `readonly = 1` â€” no INSERT/ALTER/CREATE/DROP; and make it non-overridable by the client (the `READONLY`/`CONST` constraint form) so a crafted query cannot raise its own ceiling via a `SETTINGS` clause.
  - `max_execution_time = 5` with a `MAX` constraint (the 5s timeout of NLQ-02).
  - `max_result_rows = 10000` with a `MAX` constraint (the row cap of NLQ-02).
  - `result_overflow_mode = 'throw'` â€” the cap must ERROR, not silently truncate. A truncated result that looks complete is worse than a rejection.
- A user `nlq_readonly` IDENTIFIED BY the password from `${CLICKHOUSE_READONLY_PASSWORD}`, assigned that profile.
- Grants: `GRANT SELECT ON clickhouse_analytics.sales_order_facts TO nlq_readonly` (and the same for `sales_item_facts`, `purchase_tax_facts`, `till_session_facts`) â€” **per-table, never `ON clickhouse_analytics.*`**. This makes the table allowlist enforced at the database layer too, so even a total failure of the AST validator's stage-3 allowlist cannot read a table we never granted. Defence in depth, exactly as PROJECT.md's Security constraint requires.
- Explicitly do NOT grant `INSERT`, `CREATE`, `DROP`, `ALTER`, or access to `system.*`.

Use `CREATE ... IF NOT EXISTS` (or `CREATE OR REPLACE`) forms so the file is re-runnable.
Password interpolation: the applier script (Task 3) substitutes `${CLICKHOUSE_READONLY_PASSWORD}` from the environment â€” the SQL file must contain the placeholder, never a literal secret.
  </action>

`apply.sh`: a bash script that
