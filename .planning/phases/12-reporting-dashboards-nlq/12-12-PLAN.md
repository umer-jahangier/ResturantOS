<objective>
Close GAP A (12-10-E2E §1h) — the load-bearing RPT-02 blocker. The real gateway's
`JwtGlobalFilter` requires a `Bearer` Authorization HEADER and has no query-param fallback, and
neither `/api/v1/reporting/dashboard` nor `/api/v1/kitchen` is public. A browser's native
`WebSocket` API cannot set an Authorization header on the handshake, so the gateway 401s the
upgrade for ANY caller/token before it ever reaches reporting-service (or kitchen-service). The
push mechanism itself is proven (<5s, direct to reporting-service); only the gateway hop is broken.

Purpose: RPT-02 cannot be Complete until a real browser can reach the dashboard socket through the
gateway. This fix also unblocks KDS, which shares the exact root cause.

Output: A gateway change that lets a WebSocket handshake carry its JWT as `?token=`, validates that
token at the edge with the same RS256/JWKS machinery used for headers, injects the same downstream
identity headers, and forwards the upgrade — while leaving ordinary REST auth (header-only)
untouched. Proven with a real handshake through port 8080 returning 101, not 401.
</objective>

<task type="auto">
  <name>Task 1: Add a WS-upgrade query-param JWT fallback to JwtGlobalFilter (edge-security-preserving)</name>
  <files>gateway/src/main/java/io/restaurantos/gateway/filter/JwtGlobalFilter.java</files>
  <action>
In `JwtGlobalFilter.filter(...)`, after the existing `isPublicPath` short-circuit and BEFORE the
existing `Authorization` header check, insert a WebSocket-upgrade fallback branch:

1. Add a private constant list `WS_UPGRADE_PATHS = List.of("/api/v1/reporting/dashboard/", "/api/v1/kitchen/")`.

2. Detect a WebSocket upgrade request: read the `Upgrade` request header
   (`exchange.getRequest().getHeaders().getUpgrade()` or
   `getFirst(HttpHeaders.UPGRADE)`) and test `equalsIgnoreCase("websocket")`.

3. Only when BOTH the request is a websocket upgrade AND `path` starts with one of
   `WS_UPGRADE_PATHS`, take the query-param path:
   - Read the `token` query param via
     `exchange.getRequest().getQueryParams().getFirst("token")`.
   - If it is null/blank → `return writeError(exchange, HttpStatus.UNAUTHORIZED, UNAUTHENTICATED_BODY)`.
   - Validate it with the EXISTING `validateAndParse(token)` (same RS256 + JWKS + expiry path used
     for header tokens). On any exception → 401 via `writeError`.
   - On success, feed the parsed `JwtClaims` through the SAME
     `tenantResolutionSupport.resolve(exchange, claims).flatMap(...)` block the header path uses,
     injecting `X-Tenant-Id`, `X-User-Id`, and (when present) `X-Impersonated-By`, then
     `chain.filter(mutated)`. Reuse the existing mutation logic — extract it into a private helper
     `Mono<Void> authorizeAndForward(ServerWebExchange exchange, JwtClaims claims, GatewayFilterChain chain)`
     and call it from BOTH the header path and this new WS path to avoid duplication.

4. Leave the existing header path (for all non-upgrade requests, and for any request that is NOT a
   WS upgrade on the two socket prefixes) exactly as-is. Do NOT add the socket paths to
   `PUBLIC_PATHS` — an unauthenticated upgrade must still be rejected.

Why this shape (edge security): the query-param fallback is deliberately restricted to genuine
WebSocket upgrades on the two known socket path prefixes. Ordinary REST calls (no `Upgrade` header)
continue to require the `Authorization` header, so a JWT is never accepted from a URL/query string
for normal API traffic (where it would leak into logs/history). The token is fully validated at the
gateway (not merely allow-listed through), and `DashboardWebSocketHandler`/`KdsWebSocketHandler`
independently re-validate the same `?token=` at the handler — defense in depth. This is the same
JWT-in-query-param contract both handlers already expect.
  </action>

`PUBLIC_PATHS` still does NOT contain `/api/v1/reporting/dashboard` or `/api/v1/kitchen`
(`grep -n "dashboard\|kitchen" gateway/src/main/java/io/restaurantos/gateway/filter/JwtGlobalFilter.java` returns nothing inside PUBLIC_PATHS).
  </verify>
  <done>
The gateway validates a `?token=` JWT for WS upgrades on the two socket prefixes using the same

request path is unchanged; no socket path is public.
