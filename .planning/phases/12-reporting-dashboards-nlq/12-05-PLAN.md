**`ReportDefinition`** (record): `String code, String title, String category, String sql, List<String> columns`. The `sql` is a parameterised ClickHouse query with `?` binds — NEVER string-interpolated values.

**`ReportCatalog`** — a `@Component` holding the named reports. Build the set that today's facts can honestly support. Do NOT pad the list to reach the spec's "40 named reports" — a report backed by no data is a lie. Ship these, each with real SQL against the 12-02 fact tables:

*Sales (from `sales_order_facts` / `sales_item_facts`):*
- `sales-by-day` — revenue, tax, discount, order count per `business_date`
- `sales-by-item` — qty, gross revenue per `menu_item_id`/`item_name`. **Include `cogs_paisa` / `gross_margin_paisa` columns in the SELECT but they will be NULL until Phase 8** — the DTO must carry them as `Long` (nullable), and the API must return `null`, not `0`. Add a `dataNotes` field on the response saying `"COGS and margin require Inventory (Phase 8) and are not yet available"` so the frontend can render "—" with an explanation instead of a misleading zero.
- `sales-by-hour` — order count + revenue bucketed by hour of `closed_at` (peak-hours view)
- `sales-by-order-type` — dine-in vs takeaway vs delivery split
- `discount-summary` — discount total and rate per business day

*Cash (from `till_session_facts`):*
- `till-sessions` — per-session opening/closing/variance for a date range

*Purchasing (from `purchase_tax_facts`):*
- `purchases-by-vendor` — spend + input tax per vendor for a date range

**`ReportService.run(String code, ReportRequest req)`**:
- Resolve the definition from the catalog; unknown code → 404 (a typed exception the shared `GlobalExceptionHandler` maps).
- **Tenant/branch binds are SERVER-DERIVED**: `TenantContext.requireTenantId()` for tenant; branch from `TenantContext` / the caller's JWT. `tenant_id` is NEVER a request parameter — this is the same structural rule decision 10-10-B established for purchasing's list endpoints ("impossible-by-construction tenant isolation, not just test coverage"). `branchId` MAY be an optional request param, but it must be validated against the caller's accessible branches, and an OWNER-role caller may pass `null` to mean "all my branches".
- Every catalog SQL has `WHERE tenant_id = ? AND (? IS NULL OR branch_id = ?) AND business_date BETWEEN ? AND ?` as its leading predicate. ClickHouse's `nullIf`/coalesce semantics differ from Postgres — verify the "optional branch" bind actually works against the live container rather than assuming; if it's awkward, build two SQL variants (branch-scoped and tenant-wide) in the definition instead of one clever one. Clarity beats cleverness in a query that enforces isolation.
- Cap `LIMIT` (e.g. 10000) on every report; reject date ranges longer than a configured max (e.g. 400 days) to keep P95 honest.
- Time the run with `System.nanoTime()` and INSERT a `report_run_log` row (tenant, branch, code, params JSON, row_count, duration_ms, run_by). This log IS the latency evidence for RPT-01 — the P95 claim must be computable from data, not asserted in prose.

**`ReportResultDto`**: `{ code, title, columns: List<String>, rows: List<Map<String,Object>>, rowCount, durationMs, dataNotes: List<String> }`.

**`FbrTaxSummaryDto`**:
```
{
  branchId, branchName, ntn, fbrStrn,          // header, from user-service
  periodFrom, periodTo,                         // business-date range
  outputTaxPaisa,                               // SUM(tax_paisa) FROM sales_order_facts
  taxableSalesPaisa,                            // SUM(subtotal_paisa) FROM sales_order_facts
  inputTaxPaisa,                                // SUM(input_tax_paisa) FROM purchase_tax_facts
  taxablePurchasesPaisa,                        // SUM(subtotal_paisa) FROM purchase_tax_facts
  netPayablePaisa,                              // outputTaxPaisa - inputTaxPaisa (MAY be negative = refundable credit; do NOT clamp at 0)
  salesOrderCount, purchaseInvoiceCount
}
```
All money is `Long` paisa (PROJECT.md: never Double/Float/BigDecimal for money).

**`FbrTaxSummaryService.summary(branchId, from, to)`**:
- Two aggregate ClickHouse queries (one per fact table), each with the server-derived `tenant_id` bind and the `business_date BETWEEN ? AND ?` range. The range uses BUSINESS dates (the 4-hour-offset dates the ETL already computed) — a tax period boundary that disagrees with the sales-day boundary would produce numbers that don't reconcile with the sales reports.
- `netPayablePaisa = outputTaxPaisa - inputTaxPaisa`. If input exceeds output the result is negative — that is a legitimate refundable input-tax credit. **Do not clamp it to zero.** Silently flooring a refund to zero is a financial misstatement.
- Header: Feign-call user-service's `GET /internal/users/branches/{branchId}` (confirmed real, `Docs/agent-specs/04-internal-api-contracts.md:65`) for `name`, `ntn`, `fbrStrn` (these fields exist on the `branches` table per the spec). Use `UserInternalClient` from 12-03, which is wired to the `FeignClientConfig` carrying the verbatim `forwardCallerJwt()` from commit `990026a`.
  **This call runs inside a real HTTP request context, so the caller's `Authorization` header IS present and WILL be forwarded.** That is the difference from 12-03's ETL consumer path (AMQP thread, no inbound request, no JWT to forward). If this call 401s or 503s, the cause is almost certainly a missing JWT forward — check `FeignClientConfig` before checking anything else. That exact bug (commit `990026a`) took down every PO approval in Phase 10 and was invisible to the ITs because they stubbed the client.
- If user-service is unreachable, return the report with the tax numbers and a `null` NTN/STRN plus a `dataNotes` entry — the tax figures are the point; a missing header should degrade the report, not fail it.

Scope note for the SUMMARY: this is a named INTERNAL report shaped for FBR bookkeeping. There is no FBR/IRIS e-filing API integration anywhere in the specs and none is being built. Say so plainly so no one later assumes a filing pipeline exists.
