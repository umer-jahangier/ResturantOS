**`NlqAskBox`** — a textarea + submit. Show example questions ("What was revenue last week?", "Which items sold most yesterday?") — an empty text box with no affordances gets no usage. Disable submit while the mutation is pending; show the `.skeleton` shimmer (DS-02) while waiting.

**`NlqResultPanel`** — on success:
- The narrative (if present) as a lead paragraph.
- The result table (money via `MoneyDisplay` — paisa → PKR; `null` → `—`).
- **A collapsible "Show the SQL that ran" disclosure containing the EXECUTED sql.** This is deliberate: the SQL is post-validation and tenant-scoped, so it is safe to show, and showing it is what makes an AI-generated answer trustworthy. A number with no provenance from a language model is not something a restaurant owner should act on. Do not hide it.
- If `cacheHit`, a small "cached result" badge — users need to know when they're seeing a 60-second-old answer.
- `rowCount` + `durationMs` in a footer.

**`NlqRejectionNotice`** — the piece that decides whether this feature feels intelligent or broken. Map each `RejectionCode` to a plain-English explanation and a suggested next action. Never show the raw code alone, never show a stack trace, never show the rejected SQL (12-07 deliberately does not return it — it would be an oracle for probing the validator).
Suggested copy:
- `TENANT_FILTER_MISSING` / `BRANCH_FILTER_MISSING` → "That question produced a query we couldn't safely limit to your restaurant's data, so we didn't run it. Try being more specific — for example, name a date range or a branch."
- `SHAPE_INVALID` → "This tool can only read data, never change it. Try rephrasing as a question."
- `TABLE_NOT_ALLOWED` → "That question needs data your role can't access."
- `PII_COLUMN_DENIED` → "That question asks for personal customer information, which this tool won't return."
- `PARSE_FAILED` → "We couldn't turn that into a valid query. Try rephrasing."
- `LIMIT_INVALID` → "That query asked for too many rows. Narrow it down."
- `QUERY_TIMEOUT` → "That query took too long. Try a shorter date range."
- `QUOTA_EXCEEDED` (429) → "You've used all your questions for this hour (or month). Try again later." Read the message body for which quota it was.
- 503 → "The question service is temporarily unavailable. Please try again."
Note the framing on the tenant/branch cases: the honest message is "we refused because we couldn't PROVE it was safe" — not "you did something wrong". The validator rejecting an exotic-but-legitimate query is a known, accepted cost of the design (12-04), and the copy should reflect that rather than blaming the user.
