  <action>
Follow the four-layer abstraction exactly (it is ESLint-enforced: `components/**` may not import `@/lib/api-client` — decision 04-01-E). Layers: Zod schema → adapter → repository (always `.parse()`, never the non-throwing variant) → TanStack Query hook → component.

**`reporting.schema.ts`** — Zod schemas for `ReportDefinition`, `ReportResultDto`, `FbrTaxSummaryDto`, `DashboardTileDto`.
**BUILD THESE FROM THE REAL JAVA DTOs, NOT FROM PROSE.** Open `services/reporting-service/src/main/java/io/restaurantos/reporting/dto/*.java` and mirror them field-for-field. This codebase has a documented, repeated failure mode where plan prose disagreed with the actual DTO — decisions 10-10-A, 10-12-A (a `BigDecimal` serialises as a JSON *number*, not a string), 10-13-A (field names and shapes both wrong), 10-14-A (`rejectReason` not `rejectionReason`). The Java source is the wire contract. Read it.
Specific traps to check for:
- Money is `Long` paisa on the Java side and arrives as a JSON **number**. If any value could exceed `Number.MAX_SAFE_INTEGER` it must be handled as a string/BigInt — check how `finance.schema.ts` / `purchasing.schema.ts` already model paisa and be consistent with them.
- `cogsPaisa` / `grossMarginPaisa` are **nullable** (`z.number().nullable()`), NOT optional-with-default-0. A `.default(0)` here would manufacture a false "zero cost" and defeat the whole point of 12-03's NULL contract.
- `ReportResultDto.rows` is `List<Map<String,Object>>` → `z.array(z.record(z.string(), z.unknown()))`.
- `dataNotes` is a `string[]`.

**`reporting.adapter.ts`** — API shape → domain model. Keep money as paisa in the model; format only at render time via `MoneyDisplay`/`MoneyUtils` (PROJECT.md: display only via MoneyUtils).

**`reporting.repository.ts`** — `listReports()`, `runReport(code, {branchId?, from, to})`, `fbrTaxSummary({branchId, from, to})`, `dashboardTiles(branchId)`. Every method `.parse()`s the response before adapting.

**`use-reports.ts`** — TanStack hooks: `useReports()`, `useRunReport(code)`, `useFbrTaxSummary(params)`, `useDashboardTiles(branchId)`. Add the query keys to `lib/hooks/query-keys.ts` following the existing branch-scoped convention (decision 04-02-B: a branch switch does `queryClient.clear()`, so all keys are branch-scoped by convention).

**`frontend/mocks/reporting.ts`** — MSW handlers with fixture data. **All UUIDs in fixtures must be REAL hex UUIDs** — decision 10-03-D records that mock ids with non-hex letter prefixes (`v-1`, `p-1`) silently failed `z.string().uuid()` and no test caught it.
**Register the handlers in `frontend/mocks/server.ts`, NOT in `handlers.ts`** — decision 10-14-B: `server.ts` is the real registration point; `handlers.ts` only holds auth/feature-flag fixtures.
  </action>

**`/app/reports`** — lists the named reports from `useReports()`, grouped by category, each linking to `/app/reports/{code}`.

**`/app/reports/[code]`** — a date-range picker (`from`/`to`) + a branch selector, then `ReportTable`.
Note decision 10-15-A: there is **no shadcn `Select` primitive and no date-picker** in this frontend (no `@radix-ui/react-select` dependency). Use native `<select>` and `<input type="date">` styled to match the existing `Input` component, exactly as `PeriodPicker` and the purchasing analytics selector already do. Do not add a new UI library for this.

**`ReportTable`** — renders `columns` + `rows` generically (the report shape is dynamic).
- Money columns render via `MoneyDisplay` (paisa → PKR).
- **A `null` cell renders as `—`, never as `0` and never as blank.** For `cogsPaisa`/`grossMarginPaisa` specifically, render `—` and surface the `dataNotes` string ("COGS and margin require Inventory (Phase 8) and are not yet available") as a small info banner above the table. A user seeing `0` margin would reasonably conclude the business is selling at cost. That is worse than showing nothing.
- Loading → the `.skeleton` shimmer primitive (DS-02, decision 04-05-A), not `animate-pulse`.
- Empty result → an explicit empty state, not a blank table.

**`/app/reports/fbr`** — period picker + `FbrTaxSummaryCard`.
`FbrTaxSummaryCard` shows: the branch header (name, NTN, FBR STRN), taxable sales + output tax, taxable purchases + input tax, and **net payable** as the headline figure.
- If `netPayablePaisa` is NEGATIVE, label it as a **refundable input-tax credit**, not as a negative payable. Do not render `-PKR 1,234.00` under a "Net Payable" heading — that reads as a bug. It is a real, meaningful business state.
- If `ntn`/`fbrStrn` are null (user-service unreachable), show the tax figures with a muted "branch tax registration unavailable" note rather than failing the page. The numbers are the point.

**`sidebar-nav-items.ts`** — add `Reports` (→ `/app/reports`) and `Realtime Dashboard` (→ `/app/dashboard/realtime`).
**CRITICAL — the `feature` field:** `NavItem.feature` is typed as the `FeatureFlag` union (decision 10-11-A), so a phantom flag is now a `tsc` compile error rather than a silently-invisible nav item. Reporting routes are deliberately NOT feature-gated at the gateway (12-01 left `/api/v1/reporting/` out of `RouteFeatureMap` on purpose, because `FEATURE_REPORTING_ADVANCED` is GROWTH-and-above and gating basic reports would 403 every STARTER tenant). So **leave `feature` undefined on these nav items** — do not attach `FEATURE_REPORTING_ADVANCED` to them, or STARTER tenants will lose the nav entry for a route they are fully entitled to use.
Gate nav visibility on the `reporting.report.view` / `reporting.dashboard.view` permissions instead, using the existing `useNavVisibility` mechanism if it supports permissions; if it only supports feature flags, leave the items ungated and let the API's 403 speak (decision 10-13-G notes there is no `usePermission(code)` hook yet — do not build one here, that is scope creep; just note it).
  </action>

**`use-dashboard-socket.ts`** — clone `frontend/lib/hooks/kds/use-kds-socket.ts` (read it fully). Keep verbatim:
- All WebSocket state lives inside the `useEffect` as local closure variables (the file's own comment explains why: it avoids both the self-referential `useCallback` lint issue AND the project's `react-hooks/set-state-in-effect` rule — decision 04-04-A).
- The `?token=${accessToken}` query param (browsers cannot set WS headers).
- Exponential backoff reconnect: `INITIAL_BACKOFF_MS = 1000` → `MAX_BACKOFF_MS = 30000`.
- `window.location.host` (proxying through the gateway).
Change:
- URL → `${protocol}//${host}/api/v1/reporting/dashboard/${branchId}?token=${accessToken}`
- Signature → `useDashboardSocket({ branchId })`, returning `{ isConnected, tiles }`.
- On each frame: parse with the `DashboardTileDto` Zod schema and merge into the TanStack Query cache under the `dashboardTiles(branchId)` key, so the WS push and the REST snapshot are ONE source of truth in the UI rather than two competing states.
- Drop the KDS beep (`playNewTicketBeep`) — a dashboard that beeps on every order during a dinner rush is a hostile UI.

**`/app/dashboard/realtime/page.tsx`**:
- On mount, `useDashboardTiles(branchId)` fetches the REST snapshot (12-06's `GET /dashboard/{branchId}/tiles`) so the page paints immediately.
- `useDashboardSocket({branchId})` then keeps it live. A realtime dashboard that is blank until the next order closes looks broken — the snapshot is what prevents that.
- Show a connection indicator (`isConnected` → a small green/amber dot with a text label; do not rely on colour alone — a11y).

**`DashboardTileGrid`** — a responsive grid of tiles (3 cols on lg, 1 on mobile). Each tile: title, big value (money via `MoneyDisplay`, counts plain), and a subtle "updated {n}s ago" from `computedAt`. Guard the divide-by-zero tile (`average-order-value`) — a `null` value renders `—`.

- `fbrSummary_roundTrips` — including a NEGATIVE `netPayablePaisa` fixture, asserting it survives (not clamped anywhere in the FE either).
