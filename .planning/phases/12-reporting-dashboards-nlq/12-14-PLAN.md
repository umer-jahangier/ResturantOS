the RLS lookup. This lets the real end-to-end impersonation flow work without the self-signed-JWT
workaround.

Output: auth-service sets the tenant GUC (established `set_config` pattern) before `findById`; the
real issuance endpoint returns a token; the stamp lands in `nlq_query_log` end-to-end on the real
stack.

  <action>
Three coordinated edits (no shared file ÔÇö safe to do in one plan):

A. **platform-admin `ImpersonationService.impersonate`** (it already has `tenantId`): add the
   tenant to the Feign request body Map it sends to auth-service:
   ```java
   authClient.impersonate(targetUserId,
       Map.of("tenantId", tenantId.toString(),
              "impersonatedBy", adminUserId.toString(),
              "expiresInSeconds", DEFAULT_TTL_SECONDS));
   ```
   (No change needed to `AuthInternalClient` ÔÇö it already takes `Map<String,Object>`.)

B. **auth-service `AuthProvisioningInternalController`**: extend the `ImpersonateRequest` record to
   `record ImpersonateRequest(UUID tenantId, UUID impersonatedBy, int expiresInSeconds) {}` and pass
   `request.tenantId()` into the service call:
   `provisioningAdminService.impersonate(request.tenantId(), userId, request.impersonatedBy(), request.expiresInSeconds())`.

C. **auth-service `ProvisioningAdminService.impersonate`**: change the signature to
   `impersonate(UUID tenantId, UUID targetUserId, UUID impersonatedBy, int expiresInSeconds)`.

   **CRITICAL ÔÇö add `@Transactional` to `impersonate` itself.** Today only `provisionAdmin`
   (line 40) carries `@Transactional`; `impersonate` (line 72) has NO transactional annotation.
   Annotate the `impersonate(...)` method with
   `@org.springframework.transaction.annotation.Transactional` (mirroring how `provisionAdmin`
   already declares it ÔÇö the import is already present). Without this boundary, each JPA
   native-query/repository call can check out and release its OWN pooled connection
   (autocommit-per-statement), so the `set_config` GUC may land on a different physical connection
   than `findById`'s later SELECT ÔÇö silently reproducing the exact "GUC set but RLS SELECT sees no
   tenant" failure (the 2099ac0 bug class) this plan exists to close. The `@Transactional` boundary
   is what guarantees `set_config` and `findById` share one connection.

   Then, as the FIRST statement inside that now-`@Transactional` method, set the RLS tenant GUC
   before `findById`, reusing the established `set_config` pattern already used elsewhere in
   auth-service (`AuthServiceImpl` sets it via
   `entityManager.createNativeQuery("SELECT set_config('app.current_tenant_id', :tid, true)")`):
   - Inject `jakarta.persistence.EntityManager` (add a constructor field) into
     `ProvisioningAdminService`.
   - At method start (first statement, inside the transaction):
     ```java
     entityManager.createNativeQuery("SELECT set_config('app.current_tenant_id', :tid, true)")
         .setParameter("tid", tenantId.toString())
         .getSingleResult();
     ```
   - Then the existing `userRepository.findById(targetUserId)` (and the subsequent
     `permissionResolver.resolveDefault(targetUserId)`, which reads the same RLS-scoped tables) both
     run under the correct tenant on the same connection. Because the GUC is transaction-local
     (`set_config(..., true)`) and the method is now `@Transactional`, the GUC is set on the same
     connection/transaction as the query ÔÇö do NOT set it outside the transaction (avoid the
     historical GUC-before-BEGIN bug), and do NOT rely on an implicit boundary that is not there.
   - Do NOT re-roll `TenantAwareDataSource`; do NOT weaken RLS.

   Guard: if `tenantId` is null (older caller), keep prior behaviour (no GUC) so nothing regresses ÔÇö
   but the real caller (platform-admin) now always sends it.
  </action>
