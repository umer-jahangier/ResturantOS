  <name>Task 1: Derive the tenant GUC from the forwarded JWT claim in getBranch</name>
  <files>services/user-service/src/main/java/io/restaurantos/user/controller/BranchInternalController.java</files>
  <action>
Modify `getBranch(UUID branchId, @RequestHeader(value="X-Tenant-Id", required=false) UUID tenantId)`
so the RLS tenant GUC is set from the authenticated JWT when the header is absent:

```java
UUID effectiveTenant = (tenantId != null)
        ? tenantId
        : tenantContext.getTenantId().orElse(null);
if (effectiveTenant != null) {
    setTenantGuc(effectiveTenant);
}
BranchEntity branch = branchService.get(branchId);
return ResponseEntity.ok(branch);
```

Rationale / why this is correct and safe:
- `UserSecurityConfig` registers `JwtAuthenticationFilter` on the user-service chain
  (`addFilterAfter(jwtAuthenticationFilter, UserInternalServiceFilter.class)`). For an internal
  call that forwards a valid caller Bearer JWT (reporting-service's `forwardCallerJwt()`), that
  filter parses the token and calls `tenantContext.set(tenantId, ...)` on the request thread BEFORE
  the controller runs — so `tenantContext.getTenantId()` is the caller's real tenant here.
- Prefer the explicit `X-Tenant-Id` header when present (back-compat with provisioning-saga callers
  that DO send it), else fall back to the JWT-derived tenant. This removes the seam's dependence on
  a header a caller may omit — the exact §1f failure.
- Keep the existing `setTenantGuc(UUID)` helper (it already does
  `set_config('app.current_tenant_id', ...)` + `tenantContext.set(...)`, the established 2099ac0
  GUC pattern) — do NOT re-roll `TenantAwareDataSource`.

Do NOT change `createBranch` or `getBranchesByTenant` (they already receive an explicit tenant).
Confirm `TenantContext` is already a constructor field (it is) — no new wiring needed.
  </action>

shows the JWT fallback. `createBranch`/`getBranchesByTenant` unchanged.
  </verify>
