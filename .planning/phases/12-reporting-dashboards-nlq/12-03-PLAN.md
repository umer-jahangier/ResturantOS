<objective>
Build reporting-service's ETL: consume the events that ACTUALLY EXIST today and land them as ClickHouse analytics facts, bucketed by the business-day boundary formula, idempotently.

Purpose: Without facts there are no reports, no dashboard and nothing for NLQ to query. This is the spine of the phase.

Output: RabbitMQ topology + three consumers + three fact writers + the business-day util + Postgres idempotency side-tables, proven by a Testcontainers IT that publishes a real event and asserts the row appears in a real ClickHouse.

SCOPE FENCE — READ THIS BEFORE WRITING ANY CONSUMER:
Phase 8 (Inventory) and Phase 9 are NOT STARTED. `STOCK_DEPLETED`, `LOW_STOCK_ALERT`, `WASTAGE_RECORDED`, `COUNT_VARIANCE_POSTED`, `TRANSFER_*` **do not exist as running code** — `grep -rn "STOCK_DEPLETED" services/` returns zero matches. Do NOT write a consumer, a queue, a binding, or a fact writer for any of them. Do NOT bind to `inventory.topic` for depletion. This phase consumes exactly three events: `ORDER_CLOSED` (`pos.topic` / `pos.order.closed`), `TILL_CLOSED` (`pos.topic` / `pos.till.closed`), `VENDOR_INVOICE_MATCHED` (`purchasing.topic` / `purchasing.invoice.matched`). All three are verified-real publish call sites in today's codebase.

`JOURNAL_POSTED` is deliberately deferred: FBR output tax comes from `ORDER_CLOSED.taxPaisa` and input tax from `VENDOR_INVOICE_MATCHED`, so the phase's success criteria are fully met without a journal fact table. Adding it here would be scope creep.
</objective>

**ClickHouseConfig.java** — a SECOND DataSource, entirely separate from the Spring-managed Postgres one:

ClickHouse has no RLS and no transactions — tenant isolation there is enforced purely by explicit `WHERE tenant_id = ?` predicates in every query. Never route the ClickHouse DataSource through the tenant-aware wrapper, and never hand-roll a new one.

**Consumers** — clone `services/kitchen-service/.../consumer/OrderClosedConsumer.java`'s exact shape:
```java
@RabbitListener(queues = ReportingRabbitConfig.REPORTING_ORDER_CLOSED_QUEUE)
public void onMessage(Message message) {
    EventEnvelope<OrderClosedPayload> envelope = deserialize(message);
    if (envelope == null) { log.warn("could not deserialize — skipping"); return; }
    processedEventService.tryProcess(CONSUMER_NAME, envelope.eventId(), () ->
        tenantAwareMessageProcessor.process(envelope, env -> {
            ZoneId zone = branchTimeZoneResolver.resolve(env.branchId());
            LocalDate businessDate = businessDay.businessDate(env.payload().closedAt(), zone);
            salesFactWriter.write(env, businessDate);
        })
    );
}
```
Notes that matter:
- `tenantAwareMessageProcessor` (shared-lib `io.restaurantos.shared.tenant.TenantAwareMessageProcessor`) sets the tenant context from the envelope for the duration of the work — required so the `processed_events` write lands under the right RLS tenant. This is the mechanism that commit `2099ac0` fixed; use it, don't reinvent it.
- Use the payload's own timestamp (`closedAt` for ORDER_CLOSED, the till's closed timestamp for TILL_CLOSED, the invoice/match timestamp for VENDOR_INVOICE_MATCHED) for the business-day bucket — NOT `envelope.occurredAt()` and definitely not `Instant.now()`. The business day of a sale is a property of the sale.
- Each consumer gets a distinct `CONSUMER_NAME` constant (e.g. `reporting.order-closed`) — that's the `processed_events.consumer` key, and it's what makes idempotency per-consumer rather than global.
