<objective>
Build the 7-stage SQL AST validation pipeline that stands between Claude's generated SQL and the database. This is the single highest-risk surface in the entire project: it is the thing that makes executing model-generated SQL survivable.

Purpose: NLQ-01 / ROADMAP success criterion 3. Claude can hallucinate, and the natural-language question itself is an untrusted prompt-injection vector. The pipeline must assume the SQL is hostile and prove it safe, not check that it "looks fine."

Output: A pure, I/O-free, exhaustively-tested validation pipeline (JSqlParser AST in, safe SQL string out, or a typed rejection). No Claude call, no ClickHouse execution ŌĆö those are 12-07. This plan is deliberately isolated so its adversarial test suite gets full attention.

WHY TDD: every stage is `expect(validate(sql, ctx)).toBe(safeSql | rejection)` ŌĆö a pure function with a fixed contract. The negative controls (a query that MUST be rejected) are the deliverable. Writing them after the implementation would mean writing tests that pass against whatever you happened to build; writing them first means the implementation has to earn them.

BINDING DECISION: JSqlParser (Java). NOT Python sqlglot. 12-RESEARCH's Python/sqlglot recommendation is SUPERSEDED by the user's decision ŌĆö nlq-service is Java/Spring Boot.
</objective>

**`RejectionCode`** (enum) ŌĆö one per failure mode, because the API must tell the user WHY without leaking internals:
`SHAPE_INVALID, PARSE_FAILED, TABLE_NOT_ALLOWED, PII_COLUMN_DENIED, TENANT_FILTER_MISSING, BRANCH_FILTER_MISSING, LIMIT_INVALID`.

**`NlqRejectedException`** ŌĆö carries a `RejectionCode` + a safe human message. Never carries the offending SQL back to the client verbatim (that's an oracle for an attacker probing the validator); it MAY be logged server-side.

**`QueryContext`** (record) ŌĆö `UUID tenantId, UUID branchId, String roleCode, boolean isOwner, UUID userId, UUID impersonatedBy`. Built from `JwtClaims` at the controller in 12-07. Note `isOwner` ŌĆö OWNER-role users may query across branches; everyone else is branch-pinned.

**`SqlValidationPipelineTest`** ŌĆö the happy-path + contract tests:
1. A clean `SELECT sum(total_paisa) FROM sales_order_facts WHERE business_date >= '2026-07-01'` for a non-OWNER returns SQL that contains BOTH `tenant_id = '<the ctx tenant uuid>'` AND `branch_id = '<the ctx branch uuid>'` AND a `LIMIT`.
2. Same query for an OWNER returns SQL with `tenant_id` and a `LIMIT` but is NOT forced to a single branch.
3. A query that ALREADY has `WHERE tenant_id = '<ctx tenant>'` is not double-filtered into something contradictory (it still returns rows for that tenant).
4. A query that has `WHERE tenant_id = '<SOME OTHER TENANT>'` is REJECTED (or rewritten so the caller's tenant is ANDed in and the other-tenant predicate cannot widen it ŌĆö assert the caller's tenant predicate is present and conjunctive, never `OR`). **A caller must never be able to name another tenant's id and have it honoured.**
5. `SELECT ... LIMIT 999999` is clamped to the configured max (`nlq.max-result-rows`).
6. `SELECT ...` with no LIMIT gets `LIMIT 1000` (`nlq.default-limit`) injected.
7. A `JOIN` across two allowed fact tables gets the tenant predicate applied such that BOTH table references are constrained (or the query is rejected ŌĆö assert one or the other explicitly; a JOIN where only one side is tenant-filtered is a cross-tenant leak).

**`SqlInjectionAttackTest`** ŌĆö the adversarial suite. Each case asserts `NlqRejectedException` with a SPECIFIC `RejectionCode` (assert the code, not just "it threw"). These ARE the security evidence for this phase:

- `SELECT 1; DROP TABLE sales_order_facts` (multi-statement ŌĆö must be rejected, not "first statement only")

- `SELECT 1 -- \n; DROP TABLE x` (line-comment obfuscation)

*Tenant/branch evasion ŌĆö the ones that matter most:*
- `SELECT * FROM sales_order_facts` from a non-OWNER ŌĆö must come back carrying BOTH tenant and branch predicates, or be rejected. Assert on the OUTPUT SQL, not just "no exception".
- `SELECT * FROM sales_order_facts WHERE 1=1 OR tenant_id = '<ctx>'` ŌĆö an `OR` that widens the tenant predicate to everything. Must NOT execute as-is: assert either rejection, or that the caller's tenant predicate is ANDed at the OUTERMOST level so the `OR` cannot escape it.

- A CTE: `WITH x AS (SELECT * FROM sales_order_facts) SELECT * FROM x` ŌĆö the inner select is unfiltered. Must be rejected or filtered.
- A subquery: `SELECT * FROM (SELECT * FROM sales_order_facts) t` ŌĆö same.
- A correlated/IN subquery reaching another table.
- **The governing rule (12-RESEARCH Pitfall 3, and ROADMAP success criterion 3 verbatim: "a query missing the tenant or branch filter is rejected"): if the injector cannot PROVE, by re-parsing its own output, that every table reference is constrained by the caller's tenant (and, for non-OWNERs, branch), it MUST throw `TENANT_FILTER_MISSING` / `BRANCH_FILTER_MISSING`. "Best effort, probably fine" is a cross-tenant data breach.** Rejecting a legitimate-but-exotic query is an acceptable cost; executing an unfiltered one is not.

  <action>
Implement minimally, in stage order. `SqlValidationPipeline.validate(String sql, QueryContext ctx) -> String safeSql` runs all 7 in sequence; any stage may throw `NlqRejectedException`.

**Stage 1 ŌĆö ShapeCheckStage.** Cheap structural gate BEFORE parsing.
- Reject if the trimmed SQL does not start with `SELECT` or `WITH` (case-insensitive).
- Reject if it contains more than one statement. Do this by parsing with `CCJSqlParserUtil.parseStatements(sql)` and rejecting when the statement list size != 1 ŌĆö **not** by counting semicolons (a `;` inside a string literal would false-positive, and a comment-hidden one would false-negative).
- Reject any statement whose parsed type is not `Select`. Explicitly reject `Insert`, `Update`, `Delete`, `Drop`, `Truncate`, `Alter`, `CreateTable`, `Grant`, `Execute`, and anything else JSqlParser yields that is not a `Select`. Whitelist, never blacklist: `if (!(stmt instanceof Select)) throw SHAPE_INVALID`. A blacklist will always miss a verb; ClickHouse has verbs like `OPTIMIZE`/`SYSTEM` you have not thought of.
- Reject if length exceeds a sane cap (e.g. 4000 chars).

**Stage 2 ŌĆö AstParseStage.** `CCJSqlParserUtil.parse(sql)` into a `Select`. Any `JSQLParserException` ŌåÆ `PARSE_FAILED`. Set a parser timeout if JSqlParser exposes one (it does: `CCJSqlParserUtil.parse(sql, parser -> parser.withTimeOut(1000))`) ŌĆö a pathological input must not hang a request thread.

**Stage 3 ŌĆö TableAllowlistStage.** Walk the AST for EVERY table reference (`TablesNamesFinder` gives you all of them, including inside CTEs, subqueries and UNION arms ŌĆö this is exactly why we parse instead of regex). Normalise each name: strip an optional `clickhouse_analytics.` database qualifier, lowercase. Every name must be in `allowedTableService.allowedFor(roleCode)`. Otherwise `TABLE_NOT_ALLOWED`. If `TablesNamesFinder` returns an empty set for a non-trivial query, REJECT (fail closed ŌĆö an unparseable table list means you cannot prove safety).

**Stage 4 ŌĆö PiiDenylistStage.** A configured deny-list of `table.column` pairs (property `restaurantos.nlq.pii-denylist`, seeded with at least the customer-identifying columns present on the fact tables ŌĆö read 12-02-SUMMARY's final column lists and pick the real ones; do not deny columns that don't exist). Walk all `Column` nodes and reject a match. **Also reject `SELECT *` (and `t.*`) on any table that contains a deny-listed column** ŌĆö star-expansion is the obvious bypass. Simplest correct rule: if the query star-selects from a table with any deny-listed column, reject with `PII_COLUMN_DENIED` and let the user name their columns.

**Stage 5 ŌĆö TenantFilterStage.** THE critical stage.
- For each table reference in the AST, AND `tenant_id = '<ctx.tenantId>'` into the applicable WHERE clause ŌĆö walking into CTEs, subqueries and every UNION arm, not just the top-level SELECT.
- **Then PROVE it.** Re-serialise the mutated AST to SQL, RE-PARSE it, and re-walk: assert that every table reference is governed by a conjunctive `tenant_id = '<ctx.tenantId>'` predicate. If the proof fails ŌĆö for ANY reason, including "this query shape is one my injector doesn't handle" ŌĆö throw `TENANT_FILTER_MISSING`. Do not execute anything you cannot prove. This is 12-RESEARCH Pitfall 3, and it is the literal wording of ROADMAP success criterion 3.
- A pragmatic, defensible way to make the proof tractable: restrict the supported query shapes (single SELECT, optional JOINs over allowed tables, optional GROUP BY/ORDER BY, optional simple CTEs) and reject everything else at this stage. A smaller provable surface beats a larger unprovable one. Document the supported shape set in the SUMMARY ŌĆö 12-07's Claude system prompt will instruct the model to generate only within it, which is how you keep the rejection rate low without weakening the gate.

**Stage 6 ŌĆö BranchFilterStage.** Identical mechanism, `branch_id = '<ctx.branchId>'`, applied ONLY when `!ctx.isOwner()`. Same re-parse proof; failure ŌåÆ `BRANCH_FILTER_MISSING`. If `!isOwner` and `ctx.branchId()` is null, reject (a non-OWNER with no branch cannot be scoped, so they get nothing).

**Stage 7 ŌĆö LimitInjectStage.** No LIMIT ŌåÆ inject `nlq.default-limit` (1000). LIMIT present but > `nlq.max-result-rows` (10000) ŌåÆ clamp DOWN to the max. Negative/zero/non-numeric ŌåÆ `LIMIT_INVALID`. The ClickHouse `nlq_readonly` profile from 12-02 also caps result rows server-side ŌĆö this stage is the polite first line, the profile is the hard one. Both exist on purpose.

**Allowlist storage ŌĆö RESOLVES 12-RESEARCH Open Question 4 with a concrete call:**
It lives in **nlq-service's own Postgres (`nlq_db`), table `nlq_allowed_tables`, keyed by ROLE (not by tenant)**, Redis-cached with a 10-minute TTL (the TTL the spec's M6.2 already calls for). Rationale: the spec (M6.3 stage 3) says "the role's `nlq_allowed_tables`" ŌĆö it is role-scoped. Keeping it in nlq_db (not reporting_db, not a config file) means it is administrable at runtime without a redeploy, and it is the service's own data so there is no cross-service SQL (PROJECT.md Boundaries constraint).

`V1__nlq_schema.sql` (Flyway):
- `nlq_allowed_tables (role_code VARCHAR NOT NULL, table_name VARCHAR NOT NULL, PRIMARY KEY (role_code, table_name))` ŌĆö a PLATFORM-level table, NOT tenant-scoped, so **no RLS and no tenant_id column** (same category as auth's non-RLS lookup tables). Add a comment saying so, or a future reviewer will "helpfully" add RLS and break it.
- Seed it: OWNER / TENANT_ADMIN / MANAGER / ACCOUNTANT get the four fact tables; roles like CASHIER/CHEF get none (an empty allowlist means every query they send is rejected at stage 3 ŌĆö correct: they have no business running ad-hoc analytics).
- `nlq_query_log (id UUID PK, tenant_id UUID NOT NULL, branch_id UUID, user_id UUID NOT NULL, impersonated_by UUID, role_code VARCHAR, question TEXT NOT NULL, generated_sql TEXT, executed_sql TEXT, rejection_code VARCHAR, row_count INT, duration_ms INT, cache_hit BOOLEAN NOT NULL DEFAULT false, created_at TIMESTAMPTZ NOT NULL DEFAULT now())` ŌĆö tenant-scoped, so **RLS ENABLED + FORCED** with the standard `tenant_id = current_setting('app.tenant_id')::uuid` policy copied verbatim from purchasing's V1. `impersonated_by` is the NLQ-02 impersonation stamp; 12-07 fills it from `JwtClaims.impersonatedBy()`.

`AllowedTableService.allowedFor(roleCode)` ŌåÆ `Set<String>`, Redis-cached 10 min. Cache MISS must hit Postgres; a Redis outage must fall through to Postgres, never to "allow everything".

Iterate until both suites are green. Commit: `feat(12-04): implement 7-stage SQL AST validation pipeline`.
