<objective>
Seed the four new permissions Phase 12 depends on, and — critically — WIRE THEM INTO THE MASTER CHANGELOG so Liquibase actually runs them.

Purpose: `@PreAuthorize` on a permission nobody holds is a 403 for everyone. Every report endpoint (12-05), the dashboard WebSocket (12-06), and the entire NLQ service (12-07) are gated on permissions that do not exist yet. This plan is the auth-layer foundation for all three, and it runs in Wave 1 so those plans can verify against a real, granted permission rather than discovering the 403 storm at the end of the phase.

Output: Two Liquibase changesets + the two `<include>` lines that make them real.

WHY THIS IS ITS OWN PLAN: `db.changelog-master.xml` is a shared file. Three separate plans (12-05, 12-06, 12-07) each need permissions seeded. Threading a shared XML file through three concurrent wave-3 plans is a guaranteed write race (decisions 10-12-E / 10-14-D document exactly that going wrong during Phase 10's concurrent gap-closure wave). One plan owns the file; nobody else touches it.
</objective>

**This task is the entire reason this plan exists.** `db.changelog-master.xml` lists every changeset with an explicit `<include file="db/changelog/v1.0.0/NNN-....xml"/>` line. It does **not** auto-discover files in the directory. A changeset that is not included is dead code — it will sit on disk looking correct, Liquibase will never execute it, the permissions will never exist, and every `@PreAuthorize`-gated reporting/NLQ endpoint plus the dashboard WebSocket will 403 for every user. The symptom (a 403 storm) is many layers away from the cause (one missing XML line), which is what makes this worth its own task.

This is the same class of bug as the phantom `FEATURE_NLQ` flag caught in 12-01 — a declaration that exists in one place and is never registered in the place that actually reads it.

Add, in the permissions block (after `044-finance-period-open-permission.xml`, keeping the file's existing grouping):
```xml
    <include file="db/changelog/v1.0.0/045-reporting-permissions.xml"/>
    <include file="db/changelog/v1.0.0/046-nlq-permissions.xml"/>
```

Then PROVE it ran — do not stop at "the line is in the file":
1. Start auth-service against a real Postgres (`docker compose -f deploy/docker-compose.yml up -d postgres`, then boot auth-service).
2. Query the DB directly:
   `SELECT code FROM permissions WHERE code IN ('reporting.report.view','reporting.report.fbr','reporting.dashboard.view','nlq.query.run');` → 4 rows.
   `SELECT r.code, p.code FROM role_permissions rp JOIN roles r ON ... JOIN permissions p ON ... WHERE p.code LIKE 'reporting.%' OR p.code = 'nlq.query.run';` → 16 rows (4 permissions x 4 roles).
3. Log in as a seeded OWNER via the real auth-service and **decode the returned JWT**. Assert its `permissions` claim contains all four codes. The JWT is what `@PreAuthorize` and the WebSocket handler actually read — a permission that is in the database but absent from the token is still a 403. This assertion is the one that matters.
