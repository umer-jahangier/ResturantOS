**`TilePushThrottle`** — the net-new piece. `KdsWebSocketHandler` has NO throttle: every mutation pushes immediately. That is fine for a KDS (a handful of tickets), and catastrophic for a dashboard during a dinner rush (100 orders closing in a minute = 100 recompute+push cycles per tile).

Contract: `boolean tryAcquire(String tileKey)` where `tileKey` = `branchId + ":" + tileId`. Backed by a `ConcurrentHashMap<String, Instant> lastPushAt`. Returns true (and records the timestamp) if at least `minIntervalMs` has elapsed since the last push for that tile; false otherwise.

**Critical design point — the throttle must not violate the 5s criterion.** A naive "drop if too soon" throttle can starve: if events keep arriving every 100ms, a leading-edge-only throttle pushes at t=0 and then again at t=interval, which is fine — but if the LAST event of a burst is dropped, the client's final state is stale forever. So:
- `minIntervalMs` = 1000 (configurable, `restaurantos.dashboard.tile-throttle-ms`), well inside the 5s budget.
- Implement **leading-edge push + trailing-edge flush**: if a push is throttled, mark the tile dirty; a scheduled sweeper (`@Scheduled(fixedDelay = 1000)`) pushes any dirty tile whose interval has since elapsed. This guarantees the client converges to the true state within ~2s worst case — comfortably under 5s — while a 100-event burst produces ~2 pushes, not 100.
- A pure drop-the-tail throttle would satisfy "doesn't flood" while silently BREAKING "updates within 5 seconds". Do not ship that.

**`TilePushThrottleTest`** (plain unit test, use an injectable `Clock` so it does not sleep):
- First call for a tile → true.
- Immediate second call → false, and the tile is marked dirty.
- After `minIntervalMs` elapses (advance the fake Clock) → the dirty tile is flushed exactly once.
- 100 rapid calls within one interval → exactly 1 leading push + 1 trailing flush, not 100.
- Two different tiles do not throttle each other.

**`DashboardTileDto`**: `{ tileId, title, valuePaisa (Long, nullable), valueNumber (Long, nullable), unit, businessDate, computedAt }` — a tile is either a money value or a count; use whichever field applies and leave the other null.

**`DashboardTileService.computeTiles(tenantId, branchId, businessDate)`** — aggregate queries against the ClickHouse facts for TODAY's business date (using `BusinessDay` from 12-03 to determine what "today" is for that branch — a dashboard that flips to a new day at midnight UTC instead of at the 4am business boundary would zero out the tiles mid-shift):
- `todays-revenue` — `SUM(total_paisa) FROM sales_order_facts WHERE tenant_id=? AND branch_id=? AND business_date=?`
- `todays-orders` — `count()` same predicate
- `todays-tax` — `SUM(tax_paisa)` same predicate
- `average-order-value` — revenue / orders (integer paisa division; guard divide-by-zero → null, not 0)
- `open-tills` — count of till sessions opened but not closed today (from `till_session_facts`; if the fact table only records CLOSED sessions, this tile is not computable — in that case DROP the tile rather than faking it, and say so in the SUMMARY)

Cache the computed tiles in Redis under `dashboard:tiles:{tenantId}:{branchId}:{businessDate}` (short TTL, e.g. 10s). The spec (M5.4) calls for Redis-precomputed tiles; this also means the REST snapshot endpoint and the WS push serve the same numbers rather than racing each other to slightly different ones.

**`DashboardController`** — `GET /api/v1/reporting/dashboard/{branchId}/tiles` → `ApiResponse<List<DashboardTileDto>>`, `@PreAuthorize("hasAuthority('reporting.dashboard.view')")`, tenantId server-derived. This exists so a freshly-connected client renders immediately instead of sitting blank until the next order closes. A realtime dashboard that is empty until something happens looks broken.

**Wire the push into the existing consumers** (owned by 12-03, already landed):
In `OrderClosedConsumer` and `TillClosedConsumer`, AFTER the fact writer succeeds and INSIDE the same `tenantAwareMessageProcessor.process(...)` block:
```java
dashboardTileService.recomputeAndPush(env.tenantId(), env.branchId(), businessDate);
```
`recomputeAndPush` recomputes the tiles, then for each tile calls `tilePushThrottle.tryAcquire(key)` and, if granted, `dashboardWebSocketHandler.notifySubscribers(branchId, tiles)`; if not granted, marks it dirty for the trailing flush.

**Do not let a dashboard-push failure poison the ETL.** Wrap the `recomputeAndPush` call in a try/catch that logs at WARN and swallows. The fact row is the durable truth; a failed WS push is a cosmetic problem. If an exception escaped here it would roll back the `processed_events` guard and the event would be redelivered forever — a broken dashboard would take out the ETL. Add a comment saying exactly that, so nobody "cleans up" the catch later.
